# Design Patterns

sidebar_position: 2

## Overview

This document covers battle-tested design patterns used at scale in production environments. These patterns solve recurring problems in distributed systems and are implemented using GCP services.

## When To Use

- Building services that need to handle varying loads gracefully
- Designing systems that require high availability
- Implementing communication between microservices
- Building systems that must maintain data consistency
- Creating observable and debuggable distributed systems

## When NOT To Use

- Simple CRUD applications with predictable load
- Single-service deployments without scaling requirements
- Prototypes where patterns add unnecessary complexity
- Systems where operational overhead exceeds benefits

## Core Concepts

### Pattern Categories
1. **Structural Patterns**: How components are organized
2. **Communication Patterns**: How services exchange data
3. **Resilience Patterns**: How systems handle failures
4. **Data Patterns**: How data is stored and accessed
5. **Deployment Patterns**: How services are delivered

## Real Production Example

### Netflix-Style Circuit Breaker Implementation

Netflix processes billions of API requests daily. Their circuit breaker pattern prevents cascade failures when dependent services degrade.

```typescript
// typescript implementation of circuit breaker
interface CircuitBreakerState {
  CLOSED: 'closed';
  OPEN: 'open';
  HALF_OPEN: 'half_open';
}

class CircuitBreaker {
  private state: CircuitBreakerState['CLOSED' | 'OPEN' | 'HALF_OPEN'] = 'closed';
  private failureCount = 0;
  private lastFailureTime?: number;

  constructor(
    private readonly threshold: number = 5,
    private readonly timeout: number = 60000
  ) {}

  async execute<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === 'OPEN') {
      if (Date.now() - this.lastFailureTime! >= this.timeout) {
        this.state = 'HALF_OPEN';
      } else {
        throw new Error('Circuit breaker is OPEN');
      }
    }

    try {
      const result = await fn();
      this.onSuccess();
      return result;
    } catch (error) {
      this.onFailure();
      throw error;
    }
  }

  private onSuccess(): void {
    this.failureCount = 0;
    this.state = 'CLOSED';
  }

  private onFailure(): void {
    this.failureCount++;
    this.lastFailureTime = Date.now();
    if (this.failureCount >= this.threshold) {
      this.state = 'OPEN';
    }
  }
}
```

### GCP Implementation with Cloud Functions

```typescript
// Cloud Function with circuit breaker pattern
import { CloudEvent, CloudFunctionsEvent } from '@google-cloud/functions-framework';

interface RateLimitState {
  failureCount: number;
  lastFailure: number;
  state: 'CLOSED' | 'OPEN' | 'HALF_OPEN';
}

const circuitBreakerState: RateLimitState = {
  failureCount: 0,
  lastFailure: 0,
  state: 'CLOSED'
};

const FAILURE_THRESHOLD = 5;
const TIMEOUT_MS = 60000;
const RESET_TIMEOUT_MS = 30000;

exportconst checkRateLimit = async (
  req: CloudFunctionsEvent
): Promise<{ allowed: boolean; remaining: number }> => {
  const now = Date.now();

  // Circuit breaker logic
  if (circuitBreakerState.state === 'OPEN') {
    if (now - circuitBreakerState.lastFailure > TIMEOUT_MS) {
      circuitBreakerState.state = 'HALF_OPEN';
    } else {
      throw new Error('Circuit breaker OPEN - rate limit service unavailable');
    }
  }

  try {
    // Check rate limit against Redis
    const { Memorystore } = await import('@google-cloud/memorystore');
    const memstore = new Memorystore(1, {
      host: process.env.REDIS_HOST,
      port: parseInt(process.env.REDIS_PORT || '6379')
    });

    const key = `ratelimit:${req.data.userId}`;
    const current = await memstore.get(key) || 0;

    if (current >= 1000) {
      circuitBreakerState.failureCount++;
      circuitBreakerState.lastFailure = now;

      if (circuitBreakerState.failureCount >= FAILURE_THRESHOLD) {
        circuitBreakerState.state = 'OPEN';
      }

      throw new Error('Rate limit exceeded');
    }

    await memestore.set(key, current + 1, 'EX', 60);

    // Reset circuit breaker on success
    circuitBreakerState.failureCount = 0;
    circuitBreakerState.state = 'CLOSED';

    return { allowed: true, remaining: 1000 - current - 1 };
  } catch (error) {
    console.error('Rate limit check failed:', error);
    throw error;
  }
};
```

## Common Patterns / Anti-patterns

### Pattern 1: Circuit Breaker

```typescript
// Cloud Run service with circuit breaker
import express from 'express';

const app = express();

// Half-open state allows limited requests through
const HALF_OPEN_MAX_REQUESTS = 3;
let halfOpenRequests = 0;

interface ExternalServiceClient {
  call(): Promise<Response>;
  isHealthy(): boolean;
}

const externalClient: ExternalServiceClient = {
  call: async () => {
    const response = await fetch('https://external-api.example.com/data');
    if (!response.ok) throw new Error('External service failed');
    return response;
  },
  isHealthy: () => true
};

// Middleware implementing circuit breaker
app.use('/api/data', async (req, res) => {
  const maxFailures = 5;
  const resetTimeout = 60000;

  // In production, store state in Memorystore
  const state = await getCircuitBreakerState('external-api');

  if (state === 'OPEN') {
    const timeSinceFailure = Date.now() - state.lastFailure;
    if (timeSinceFailure < resetTimeout) {
      return res.status(503).json({ error: 'Service temporarily unavailable' });
    }
    state.state = 'HALF_OPEN';
    await saveCircuitBreakerState('external-api', state);
  }

  if (state === 'HALF_OPEN') {
    if (halfOpenRequests >= HALF_OPEN_MAX_REQUESTS) {
      return res.status(503).json({ error: 'Probe requests exhausted' });
    }
    halfOpenRequests++;
  }

  try {
    const data = await externalClient.call();
    resetCircuitBreaker('external-api');
    return res.json(data);
  } catch (error) {
    incrementFailure('external-api');
    if (getFailureCount('external-api') >= maxFailures) {
      openCircuit('external-api');
    }
    return res.status(500).json({ error: 'Upstream service error' });
  }
});
```

### Pattern 2: Retry with Exponential Backoff

```typescript
// GCP Cloud Functions retry handler
interface RetryConfig {
  maxAttempts: number;
  initialDelayMs: number;
  maxDelayMs: number;
  backoffMultiplier: number;
}

const defaultRetryConfig: RetryConfig = {
  maxAttempts: 3,
  initialDelayMs: 1000,
  maxDelayMs: 30000,
  backoffMultiplier: 2
};

async function withRetry<T>(
  fn: () => Promise<T>,
  config: RetryConfig = defaultRetryConfig
): Promise<T> {
  let lastError: Error;
  let delay = config.initialDelayMs;

  for (let attempt = 1; attempt <= config.maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (error) {
      lastError = error as Error;

      if (attempt === config.maxAttempts) {
        throw error;
      }

      // Check if error is retryable
      if (!isRetryableError(error)) {
        throw error;
      }

      // Add jitter to prevent thundering herd
      const jitter = Math.random() * 0.3 * delay;
      await sleep(delay + jitter);

      delay = Math.min(delay * config.backoffMultiplier, config.maxDelayMs);
    }
  }

  throw lastError!;
}

function isRetryableError(error: Error): boolean {
  const retryableCodes = ['ETIMEDOUT', 'ECONNRESET', 'ECONNREFUSED', '503'];
  return retryableCodes.some(code => error.message.includes(code));
}

function sleep(ms: number): Promise<void> {
  return new Promise(resolve => setTimeout(resolve, ms));
}

// Usage in Cloud Function
exportconst processPayment = async (event: CloudEvent<any>) => {
  const paymentData = JSON.parse(Buffer.from(event.data.message.data, 'base64').toString());

  return withRetry(async () => {
    // Call payment gateway with retry logic
    const result = await fetch('https://payment-gateway.example.com/charge', {
      method: 'POST',
      body: JSON.stringify(paymentData),
      headers: { 'Authorization': `Bearer ${process.env.PAYMENT_API_KEY}` }
    });

    if (!result.ok) {
      throw new Error(`Payment failed: ${result.status}`);
    }

    return result.json();
  });
};
```

### Pattern 3: Saga Pattern for Distributed Transactions

```typescript
// Saga orchestrator for order processing
interface SagaStep {
  name: string;
  execute: () => Promise<void>;
  compensate: () => Promise<void>;
}

class OrderSaga {
  private steps: SagaStep[] = [
    {
      name: 'reserve-inventory',
      execute: this.reserveInventory.bind(this),
      compensate: this.releaseInventory.bind(this)
    },
    {
      name: 'process-payment',
      execute: this.processPayment.bind(this),
      compensate: this.refundPayment.bind(this)
    },
    {
      name: 'create-shipment',
      execute: this.createShipment.bind(this),
      compensate: this.cancelShipment.bind(this)
    }
  ];

  async execute(orderId: string): Promise<void> {
    const completedSteps: SagaStep[] = [];

    for (const step of this.steps) {
      try {
        await step.execute();
        completedSteps.push(step);
      } catch (error) {
        console.error(`Step ${step.name} failed for order ${orderId}`);

        // Compensate in reverse order
        for (const completed of completedSteps.reverse()) {
          try {
            await completed.compensate();
          } catch (compensateError) {
            console.error(`Compensation failed for ${completed.name}:`, compensateError);
            // Alert operations team - manual intervention may be needed
            await this.alertOperations(orderId, completed.name, compensateError);
          }
        }

        throw error;
      }
    }
  }

  private async reserveInventory(orderId: string): Promise<void> {
    // GCP Cloud SQL transaction
    await db.transaction(async (tx) => {
      await tx.query(
        'UPDATE inventory SET reserved = reserved + 1 WHERE product_id = ?',
        [orderId.productId]
      );
      await tx.query(
        'INSERT INTO reservations (order_id, product_id, reserved_at) VALUES (?, ?, NOW())',
        [orderId, orderId.productId]
      );
    });
  }

  private async releaseInventory(orderId: string): Promise<void> {
    await db.query(
      'UPDATE inventory SET reserved = reserved - 1 WHERE product_id = ?',
      [orderId.productId]
    );
  }

  private async processPayment(orderId: string): Promise<void> {
    const response = await fetch('https://payment.example.com/charge', {
      method: 'POST',
      body: JSON.stringify({ orderId, amount: orderId.amount })
    });

    if (!response.ok) throw new Error('Payment failed');
    const { transactionId } = await response.json();

    await db.query(
      'UPDATE orders SET payment_transaction_id = ? WHERE id = ?',
      [transactionId, orderId]
    );
  }

  private async refundPayment(orderId: string): Promise<void> {
    const order = await db.query('SELECT payment_transaction_id FROM orders WHERE id = ?', [orderId]);
    await fetch('https://payment.example.com/refund', {
      method: 'POST',
      body: JSON.stringify({ transactionId: order.payment_transaction_id })
    });
  }

  private async createShipment(orderId: string): Promise<void> {
    // Create shipment via carrier API
  }

  private async cancelShipment(orderId: string): Promise<void> {
    // Cancel shipment
  }

  private async alertOperations(orderId: string, step: string, error: any): Promise<void> {
    // Send to PagerDuty, Slack, etc.
  }
}
```

### Pattern 4: CQRS (Command Query Responsibility Segregation)

```typescript
// Write side - Command handler
app.post('/orders', async (req, res) => {
  const command: CreateOrderCommand = req.body;

  // Validate command
  if (!command.userId || !command.items?.length) {
    return res.status(400).json({ error: 'Invalid command' });
  }

  // Execute command
  const order = await orderRepository.create({
    id: generateUUID(),
    userId: command.userId,
    items: command.items,
    status: 'PENDING',
    createdAt: new Date()
  });

  // Publish event for read model update
  await pubsub.topic('order-events').publish({
    type: 'ORDER_CREATED',
    orderId: order.id,
    timestamp: new Date().toISOString()
  });

  return res.status(201).json(order);
});

// Read side - Query handler (separate service/scaled independently)
app.get('/orders/:id', async (req, res) => {
  // Query from denormalized read model
  const orderReadModel = await readModelDb.query(
    'SELECT * FROM order_read_model WHERE id = ?',
    [req.params.id]
  );

  if (!orderReadModel) {
    return res.status(404).json({ error: 'Order not found' });
  }

  return res.json(orderReadModel);
});

// Projection service ( Cloud Run / Cloud Functions)
exportconst projectOrderCreated = async (event: CloudEvent<any>) => {
  const { orderId, userId, items, createdAt } = JSON.parse(
    Buffer.from(event.data.message.data, 'base64').toString()
  );

  // Build denormalized read model
  const total = items.reduce((sum: number, item: any) => sum + item.price * item.quantity, 0);
  const itemCount = items.reduce((sum: number, item: any) => sum + item.quantity, 0);

  // Upsert to read model
  await readModelDb.query(`
    INSERT INTO order_read_model (id, user_id, total, item_count, status, created_at)
    VALUES (?, ?, ?, ?, 'PENDING', ?)
    ON DUPLICATE KEY UPDATE total = VALUES(total)
  `, [orderId, userId, total, itemCount, createdAt]);
};
```

### Anti-patterns to Avoid

| Anti-pattern | Problem | Solution |
|-------------|---------|----------|
| Retry storm | Retries overwhelm recovering service | Use jitter + circuit breaker |
| Shared state | Coupling between services | Stateless design, external state store |
| Synchronous chains | Latency multiplication, cascade failures | Async messaging, event-driven |
| Mega-service | Complexity, slow deploys | Split by domain boundary |
| N+1 queries | Performance degradation | Batch queries, data loader |

## Interview Questions

### Q1: When would you NOT use a circuit breaker?
**Answer**: Circuit breakers add complexity. For low-traffic services where failures are acceptable, or when the downstream service is critical and there's no fallback, a circuit breaker may not provide value. Also, if the downstream call is always fast (&lt;10ms&gt;) and highly available, the overhead may not be justified.

### Q2: Explain the trade-offs of the Saga pattern vs 2-phase commit
**Answer**: 2-phase commit is blocking and has availability issues (coordinator failure). Saga provides better availability but requires compensating actions that may also fail. Saga is eventually consistent while 2PC is strongly consistent. Choose Saga for distributed systems where availability matters more than strong consistency.

### Q3: How do you handle partial failures in a saga?
**Answer**: Each step must have a compensating action. If a compensation fails, alert operations immediately for manual intervention. Consider idempotent compensations. Track saga state in persistent storage. Implement retry logic for transient failures in compensations.

### Q4: When should you use CQRS vs simple CRUD?
**Answer**: CQRS adds significant complexity. Use it when you have distinct read/write patterns with different performance requirements, when reads vastly outnumber writes, when you need different data models for different audiences, or when building event sourcing infrastructure. For most applications, simple CRUD with database views is sufficient.

### Q5: How do you prevent retry storms?
**Answer**: Implement exponential backoff with jitter. Cap the maximum number of retries. Use circuit breakers to stop retrying when downstream is unhealthy. Consider using a message queue for async retries instead of immediate client-side retries.
