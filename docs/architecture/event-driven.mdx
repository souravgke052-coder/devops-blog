# Event-Driven Architecture

sidebar_position: 5

## Overview

Event-driven architecture (EDA) decouples services through asynchronous event communication. Rather than services calling each other directly, they emit events and react to events from other services. GCP provides Pub/Sub for messaging, Eventarc for event routing, and Cloud Functions/Cloud Run for event processing.

## When To Use

- Building loosely coupled microservices
- Processing asynchronous workflows (order fulfillment, notifications)
- Handling variable traffic with buffering
- Maintaining eventual consistency across services
- Building real-time data pipelines
- Implementing audit logs and audit trails

## When NOT To Use

- Operations requiring immediate response (not eventual consistency tolerance)
- Simple request-response APIs where async doesn't add value
- Transactions requiring strong consistency (consider synchronous patterns)
- Low-latency requirements where queue overhead is unacceptable
- When debugging complexity exceeds benefits

## Core Concepts

### Event Types
- **Domain Events**: Business significant occurrences (OrderPlaced, PaymentSucceeded)
- **Integration Events**: Cross-service communication (UserCreated triggers EmailService)
- **System Events**: Infrastructure notifications (Cloud Storage ObjectFinalized)

### Messaging Patterns
- **Pub/Sub**: One-to-many fan-out messaging
- **Event Streaming**: Ordered, durable event log (via Pub/Sub Lite or Dataflow)
- **Command Messages**: Request-response style via reply topic

## Real Production Example

### Uber-Style Event-Driven Order Processing

```typescript
// Order Service - Publishes domain events
import { PubSub } from '@google-cloud/pubsub';
import express from 'express';

const pubsub = new PubSub();

const app = express();

// POST /orders
app.post('/orders', async (req, res) => {
  const { items, customerId, deliveryAddress } = req.body;

  const db = await getPool().connect();

  try {
    await db.query('BEGIN');

    // Create order
    const orderResult = await db.query(`
      INSERT INTO orders (customer_id, status, delivery_address, created_at)
      VALUES ($1, 'PENDING', $2, NOW())
      RETURNING *
    `, [customerId, JSON.stringify(deliveryAddress)]);

    const order = orderResult.rows[0];

    // Create order items
    for (const item of items) {
      await db.query(`
        INSERT INTO order_items (order_id, product_id, quantity, price)
        VALUES ($1, $2, $3, $4)
      `, [order.id, item.productId, item.quantity, item.price]);
    }

    await db.query('COMMIT');

    // Publish domain event (async, don't await)
    publishEvent('order.created', {
      orderId: order.id,
      customerId,
      items,
      total: calculateTotal(items),
      timestamp: new Date().toISOString()
    }).catch(err => console.error('Failed to publish event:', err));

    res.status(201).json(order);
  } catch (error) {
    await db.query('ROLLBACK');
    throw error;
  } finally {
    db.release();
  }
});

// PATCH /orders/:id/confirm - confirmation flow
app.patch('/orders/:id/confirm', async (req, res) => {
  const { id } = req.params;
  const { paymentMethodId } = req.body;

  const db = await getPool().connect();

  try {
    await db.query('BEGIN');

    // Lock order row
    const orderResult = await db.query(
      'SELECT * FROM orders WHERE id = $1 FOR UPDATE',
      [id]
    );

    if (orderResult.rows.length === 0) {
      return res.status(404).json({ error: 'Order not found' });
    }

    const order = orderResult.rows[0];

    if (order.status !== 'PENDING') {
      return res.status(400).json({ error: 'Invalid order state' });
    }

    // Process payment (external call)
    const paymentResult = await processPayment(order.id, paymentMethodId);

    if (paymentResult.success) {
      await db.query(`
        UPDATE orders SET status = 'CONFIRMED', updated_at = NOW()
        WHERE id = $1
      `, [id]);

      // Publish events
      await Promise.all([
        publishEvent('order.confirmed', { orderId: id, paymentId: paymentResult.paymentId }),
        publishEvent('payment.captured', {
          orderId: id,
          amount: paymentResult.amount,
          paymentId: paymentResult.paymentId
        })
      ]);

      res.json({ status: 'CONFIRMED', paymentId: paymentResult.paymentId });
    } else {
      await db.query(`
        UPDATE orders SET status = 'PAYMENT_FAILED', updated_at = NOW()
        WHERE id = $1
      `, [id]);

      await publishEvent('order.payment_failed', {
        orderId: id,
        reason: paymentResult.error
      });

      res.status(400).json({ error: 'Payment failed', reason: paymentResult.error });
    }

    await db.query('COMMIT');
  } catch (error) {
    await db.query('ROLLBACK');
    throw error;
  } finally {
    db.release();
  }
});

// Event publishing helper
async function publishEvent(type: string, data: any): Promise<void> {
  const topic = pubsub.topic(getTopicForEvent(type));

  await topic.publishMessage({
    attributes: {
      eventType: type,
      timestamp: new Date().toISOString(),
      source: 'order-service',
      version: '1.0'
    },
    data: Buffer.from(JSON.stringify(data))
  });
}

function getTopicForEvent(type: string): string {
  const topicMap: Record<string, string> = {
    'order.created': 'order-events',
    'order.confirmed': 'order-events',
    'order.payment_failed': 'order-events',
    'payment.captured': 'payment-events'
  };
  return topicMap[type] || 'generic-events';
}
```

## Common Patterns / Anti-patterns

### Pattern 1: Event Sourcing

```typescript
// Event Store implementation
interface Event<T = any> {
  id: string;
  aggregateId: string;
  type: string;
  payload: T;
  metadata: {
    timestamp: string;
    userId?: string;
    correlationId?: string;
  };
  version: number;
}

class EventStore {
  constructor(private db: Pool) {}

  async append<T>(aggregateId: string, event: Omit<Event<T>, 'id' | 'version'>): Promise<Event<T>> {
    const client = await this.db.connect();

    try {
      await client.query('BEGIN');

      // Get current version
      const versionResult = await client.query(`
        SELECT COALESCE(MAX(version), 0) as version
        FROM events WHERE aggregate_id = $1
      `, [aggregateId]);

      const newVersion = versionResult.rows[0].version + 1;

      // Insert event
      const insertResult = await client.query(`
        INSERT INTO events (aggregate_id, type, payload, metadata, version, created_at)
        VALUES ($1, $2, $3, $4, $5, NOW())
        RETURNING *
      `, [
        aggregateId,
        event.type,
        JSON.stringify(event.payload),
        JSON.stringify(event.metadata),
        newVersion
      ]);

      await client.query('COMMIT');

      return {
        id: insertResult.rows[0].id,
        aggregateId,
        type: event.type,
        payload: event.payload,
        metadata: event.metadata,
        version: newVersion
      };
    } catch (error) {
      await client.query('ROLLBACK');
      throw error;
    } finally {
      client.release();
    }
  }

  async getEvents(aggregateId: string): Promise<Event[]> {
    const result = await this.db.query(`
      SELECT * FROM events
      WHERE aggregate_id = $1
      ORDER BY version ASC
    `, [aggregateId]);

    return result.rows.map(row => ({
      id: row.id,
      aggregateId: row.aggregate_id,
      type: row.type,
      payload: row.payload,
      metadata: row.metadata,
      version: row.version
    }));
  }

  // Rebuild aggregate state
  async rebuildState<T>(aggregateId: string): Promise<T> {
    const events = await this.getEvents(aggregateId);

    return events.reduce((state, event) => {
      return applyEvent(state, event);
    }, {} as T);
  }
}
```

### Pattern 2: Outbox Pattern for Reliable Events

```typescript
// Outbox pattern implementation
class OrderService {
  constructor(
    private db: Pool,
    private pubsub: PubSub
  ) {}

  async createOrder(orderData: OrderData): Promise<Order> {
    const client = await this.db.connect();

    try {
      await client.query('BEGIN');

      // Insert order
      const orderResult = await client.query(`
        INSERT INTO orders (customer_id, total, status)
        VALUES ($1, $2, 'PENDING')
        RETURNING *
      `, [orderData.customerId, orderData.total]);

      const order = orderResult.rows[0];

      // Insert into outbox table (transactional outbox)
      await client.query(`
        INSERT INTO outbox (aggregate_type, aggregate_id, event_type, payload)
        VALUES ('Order', $1, 'OrderCreated', $2)
      `, [order.id, JSON.stringify({
        orderId: order.id,
        customerId: orderData.customerId,
        total: orderData.total
      })]);

      await client.query('COMMIT');

      return order;
    } finally {
      client.release();
    }
  }
}

// Outbox processor (runs as separate service)
exportconst processOutbox = async () => {
  const outboxDb = await getOutboxPool();
  const pubsub = new PubSub();

  while (true) {
    const events = await outboxDb.query(`
      SELECT * FROM outbox
      WHERE processed = false
      ORDER BY created_at ASC
      LIMIT 100
      FOR UPDATE SKIP LOCKED
    `);

    for (const event of events.rows) {
      try {
        const topic = getTopicForEvent(event.event_type);

        await pubsub.topic(topic).publishMessage({
          attributes: {
            eventType: event.event_type,
            aggregateId: event.aggregate_id
          },
          data: Buffer.from(event.payload)
        });

        await outboxDb.query(
          'UPDATE outbox SET processed = true, processed_at = NOW() WHERE id = $1',
          [event.id]
        );

        console.log(`Published event ${event.id}`);
      } catch (error) {
        console.error(`Failed to publish event ${event.id}:`, error);
      }
    }

    // Process every 100ms
    await sleep(100);
  }
};
```

### Anti-patterns

| Anti-pattern | Problem | Solution |
|-------------|---------|----------|
| Chatty events | Many small messages | Batch events, use message schemas |
| Event versioning | Schema changes break consumers | Version in payload, backward compatibility |
| Synchronous events | Tight coupling, cascade failures | Use async publishing |
| No dead letter handling | Poison messages block queue | Configure DLQ, alerting |
| Eventual consistency confusion | Data appears stale | Document consistency guarantees |

## Interview Questions

### Q1: How do you ensure exactly-once delivery in event-driven systems?
**Answer**: Enable `enable_exactly_once_delivery` on Pub/Sub subscriptions. On the consumer side, implement idempotency using deduplication keys (store processed event IDs with TTL). Use transactional outbox pattern to guarantee event publication alongside business data.

### Q2: What is the outbox pattern and why use it?
**Answer**: The outbox pattern solves the dual-write problem where updating a database and publishing an event aren't atomic. By writing events to an "outbox" table in the same transaction as business data, and having a separate process poll and publish those events, you guarantee at-least-once delivery without distributed transactions.

### Q3: How do you handle schema evolution in events?
**Answer**: Use versioning in the event type or payload. Maintain backward compatibility by adding optional fields rather than required. Version migrations should be handled by upcasters. Document schema changes and support old versions for a migration period. Consider schema registry for validation.

### Q4: When would you use Eventarc over Pub/Sub directly?
**Answer**: Eventarc for GCP event sources (Cloud Storage, Cloud Build, Cloud Scheduler) where you want Cloud Events standard format. Direct Pub/Sub for application-level events, cross-project communication, or when you need more control over message delivery.

### Q5: How do you debug event flow issues?
**Answer**: Use correlation IDs across all events. Enable Cloud Audit logs. Check subscription metrics (backlog, age). Use Cloud Trace for distributed tracing. Implement event replay testing. Monitor DLQ depth.
