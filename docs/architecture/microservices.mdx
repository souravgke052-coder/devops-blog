# Microservices Architecture

sidebar_position: 3

## Overview

Microservices architecture decomposes applications into small, autonomous services that own their data and communicate via well-defined APIs. On GCP, we leverage Cloud Run, Cloud SQL, and Pub/Sub to build resilient, scalable microservices.

## When To Use

- Building applications that need independent deployability
- Teams that need autonomy in technology choices
- Services with varying scaling requirements
- Systems requiring fault isolation
- Applications expected to grow in complexity

## When NOT To Use

- Simple CRUD applications with predictable load
- Small teams (< 5 developers) without DevOps maturity
- When time-to-market is critical and complexity is manageable
- Services that are tightly coupled by nature
- When operational overhead exceeds benefits

## Core Concepts

### Service Design Principles
- **Single Responsibility**: Each service owns one domain
- **API-First**: Well-defined contracts using OpenAPI
- **Database per Service**: No shared databases
- **Smart Endpoints, Dumb Pipes**: Business logic in services
- ** Decentralized Data**: Each service manages its own data

### Boundaries
- **Domain Boundaries**: Align with business capabilities
- **Team Boundaries**: Inverse Conway Maneuver
- **Deployment Boundaries**: Independent release cycles

## Real Production Example

### Stripe-Style Payment Microservices

Stripe processes millions of transactions daily using microservices architecture:

```typescript
// Payment Service - Cloud Run
import express from 'express';
import { Pool } from 'pg';

const app = express();
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 20,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
});

// Health check
app.get('/healthz', async (req, res) => {
  try {
    await pool.query('SELECT 1');
    res.json({ status: 'healthy' });
  } catch (error) {
    res.status(503).json({ status: 'unhealthy' });
  }
});

// Create payment intent
app.post('/v1/payment_intents', async (req, res) => {
  const { amount, currency, customer_id, metadata } = req.body;

  // Validate request
  if (!amount || !currency) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Create payment intent record
    const result = await client.query(`
      INSERT INTO payment_intents (
        id, amount, currency, customer_id, metadata,
        status, created_at
      ) VALUES (
        gen_random_uuid(), $1, $2, $3, $4, 'requires_payment_method', NOW()
      ) RETURNING *
    `, [amount, currency, customer_id, JSON.stringify(metadata)]);

    const paymentIntent = result.rows[0];

    // Publish event for other services
    await pubsub.topic('payment-events').publish({
      type: 'PAYMENT_INTENT_CREATED',
      data: {
        id: paymentIntent.id,
        customer_id,
        amount,
        currency
      }
    });

    await client.query('COMMIT');

    return res.status(201).json({
      id: paymentIntent.id,
      object: 'payment_intent',
      amount,
      currency,
      status: paymentIntent.status
    });
  } catch (error) {
    await client.query('ROLLBACK');
    console.error('Payment intent creation failed:', error);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
});

// Confirm payment
app.post('/v1/payment_intents/:id/confirm', async (req, res) => {
  const { payment_method_id } = req.body;
  const { id } = req.params;

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Get payment intent
    const intentResult = await client.query(
      'SELECT * FROM payment_intents WHERE id = $1 FOR UPDATE',
      [id]
    );

    if (intentResult.rows.length === 0) {
      return res.status(404).json({ error: 'Payment intent not found' });
    }

    const intent = intentResult.rows[0];

    if (intent.status !== 'requires_payment_method') {
      return res.status(400).json({ error: 'Invalid payment intent status' });
    }

    // Process payment (call payment processor, Stripe, etc.)
    const processorResult = await processWithPaymentProcessor({
      amount: intent.amount,
      paymentMethodId: payment_method_id
    });

    if (processorResult.success) {
      await client.query(`
        UPDATE payment_intents
        SET status = 'succeeded', updated_at = NOW()
        WHERE id = $1
      `, [id]);

      await pubsub.topic('payment-events').publish({
        type: 'PAYMENT_SUCCEEDED',
        data: { payment_intent_id: id, processor_id: processorResult.id }
      });
    } else {
      await client.query(`
        UPDATE payment_intents
        SET status = 'requires_payment_method', updated_at = NOW()
        WHERE id = $1
      `, [id]);
    }

    await client.query('COMMIT');

    return res.json({ status: 'confirmed' });
  } catch (error) {
    await client.query('ROLLBACK');
    throw error;
  } finally {
    client.release();
  }
});

export default app;
```

## Common Patterns / Anti-patterns

### Service Communication Patterns

#### Synchronous: REST/gRPC

```typescript
// gRPC service definition
syntax = "proto3";

package payments;

service PaymentService {
  rpc CreatePaymentIntent (CreatePaymentIntentRequest) returns (PaymentIntent);
  rpc ConfirmPayment (ConfirmPaymentRequest) returns (PaymentConfirmation);
  rpc GetPayment (GetPaymentRequest) returns (Payment);
}

message CreatePaymentIntentRequest {
  int64 amount = 1;
  string currency = 2;
  string customer_id = 3;
  map<string, string> metadata = 4;
}
```

#### Asynchronous: Pub/Sub

```typescript
// Publishing events
import { PubSub } from '@google-cloud/pubsub';

const pubsub = new PubSub();

app.post('/orders', async (req, res) => {
  const order = await orderService.create(req.body);

  // Async event - don't wait for consumers
  await pubsub.topic('order-events').publish({
    attributes: {
      eventType: 'ORDER_CREATED',
      timestamp: new Date().toISOString()
    },
    messages: [{
      data: Buffer.from(JSON.stringify(order))
    }]
  });

  // Return immediately
  res.status(201).json(order);
});
```

### Anti-patterns

| Anti-pattern | Problem | Solution |
|-------------|---------|----------|
| Distributed Monolith | Services are deployable but tightly coupled | Define clear domain boundaries |
| Shared Database | Coupling, scaling limits | Database per service |
| Chatty Communication | High latency, network overhead | Batch requests, BFF pattern |
| Shared Libraries | Coupling, version conflicts | API contracts, sidecar pattern |
| No Ownership | Unclear responsibility | Team-aligned service ownership |

## Monitoring

### Distributed Tracing with Cloud Trace

```typescript
import { TraceAgent } from '@google-cloud/trace-agent';

// Initialize tracing
const traceAgent = new TraceAgent({
  projectId: process.env.GCP_PROJECT,
  serviceName: 'payment-service',
  serviceVersion: process.env.VERSION
});

// Propagate trace context
app.use((req, res, next) => {
  const traceHeader = req.headers['x-cloud-trace-context'];
  if (traceHeader) {
    traceAgent.setCurrentContext(traceHeader);
  }
  next();
});

// Trace outgoing requests
app.get('/v1/orders/:id', async (req, res) => {
  const span = traceAgent.createChildSpan({ name: 'getOrder' });

  try {
    const order = await orderService.findById(req.params.id);
    span.end();
    res.json(order);
  } catch (error) {
    span.end({
      labels: { error: error.message }
    });
    throw error;
  }
});
```

## Troubleshooting

### Common Microservices Issues

| Symptom | Likely Cause | Resolution |
|---------|--------------|------------|
| 503 after deployment | Service not ready | Check min instances, readiness probe |
| Intermittent timeouts | Cold starts | Set min instances > 0 |
| High latency | Database connection pool | Increase pool size or use connection proxy |
| Memory spikes | Memory limit too low | Increase memory, check for leaks |
| Failed pub/sub messages | Dead letter queue overflow | Check subscription errors |

## Interview Questions

### Q1: How do you handle data consistency across microservices?
**Answer**: Use eventual consistency with event-driven patterns. Each service owns its data and publishes events. For transactions spanning services, use the Saga pattern with compensating transactions. Avoid distributed transactions (2PC) due to availability implications.

### Q2: What are the key differences between REST and gRPC for service communication?
**Answer**: REST uses JSON over HTTP/1.1, is human-readable, and widely supported. gRPC uses Protocol Buffers over HTTP/2, is binary and more efficient, supports bidirectional streaming, and provides type safety. gRPC is better for internal service-to-service communication; REST for external APIs.

### Q3: How do you prevent cascading failures?
**Answer**: Implement circuit breakers, bulkheads (resource isolation), timeouts on all outgoing calls, retry with exponential backoff, and graceful degradation. Load shedding helps protect the system when overwhelmed.

### Q4: What is the strangler fig pattern and when would you use it?
**Answer**: Gradually migrate a monolith by routing specific features to new microservices while keeping the old system running. Use it when you need to migrate incrementally, can't do a big-bang rewrite, and want to reduce risk by running both systems in parallel.

### Q5: How do you manage API versioning across services?
**Answer**: Use URL versioning (v1, v2) for major changes, header versioning for minor changes. Maintain backward compatibility within a version. Implement feature flags for gradual rollouts. Document changes via OpenAPI specifications.
