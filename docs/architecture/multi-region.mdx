# Multi-Region Architecture

sidebar_position: 6

## Overview

Multi-region architecture distributes application resources across multiple geographic locations to achieve high availability, reduce latency for global users, and provide disaster recovery capabilities. GCP offers various multi-region options ranging from simple load balancing to active-active deployments with data replication.

## When To Use

- Applications requiring 99.99%+ availability SLAs
- Global user bases requiring low-latency access
- Regulatory requirements for data residency
- Disaster recovery planning for critical systems
- High availability for stateless and stateful workloads

## When NOT To Use

- Applications serving primarily local users
- When additional latency of cross-region communication is unacceptable
- When cost of multi-region exceeds business value
- When RPO < 1 second is required (consider synchronous replication)
- Simple applications that don't justify operational complexity

## Core Concepts

### Multi-Region Deployment Models

| Model | Description | RPO | RTO | Use Case |
|-------|-------------|-----|-----|----------|
| Active-Active | All regions serve traffic | ~0 | Minutes | Global applications |
| Active-Passive | Primary region serves, secondary standby | Seconds-Hours | Minutes-Hours | DR-focused |
| Pilot Light | Minimal warm resources in secondary | Minutes | Minutes-Hours | Cost-optimized DR |
| Warm Standby | Scaled-down full stack | Seconds | Minutes | Balanced DR |

### Data Replication Strategies

```
Replication Types:

Synchronous (RPO=0)
┌──────────┐ ───────► ┌──────────┐
│  Write   │          │  Read    │
│  Primary │ ───────► │  Replica │
└──────────┘          └──────────┘
   Same time ───────► (Writes wait)

Asynchronous (RPO=Seconds)
┌──────────┐          ┌──────────┐
│  Write   │ ───────► │  Read    │
│  Primary │          │  Replica │
└──────────┘          └──────────┘
   Write ───────► (Delayed)

Eventual Consistency (RPO=Minutes)
┌──────────┐          ┌──────────┐
│  Write   │ ───────► │  Read    │
│  Primary │ Event    │  Replica │
└──────────┘ ───────► └──────────┘
```

## Real Production Example

### Netflix-Style Multi-Region API Gateway

```typescript
// Multi-region Cloud Run deployment with regional affinity
import express from 'express';
import { CloudTrace } from '@google-cloud/trace-agent';

const app = express();
const trace = new CloudTrace();

// Request tracing
app.use((req, res, next) => {
  const span = trace.startSpan(`request:${req.path}`);
  span.addLabel('method', req.method);
  span.addLabel('region', process.env.GCP_REGION || 'unknown');

  res.on('finish', () => {
    span.end();
  });

  next();
});

// Health check for load balancer
app.get('/healthz', async (req, res) => {
  const checks = await Promise.allSettled([
    checkDatabase(),
    checkCache(),
    checkDependencies()
  ]);

  const healthy = checks.every(c => c.status === 'fulfilled');

  if (healthy) {
    res.json({
      status: 'healthy',
      region: process.env.GCP_REGION,
      timestamp: new Date().toISOString(),
      version: process.env.VERSION
    });
  } else {
    res.status(503).json({
      status: 'unhealthy',
      region: process.env.GCP_REGION,
      checks: checks.map((c, i) => ({
        name: ['database', 'cache', 'dependencies'][i],
        status: c.status
      }))
    });
  }
});

// Regional API endpoint
app.get('/api/content/:id', async (req, res) => {
  const { id } = req.params;

  // Check cache first (local Memorystore)
  const cached = await redis.get(`content:${id}`);
  if (cached) {
    return res.json({ source: 'cache', data: JSON.parse(cached) });
  }

  // Fetch from primary database
  const content = await db.query(
    'SELECT * FROM content WHERE id = $1',
    [id]
  );

  if (!content.rows.length) {
    return res.status(404).json({ error: 'Content not found' });
  }

  // Store in cache with TTL
  await redis.setex(`content:${id}`, 300, JSON.stringify(content.rows[0]));

  res.json({ source: 'database', data: content.rows[0] });
});

// Geolocation-aware content
app.get('/api/local-content', async (req, res) => {
  const { lat, lng } = req.query;

  // Get user's approximate region from headers or IP
  const userRegion = getRegionFromLatLng(parseFloat(lat), parseFloat(lng));

  // For EU users, prefer EU content sources
  const preferredContent = userRegion.startsWith('eu')
    ? await getEUPreferredContent()
    : await getUSPreferredContent();

  res.json({
    userRegion,
    content: preferredContent,
    servedFromRegion: process.env.GCP_REGION
  });
});

// Cross-region data sync status
app.get('/api/sync-status', async (req, res) => {
  const syncStatus = await checkCrossRegionSync();

  res.json({
    status: syncStatus.synced ? 'in_sync' : 'lagging',
    lagSeconds: syncStatus.lagSeconds,
    lastSyncTimestamp: syncStatus.lastSync,
    regions: {
      'us-central1': syncStatus.regions['us-central1'],
      'europe-west1': syncStatus.regions['europe-west1'],
      'asia-east1': syncStatus.regions['asia-east1']
    }
  });
});

function getRegionFromLatLng(lat: number, lng: number): string {
  // Simplified region determination
  if (lng < -30 && lng > -60 && lat > 35 && lat < 70) return 'eu-west';
  if (lat > -60 && lat < 60 && lng > 100 && lng < 150) return 'asia-east';
  return 'us-central';
}
```

## Common Patterns / Anti-patterns

### RPO/RTO Targets by Service Type

| Service | Typical RPO | Typical RTO | Implementation |
|---------|-------------|-------------|----------------|
| Static Content | 0 (CDN) | Minutes | Cloud CDN, multi-region GCS |
| API Services | Seconds | Minutes | Cloud Run multi-region, health checks |
| User Data | Seconds | Minutes | Cloud SQL replicas, Spanner |
| Transaction Data | 0 | Minutes | Spanner multi-region,同步复制 |
| Analytics | Minutes-Hours | Hours | BigQuery multi-region |
| Session State | N/A (stateless) | N/A | Redis in each region |

### Active-Active Pattern

```typescript
// Conflict resolution for active-active writes
class ConflictResolver<T> {
  constructor(
    private strategy: 'last-write-wins' | 'server-wins' | 'manual-merge'
  ) {}

  resolve(local: T, remote: T, timestampLocal: Date, timestampRemote: Date): T {
    switch (this.strategy) {
      case 'last-write-wins':
        return timestampLocal > timestampRemote ? local : remote;

      case 'server-wins':
        return remote; // Always prefer server data

      case 'manual-merge':
        return this.merge(local, remote); // Custom merge logic

      default:
        throw new Error(`Unknown strategy: ${this.strategy}`);
    }
  }
}
```

### Anti-patterns

| Anti-pattern | Problem | Solution |
|-------------|---------|----------|
| Synchronous cross-region calls | High latency, cascade failures | Async events, local reads |
| Single-region writes everywhere | Write bottleneck, latency | Geographically distributed writes |
| No conflict resolution | Data corruption | Implement CRDT or conflict resolution |
| Inconsistent cache invalidation | Stale data | Use regional caches with appropriate TTL |
| No regional affinity | Unnecessary cross-region traffic | Route to nearest region |

## Interview Questions

### Q1: What is the difference between RPO and RTO?
**Answer**: RPO (Recovery Point Objective) is the maximum acceptable data loss measured in time - how far back in time data might need to be recovered from. RTO (Recovery Time Objective) is the maximum acceptable downtime before systems must be restored. RPO determines backup frequency; RTO determines DR infrastructure investment.

### Q2: How do you handle split-brain scenarios in active-active?
**Answer**: Use consensus-based locking for critical operations, implement fencing tokens to prevent split-brain writes, use eventual consistency with conflict resolution (CRDTs or last-write-wins), and have clear manual override procedures. For truly critical data, active-active may not be appropriate.

### Q3: What are the trade-offs of Cloud Spanner vs multi-region Cloud SQL?
**Answer**: Spanner provides strong consistency globally with automatic replication and no read replicas needed. Cloud SQL is simpler and cheaper for regional workloads with read replicas for read scaling. Spanner excels for globally distributed writes with strong consistency; Cloud SQL is better for read-heavy workloads in specific regions.

### Q4: How do you test multi-region failover?
**Answer**: Conduct regular chaos engineering exercises, test by manually failing over primary databases, verify DNS TTL settings allow quick failover, test with actual user traffic simulation, and validate RTO/RPO during tests. Use Runbook to document failover procedures and test them quarterly.

### Q5: What is the minimum latency you can achieve cross-region?
**Answer**: Physical distance determines minimum latency. Approximate round-trip times: US-East to US-West ~60-80ms, US to Europe ~80-100ms, US to Asia ~150-200ms. This is before network overhead. Design for these latencies by minimizing cross-region calls and using local reads.
