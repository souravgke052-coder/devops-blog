# Security Architecture

sidebar_position: 9

## Overview

Security architecture for the DevOps Blog platform implements defense-in-depth principles using GCP's native security services. This document covers zero trust networking, identity and access management, data protection, compliance, and security monitoring.

## When To Use

- Designing network security for services
- Implementing identity and access controls
- Planning data protection and encryption
- Setting up security monitoring and alerting
- Meeting compliance requirements (SOC2, HIPAA, PCI-DSS)

## When NOT to Use

- Security is not optional - all architectures must consider security
- Skipping security for speed undermines the platform
- Security should be built-in, not bolted-on

## Core Concepts

### Zero Trust Principles

1. **Never Trust, Always Verify**: Every request is authenticated and authorized
2. **Least Privilege Access**: Grant minimum permissions required
3. **Assume Breach**: Design for containment and detection
4. **Verify Explicitly**: Use all available data points for decisions

## Real Production Example

### Zero Trust Service-to-Service Authentication

```typescript
// Service mesh authentication using Workload Identity
// Cloud Run service with secure service communication

import { GoogleAuth } from 'google-auth-library';
import express from 'express';

const app = express();

// Middleware for service-to-service authentication
const serviceAuth = async (req: express.Request, res: express.Response, next: express.NextFunction) => {
  const authHeader = req.headers.authorization;

  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing authorization token' });
  }

  const token = authHeader.substring(7);

  try {
    // Verify JWT token
    const decoded = await verifyServiceToken(token);

    // Add service identity to request context
    req.serviceIdentity = {
      serviceAccount: decoded.service_account,
      audience: decoded.aud,
      issuer: decoded.iss
    };

    // Log for audit
    console.log(JSON.stringify({
      event: 'service_auth_success',
      caller: decoded.service_account,
      resource: req.path,
      timestamp: new Date().toISOString()
    }));

    next();
  } catch (error) {
    console.log(JSON.stringify({
      event: 'service_auth_failure',
      error: (error as Error).message,
      timestamp: new Date().toISOString()
    }));
    return res.status(403).json({ error: 'Invalid token' });
  }
};

// Verify service account JWT
async function verifyServiceToken(token: string): Promise<JWTPayload> {
  const auth = new GoogleAuth();

  // Verify token using Google's public keys
  const ticket = await auth.verifyIdToken({
    idToken: token,
    audience: process.env.SERVICE_AUDIENCE
  });

  const payload = ticket.getPayload();

  if (!payload) {
    throw new Error('Invalid token payload');
  }

  // Verify issuer is Google
  if (!['https://accounts.google.com', 'https://cloud.google.com/'].includes(payload.iss)) {
    throw new Error('Invalid token issuer');
  }

  return payload as JWTPayload;
}

// Example: Calling another service with authentication
async function callDownstreamService(serviceUrl: string, data: any): Promise<any> {
  const auth = new GoogleAuth({
    scopes: ['https://www.googleapis.com/auth/cloud-platform']
  });

  const client = await auth.getClient();

  // Get access token for service account
  const tokenResponse = await client.getAccessToken();

  const response = await fetch(serviceUrl, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${tokenResponse.token}`
    },
    body: JSON.stringify(data)
  });

  if (!response.ok) {
    throw new Error(`Downstream service error: ${response.status}`);
  }

  return response.json();
}

// API endpoints with service authentication
app.get('/api/internal/metrics', serviceAuth, async (req, res) => {
  const metrics = await getMetrics();
  res.json(metrics);
});

app.post('/api/internal/sync', serviceAuth, async (req, res) => {
  const result = await callDownstreamService(
    'https://sync-service.example.com/api/sync',
    req.body
  );
  res.json(result);
});

// Health endpoint (no auth for load balancer)
app.get('/healthz', (req, res) => {
  res.json({ status: 'healthy' });
});
```

### Secret Management Implementation

```typescript
// Secret Manager integration for secure secret storage
import { SecretManagerServiceClient } from '@google-cloud/secret-manager';

const secretClient = new SecretManagerServiceClient();

interface SecretConfig {
  secretId: string;
  version?: string;
  addToEnv?: boolean;
}

// Cache secrets in memory (per-instance)
const secretCache = new Map<string, { value: string; expires: number }>();
const CACHE_TTL_MS = 5 * 60 * 1000; // 5 minutes

async function getSecret(config: SecretConfig): Promise<string> {
  const { secretId, version = 'latest', addToEnv = false } = config;

  // Check cache first
  const cached = secretCache.get(secretId);
  if (cached && cached.expires > Date.now()) {
    return cached.value;
  }

  // Fetch from Secret Manager
  const [accessResponse] = await secretClient.accessSecretVersion({
    name: `projects/${process.env.PROJECT_ID}/secrets/${secretId}/versions/${version}`
  });

  const secretValue = accessResponse.payload?.data?.toString();

  if (!secretValue) {
    throw new Error(`Failed to access secret: ${secretId}`);
  }

  // Cache the secret
  secretCache.set(secretId, {
    value: secretValue,
    expires: Date.now() + CACHE_TTL_MS
  });

  // Optionally add to process.env
  if (addToEnv) {
    process.env[secretId] = secretValue;
  }

  return secretValue;
}

// Rotating secret handler
async function rotateSecret(secretId: string, newValue: string): Promise<void> {
  // Add new version
  const [addedVersion] = await secretClient.addSecretVersion({
    parent: `projects/${process.env.PROJECT_ID}/secrets/${secretId}`,
    payload: {
      data: Buffer.from(newValue)
    }
  });

  const versionId = addedVersion.name?.split('/').pop();

  // Update service to use new version (requires service restart in Cloud Run)
  console.log(`Secret ${secretId} rotated to version ${versionId}`);

  // Clear cache to force reload
  secretCache.delete(secretId);
}

// Initialize secrets on startup
exportasync function initializeSecrets(): Promise<void> {
  const requiredSecrets = [
    { secretId: 'database-url', addToEnv: true },
    { secretId: 'api-key', addToEnv: false },
    { secretId: 'jwt-secret', addToEnv: true }
  ];

  await Promise.all(
    requiredSecrets.map(secret => getSecret(secret))
  );

  console.log('All secrets initialized successfully');
}
```

### Security Headers and Input Validation

```typescript
// Security middleware for all Cloud Run services
import helmet from 'helmet';
import rateLimit from 'express-rate-limit';
import { body, validationResult } from 'express-validator';

// Helmet for security headers
app.use(helmet());

// Content Security Policy
app.use((req, res, next) => {
  res.setHeader(
    'Content-Security-Policy',
    "default-src 'self'; " +
    "script-src 'self'; " +
    "object-src 'none'; " +
    "frame-ancestors 'none'; " +
    "base-uri 'self'; " +
    "form-action 'self';"
  );
  next();
});

// Rate limiting
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100, // limit each IP to 100 requests per windowMs
  message: { error: 'Too many requests, please try again later' },
  standardHeaders: true,
  legacyHeaders: false
});

app.use('/api/', limiter);

// Stricter rate limit for authentication endpoints
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5, // Only 5 attempts per 15 minutes
  message: { error: 'Too many authentication attempts' }
});

app.use('/api/auth/', authLimiter);

// Input validation
exportconst validateCreateArticle = [
  body('title')
    .trim()
    .isLength({ min: 1, max: 200 })
    .withMessage('Title must be 1-200 characters'),

  body('content')
    .trim()
    .isLength({ min: 1, max: 50000 })
    .withMessage('Content must be 1-50000 characters'),

  body('tags')
    .isArray({ max: 10 })
    .withMessage('Maximum 10 tags allowed'),

  body('tags.*')
    .isString()
    .trim()
    .isLength({ min: 1, max: 50 })
    .withMessage('Each tag must be 1-50 characters'),

  (req: express.Request, res: express.Response, next: express.NextFunction) => {
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
      return res.status(400).json({ errors: errors.array() });
    }
    next();
  }
];

app.post('/api/articles', validateCreateArticle, async (req, res) => {
  // Request is validated
});
```

## Common Patterns / Anti-patterns

### Security Patterns

| Pattern | Description | Implementation |
|---------|-------------|----------------|
| Defense in Depth | Multiple security layers | Network + IAM + App + Data |
| Zero Trust | Never trust, always verify | mTLS, JWT validation |
| Least Privilege | Minimum required permissions | Granular IAM roles |
| Secrets Rotation | Automated credential rotation | Secret Manager + Cloud Run restart |
| Immutable Infrastructure | No changes to running instances | Terraform + container images |

### Anti-patterns

| Anti-pattern | Problem | Solution |
|-------------|---------|----------|
| Overly permissive IAM | Lateral movement risk | Granular roles, conditions |
| Hardcoded secrets | Secret exposure | Secret Manager |
| Public cloud resources | Attack surface | Private networking |
| Disabled audit logs | No forensics capability | Cloud Audit Logs |
| No network segmentation | Lateral movement | VPC, firewall rules |

### SIEM Integration

```typescript
// Cloud Logging export to SIEM
exportconst exportSecurityLogs = async (event: CloudFunctionsEvent<any>) => {
  const logEntry = JSON.parse(
    Buffer.from(event.data.message.data, 'base64').toString()
  );

  // Parse and enrich security events
  const securityEvent = {
    timestamp: logEntry.timestamp,
    severity: logEntry.severity,
    service: logEntry.resource?.labels?.service,
    event: parseSecurityEvent(logEntry),
    metadata: {
      environment: process.env.ENVIRONMENT,
      version: process.env.VERSION
    }
  };

  // Send to SIEM (Splunk, Chronicle, etc.)
  await sendToSIEM(securityEvent);
};

function parseSecurityEvent(logEntry: any): ParsedEvent | null {
  const textPayload = logEntry.textPayload || '';
  const jsonPayload = logEntry.jsonPayload || {};

  if (textPayload.includes('authentication failure') || jsonPayload.code === 'AUTH_FAILURE') {
    return {
      type: 'AUTH_FAILURE',
      source: jsonPayload.caller_ip,
      user: jsonPayload.user
    };
  }

  if (textPayload.includes('SQL injection') || textPayload.includes('XSS')) {
    return {
      type: 'ATTACK_DETECTED',
      attackType: textPayload.includes('SQL') ? 'SQL_INJECTION' : 'XSS',
      source: jsonPayload.caller_ip
    };
  }

  return null;
}
```

## Interview Questions

### Q1: Explain zero trust and how you implement it on GCP.
**Answer**: Zero trust means never trusting and always verifying regardless of network location. On GCP, we implement it through: Workload Identity for service-to-service auth, mTLS for internal communication, VPC Service Controls for data exfiltration prevention, Cloud Armor for API protection, and continuous authentication/authorization checks without relying on network location.

### Q2: How do you handle secrets management at scale?
**Answer**: Use Secret Manager with versioned secrets and automatic rotation. Environment variables reference secrets at runtime, never store secrets in code or config files. Implement secret scanning in CI/CD to catch accidental commits. Use Workload Identity to avoid service account keys entirely.

### Q3: What is your approach to incident response?
**Answer**: 1) Detection: Cloud Armor, audit logs, SIEM integration. 2) Containment: Isolate affected services, revoke credentials, block IPs. 3) Investigation: Analyze logs, trace attack path. 4) Eradication: Remove malware, patch vulnerabilities. 5) Recovery: Restore from clean backups. 6) Post-incident: Document, implement preventive measures.

### Q4: How do you ensure compliance with security standards?
**Answer**: Implement security controls mapped to compliance frameworks (SOC2, PCI-DSS). Use binary authorization to ensure only approved container images deploy. Enable comprehensive audit logging with immutable storage. Conduct regular security reviews and penetration testing. Document controls and maintain evidence for audits.

### Q5: What are the most critical security considerations for serverless?
**Answer**: Serverless introduces unique challenges: 1) Function permissions - apply least privilege. 2) Secret management - use Secret Manager. 3) Cold starts - don't skip security headers for speed. 4) Concurrency - understand and limit to prevent runaway costs/attacks. 5) Vendor lock-in security - understand shared responsibility model.
