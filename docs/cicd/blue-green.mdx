# Blue-Green Deployments

## Overview

Blue-green deployment maintains two identical production environments (blue and green), where one serves live traffic while the other remains idle. At deployment time, the new version is deployed to the idle environment, validated, then all traffic is switched at once. This pattern enables instant rollback and zero-downtime releases.

## When To Use

- Applications requiring instant traffic switch
- Environments with compliance requiring validated readiness
- Services with database migrations requiring specific versions
- High-stakes applications where gradual rollout is too risky
- Services behind load balancers with weight control

## When NOT To Use

- Extremely resource-constrained environments (requires double capacity)
- Stateful applications with local storage that's hard to replicate
- Rapid hotfix scenarios where environment switch is too slow

## Core Concepts

### Blue-Green Architecture

```
                    Load Balancer
                         │
                    ┌────┴────┐
                    │         │
                    ▼         ▼
              ┌─────────┐ ┌─────────┐
              │  BLUE   │ │  GREEN  │
              │ (Live)  │ │ (Idle)  │
              │  v1.0   │ │  v1.1   │
              └─────────┘ └─────────┘
                    │         │
                    │    Deploy to idle
                    │    Test & validate
                    │         │
                    └────┬────┘
                         │
                    Switch selector
                    (Instant traffic)
                         │
                         ▼
                   ┌─────────┐
                   │  GREEN  │
                   │ (Live)  │
                   │  v1.1   │
                   └─────────┘
```

### Key Components

| Component | Purpose |
|-----------|---------|
| **Blue Environment** | Currently serving production traffic |
| **Green Environment** | Idle, receives new deployment |
| **Load Balancer** | Routes traffic to active environment |
| **Health Check** | Validates new deployment before switch |
| **Connection Drain** | Graceful handling of in-flight requests |

## Real Production Example

### Complete Blue-Green Deployment Pipeline

```yaml
name: Blue-Green Production Deploy

on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      skip_tests:
        type: boolean
        default: false

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ============================================
  # PRE-DEPLOYMENT
  # ============================================
  pre-deploy:
    name: Pre-deployment Checks
    runs-on: ubuntu-latest
    
    outputs:
      image_tag: ${{ steps.meta.outputs.tags }}
      active_slot: ${{ steps.state.outputs.active_slot }}
      target_slot: ${{ steps.state.outputs.target_slot }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Get current active slot
        id: state
        run: |
          gcloud container clusters get-credentials ${{ vars.PROD_CLUSTER }} --region us-central1
          
          # Get current active from service selector
          ACTIVE=$(kubectl get service api -n production -o jsonpath='{.spec.selector.slot}')
          
          if [ "$ACTIVE" == "blue" ]; then
            TARGET="green"
          else
            TARGET="blue"
          fi
          
          echo "active_slot=$ACTIVE" >> $GITHUB_OUTPUT
          echo "target_slot=$TARGET" >> $GITHUB_OUTPUT
          echo "Deploying to $TARGET slot, currently $ACTIVE is active"
      
      - name: Docker meta
        id: meta
        run: |
          TAG="sha-${{ github.sha }}"
          if [[ "${{ github.event_name }}" == "tag" ]]; then
            TAG=${GITHUB_REF#refs/tags/}
          fi
          echo "tags=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:$TAG" >> $GITHUB_OUTPUT

  # ============================================
  # DEPLOY TO IDLE ENVIRONMENT
  # ============================================
  deploy-idle:
    name: Deploy to ${{ needs.pre-deploy.outputs.target_slot }}
    runs-on: ubuntu-latest
    needs: pre-deploy
    
    permissions:
      id-token: write
      contents: read
    
    steps:
      - uses: actions/checkout@v4
      
      - name: GCP Auth
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ vars.GCP_WIP }}
          service_account: ${{ vars.PROD_SA }}
      
      - name: Deploy to idle slot
        run: |
          gcloud container clusters get-credentials ${{ vars.PROD_CLUSTER }} --region us-central1
          
          DEPLOYMENT=api-${{ needs.pre-deploy.outputs.target_slot }}
          
          # Deploy to idle slot
          kubectl set image deployment/$DEPLOYMENT api=${{ needs.pre-deploy.outputs.image_tag }} --namespace=production
          
          # Update annotation for rollback
          kubectl annotate deployment/$DEPLOYMENT -n production \
            previous-image=$(kubectl get deployment/$DEPLOYMENT -n production -o jsonpath='{.spec.template.spec.containers[0].image}') \
            --overwrite
          
          # Wait for rollout
          kubectl rollout status deployment/$DEPLOYMENT --namespace=production --timeout=15m
          
          echo "Deployed to $DEPLOYMENT: ${{ needs.pre-deploy.outputs.image_tag }}"

  # ============================================
  # VALIDATION
  # ============================================
  validate:
    name: Validate Deployment
    runs-on: ubuntu-latest
    needs: deploy-idle
    if: ${{ !github.event.inputs.skip_tests }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Run validation tests
        run: |
          gcloud container clusters get-credentials ${{ vars.PROD_CLUSTER }} --region us-central1
          
          TARGET=api-${{ needs.pre-deploy.outputs.target_slot }}
          
          echo "Validating deployment: $TARGET"
          
          # Get pod IP
          POD=$(kubectl get pods -n production -l deployment=$TARGET --field-selector=status.phase=Running -o jsonpath='{.items[0].metadata.name}')
          
          # Port forward for testing
          kubectl port-forward -n production pod/$POD 8080:8080 &
          PORTFORWARD_PID=$!
          
          sleep 5
          
          # Run smoke tests
          curl -f http://localhost:8080/health || exit 1
          curl -f http://localhost:8080/ready || exit 1
          
          # Run integration tests
          curl -f http://localhost:8080/api/v1/users || exit 1
          
          kill $PORTFORWARD_PID 2>/dev/null || true
          
          echo "Validation passed"

  # ============================================
  # TRAFFIC SWITCH
  # ============================================
  switch-traffic:
    name: Switch Traffic
    runs-on: ubuntu-latest
    needs: validate
    
    environment:
      name: production
      url: https://myapp.com
    
    permissions:
      id-token: write
      contents: read
    
    steps:
      - uses: actions/checkout@v4
      
      - name: GCP Auth
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ vars.GCP_WIP }}
          service_account: ${{ vars.PROD_SA }}
      
      - name: Switch traffic to new slot
        run: |
          gcloud container clusters get-credentials ${{ vars.PROD_CLUSTER }} --region us-central1
          
          TARGET=${{ needs.pre-deploy.outputs.target_slot }}
          
          echo "Switching traffic to $TARGET..."
          
          # Switch service selector
          kubectl patch service api -n production \
            --type='json' \
            -p="[{\"op\":\"replace\",\"path\":\"/spec/selector/slot\",\"value\":\"$TARGET\"}]"
          
          # Wait for DNS propagation
          sleep 5
          
          # Verify service selector
          SELECTOR=$(kubectl get service api -n production -o jsonpath='{.spec.selector.slot}')
          echo "Service selector now points to: $SELECTOR"
          
          if [ "$SELECTOR" != "$TARGET" ]; then
            echo "Traffic switch failed!"
            exit 1
          fi
      
      - name: Verify switch
        run: |
          # Wait for all pods to receive traffic
          sleep 30
          
          # Health check
          curl -f https://myapp.com/health || exit 1
          
          # Check active pods
          kubectl get pods -n production -l slot=${{ needs.pre-deploy.outputs.target_slot }} --field-selector=status.phase=Running
      
      - name: Update active slot annotation
        run: |
          # Store previous image for future rollback
          PREVIOUS=$(kubectl get deployment/api-${{ needs.pre-deploy.outputs.active_slot }} -n production -o jsonpath='{.spec.template.spec.containers[0].image}')
          kubectl annotate service api -n production last-successful-image=$PREVIOUS --overwrite

  # ============================================
  # CLEANUP OLD ENVIRONMENT
  # ============================================
  cleanup:
    name: Cleanup
    runs-on: ubuntu-latest
    needs: switch-traffic
    if: always()
    
    permissions:
      id-token: write
    
    steps:
      - uses: actions/checkout@v4
      
      - name: GCP Auth
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ vars.GCP_WIP }}
          service_account: ${{ vars.PROD_SA }}
      
      - name: Scale down old deployment
        if: success()
        run: |
          gcloud container clusters get-credentials ${{ vars.PROD_CLUSTER }} --region us-central1
          
          OLD=api-${{ needs.pre-deploy.outputs.active_slot }}
          
          echo "Scaling down $OLD to 0 (keeping for rollback)"
          kubectl scale deployment/$OLD --namespace=production --replicas=0
          
          echo "Cleanup complete. Old version kept at 0 replicas for emergency rollback."
      
      - name: Notify
        if: always()
        uses: slackapi/slack-github-action@v1.25.0
        with:
          payload: |
            {
              "text": "${{ needs.switch-traffic.result == 'success' && ':white_check_mark:' || ':x:' }} Blue-Green Deploy",
              "blocks": [{
                "type": "section",
                "text": {
                  "type": "mrkdwn",
                  "text": "*Blue-Green Deployment*\nStatus: ${{ needs.switch-traffic.result }}\nNew Version: ${{ needs.pre-deploy.outputs.image_tag }}\nActive Slot: ${{ needs.pre-deploy.outputs.target_slot }}\n<${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}|View Details>"
                }
              }]
            }
        env:
          SLACK_WEBHOOK_TYPE: INCOMING_WEBHOOK
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```

      --name=api.myapp.com \

## Common Commands / Steps

### Blue-Green Operations

```bash
# Check current state
kubectl get deployment -n production -l app=api
kubectl get service api -n production -o jsonpath='{.spec.selector.slot}'

# Deploy to idle slot
kubectl set image deployment/api-green api=$IMAGE_TAG -n production
kubectl rollout status deployment/api-green -n production

# Switch traffic
kubectl patch service api -n production -p '{"spec":{"selector":{"slot":"green"}}}'

# Verify switch
kubectl get pods -n production -l slot=green
curl -f https://myapp.com/health

# Instant rollback (switch back)
kubectl patch service api -n production -p '{"spec":{"selector":{"slot":"blue"}}}'

# Scale down old deployment
kubectl scale deployment/api-blue --replicas=0 -n production
```

## Security

### Blue-Green Security Considerations

| Concern | Mitigation |
|---------|------------|
| Double blast radius | Both environments must be secured |
| Credentials on both | Rotate after full switch |
| Traffic interception | Use TLS throughout |
| Configuration drift | Sync configs, use GitOps |

### Resource Optimization

## Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Traffic not switching | DNS TTL too long | Lower TTL before switch, wait |
| Pods not ready | Readiness probe failing | Check logs, increase timeout |
| Connection drops | No graceful drain | Add connection draining |
| Session lost | Session affinity change | Use sticky sessions |
| Both slots active | Selector not updated | Verify service selector |

## Interview Questions

### Q1: What are the advantages of blue-green over rolling deployments?

**Answer:** Blue-green offers instant traffic switch (< 1 minute) versus rolling's gradual approach. Rollback is simply switching back, while rolling requires undoing each pod update. Blue-green allows complete validation before any user impact. However, blue-green requires double the resources (two full environments running) while rolling uses the same resources.

### Q2: How do you handle database migrations in blue-green deployments?

**Answer:** Database migrations are the main challenge. Options: 1) Make migrations backward-compatible (expand-contract pattern), 2) Run migrations before traffic switch as a separate step, 3) Use feature flags to hide incompatible features, 4) For schema changes, run migration during the validation phase, 5) Accept that rollback may not revert schema changes.

### Q3: How does DNS-based traffic switching work?

**Answer:** DNS switching updates A records to point to the new environment's IP. DNS TTL determines how fast propagation occurs. For instant switch, set TTL to 60 seconds or lower before switch, wait for cache expiration, then update. Alternative: use weighted records (Route 53) for gradual shift, or pointer records (CNAME) that swap entirely.

### Q4: What is connection draining and why is it important?

**Answer:** Connection draining allows existing requests to complete before removing instances. Without it, in-flight requests to the old slot get terminated mid-request, causing errors. Implementation: set `terminationGracePeriodSeconds` and configure load balancer drain timeout. Most cloud load balancers have built-in connection draining.

### Q5: How do you validate the idle environment before traffic switch?

**Answer:** 1) Automated smoke tests against idle deployment (health, API, critical paths), 2) Load testing to ensure handles expected traffic, 3) Manual QA verification, 4) Staging environment mirroring production, 5) Synthetic monitoring checks, 6) Database connectivity verification, 7) Downstream dependency validation.
