# Git Workflows & Commands

## Overview

Git is the foundation of modern CI/CD. Every pipeline starts with a code push, and understanding Git workflows is critical for building effective automated pipelines. This guide covers production-grade Git usage, from branching strategies to advanced commands that power enterprise DevOps practices.

## When To Use

- Every software project using version control
- Teams of any size collaborating on code
- CI/CD pipeline integration
- Compliance-required audit trails
- Parallel feature development

## When NOT To Use

- Single-file, throwaway scripts without version history needs
- Large binary files that benefit from specialized version control (use LFS or external storage)
- Projects with no source code (pure data pipelines)

## Core Concepts

### Branching Strategies

#### GitHub Flow (Simple, Trunk-Based)

```
main ────────────────────────────────────────────────────────▶
        │                    │                    │
        ▼                    ▼                    ▼
    feature/login ──────▶ merge ───────────▶ hotfix/critical ──▶ merge
```

Best for: Continuous deployment teams, small teams, web applications.

**Rules:**
1. `main` is always deployable
2. Feature branches from `main`
3. Rebase or merge PR when ready
4. Immediate deployment after merge

#### GitFlow

```
develop ─────────▶ merge ────────▶ release/v1.0 ────▶
     │               │                  │              │
     ▼               ▼                  │              ▼
feature/auth ────▶ merge           merge ◀───────────
                                                      
                                         ▼
main ───────────────────────────────────────────────────────▶
                                             │
                                             ▼
                                       tag: v1.0.0
```

Best for: Scheduled releases, multiple versions in production, enterprise.

**Branches:**
- `main`: Production-ready code
- `develop`: Integration branch for features
- `feature/*`: New features
- `release/*`: Release preparation
- `hotfix/*`: Production fixes

#### Trunk-Based Development

```
main ───▶ commit ───▶ commit ───▶ commit ───▶ commit ───▶ ...
              │            │            │
              ▼            ▼            ▼
          feature/a    feature/b    feature/c  (short-lived, < 2 days)
```

Best for: High-velocity teams, continuous deployment, microservices.

**Rules:**
1. All developers work off `main` (trunk)
2. Feature flags for incomplete features
3. Branches live < 2 days
4. Multiple deploys per day

### Key Git Principles

1. **Atomic Commits**: Each commit represents one logical change
2. **Descriptive Messages**: Follow conventional commits format
3. **Protected Branches**: Never force-push to main
4. **Required Reviews**: All changes go through PR review
5. **Linear History**: Prefer rebase over merge commits on feature branches

## Real Production Example

### Multi-Environment Release Flow

```
1. Developer creates feature branch
   git checkout -b feature/user-auth
   git commit -m "feat(auth): add OAuth2 login"

2. Push and create PR
   git push -u origin feature/user-auth
   gh pr create --title "feat(auth): add OAuth2 login" --reviewer @team

3. CI runs on PR (automatic)
   ├── Lint & Type Check
   ├── Unit Tests
   ├── Integration Tests
   └── Security Scan

4. Code review and approval
   ├── 2 approvals required
   ├── Branch must be up to date with main
   └── Status checks must pass

5. Merge to main (automatic deployment to staging)
   └── Pipeline: build → test → deploy-staging → e2e

6. Create release tag (production deployment)
   git tag v1.2.3 -m "Release v1.2.3"
   git push origin v1.2.3

7. Production deployment triggered
   └── Pipeline: build → deploy-production (with approval)
```

## Common Commands / Steps

### Essential Git Commands

```bash
# ============================================
# Basics
# ============================================
git init                              # Initialize repo
git clone <url>                       # Clone repository
git status                            # Check status
git log --oneline --graph --all       # Visual history

# ============================================
# Branching
# ============================================
git branch                            # List branches
git branch -a                         # List all branches
git checkout -b feature/new-feature   # Create and switch
git switch -c feature/new-feature     # Modern switch
git branch -d feature/old-feature     # Delete local
git push origin --delete feature/old  # Delete remote

# ============================================
# Staging & Committing
# ============================================
git add <file>                        # Stage file
git add -p                            # Stage patches (interactively)
git add .                              # Stage all
git commit -m "feat: add feature"     # Commit with message
git commit --amend                    # Modify last commit
git commit --amend --no-edit          # Add staged to last commit
git commit --fixup <hash>             # Create fixup commit
git commit -s                         # Sign off commit

# ============================================
# Syncing
# ============================================
git fetch                             # Fetch updates
git pull                              # Fetch and merge
git pull --rebase                     # Fetch and rebase
git push                              # Push commits
git push --force-with-lease           # Safe force push
git push --follow-tags                # Push annotated tags

# ============================================
# Inspection & Recovery
# ============================================
git diff                              # Working tree changes
git diff --staged                     # Staged changes
git diff <branch1> <branch2>          # Branch differences
git log                               # Commit history
git log --oneline -10                 # Last 10 commits
git log --author="name"               # Filter by author
git log --since="2 weeks ago"         # Time-based filter
git blame <file>                      # Line-by-line history
git reflog                            # All reference updates
git stash                             # Temporarily stash
git stash pop                         # Apply and remove stash
git stash list                        # List stashes
git stash -p                          # Stash specific changes

# ============================================
# Rewriting History
# ============================================
git rebase -i HEAD~3                  # Interactive rebase
git rebase main                       # Rebase on main
git rebase --onto main feature/a feature/b
git merge --no-ff feature-branch      # Merge with merge commit
git cherry-pick <commit-hash>         # Apply specific commit

# ============================================
# Tags
# ============================================
git tag v1.0.0                        # Lightweight tag
git tag -a v1.0.0 -m "Release v1.0.0" # Annotated tag
git tag -a v1.0.0 <commit-hash>       # Tag specific commit
git push origin --tags                # Push all tags
git tag -d v1.0.0                     # Delete local tag
git push origin :refs/tags/v1.0.0     # Delete remote tag

# ============================================
# Worktree (Isolated Workspaces)
# ============================================
git worktree list                     # List worktrees
git worktree add ../hotfix hotfix/*   # Create worktree
git worktree prune                    # Clean stale worktrees
```

### Advanced Git Operations

```bash
# ============================================
# Bisect (Find Bugs)
# ============================================
git bisect start
git bisect bad                        # Current commit is bad
git bisect good v1.0.0                # Known good commit
# Git checks out middle commit
# Test and mark good or bad
# Repeat until found
git bisect reset                      # End bisect

# ============================================
# Blame (Code History)
# ============================================
git blame src/main.py                 # Show author per line
git blame -L 10,20 src/main.py        # Specific lines
git blame -e src/main.py              # Show emails

# ============================================
# Submodules
# ============================================
git submodule add <url> <path>        # Add submodule
git submodule update --init           # Clone submodules
git submodule update --remote         # Update to latest

# ============================================
# Bundle (Offline Transfer)
# ============================================
git bundle create repo.bundle --all   # Create bundle
git clone repo.bundle repo-copy       # Clone from bundle

# ============================================
# Clean (Remove Untracked)
# ============================================
git clean -n                          # Preview what would be deleted
git clean -f                          # Remove untracked files
git clean -fd                         # Remove directories too
git clean -fX                         # Remove ignored only
git clean -fx                         # Remove ignored and untracked
```

## Monitoring

### Git Metrics

| Metric | Description | Target |
|--------|-------------|--------|
| Commit Frequency | Commits per developer per day | 3-10 |
| PR Review Time | Time from PR open to merge | < 24 hours |
| Merge Conflicts | Frequency of merge conflicts | Minimized via rebase |
| Branch Lifetime | How long feature branches live | < 2 days (trunk) |
| Rebase Rate | Feature branches rebased on main | Frequent |

### Git Hooks for Monitoring

```bash
#!/bin/bash
# .git/hooks/post-commit

# Log commit metrics
echo "$(date -Iseconds),$(git rev-parse HEAD),$(git config user.email),$(git diff-tree --no-commit-id --name-only -r HEAD | wc -l)" >> .git/metrics.log

# Notify Slack (optional)
if [ -n "$SLACK_WEBHOOK" ]; then
  curl -X POST "$SLACK_WEBHOOK" \
    -H 'Content-Type: application/json' \
    -d "{\"text\": \"New commit: $(git log -1 --format=%s)\"}"
fi
```

## Security

### GPG Signing

```bash
# Generate GPG key
gpg --full-generate-key  # Choose RSA 4096, no expiration

# Get key ID
gpg --list-secret-keys --keyid-format=long

# Configure Git
git config --global commit.gpgsign true
git config --global tag.gpgsign true
git config --global user.signingkey <YOUR_KEY_ID>

# Export public key for GitHub
gpg --armor --export <YOUR_KEY_ID> | pbcopy
# Add to GitHub Settings > SSH and GPG keys
```

### Signing Commits

```bash
# Sign a commit
git commit -S -m "feat: secure feature"

# Sign a tag
git tag -s v1.0.0 -m "Signed release v1.0.0"

# Verify
git log --show-signature
git tag -v v1.0.0
```

### Security Best Practices

1. **Disable force-push on main branches**
2. **Require signed commits**
3. **Use branch protection rules**
4. **Implement pre-commit hooks for secrets**
5. **Never commit credentials or keys**

```yaml
# .git/hooks/pre-commit (detect secrets)
#!/bin/bash
# Using GitGuardian or similar
git diff --staged -U0 | grep -iE "(password|secret|api_key|token)" && {
  echo "Potential secret detected. Review and commit with --no-verify if intentional."
  exit 1
}
```

## Cost Optimization

Git operations themselves are free, but related CI/CD costs can be optimized:

| Optimization | Description |
|--------------|-------------|
| Shallow Clones | `git clone --depth 1` for faster checkout |
| Sparse Checkout | Only clone needed directories |
| Fetch vs Clone | Use fetch for repeated operations |
| LFS for Large Files | Use Git LFS instead of storing in repo |
| Commit Frequency | More, smaller commits = better diffs |

```bash
# Shallow clone for CI
git clone --depth 1 --branch main https://github.com/org/repo.git

# Shallow fetch for updates
git fetch --depth 1 origin main

# Sparse checkout
git sparse-checkout init --cone
git sparse-checkout set src/tests
```

## Troubleshooting

### Common Git Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Detached HEAD | Checkout without branch | `git checkout -b <branch>` to save |
| Lost commits | Reset before push | `git reflog` to find commit |
| Merge conflicts | Concurrent changes | `git mergetool` or manual resolve |
| Large repo clone | Full history | Shallow clone or LFS |
| Push rejected | Remote has new commits | `git pull --rebase` first |
| Wrong branch commit | Committed to wrong branch | Cherry-pick or rebase |

### Recovery Commands

```bash
# Recover lost commits
git reflog                              # Find the commit
git checkout -b recovery <commit-hash>  # Create branch with it

# Undo last commit (keep changes)
git reset --soft HEAD~1

# Undo last commit (discard changes)
git reset --hard HEAD~1

# Revert a commit (safe undo)
git revert <commit-hash>
git revert --no-commit HEAD~3..HEAD     # Revert range

# Find dangling commits
git fsck --unreachable --no-reflogs

# Clean up stale remote tracking branches
git fetch --prune
git remote prune origin
```

### Resolving Merge Conflicts

```bash
# Start merge
git merge feature/other

# If conflicts, edit files manually or use mergetool
git mergetool

# After resolving
git add <resolved-files>
git commit

# Abort if needed
git merge --abort
```

## Interview Questions

### Q1: Explain the difference between merge and rebase. When would you use each?

**Answer:** Merge integrates changes from one branch into another, preserving history and creating a merge commit. Rebase rewrites commit history to apply changes on top of another branch, creating linear history. Use merge for integrating completed feature branches (preserves context). Use rebase for keeping feature branches up-to-date with main (cleaner history). Never rebase commits already pushed to shared branches.

### Q2: How do you recover from a force-push accident?

**Answer:** If `--force-with-lease` was used and rejected, the push may still have partial effect. Use `git reflog` to find the previous state of the branch and `git reset --hard` to restore it. Alternatively, if someone else pushed to the same branch, coordinate with them to restore their changes. Prevention: use `--force-with-lease` instead of `--force`, and enable branch protection.

### Q3: What is the purpose of Git hooks in CI/CD?

**Answer:** Git hooks are scripts that run at specific points in the Git workflow (pre-commit, pre-push, etc.). In CI/CD, they're used for: enforcing code style (pre-commit lint), validating commit messages (commitlint), blocking commits with secrets, running fast checks before expensive CI runs, and triggering notifications.

### Q4: How would you handle a large repository that causes slow clones?

**Answer:** 1) Use shallow clones with `--depth 1`, 2) Implement sparse checkout for partial checkouts, 3) Move large assets to Git LFS, 4) Split monolithic repo into monorepo with Git submodules, 5) Use Git bundle for offline transfer, 6) Consider alternating backends like GitHub's virtual file system.

### Q5: Explain Git's object model and why it matters for security.

**Answer:** Git stores content as objects (blobs for files, trees for directories, commits for snapshots) identified by SHA-1 hashes. This provides cryptographic integrity - any change to content changes the hash. Combined with GPG signing, this creates a verifiable chain of custody for all code changes, essential for compliance and supply chain security.
