# Dockerfile Best Practices

## Overview

A Dockerfile is a text document containing instructions to build a Docker image. Each instruction creates a layer in the image. Well-crafted Dockerfiles result in smaller, more secure images that build faster and are easier to maintain.

**Key Benefits of Good Dockerfile Practices:**
- Faster build times through layer caching
- Smaller image sizes reducing storage and transfer costs
- Improved security through minimal attack surface
- Better reproducibility across environments
- Easier maintenance and debugging

## When To Use

- Building custom application images
- Creating reproducible build environments
- Setting up CI/CD pipelines
- Standardizing development environments
- Packaging applications for distribution
- Creating minimal runtime images

## When NOT To Use

- When a pre-built image meets your needs
- For simple static files that can use existing images
- When build complexity outweighs benefits (single-use scripts)
- For development environments that need rapid iteration (use bind mounts)

## Core Concepts

### Dockerfile Instruction Types

| Instruction | Purpose | Cache Behavior |
|-------------|---------|----------------|
| `FROM` | Base image | No cache (always new) |
| `RUN` | Execute command | Creates layer, cached |
| `COPY` | Copy files | Creates layer, cached |
| `ADD` | Copy with URL/tar | Creates layer, cached |
| `WORKDIR` | Set directory | No layer, metadata |
| `ENV` | Environment var | No layer, metadata |
| `ARG` | Build argument | No layer, metadata |
| `LABEL` | Metadata | No layer, metadata |
| `EXPOSE` | Document port | No layer, metadata |
| `USER` | Set user | No layer, metadata |
| `VOLUME` | Create mount | No layer, metadata |
| `ONBUILD` | Trigger | Adds metadata |
| `HEALTHCHECK` | Health check | No layer, metadata |
| `CMD` | Default command | Overrides entrypoint |
| `ENTRYPOINT` | Executable | Command form |

### Layer Caching Principles

```
┌─────────────────────────────────────────────────────────────────────┐
│  Docker Layer Cache Decision                                        │
│                                                                      │
│  1. Has this instruction been run before with same context?         │
│  2. Is the parent layer cache valid?                                │
│  3. If YES to both → Use cache                                      │
│  4. If NO → Execute instruction and create new layer                │
└─────────────────────────────────────────────────────────────────────┘
```

### Layer Optimization Rules

1. **Order from least to most frequently changing**
2. **Combine related RUN commands**
3. **Use .dockerignore to exclude files**
4. **Copy dependency files before source code**
5. **Minimize number of layers**

## Real Production Example

### Node.js Production Dockerfile (Multi-Stage)

```dockerfile
# ============================================
# Stage 1: Dependencies
# ============================================
FROM node:20-alpine AS deps

WORKDIR /app

# Copy package files
COPY package*.json ./

# Install ALL dependencies (including dev)
RUN npm ci

# ============================================
# Stage 2: Builder
# ============================================
FROM node:20-alpine AS builder

WORKDIR /app

# Copy node_modules from deps stage
COPY --from=deps /app/node_modules ./node_modules

# Copy source
COPY . .

# Build arguments
ARG NEXT_PUBLIC_API_URL
ARG NEXT_PUBLIC_SENTRY_DSN

ENV NEXT_PUBLIC_API_URL=${NEXT_PUBLIC_API_URL}
ENV NEXT_PUBLIC_SENTRY_DSN=${NEXT_PUBLIC_SENTRY_DSN}

# Build application
RUN npm run build

# ============================================
# Stage 3: Production
# ============================================
FROM node:20-alpine AS production

# Security: Add labels
LABEL maintainer="devops@example.com"
LABEL version="1.0.0"
LABEL description="Production Next.js application"

# Create non-root user with group
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodeapp -u 1001

WORKDIR /app

# Copy package files
COPY package*.json ./

# Install production dependencies only
RUN npm ci --only=production && npm cache clean --force

# Copy built application
COPY --from=builder --chown=nodeapp:nodejs /app/.next ./.next
COPY --from=builder --chown=nodeapp:nodejs /app/public ./public
COPY --from=builder --chown=nodeapp:nodejs /app/package.json ./
COPY --from=builder --chown=nodeapp:nodejs /app/next.config.js ./

# Set ownership
USER nodeapp

# Environment
ENV NODE_ENV=production
ENV PORT=3000

# Expose port
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD node -e "require('http').get('http://localhost:3000/health', (r) => process.exit(r.statusCode === 200 ? 0 : 1)).on('error', () => process.exit(1))"

# Use exec form for proper signal handling
CMD ["node_modules/.bin/next", "start"]
```

### Go Production Dockerfile

```dockerfile
# ============================================
# Stage 1: Builder
# ============================================
FROM golang:1.21-alpine AS builder

# Install build dependencies
RUN apk add --no-cache git ca-certificates

WORKDIR /build

# Copy go mod files first for caching
COPY go.mod go.sum ./
RUN go mod download

# Copy source code
COPY . .

# Build arguments
ARG VERSION=dev
ARG COMMIT_SHA=unknown

# Build binary with optimizations
ENV CGO_ENABLED=0 \
    GOOS=linux \
    GOARCH=amd64

RUN go build \
    -ldflags="-s -w -X main.Version=${VERSION} -X main.Commit=${COMMIT_SHA}" \
    -o myapp \
    ./cmd/myapp

# ============================================
# Stage 2: Production (Distroless)
# ============================================
FROM gcr.io/distroless/static:nonroot AS production

# Copy binary from builder
COPY --from=builder /build/myapp /myapp

# Copy TLS certificates
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

# Set user (distroless nonroot user)
USER nonroot:nonroot

# No shell, no additional packages
ENTRYPOINT ["/myapp"]
```

### Python/FastAPI Production Dockerfile

```dockerfile
# ============================================
# Stage 1: Builder
# ============================================
FROM python:3.12-slim AS builder

WORKDIR /app

# Install build dependencies
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# Create virtual environment
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# Install dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir --upgrade pip && \
    pip install --no-cache-dir -r requirements.txt

# ============================================
# Stage 2: Production
# ============================================
FROM python:3.12-slim AS production

# Security labels
LABEL maintainer="devops@example.com"
LABEL version="1.0.0"

# Install runtime dependencies only
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/* \
    && useradd --create-home --shell /bin/bash appuser

WORKDIR /app

# Copy virtual environment from builder
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# Copy application
COPY --chown=appuser:appuser . .

# Switch to non-root user
USER appuser

# Environment
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PYTHONFAULTHANDLER=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1

# Health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

# Expose port
EXPOSE 8000

# Run with uvicorn
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Nginx Production Dockerfile

```dockerfile
FROM nginx:1.25-alpine

# Remove default nginx config
RUN rm /etc/nginx/conf.d/default.conf

# Copy custom config
COPY nginx.conf /etc/nginx/nginx.conf
COPY conf.d/ /etc/nginx/conf.d/

# Create non-root user for nginx
RUN chown -R nginx:nginx /var/cache/nginx /var/log/nginx /etc/nginx/conf.d && \
    chown -R nginx:nginx /var/run && \
    chmod 755 /var/cache/nginx /var/log/nginx /etc/nginx/conf.d

# Use nginx user
USER nginx

EXPOSE 80 443

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost/health || exit 1

CMD ["nginx", "-g", "daemon off;"]
```

## CLI Examples

### Building Images

```bash
# Basic build
docker build -t myapp:latest .

# Build with tag
docker build -t myapp:v1.2.3 -t myapp:latest .

# Build with build args
docker build --build-arg VERSION=1.2.3 --build-arg API_KEY=xxx .

# Build specific target (multi-stage)
docker build --target production -t myapp:prod .

# Build with no cache
docker build --no-cache -t myapp:fresh .

# Build with build secrets (BuildKit)
docker build --secret id=npmrc,src=.npmrc .

# Build with SSH agent (BuildKit)
docker build --ssh default .

# Build with inline cache
docker build --build-arg BUILDKIT_INLINE_CACHE=1 -t myapp .

# Build with custom Dockerfile
docker build -f Dockerfile.prod -t myapp:prod .

# Build with label
docker build -t myapp --label version=1.0 .

# Build and push directly
docker build -t myregistry.io/myapp:v1 -t myregistry.io/myapp:latest .
docker push myregistry.io/myapp
```

### BuildKit Features

```bash
# Enable BuildKit (recommended)
export DOCKER_BUILDKIT=1

# Build with cache mount
docker build --mount=type=cache,target=/root/.cache/pip .

# Build with secret mount (not stored in image layer)
docker build --secret id=aws_credentials,src=$HOME/.aws/credentials .

# Build with SSH agent
docker build --ssh default .

# Multi-platform build
docker buildx create --use
docker buildx build --platform linux/amd64,linux/arm64 -t myapp .

# Inspect build cache
docker builder prune  # Clean build cache
docker buildx du      # Show disk usage
```

## Common Commands

### Dockerfile Instructions Reference

| Instruction | Syntax | Notes |
|-------------|--------|-------|
| `FROM` | `FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]` | First instruction, can use multi-stage |
| `RUN` | `RUN <command>` (shell) or `RUN ["executable", "param"]` (exec) | Exec form preferred for signal handling |
| `COPY` | `COPY [--chown=<user>:<group>] <src>... <dest>` | From build context |
| `ADD` | `ADD [--chown=<user>:<group>] <src>... <dest>` | Can fetch URLs and extract tar |
| `WORKDIR` | `WORKDIR /path/to/workdir` | Creates directory if needed |
| `ENV` | `ENV <key>=<value> ...` | Persists in container |
| `ARG` | `ARG <name>[=<default>]` | Not persisted in image |
| `LABEL` | `LABEL <key>=<value> ...` | Metadata |
| `EXPOSE` | `EXPOSE <port> [<port>/<protocol>]` | Documentation only |
| `USER` | `USER <user>[:<group>]` | Switch user |
| `VOLUME` | `VOLUME ["/path"]` | Create mount point |
| `CMD` | `CMD ["executable", "param"]` (exec) or `CMD command param` (shell) | Can be overridden |
| `ENTRYPOINT` | `ENTRYPOINT ["executable", "param"]` | Configured as executable |
| `HEALTHCHECK` | `HEALTHCHECK [OPTIONS] CMD <command>` | Container health |
| `ONBUILD` | `ONBUILD <instruction>` | Trigger for child images |
| `STOPSIGNAL` | `STOPSIGNAL <signal>` | Signal to stop |

## Security

### Security Hardening Dockerfile

```dockerfile
# Start from minimal base
FROM python:3.12-slim

# Add metadata
LABEL maintainer="security@example.com"
LABEL security-check="scanned"

# Use specific version tags, not 'latest'
# FROM node:20-alpine (bad)
# FROM node:20.15.0-alpine (good)

# Create non-root user
RUN groupadd --gid 1001 appgroup && \
    useradd --uid 1001 --gid appgroup --shell /bin/bash appuser

# Install packages in single layer
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# Copy files with ownership
COPY --chown=appuser:appgroup . .

# Switch to non-root
USER appuser

# Set secure defaults
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PYTHONFAULTHANDLER=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1 \
    PIP_ROOT_USER_ACTION=prevent

# Read-only filesystem
# Note: Some applications need temp directory
# RUN mkdir -p /tmp && chmod 1777 /tmp

# No new privileges
# (set via docker run: --security-opt=no-new-privileges)

# Expose only needed ports
EXPOSE 8000

# Health check
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

CMD ["python", "app.py"]
```

## Cost Optimization

### Multi-Stage Build Pattern

```dockerfile
# Build stage (large)
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn package -DskipTests

# Runtime stage (small)
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/myapp.jar ./myapp.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "myapp.jar"]
```

### Layer Optimization

```dockerfile
# Bad: Multiple RUN commands
RUN apt-get update
RUN apt-get install -y git
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# Good: Combine related commands
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        git \
        curl \
    && rm -rf /var/lib/apt/lists/*

# Bad: Copy everything then build
COPY . .
RUN npm run build

# Good: Copy only what's needed for build
COPY package*.json ./
RUN npm ci
COPY src/ ./src/
RUN npm run build
```

## Troubleshooting

### Common Dockerfile Mistakes

| Issue | Cause | Solution |
|-------|-------|----------|
| Build fails on COPY | File not in context | Check .dockerignore, verify path |
| Layer not updating | Cache used | Add `--no-cache` or invalidate cache |
| Image too large | Unnecessary files | Use .dockerignore, multi-stage build |
| Permission denied | Wrong user | Add `USER` instruction with proper uid |
| CMD not running | Entrypoint conflict | Check Entrypoint/CMD interaction |
| Environment variables empty | ENV not set | Verify ENV instructions order |
| apt-get fails | Missing update | Add `apt-get update` before install |

### Debugging Builds

```bash
# Build with verbose output
DOCKER_BUILDKIT=0 docker build -t myapp --progress=plain .

# Build interactive (for debugging)
RUN --mount=type=cache,target=/root/.cache/bash \
    bash -c "set -x; ls -la"

# Check intermediate containers
docker ps -a | grep myapp

# Inspect layer by layer
docker history myapp

# Shell into build container (BuildKit)
docker build --progress=plain --target build-stage .
docker run -it <image_id> /bin/sh
```

### Build Cache Invalidation

```dockerfile
# Explicitly invalidate cache
COPY package*.json /app/
RUN npm ci
COPY . /app/   # This invalidates npm ci cache when any file changes

# Better: Use specific COPY commands
COPY src /app/src   # Invalidate only when src changes
COPY public /app/public
```

## Interview Questions

### Q1: What is the difference between COPY and ADD?
**A:** `COPY` is preferred for simplicity. It copies files from build context. `ADD` has additional capabilities: it can fetch remote URLs and extract tar files. For most cases, `COPY` is clearer and recommended. Use `ADD` only when you specifically need URL fetching or tar extraction.

### Q2: Why use multi-stage builds?
**A:** Multi-stage builds separate build and runtime environments. The final image contains only runtime artifacts, excluding build tools, source code, and intermediate files. This results in smaller, more secure images without build dependencies. The build stage can use full SDK images while the production stage uses minimal runtime images.

### Q3: How do you handle secrets in Docker builds?
**A:** Never put secrets in Dockerfiles or images. Use BuildKit secret mounting (`--secret`) to pass secrets at build time without storing them in layers. External secret management (Vault, cloud secret services) should inject secrets at runtime. Never commit credentials to version control or Docker context.

### Q4: What is the difference between CMD and ENTRYPOINT?
**A:** `CMD` provides default arguments to the container (can be overridden at runtime). `ENTRYPOINT` configures the container to run as an executable. When both are used, `CMD` provides default arguments to `ENTRYPOINT`. The exec form `["executable", "param"]` is preferred for proper signal handling and avoiding shell interpretation.

### Q5: How do you optimize Dockerfile layer caching?
**A:** Order instructions from least to most frequently changing. Copy dependency files before source code. Combine related RUN commands. Use .dockerignore to exclude files. Each instruction's cache is invalidated only when files that COPY/ADD reference change, not when later instructions modify the filesystem.

### Q6: What security considerations for Dockerfiles?
**A:** Use minimal base images (alpine, distroless), create and use non-root users, drop all capabilities, set read-only filesystem where possible, avoid installing unnecessary packages, scan for vulnerabilities, use specific version tags, and avoid secrets in build context.

### Q7: Why use distroless or scratch images?
**A:** Distroless and scratch images have minimal or no packages, shells, or utilities. This reduces attack surface dramatically. Distroless images are multi-arch and contain only the application and its runtime dependencies. Scratch is literally empty—only the binary, useful for statically compiled languages like Go.

### Q8: How does HEALTHCHECK work?
**A:** `HEALTHCHECK` runs a command periodically. If it fails, the container is marked unhealthy. Docker reports health status in `docker ps`. Exit code 0 = healthy, 1 = unhealthy. This enables orchestrators to make routing decisions based on actual application health, not just process existence.

### Q9: What is the difference between ARG and ENV?
**A:** `ARG` defines build-time variables (not persisted in image). `ENV` defines environment variables (persisted in image, available in running container). Use `ARG` for build configuration, `ENV` for runtime configuration. `ENV` values can be overridden at runtime with `docker run -e`.

### Q10: How do you reduce Docker image size?
**A:** Use minimal base images (alpine, slim, distroless), implement multi-stage builds to exclude build tools, combine RUN commands to minimize layers, use .dockerignore to exclude unnecessary files, order instructions for better caching, and consider distroless/scratch for compiled languages.
