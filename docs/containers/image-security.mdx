# Container Image Security

## Overview

Container image security encompasses practices and tools to ensure container images are free from vulnerabilities, properly authenticated, and safely configured. With containers being the primary deployment artifact in modern cloud-native applications, securing the image supply chain is critical.

**Key Security Areas:**
- Vulnerability scanning and management
- Image signing and verification
- Supply chain security (SBOMs)
- Base image hardening
- Runtime security policies
- Secret management
- Access control and authentication

## When To Use

- **CI/CD pipelines** requiring automated security checks
- **Production deployments** needing verified, trusted images
- **Compliance requirements** (SOC 2, HIPAA, PCI-DSS)
- **Multi-tenant environments** requiring strong isolation
- **Supply chain security** requirements (SBOM generation, signing)
- **Regulatory environments** with strict security mandates

## When NOT To Use

- **Development/learning environments** where overhead isn't justified
- **Simple internal tools** with no external exposure
- **Non-production workloads** where rapid iteration matters more than security

## Real Production Example

### Secure Image Pipeline

**CI/CD Pipeline with Security Gates:**

```yaml
# .github/workflows/secure-build.yml

name: Secure Container Build

on:
  push:
    branches: [main]
    tags: ['v*']
  pull_request:
    branches: [main]

env:
  REGISTRY: myregistry.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ============================================
  # Stage 1: Build
  # ============================================
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      - name: Build image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: false
          tags: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ============================================
  # Stage 2: Vulnerability Scan
  # ============================================
  scan:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      
      - name: Pull image for scanning
        run: |
          docker pull ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
      
      - name: Run Trivy scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: '${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}'
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'  # Fail on CRITICAL/HIGH
      
      - name: Upload Trivy results
        uses: github/codeql-action/upload-sarif@v2
        with:
          sarif_file: 'trivy-results.sarif'
      
      - name: Scan for secrets
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: '${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}'
          scan-type: 'secret'

  # ============================================
  # Stage 3: Generate SBOM
  # ============================================
  sbom:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      
      - name: Pull image
        run: docker pull ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
      
      - name: Generate SBOM with Syft
        uses: anchore/sbft-action@main
        with:
          image: '${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}'
          format: spdx-json
          output-file: sbom.spdx.json
      
      - name: Upload SBOM
        uses: actions/upload-artifact@v3
        with:
          name: sbom
          path: sbom.spdx.json

  # ============================================
  # Stage 4: Sign Image
  # ============================================
  sign:
    runs-on: ubuntu-latest
    needs: [build, scan]
    if: github.event_name == 'push'
    permissions:
      id-token: write  # For OIDC
      actions: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      
      - name: Log in to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ secrets.REGISTRY_USER }}
          password: ${{ secrets.REGISTRY_TOKEN }}
      
      - name: Sign image with Cosign
        uses: sigstore/cosign-installer@v3
      
      - name: Sign image
        run: |
          cosign sign \
            --yes \
            --bbox-support \
            --tlog-upload \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
        env:
          COSIGN_EXPERIMENTAL: "1"

  # ============================================
  # Stage 5: Push and Deploy
  # ============================================
  push:
    runs-on: ubuntu-latest
    needs: sign
    if: startsWith(github.ref, 'refs/tags/')
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      - name: Extract version
        id: version
        run: echo "VERSION=${GITHUB_REF#refs/tags/}" >> $GITHUB_OUTPUT
      
      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ steps.version.outputs.VERSION }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
```

### Kubernetes ImagePolicyWebhook Configuration

```yaml
# admission-configuration.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: AdmissionConfiguration
plugin:
- name: ImagePolicyWebhook
  configuration:
    apiVersion: apiserver.config.k8s.io/v1
    kind: ImagePolicyWebhook
   kubeConfigFile: /etc/kubernetes/kubeconfig
    defaultAllow: false  # Deny by default
    allowedMarkupHook:
      - version: v1
        request:
          operation: CREATE
          resource: pods
        imagePatterns:
          - pattern: 'myregistry.io/*:*'
            optional: true
          - pattern: 'registry.k8s.io/*:*'
            optional: true
```

## CLI Examples

### Trivy Vulnerability Scanning

```bash
# Install Trivy
brew install trivy
# or
apt-get install trivy

# Scan local image
trivy image nginx:alpine

# Scan with specific severity
trivy image --severity HIGH,CRITICAL myapp:latest

# Scan with policy violation
trivy image --exit-code 1 --severity CRITICAL myapp:latest

# Scan filesystem
trivy fs --security-checks vuln,config,secret ./

# Scan Docker config (for leaked credentials)
trivy config --security-checks secret ./

# Generate report in different formats
trivy image -f json -o report.json myapp:latest
trivy image -f sarif -o report.sarif myapp:latest
trivy image -f table myapp:latest  # default

# Scan Git repository
trivy repo https://github.com/myorg/myrepo

# Ignore unfixed vulnerabilities
trivy image --ignore-unfixed myapp:latest

# Skip specific files/directories
trivy fs --exclude-images --exclude-dir '.git' --exclude-dir 'node_modules' ./

# Clear cache
trivy image --clear-cache myapp:latest
trivy cache --refresh
```

### Snyk Container Scanning

```bash
# Install Snyk CLI
npm install -g snyk

# Authenticate
snyk auth

# Test container image
snyk container test myapp:latest

# Test with severity threshold
snyk container test myapp:latest --severity-threshold=high

# Monitor image for new vulnerabilities
snyk container monitor myapp:latest

# Test Dockerfile
snyk container test --docker myapp:latest --file=Dockerfile

# Output in JSON format
snyk container test myapp:latest --json > results.json
```

### Cosign Image Signing

```bash
# Install Cosign
brew install cosign
# or
go install github.com/sigstore/cosign/cmd/cosign@latest

# Generate key pair
cosign generate-key-pair

# Sign image
cosign sign myregistry.io/myapp:v1.0.0

# Sign with key
cosign sign --key cosign.key myregistry.io/myapp:v1.0.0

# Sign with KMS key
cosign sign --key kms://my-kms-url myregistry.io/myapp:v1.0.0

# Sign with OIDC (keyless signing)
COSIGN_EXPERIMENTAL=1 cosign sign myregistry.io/myapp:v1.0.0

# Verify image
cosign verify myregistry.io/myapp:v1.0.0

# Verify with public key
cosign verify --key cosign.pub myregistry.io/myapp:v1.0.0

# Verify with attestation
cosign verify --attestation-policy policy.json myregistry.io/myapp:v1.0.0

# Upload SBOM
cosign attest --predicate sbom.spdx.json myregistry.io/myapp:v1.0.0

# Verify attestation
cosign verify-attestation --type spdxjson myregistry.io/myapp:v1.0.0
```

### Syft SBOM Generation

```bash
# Install Syft
brew install syft

# Generate SBOM from image
syft myapp:latest -o json > sbom.json
syft myapp:latest -o spdx-json > sbom.spdx
syft myapp:latest -o cyclonedx-json > sbom.cdx

# Generate SBOM from directory
syft . -o json > sbom.json

# Generate SBOM with package catalog
syft myapp:latest -o json | jq '.packages[].name'

# Scan and compare with vulnerability database
syft myapp:latest -o json | grype

# Update catalog
syft packages myapp:latest --catalogers +node-lockfile
```

### Grype Vulnerability Matching

```bash
# Install Grype
brew install grype

# Scan image for vulnerabilities
grype myapp:latest

# Scan with JSON output
grype myapp:latest -o json > vulnerabilities.json

# Scan and fail on critical
grype myapp:latest --fail-on critical

# Scan SBOM instead of image
grype sbom:sbom.spdx myapp:latest

# Show fixed vulnerabilities
grype myapp:latest --show-fixed

# Ignore specific CVEs
grype myapp:latest --ignore CVE-2021-44228
```

## Dockerfile / Compose Examples

### Hardened Dockerfile

```dockerfile
# Stage 1: Build
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# Stage 2: Production (Distroless)
FROM gcr.io/distroless/nodejs20-debian11 AS production

# Security: Non-root, no shell
USER nonroot

WORKDIR /app

# Copy built application
COPY --from=builder --chown=nonroot:nonroot /app/dist ./dist
COPY --from=builder --chown=nonroot:nonroot /app/node_modules ./node_modules
COPY --from=builder --chown=nonroot:nonroot /app/package.json ./

# Security: Read-only filesystem (need /tmp for temp)
# Note: Some apps need /tmp, add as tmpfs mount at runtime
# RUN mkdir -p /tmp && chmod 1777 /tmp

# Health check
HEALTHCHECK --interval=30s --timeout=5s --start-period=5s --retries=3 \
    CMD node -e "require('http').get('http://localhost:3000/health', (r) => process.exit(r.statusCode === 200 ? 0 : 1)).on('error', () => process.exit(1))"

EXPOSE 3000

CMD ["node", "dist/main.js"]
```

### Security-Scanned Dockerfile

```dockerfile
# This Dockerfile demonstrates security best practices
# Scanning should happen in CI/CD

FROM python:3.12-slim

# Security: Add labels
LABEL maintainer="security@example.com"
LABEL security-check="required"

# Security: Create non-root user
RUN groupadd --gid 1001 appgroup && \
    useradd --uid 1001 --gid appgroup --shell /bin/bash appuser

# Security: Install minimal packages in one layer
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        ca-certificates \
        # Only production deps
        postgresql-client \
    && rm -rf /var/lib/apt/lists/*

# Copy with ownership
COPY --chown=appuser:appgroup . .

# Security: Switch to non-root
USER appuser

# Security: Environment hardening
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PYTHONFAULTHANDLER=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1 \
    PIP_ROOT_USER_ACTION=prevent

EXPOSE 8000

# Run with explicit entrypoint
ENTRYPOINT ["python", "app.py"]
```

### Docker Compose with Security Options

```yaml
version: "3.9"

services:
  app:
    image: myapp:latest
    # Security: Read-only root filesystem
    read_only: true
    
    # Security: Temporary filesystem
    tmpfs:
      - /tmp:rw,noexec,nosuid,size=64m
      - /var/run:rw,noexec,nosuid
    
    # Security: Drop all capabilities
    cap_drop:
      - ALL
    
    # Security: Add specific capability
    cap_add:
      - NET_BIND_SERVICE
    
    # Security: No new privileges
    security_opt:
      - no-new-privileges:true
    
    # Security: Non-root user
    user: "1000:1000"
    
    # Security: Resource limits
    deploy:
      resources:
        limits:
          memory: 256M
          cpus: "0.5"
    
    # Security: Health check
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  db:
    image: postgres:15-alpine
    security_opt:
      - no-new-privileges:true
    read_only: true
    tmpfs:
      - /tmp:rw,noexec,nosuid
      - /var/run/postgresql:rw,noexec,nosuid
    deploy:
      resources:
        limits:
          memory: 1G
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

## Troubleshooting

### Common Security Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Vulnerability scan fails | No scanner installed | Install Trivy/Snyk |
| Signature verification fails | Wrong key | Update public key |
| SBOM not generated | Missing tool | Install Syft |
| Policy denies image | Blocked by admission | Whitelist registry |
| Secrets in image | Credentials in Dockerfile | Use build secrets |

### Debugging Security Issues

```bash

cosign verify --verbose myapp:latest
```

## Interview Questions

### Q1: What is an SBOM and why is it important?
**A:** A Software Bill of Materials (SBOM) is a formal record of software components and dependencies. It's important for vulnerability management (linking CVEs to your components), license compliance, and supply chain security. It enables faster response to vulnerabilities and helps satisfy compliance requirements.

### Q2: What is image signing and why use it?
**A:** Image signing cryptographically verifies the identity of the image publisher and ensures the image hasn't been tampered with. It prevents deploying malicious or modified images. Tools like Cosign use asymmetric keys or OIDC for keyless signing.

### Q3: What are the key container security best practices?
**A:** Use minimal base images (alpine, distroless), run as non-root, use read-only root filesystem, drop all capabilities, enable security options (no-new-privileges), scan for vulnerabilities in CI/CD, sign images, use Secrets management, implement network policies, and enable audit logging.

### Q4: What is Cosign and how does keyless signing work?
**A:** Cosign is a tool for signing and verifying OCI images. Keyless signing uses OIDC (OpenID Connect) to authenticate via GitHub Actions, Google, or other identity providers. It generates short-lived certificates tied to the identity, making key management simpler and more secure.

### Q5: What is the difference between Trivy, Snyk, and Grype?
**A:** Trivy is an open-source vulnerability scanner from Aqua Security. Snyk is a commercial scanner with IDE integration and remediation advice. Grype is an open-source scanner from Anchore that can consume SBOMs for faster scanning. All scan container images for CVEs but have different feature sets.

### Q6: How do admission controllers improve image security?
**A:** ImagePolicyWebhook or OPA Gatekeeper admission controllers can enforce policies on what images can be deployed. They can require images to come from trusted registries, be signed, have recent vulnerability scans, or meet other security requirements before being admitted to the cluster.

### Q7: What are distroless images and why use them?
**A:** Distroless images from Google contain only the application runtime and its dependencies, with no shell, package manager, or other utilities. This dramatically reduces the attack surface because there are fewer components that could be exploited.

### Q8: How do you handle secrets in container builds?
**A:** Never bake secrets into images. Use BuildKit secret mounting (`--secret`) for build-time secrets, external secret managers (Vault, cloud secrets) for runtime secrets, Kubernetes Secrets for in-cluster secrets, and environment variables or volume mounts at runtime.

### Q9: What is the supply chain security approach for containers?
**A:** Supply chain security includes: signing source code commits, verifying build provenance, using dependency scanning (SCA), generating and signing SBOMs, signing built images, verifying signatures before deployment, and maintaining an audit trail of what was deployed and when.

### Q10: What is Docker Content Trust?
**A:** Docker Content Trust (DCT) enables image signing using The Update Framework (TUF). It ensures you're pulling images from the expected publisher and that the image hasn't been tampered with. When enabled, Docker only pulls and runs signed images from trusted repositories.
