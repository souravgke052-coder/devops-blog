# Container Registry

## Overview

A container registry is a storage and distribution system for container images. It hosts image repositories, manages versioning through tags, and provides APIs for pushing and pulling images. Registries are the backbone of the container supply chain, enabling reliable image storage and distribution.

**Registry Types:**
- **Public registries**: Docker Hub, GitHub Container Registry, Quay.io
- **Private registries**: Self-hosted Harbor, GitLab Container Registry
- **Cloud registries**: Google Artifact Registry, AWS ECR, Azure Container Registry

**Key Concepts:**
- **Registry**: Server hosting multiple repositories
- **Repository**: Collection of related images (e.g., `myregistry.io/myapp`)
- **Tag**: Version identifier (e.g., `v1.2.3`, `latest`)
- **Manifest**: Metadata describing image layers and configuration
- **Blob**: Individual layer files stored in registry

## When To Use

- **Storing private images** not suitable for public registries
- **Compliance requirements** for data residency and audit
- **Air-gapped environments** requiring self-hosted solutions
- **Performance optimization** with regional registries
- **Enterprise authentication** with SSO/LDAP integration
- **Image scanning integration** within the registry
- **Fine-grained access control** beyond public registry permissions

## When NOT To Use

- **Simple projects** where Docker Hub suffices
- **Personal/hobby projects** where convenience outweighs control
- **Proof of concepts** before infrastructure commitment

## Real Production Example

### Google Artifact Registry Setup

```bash
# 1. Create Artifact Registry repository
gcloud artifacts repositories create my-repo \
  --repository-format=docker \
  --location=us-central1 \
  --description="Production container images"

# 2. Configure Docker authentication
gcloud auth configure-docker us-central1-docker.pkg.dev

# 3. Tag and push images
docker tag myapp:latest us-central1-docker.pkg.dev/my-project/my-repo/myapp:v1.0.0
docker push us-central1-docker.pkg.dev/my-project/my-repo/myapp:v1.0.0

# 4. Set up vulnerability scanning
gcloud artifacts repositories update my-repo \
  --location=us-central1 \
  --enable-dynamic-import \
  --docker-repo=docker

# 5. Configure cleanup policies
gcloud artifacts repositories update my-repo \
  --location=us-central1 \
  --docker-repo=docker \
  --cleanup-policy=delete-days-older-than=90 \
  --cleanup-policy-dry-run=false
```

### Amazon ECR

```bash
# Create ECR repository
aws ecr create-repository \
  --repository-name myapp \
  --image-scanning-configuration scanOnPush=true \
  --encryption-configuration encryptionType=AES256

# Get login password
aws ecr get-login-password | docker login --username AWS --password-stdin \
  123456789.dkr.ecr.us-east-1.amazonaws.com

# Tag and push
docker tag myapp:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1

# List images
aws ecr list-images --repository-name myapp

# Describe image scan findings
aws ecr describe-image-scan-findings \
  --repository-name myapp \
  --image-id imageTag=v1

# Lifecycle policy
aws ecr put-lifecycle-policy \
  --repository-name myapp \
  --lifecycle-policy-text '{
    "rules": [{
      "rulePriority": 1,
      "description": "Expire untagged images after 14 days",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 14
      },
      "action": {"type": "expire"}
    }]
  }'
```

## Dockerfile / Compose Examples

### Image Promotion Pipeline

```yaml
# docker-compose.yml for image promotion

version: "3.9"

services:
  # Build stage
  build:
    build:
      context: ./app
      dockerfile: Dockerfile
    image: ${REGISTRY}/myapp:build-${GIT_SHA}
    profiles:
      - ci

  # Scan and sign
  security-scan:
    image: aquasec/trivy:latest
    profiles:
      - ci
    environment:
      - REGISTRY=${REGISTRY}
      - IMAGE_TAG=${GIT_SHA}
    command: image --severity HIGH,CRITICAL ${REGISTRY}/myapp:build-${GIT_SHA}

  # Promote to staging
  promote-staging:
    image: google/cloud-sdk:latest
    profiles:
      - ci
    environment:
      - REGISTRY=${REGISTRY}
      - GIT_SHA=${GIT_SHA}
    command: >
      gcloud artifacts docker tags set
      ${REGISTRY}/myapp:build-${GIT_SHA}
      ${REGISTRY}/myapp:staging-${GIT_SHA}

  # Deploy to staging
  deploy-staging:
    image: bitnami/kubectl:latest
    profiles:
      - ci
    environment:
      - KUBECONFIG=/.kube/config
    volumes:
      - kubeconfig:/.kube:ro
    command: kubectl set image deployment/myapp myapp=${REGISTRY}/myapp:staging-${GIT_SHA}

  # Integration tests would go here
  test-staging:
    image: newman:latest
    profiles:
      - ci
    environment:
      - API_URL=https://staging.example.com
    command: run postman-collection.json

  # Promote to production
  promote-production:
    image: google/cloud-sdk:latest
    profiles:
      - ci
    environment:
      - REGISTRY=${REGISTRY}
      - GIT_SHA=${GIT_SHA}
    command: >
      gcloud artifacts docker tags set
      ${REGISTRY}/myapp:staging-${GIT_SHA}
      ${REGISTRY}/myapp:prod-${GIT_SHA}
      ${REGISTRY}/myapp:latest
```

## Troubleshooting

### Common Registry Issues

| Issue | Cause | Solution |
|-------|-------|----------|

## Interview Questions

### Q1: What is the difference between a registry and a repository?
**A:** A registry is the server hosting multiple repositories (e.g., Docker Hub, GCR). A repository is a named collection of related images within a registry (e.g., `myregistry.io/myapp`). An image is identified by `registry/repository:tag` or `registry/repository@sha256:digest`.

### Q2: How does Docker authenticate to a registry?
**A:** Docker uses the Docker config file (`~/.docker/config.json`) storing credentials or tokens. On login, credentials are stored base64-encoded. For GCP, `gcloud auth configure-docker` stores OAuth2 tokens. For AWS ECR, `aws ecr get-login-password` provides short-lived tokens.

### Q3: What is a multi-architecture image?
**A:** A multi-arch image uses a manifest list to reference architecture-specific images. When pulling, the Docker client automatically selects the correct image for the host's architecture (amd64, arm64, etc.). Build with `docker buildx build --platform linux/amd64,linux/arm64`.

### Q4: How do you secure a container registry?
**A:** Secure by enabling TLS, using strong authentication (SSO, robot accounts), implementing RBAC/IAM, enabling vulnerability scanning, signing images (Cosign), using network policies to restrict access, implementing audit logging, and setting up retention/cleanup policies.

### Q5: What are robot accounts and why use them?
**A:** Robot accounts are service accounts for CI/CD systems, not tied to individual users. They enable automation without sharing human credentials, can be rotated, and can have scoped permissions. In Harbor, they're called "Robot Accounts."

### Q6: What is image promotion?
**A:** Image promotion moves images through lifecycle stages (build → staging → production). It typically involves tagging or copying images between registries with security gates (scanning, testing) at each stage. Enables traceability and controlled rollouts.

### Q7: How do you handle registry rate limits?
**A:** Authenticated requests have higher limits than anonymous. Use service accounts with registry authentication. Implement caching with registry mirrors. For Docker Hub, upgrade to Teams or Business tier. Consider multi-registry distribution.

### Q8: What is a Helm chart repository vs container registry?
**A:** A container registry stores OCI images. A Helm chart repository stores Helm charts, which package Kubernetes manifests. Helm 3 supports OCI-based chart storage in container registries using the OCI distribution spec.
