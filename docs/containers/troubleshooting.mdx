# Container Troubleshooting

## Overview

Container troubleshooting is the systematic process of diagnosing and resolving issues with containers, images, networks, and related infrastructure. This guide covers the tools, techniques, and workflows for identifying and fixing common container problems.

**Troubleshooting Categories:**
- Container lifecycle issues (start/stop/restart)
- Application crashes and exit codes
- Network connectivity problems
- Storage and volume issues
- Resource exhaustion (CPU, memory, disk)
- Image pulling and authentication problems
- Permission and security errors
- Performance degradation

## When To Use

- **Application failures** in containerized environments
- **Deployment issues** in CI/CD pipelines
- **Debugging production incidents** in Kubernetes or Docker
- **Performance optimization** investigations
- **Configuration problems** with networking or storage
- **Security incident response** involving containers

## When NOT To Use

- **When infrastructure is down** (fix infrastructure first)
- **Without proper monitoring** (identify symptoms first)
- **Without logging configured** (set up logging first)

## Real Production Example

### Scenario: API Service Pod Restart Loop

**Step 1: Identify the issue**

```bash
# Check pod status
kubectl get pods -n production | grep api

# Output shows RESTARTS count increasing
# NAME    READY   STATUS             RESTARTS   AGE
# api-0   0/1     CrashLoopBackOff   5          10m

# Get detailed pod info
kubectl describe pod api-0 -n production

# Look at Events section for error messages
# Last State: Terminated, Reason: OOMKilled
```

**Step 2: Investigate the logs**

```bash
# Get previous container logs (before crash)
kubectl logs api-0 -n production --previous

# Output shows:
# FATAL: Exception: OutOfMemoryError: Java heap space
# at com.myapp.Cache.<init>(Cache.java:45)

# Check current resource usage
kubectl top pod api-0 -n production

# Output:
# NAME    CPU(cores)   MEMORY(bytes)
# api-0   850m        892Mi
```

**Step 3: Diagnose the root cause**

```bash
# Check pod spec for memory limits
kubectl get pod api-0 -n production -o jsonpath='{.spec.containers[0].resources}'

# Output shows limit of 512Mi, but app uses ~900Mi

# Check application config
kubectl exec api-0 -n production -- cat /app/config.yaml | grep heap

# Heap size configured as -Xmx512m which matches limit
```

**Step 4: Apply the fix**

```yaml
# Update the Deployment with increased memory
# api-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
  namespace: production
spec:
  template:
    spec:
      containers:
        - name: api
          resources:
            limits:
              memory: 2Gi
            requests:
              memory: 1Gi
          env:
            - name: JAVA_OPTS
              value: "-Xmx1g -Xms512m"
```

```bash
# Apply the fix
kubectl apply -f api-deployment.yaml

# Verify pod is running
kubectl get pods -n production | grep api

# Output:
# NAME    READY   STATUS    RESTARTS   AGE
# api-0   1/1     Running   0          2m
```

### Scenario: Database Connection Failures

**Debugging steps:**

```bash
# 1. Check database pod status
kubectl get pods -n database | grep postgres
# postgres-0   1/1     Running   0          1h

# 2. Check application pod logs
kubectl logs api-0 -n production | grep -i "connection\|database\|postgres"

# Output:
# ERROR: connection refused: postgres:5432
# Caused by: java.net.ConnectException: Connection refused

# 3. Check if database is accepting connections
kubectl exec -it postgres-0 -n database -- psql -U postgres -c "SELECT 1"

# 4. Check network policies
kubectl get networkpolicy -n production
kubectl describe networkpolicy allow-postgres -n production

# 5. Verify DNS resolution
kubectl exec api-0 -n production -- nslookup postgres
kubectl exec api-0 -n production -- nc -zv postgres 5432

# 6. Check service endpoints
kubectl get endpoints postgres -n database

# All looks correct, check pg_hba.conf
kubectl exec -it postgres-0 -n database -- cat /var/lib/postgresql/data/pg_hba.conf

# Found: host all all 0.0.0.0/0 reject
# Fix: Update pg_hba.conf to allow from application network
```

## CLI Examples

### Basic Container Inspection

```bash
# List all containers (running and stopped)
docker ps -a
docker ps -a --format "table {{.ID}}\t{{.Names}}\t{{.Status}}\t{{.Ports}}"

# Inspect container details
docker inspect mycontainer
docker inspect --format='{{.State.Status}}' mycontainer
docker inspect --format='{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' mycontainer
docker inspect --format='{{.Config.Env}}' mycontainer

# Show container processes
docker top mycontainer

# View filesystem changes
docker diff mycontainer
# A = Added, C = Changed, D = Deleted

# View resource usage
docker stats --no-stream mycontainer
docker stats --no-stream --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}\t{{.NetIO}}\t{{.BlockIO}}"
```

### Log Analysis

```bash
# View container logs
docker logs mycontainer
docker logs --tail 100 mycontainer
docker logs --since 1h mycontainer
docker logs --since 2024-01-15T10:00:00 mycontainer
docker logs --timestamps mycontainer

# Follow logs in real-time
docker logs -f mycontainer

# Search logs for errors
docker logs mycontainer 2>&1 | grep -i error
docker logs mycontainer --since 1h | grep -E "(ERROR|FATAL|Exception)"

# JSON logging driver logs
docker logs mycontainer | jq '.log'

# Show logs with specific format
docker logs --format '{{.ID}} - {{.Names}} - {{.CreatedAt}} - {{.Status}}' mycontainer
```

### Executing Debug Commands

```bash
# Enter container shell
docker exec -it mycontainer sh
docker exec -it mycontainer bash

# Execute single command
docker exec mycontainer ls -la /app
docker exec mycontainer cat /etc/resolv.conf
docker exec mycontainer ps aux

# Execute as root
docker exec -u root mycontainer sh

# Execute with environment
docker exec -e DEBUG=1 mycontainer env

# Copy files from container
docker cp mycontainer:/app/logs /tmp/logs
docker cp mycontainer:/app/config.yaml .

# Copy files to container
docker cp ./config.yaml mycontainer:/app/config.yaml
```

### Network Debugging

```bash
# List networks
docker network ls
docker network inspect bridge

# Inspect container network
docker inspect --format='{{json .NetworkSettings}}' mycontainer | jq

# Test network connectivity
docker exec mycontainer ping -c 3 google.com
docker exec mycontainer curl -v localhost:8080/health
docker exec mycontainer nc -zv db 5432

# Check DNS resolution
docker exec mycontainer nslookup google.com
docker exec mycontainer getent hosts db

# Show routing table
docker exec mycontainer route -n

# Check listening ports
docker exec mycontainer netstat -tlnp
docker exec mycontainer ss -tlnp

# Capture network traffic
docker exec mycontainer tcpdump -i eth0 -w /tmp/capture.pcap
docker cp mycontainer:/tmp/capture.pcap .
```

### Volume and Storage Debugging

```bash
# List volumes
docker volume ls
docker volume inspect myvolume

# Check volume mounts in container
docker inspect --format='{{json .Mounts}}' mycontainer | jq

# Check disk usage inside container
docker exec mycontainer df -h
docker exec mycontainer du -sh /app

# Check inode usage
docker exec mycontainer df -i

# Check for disk full conditions
docker exec mycontainer bash -c "df -h; echo '---'; docker system df"
```

### System Resource Debugging

```bash
# View container resource limits
docker inspect --format='{{json .HostConfig.Memory}}' mycontainer
docker inspect --format='{{json .HostConfig.CpuShares}}' mycontainer

# Check cgroup limits
docker exec mycontainer cat /sys/fs/cgroup/memory/memory.limit_in_bytes
docker exec mycontainer cat /sys/fs/cgroup/memory/memory.usage_in_bytes

# Check for OOM killer
dmesg | grep -i "killed process"
docker exec mycontainer dmesg | grep -i oom

# Check process limits
docker exec mycontainer cat /proc/self/limits

# View kernel messages
docker exec mycontainer dmesg
```

## Dockerfile / Compose Examples

### Debug Container Definition

```yaml
# docker-compose.debug.yml

version: "3.9"

services:
  app:
    image: myapp:debug
    build:
      context: .
      target: debug  # Multi-stage debug build
    ports:
      - "8080:8080"  # Debug port
      - "9090:9090"  # Metrics
    environment:
      - DEBUG=true
      - LOG_LEVEL=debug
    volumes:
      - ./src:/app/src:ro  # Source code mount
      - app-cache:/root/.cache
    command: ["npm", "run", "dev:debug"]
    privileged: true  # Debug capabilities

  # Debug utilities
  debug-tools:
    image: curlimages/curl:latest
    network_mode: "service:app"
    profiles:
      - debug
```

## Security

### Secure Debugging Practices

```bash
# Never expose debug ports in production
# Use temporary debug containers
kubectl debug -it api-0 -n production --image=curlimages/curl -- /bin/sh

# Use ephemeral debug container
kubectl debug api-0 -n production --share-processes --copy-to=api-debug

# Clean up after debugging
kubectl delete debug api-debug

# Audit debug access
# Check who accessed what and when
kubectl get events --field-selector reason=Exec --all-namespaces
```

### Container Won't Start

| Cause | Diagnosis | Solution |
|-------|-----------|----------|
| Image not found | `docker images` | Pull or build image |
| Port conflict | `docker logs` | Change port mapping |
| Environment missing | `docker inspect` | Set required ENV |
| Volume mount error | `docker logs` | Check host path exists |
| Permission denied | `docker logs` | Fix file permissions |
| OOM limit | `docker stats` | Increase memory limit |
| Invalid CMD | `docker logs` | Fix Dockerfile CMD |

## Interview Questions

### Q1: How do you debug a container that exits immediately?
**A:** Check the exit code with `docker ps -a` and `docker inspect`. Look at logs with `docker logs`. Common causes: invalid CMD/ENTRYPOINT, missing dependencies, application errors. Try running interactively with `docker run -it` to see the error output directly.

### Q2: What is CrashLoopBackOff and how do you fix it?
**A:** Kubernetes CrashLoopBackOff means the container keeps crashing and Kubernetes retries with increasing delays. Fix by: checking logs (`kubectl logs --previous`), verifying resource limits aren't too low, checking the application can start with required config, fixing any missing dependencies or secrets.

### Q3: How do you debug network issues between containers?
**A:** Check container is on the same network with `docker network inspect`. Test DNS resolution with `docker exec mycontainer nslookup servicename`. Test connectivity with ping/curl/nc. Check firewall rules and network policies (Kubernetes). Verify service is actually listening on expected port.

### Q4: How do you handle OOM (Out of Memory) issues?
**A:** Increase container memory limit. Optimize application memory usage. Profile the application to find leaks. Check JVM heap settings for Java apps. Review actual memory usage with `docker stats` and `docker exec free -h`. Set appropriate memory reservations.

### Q5: What tools do you use for container debugging?
**A:** `docker exec` for shell access, `docker logs` for application logs, `docker stats` for metrics, `docker inspect` for configuration, `docker network` for networking, `docker cp` for file access. In Kubernetes: `kubectl logs`, `kubectl exec`, `kubectl describe`, `kubectl debug`.

### Q6: How do you debug a container that can't pull images?
**A:** Verify authentication with `docker login` or cloud SDK. Check network connectivity to registry. Verify image name and tag exist. Check rate limits. Use `docker pull -v` for verbose output. Verify TLS certificates. Try pulling a public image to isolate the issue.

### Q7: What's the difference between docker logs and docker events?
**A:** `docker logs` shows application output (stdout/stderr). `docker events` shows Docker daemon events (container start/stop, image pull, etc.). Both are useful: logs for app debugging, events for infrastructure debugging.

### Q8: How do you debug permission issues in containers?
**A:** Check user with `docker exec whoami`. Verify volume ownership with `docker exec ls -la`. Use `docker exec chown` to fix. Set USER in Dockerfile. Use `docker run --user` flag. Check SELinux/AppArmor labels. Use `--privileged` for testing (not production).

### Q9: What is the debugging workflow in Kubernetes?
**A:** 1) `kubectl get pods` to see status. 2) `kubectl describe pod` for events. 3) `kubectl logs` for application logs. 4) `kubectl exec` for debugging commands. 5) `kubectl port-forward` for service testing. 6) `kubectl debug` for ephemeral debugging. 7) `kubectl top` for resource analysis.

### Q10: How do you investigate slow container performance?
**A:** Check resource limits vs usage with `docker stats`. Look for CPU throttling with `docker exec cat /sys/fs/cgroup/cpu/cpu.stat`. Check I/O with `docker exec iostat`. Profile the application. Look for network latency. Check for disk full conditions. Review application logs for slow queries/operations.
