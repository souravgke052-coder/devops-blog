# BigQuery

## Overview

Google BigQuery is a serverless, highly scalable data warehouse that enables super-fast SQL queries against massive datasets. It uses columnar storage and tree architecture for querying petabytes in seconds. BigQuery separates storage and compute, allowing independent scaling and cost optimization. ML built-in enables predictive analytics directly in the data warehouse.

## When To Use

- Data warehousing and business intelligence
- Analytics on large datasets (> 10 GB)
- Ad-hoc SQL queries on raw data
- Data lake queries (using external tables)
- Streaming analytics (with streaming insert pricing)
- Machine learning with BigQuery ML
- Time-series analytics and trends

## When NOT To Use

- Real-time transactional workloads (use Cloud SQL)
- Millisecond-latency requirements
- OLTP workloads
- Small datasets with complex joins
- Simple key-value lookups
- File storage needs (use Cloud Storage)

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                       BigQuery                               │
├─────────────────────────────────────────────────────────────┤
│  Storage:                                                   │
│  ├── Columnar storage (Capacitor format)                   │
│  ├── Automatic compression                                 │
│  ├── Managed storage (Cold/Hot tiers)                      │
│  └── External tables (Cloud Storage, Sheets)               │
├─────────────────────────────────────────────────────────────┤
│  Compute:                                                   │
│  ├── On-demand (per byte scanned)                          │
│  ├── Flat-rate (reserved slots)                            │
│  └── BI Engine (in-memory acceleration)                    │
├─────────────────────────────────────────────────────────────┤
│  ML:                                                        │
│  ├── BigQuery ML (SQL-based ML)                            │
│  └── Vertex AI integration                                 │
└─────────────────────────────────────────────────────────────┘
```

## Core Concepts

### Dataset
Container for tables and views. Acts as a namespace for tables.

### Table
Data organized in rows and columns. Types: native, external, logical views.

### Slot
Virtual CPU for query execution. On-demand uses slot-based pricing.

### Partitioning
Dividing tables into segments based on date/timestamp or integer range.

### Clustering
Sorting data by column values for improved query performance.

### Streaming Insert
Real-time data insertion at $0.010/200 MB.

## CLI Examples

```bash
# Create dataset
gcloud bigquery datasets create my_dataset \
  --location=US \
  --description="My analytics dataset"

# Create table with schema
gcloud bigquery mk-table my_dataset.my_table \
  --schema=id:INTEGER,name:STRING,created_at:TIMESTAMP

# Load data from Cloud Storage
gcloud bigquery load \
  --source_format=CSV \
  --location=US \
  my_dataset.users \
  gs://my-bucket/users.csv \
  --schema=id:INTEGER,name:STRING,email:STRING

# Query from command line
gcloud bigquery query \
  --use_legacy_sql=false \
  "SELECT * FROM my_dataset.users LIMIT 10"

# Query with parameters
gcloud bigquery query \
  --parameter=date_value:STRING:2024-01-15 \
  "SELECT * FROM events WHERE DATE(event_time) = @date_value"

# Export to Cloud Storage
gcloud bigquery export \
  my_project:my_dataset.orders \
  gs://my-bucket/exports/orders_$(date +%Y%m%d).csv

# Create partitioned table
gcloud bigquery mk-table my_dataset.events \
  --schema=event_id:STRING,event_time:TIMESTAMP,data:STRING \
  --time_partitioning-type=DAY \
  --time_partitioning-field=event_time

# Create clustered table
gcloud bigquery mk-table my_dataset.events \
  --schema=event_id:STRING,customer_id:STRING,event_type:STRING \
  --clustering-fields=customer_id,event_type

# Update dataset labels
gcloud bigquery datasets update my_dataset \
  --location=US \
  --update-labels=environment:production,cost_center:analytics

# Copy table
gcloud bigquery cp my_dataset.source_table my_dataset.dest_table

# Create view
gcloud bigquery mk-view my_dataset.active_users_view \
  "SELECT * FROM my_dataset.users WHERE status = 'active'"
```

### Column-level Security
```sql
-- Create view with column masking
CREATE VIEW safe_customers AS
SELECT
  customer_id,
  -- Mask email for non-admin users
  CASE 
    WHEN SESSION_USER() LIKE '%@admin.%' THEN email
    ELSE CONCAT(LEFT(email, 2), '***', RIGHT(email, 4))
  END AS email,
  name
FROM customers;

-- Row-level security via policy tags
CREATE TABLE customers (
  customer_id STRING,
  email STRING,
  ssn STRING OPTIONS (policy_tags = ['confidential'])
);
```

### Authorized Views
```sql
-- Create view in separate dataset for restricted access
CREATE VIEW restricted_orders AS
SELECT order_id, customer_id, total
FROM orders
WHERE customer_id IN (
  SELECT customer_id FROM allowed_customers WHERE user_email = SESSION_USER()
);
```

## Cost Optimization

### Query Cost Reduction
```sql
-- Select only needed columns
SELECT customer_id, order_date, total  -- Instead of SELECT *

-- Use partitioning filters
WHERE DATE(event_time) BETWEEN '2024-01-01' AND '2024-01-31'  -- Filters partitions

-- Use clustering fields in WHERE
WHERE customer_id = '123' AND event_type = 'purchase'  -- Uses clustering

-- Avoid SELECT *
EXPLAIN 
SELECT * FROM large_table;  -- Scans entire table

EXPLAIN 
SELECT id, name FROM large_table;  -- Scans only needed columns
```

### Partitioning Strategy
```sql
-- Partition by date for time-series data
CREATE TABLE events (
  event_date DATE,
  event_time TIMESTAMP,
  data STRING

PARTITION BY DATE(event_time)
CLUSTER BY customer_id;
```

## Interview Questions

**Q: What is the difference between partitioning and clustering in BigQuery?**
A: Partitioning splits a table into segments based on a column (date or integer), allowing queries to skip irrelevant partitions. Clustering sorts data by specified columns within each partition, improving performance for filtered queries. Partitioning is required for time-based pruning; clustering provides additional sort optimization.

**Q: How does BigQuery pricing work?**
A: On-demand pricing charges $5.00 per TB of data scanned. Flat-rate pricing provides reserved slots at ~$1,700/month per slot. Storage costs $0.02/GB/month, dropping to $0.01/GB after 90 days. Queries that don't touch tables (metadata queries) have lower costs.

**Q: When should you use external tables vs native tables?**
A: Use native tables for frequently queried data requiring BigQuery optimization. Use external tables for data that lives in other systems (Cloud Storage, Bigtable, Google Sheets), for ETL pipelines, or for data that shouldn't be duplicated. External queries have different performance characteristics.

**Q: How does BigQuery handle concurrent queries?**
A: BigQuery uses a slot-based execution model. On-demand queries share a global slot pool (2000 slots per project). Reserved slots guarantee capacity for critical workloads. Queries compete for available slots; excess queries queue.

**Q: What are the limitations on UPDATE/DELETE operations?**
A: DML operations (UPDATE, DELETE, MERGE) work within partition boundaries for partitioned tables and have size limits. Use MERGE for upsert patterns. Large-scale DML may be slow; consider recreating tables instead.

**Q: How do you implement row-level security in BigQuery?**
A: Use views with SESSION_USER() to filter rows based on the query user. For column-level security, use CASE statements to mask sensitive data. Authorized views provide another security pattern. Policy tags enable data governance integration.
