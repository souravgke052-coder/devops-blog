# Cloud SQL

## Overview

Google Cloud SQL is a fully managed relational database service supporting MySQL, PostgreSQL, and SQL Server. It provides automated backups, replication, patching, and maintenance while abstracting database administration overhead. Cloud SQL handles over 100,000 databases for GCP customers and offers up to 64 TB of storage with high availability configurations.

## When To Use

- Traditional web applications requiring SQL
- E-commerce platforms with transactional needs
- CMS and content management systems
- Mobile application backends
- Analytics dashboards with relational data
- Small to medium datasets (< 64 TB)
- Lift-and-shift from on-premises SQL workloads

## When NOT To Use

- Massive scale (> 100 TB, multi-petabyte)
- Globally distributed writes (use Spanner)
- Complex NoSQL queries (use Firestore or Bigtable)
- In-memory caching (use Memorystore)
- Time-series data at scale (use BigQuery or TimescaleDB)
- Extreme write throughput (use Cloud Bigtable)

## Core Concepts

### Instance
The core database server. Types: Primary (read-write), Read Replica (read-only), Failover Replica (HA standby).

### Database
Logical container within an instance holding tables, views, and other objects.

### Connection
Client connection to the database. Cloud SQL Proxy provides secure connectivity.

### Backup
Point-in-time recovery using automated or on-demand backups stored in Cloud Storage.

### Private IP
VPC-native connectivity without public internet exposure.

### Authorized Network
IP ranges allowed to connect directly to the instance.

### Flags
Database-specific configuration parameters (e.g., `max_connections`, `log_min_duration_statement`).

### HA Configuration
Regional configuration with automatic failover to standby in different zone.

## Real Production Example

**E-commerce Platform with Cloud SQL PostgreSQL**

```hcl
# Terraform configuration for production e-commerce database

# Enable required APIs
resource "google_project_service" "sqladmin" {
  service = "sqladmin.googleapis.com"
}

# Create VPC for private connectivity
resource "google_compute_network" "sql_vpc" {
  name = "sql-vpc-${var.environment}"
  auto_create_subnetworks = false
}

resource "google_compute_subnetwork" "sql_subnet" {
  name          = "sql-subnet-${var.environment}"
  network       = google_compute_network.sql_vpc.id
  ip_cidr_range = "10.0.1.0/24"
  region        = var.region
  
  private_ip_google_access = true
}

# Create Private Service Access for Cloud SQL
resource "google_compute_global_address" "sql_private_ip" {
  name          = "sql-private-ip-${var.environment}"
  purpose       = "VPC_PEERING"
  address_type  = "INTERNAL"
  prefix_length = 16
  network       = google_compute_network.sql_vpc.id
}

resource "google_service_networking_connection" "sql_vpc_connection" {
  network       = google_compute_network.sql_vpc.id
  service       = "servicenetworking.googleapis.com"
  reserved_peering_ranges = [google_compute_global_address.sql_private_ip.name]
}

# Create Cloud SQL instance with HA
resource "google_sql_database_instance" "ecommerce_db" {
  name             = "ecommerce-db-${var.environment}"
  database_version = "POSTGRES_15"
  region           = var.region
  
  deletion_protection  = true
  enable_binary_logging = true
  insights_config {
    query_insights_enabled  = true
    query_string_length     = 1024
    record_application_tags = true
    record_client_address   = false
  }
  
  settings {
    tier              = "db-n2-standard-4"
    availability_type = "REGIONAL"  # HA enabled
    disk_type         = "PD_SSD"
    disk_size         = 500
    disk_autoresize   = true
    storage_auto_increase_limit = 1000
    
    ip_configuration {
      ipv4_enabled    = false
      private_network = google_compute_network.sql_vpc.id
      require_ssl     = true
      
      authorized_networks {
        name  = "app-tier"
        value = "10.0.2.0/24"
      }
    }
    
    backup_configuration {
      enabled                        = true
      start_time                     = "03:00"
      point_in_time_recovery_enabled = true
      backup_retention_settings {
        retained_backups = 7
        retention_unit   = "COUNT"
      }
    }
    
    maintenance_window {
      day          = 7  # Sunday
      hour         = 4  # 4 AM
      update_track = "stable"
    }
    
    database_flags {
      name  = "max_connections"
      value = "500"
    }
    
    database_flags {
      name  = "log_min_duration_statement"
      value = "1000"
    }
    
    database_flags {
      name  = "pgaudit.log"
      value = "read,write"
    }
    
    insights_config {
      query_insights_enabled  = true
      query_string_length     = 1024
    }
    
    ip_configuration {
      dynamic "authorized_networks" {
        for_each = var.authorized_ips
        content {
          name  = authorized_networks.value.name
          value = authorized_networks.value.cidr
        }
      }
    }
  }
  
  labels = {
    environment = var.environment
    team        = "platform"
    component   = "database"
  }
}

# Create databases
resource "google_sql_database" "ecommerce" {
  name     = "ecommerce"
  instance = google_sql_database_instance.ecommerce_db.name
}

resource "google_sql_database" "analytics" {
  name     = "analytics"
  instance = google_sql_database_instance.ecommerce_db.name
}

# Create read replica in same region
resource "google_sql_database_instance" "read_replica" {
  name            = "ecommerce-db-read-replica-${var.environment}"
  database_version = "POSTGRES_15"
  region          = var.region
  
  master_instance_name = google_sql_database_instance.ecommerce_db.name
  
  deletion_protection = false
  
  replica_configuration {
    failover_target = false
  }
  
  settings {
    tier              = "db-n2-standard-4"
    availability_type = "ZONAL"  # Replica is zonal
    
    ip_configuration {
      ipv4_enabled    = false
      private_network = google_compute_network.sql_vpc.id
      require_ssl     = true
    }
    
    database_flags {
      name  = "max_connections"
      value = "500"
    }
  }
}

# Cross-region read replica for DR
resource "google_sql_database_instance" "dr_replica" {
  name             = "ecommerce-db-dr-${var.environment}"
  database_version = "POSTGRES_15"
  region           = var.dr_region  # Different region
  
  master_instance_name = google_sql_database_instance.ecommerce_db.name
  
  deletion_protection = false
  
  settings {
    tier              = "db-n2-standard-2"  # Smaller for DR
    availability_type = "ZONAL"
    
    ip_configuration {
      ipv4_enabled    = false
      private_network = google_compute_network.dr_vpc.id
      require_ssl     = true
    }
  }
}

# User management via Secret Manager (production best practice)
resource "google_secret_manager_secret" "db_password" {
  secret_id = "ecommerce-db-password"
  
  replication {
    auto {}
  }
}

resource "google_secret_manager_secret_version" "db_password" {
  secret      = google_secret_manager_secret.db_password.id
  secret_data = var.db_password
}

# IAM database authentication
resource "google_sql_database_instance" "ecommerce_db" {
  # ... previous config ...
  
  settings {
    # ... previous settings ...
    
    iam_authentication = true
  }
}
```

## Interview Questions

**Q: What is the difference between REGIONAL and ZONAL availability types?**
A: REGIONAL creates a standby replica in a different zone within the same region, providing high availability with automatic failover (~60 second RTO). ZONAL is single-zone only, lower cost but no HA. Regional adds ~50% to instance cost.

**Q: How does Cloud SQL handle point-in-time recovery?**
A: Cloud SQL maintains continuous Write-Ahead Log (WAL) archiving to Cloud Storage. When PITR is enabled, you can restore to any point within the backup retention period (1-35 days) by replaying WAL entries.

**Q: What are the connection limits for Cloud SQL?**
A: PostgreSQL default max_connections varies by tier (db-f1-micro: 25, db-g1-small: 50, larger: 100-8000). Use connection pooling (PgBouncer, Cloud SQL Auth Proxy) for high-concurrency workloads.

**Q: When should you use Cloud SQL vs Cloud Spanner?**
A: Choose Cloud SQL for: regional data, moderate scale (< 2 TB), cost-sensitive workloads, MySQL/PostgreSQL compatibility. Choose Spanner for: global distribution, > 2 TB, massive scale, strong consistency across regions.

**Q: How do you secure Cloud SQL connections?**
A: Use Private IP with VPC, enforce SSL (`require_ssl=true`), enable IAM authentication, use Cloud SQL Auth Proxy, configure authorized networks if needed, and implement principle of least privilege in IAM.

**Q: What is Cloud SQL Auth Proxy and why use it?**
A: Auth Proxy provides secure connection to Cloud SQL using Cloud SQL API, eliminating the need for allowlisting IP addresses. It handles IAM authentication and mTLS encryption automatically, working even from outside GCP.

**Q: How do you migrate from on-premises to Cloud SQL?**
A: Use Database Migration Service (DMS) for continuous migration with minimal downtime. For PostgreSQL, also use `pg_dump`/`pg_restore` for one-time migration or logical replication for ongoing sync.
