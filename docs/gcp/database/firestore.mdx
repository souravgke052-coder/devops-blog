# Firestore

## Overview

Google Cloud Firestore is a serverless, NoSQL document database that automatically scales to handle millions of clients. It offers real-time sync, offline support, and mobile SDKs, making it ideal for mobile apps, web apps, and IoT devices. Firestore organizes data in documents (JSON-like) and collections, with support for subcollections and queries.

## When To Use

- Mobile and web applications requiring real-time sync
- User profile and preference storage
- Chat messages and social feeds
- Content management with flexible schemas
- Gaming leaderboards and state
- IoT device data ingestion
- Prototyping and rapid development

## When NOT To Use

- Highly normalized relational data
- Complex queries requiring joins
- Blob/file storage (use Cloud Storage)
- Time-series data at massive scale (use Bigtable)
- Applications requiring SQL expressiveness
- Data warehousing and analytics (use BigQuery)

## Core Concepts

### Document
The basic unit of storage. Contains key-value pairs, supports nested objects, arrays, and subcollection references. Maximum size: 1 MB.

### Collection
A container for documents. Can hold documents and subcollection references.

### Subcollection
A collection nested within a document. Enables hierarchical data modeling.

### Query
Read operations that filter and order documents. Supports compound indexes.

### Real-time Listeners
SDK feature that pushes updates to clients when data changes.

### Offline Persistence
SDK feature that caches data locally, enabling offline reads and writes.

```javascript
// Firestore Security Rules (firestore.rules)
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Helper functions
    function isAuthenticated() {
      return request.auth != null;
    }
    
    function isOwner(userId) {
      return isAuthenticated() && request.auth.uid == userId;
    }
    
    function isValidMessage() {
      let data = request.resource.data;
      return data.content.size() > 0 
        && data.content.size() < 10000
        && data.createdAt == request.time;
    }
    
    // User profiles
    match /users/{userId} {
      allow read: if isAuthenticated();
      allow write: if isOwner(userId);
    }
    
    // Chat channels
    match /channels/{channelId} {
      allow read: if isAuthenticated();
      allow create: if isAuthenticated();
      allow update, delete: if isOwner(resource.data.createdBy);
    }
    
    // Messages in channels
    match /channels/{channelId}/messages/{messageId} {
      allow read: if isAuthenticated();
      allow create: if isAuthenticated() && isValidMessage();
      allow update, delete: if isOwner(resource.data.senderId);
    }
    
    // Typing indicators (temporary data)
    match /typing/{channelId}/{userId} {
      allow read: if isAuthenticated();
      allow write: if isOwner(userId);
    }
  }
}
```

## Security

### Security Rules
```javascript
// Comprehensive security rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Validate data types and constraints
    function isValidUser(userId) {
      let data = request.resource.data;
      return data.keys().hasAll(['email', 'displayName', 'createdAt'])
        && data.email is string
        && data.displayName is string
        && data.createdAt == request.time;
    }
    
    function isValidPost() {
      let data = request.resource.data;
      return data.title.size() > 0
        && data.title.size() < 500
        && data.content.size() < 50000
        && data.publishedAt == request.time
        || resource.data.publishedAt == request.resource.data.publishedAt;
    }
    
    // Users collection
    match /users/{userId} {
      allow read: if request.auth != null;
      allow create: if request.auth.uid == userId && isValidUser(userId);
      allow update: if request.auth.uid == userId 
        && request.resource.data.email == resource.data.email; // Immutable email
      allow delete: if false; // Don't allow deletion
    }
    
    // Posts collection
    match /posts/{postId} {
      allow read: if true;
      allow create: if request.auth != null && isValidPost();
      allow update: if request.auth.uid == resource.data.authorId;
      allow delete: if request.auth.uid == resource.data.authorId;
    }
    
    // Admin-only collection
    match /admin/{document=**} {
      allow read, write: if request.auth != null 
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.isAdmin == true;
    }
  }
}
```

```javascript
// Firestore quotas:
// 1M concurrent connections per database
// 10K writes/second per database
// 1M reads/second per database

// If quota exceeded:
// 1. Implement exponential backoff
const retryWithBackoff = async (operation, maxRetries = 3) => {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await operation();
    } catch (error) {
      if (error.code === 'RESOURCE_EXHAUSTED' && i < maxRetries - 1) {
        await new Promise(r => setTimeout(r, Math.pow(2, i) * 1000));
      } else {
        throw error;
      }
    }
  }
};
```

## Interview Questions

**Q: What is the difference between Firestore and Cloud Datastore?**
A: Firestore Native mode offers real-time sync, offline support, and mobile SDKs. Datastore mode provides backwards compatibility with older Datastore apps. New projects should use Firestore Native mode for better features.

**Q: How does Firestore handle real-time synchronization?**
A: Firestore uses WebSocket-based connections for real-time updates. The SDK maintains an open connection to Firestore servers, which push changes as they occur. Listeners are notified with the full document or query results on changes.

**Q: What are the Firestore security rules evaluation costs?**
A: Security rules are evaluated per document accessed. Reading 100 documents evaluates rules 100 times. This can become expensive at scale—use App Check and optimize rules to avoid unnecessary evaluations.

**Q: How do you model hierarchical data in Firestore?**
A: Use subcollections for one-to-many relationships (e.g., messages within channels). Use document references for many-to-one relationships. Consider flattening data for read-heavy patterns and normalizing for write-heavy patterns.

**Q: What are the Firestore query limitations?**
A: Queries can only filter on fields that have an index. You cannot query across subcollections with a single query. IN and array-contains-any are limited to 10 values. No support for JOINs or cross-collection queries.

**Q: How does offline persistence work in Firestore?**
A: The SDK caches writes and reads to IndexedDB (browser) or SQLite (mobile). When offline, reads come from the cache, writes queue locally. On reconnection, the SDK syncs pending writes and fetches updates from the server.

## Cheat Sheet

```javascript
// Initialize
import { initializeFirestore } from 'firebase/firestore';
const db = initializeFirestore(app, { host: 'project.firebaseio.com', ssl: true });

// CRUD Operations
import { collection, doc, getDoc, setDoc, updateDoc, deleteDoc } from 'firebase/firestore';

// Create/Update
await setDoc(doc(db, 'users', userId), { email, name, createdAt: serverTimestamp() });

// Read
const docSnap = await getDoc(doc(db, 'users', userId));
if (docSnap.exists()) console.log(docSnap.data());

// Update
await updateDoc(doc(db, 'users', userId), { lastLogin: serverTimestamp() });

// Delete
await deleteDoc(doc(db, 'users', userId));

// Query
import { query, where, orderBy, limit } from 'firebase/firestore';
const q = query(collection(db, 'posts'), 
  where('authorId', '==', userId), 
  orderBy('createdAt', 'desc'), 
  limit(20));

// Real-time listener
const unsubscribe = onSnapshot(q, (snapshot) => {
  snapshot.docChanges().forEach(change => {
    if (change.type === 'added') console.log('Added:', change.doc.data());
    if (change.type === 'modified') console.log('Modified:', change.doc.data());
    if (change.type === 'removed') console.log('Removed:', change.doc.id);
  });
});

// Cleanup
unsubscribe();
```
