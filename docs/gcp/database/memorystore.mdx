# Memorystore

## Overview

Google Cloud Memorystore is a fully managed in-memory data store service for Redis and Memcached. It provides sub-millisecond latency for caching, session management, and real-time analytics. Memorystore handles deployment, replication, failover, and patching automatically, offering 99.99% availability for the Enterprise tier.

## When To Use

- Application caching for database acceleration
- Session storage for web applications
- Real-time leaderboards and gaming leaderboards
- Rate limiting and throttling
- ML feature store for low-latency inference
- Distributed locking coordination
- Message queue for pub/sub patterns

## When NOT To Use

- Primary data store (data loss risk)
- Data requiring durability guarantees
- Full-text search (use Elasticsearch)
- Graph data (use Neo4j)
- Relational data (use Cloud SQL)
- Batch processing jobs

## Core Concepts

### Instance
A deployed Redis or Memcached server. Types: Basic (standalone), Standard (HA), Enterprise (full HA + replicas).

### Node
Individual server within a Memorystore instance. Basic has 1 node, Standard/Enterprise have 2+ nodes.

### Shards
For Memcached clusters, data is distributed across multiple shards.

### Redis AUTH
Password-based authentication for Redis instances.

### Redis ACL
Access Control Lists for fine-grained permission management (Redis 6+).

### Reserved IP Range
Memorystore instances are VPC-native and use addresses from a reserved range.

## Real Production Example

**Session Store and Cache for Web Application**

```hcl
# Terraform configuration for production Memorystore setup

# Create VPC for Memorystore
resource "google_compute_network" "memorystore_vpc" {
  name = "memorystore-vpc-${var.environment}"
  auto_create_subnetworks = false
}

resource "google_compute_subnetwork" "memorystore_subnet" {
  name          = "memorystore-subnet-${var.environment}"
  network       = google_compute_network.memorystore_vpc.id
  ip_cidr_range = "10.0.2.0/28"  # /28 provides 11 IPs, enough for most uses
  region        = var.region
  
  private_ip_google_access = true
}

# Create Private Service Access
resource "google_compute_global_address" "memorystore_private_ip" {
  name          = "memorystore-private-ip-${var.environment}"
  purpose       = "VPC_PEERING"
  address_type  = "INTERNAL"
  prefix_length = 16
  network       = google_compute_network.memorystore_vpc.id
}

resource "google_service_networking_connection" "memorystore_vpc_connection" {
  network       = google_compute_network.memorystore_vpc.id
  service       = "servicenetworking.googleapis.com"
  reserved_peering_ranges = [google_compute_global_address.memorystore_private_ip.name]
}

# Redis Instance for Session Storage (Standard tier for HA)
resource "google_redis_instance" "session_store" {
  name           = "session-redis-${var.environment}"
  memory_size_gb = 5
  region         = var.region
  location_id    = "${var.region}-a"  # Primary zone
  
  tier = "STANDARD_HA"
  
  redis_version  = "redis_7_0"
  
  display_name   = "Session Store - ${var.environment}"
  
  reserved_ip_range = "10.0.2.5"
  
  authorized_network = google_compute_network.memorystore_vpc.id
  
  connectivity_settings {
    private_service_subnet = google_compute_subnetwork.memorystore_subnet.name
    direct_path_enabled    = true
  }
  
  maintenance_policy {
    day          = 7  # Sunday
    start_hour   = 2   # 2 AM
    update_mode  = "ZONE_UPDATE"
  }
  
  auth_enabled      = true
  transit_encryption_mode = "SERVER_AUTHENTICATION"
  
  labels = {
    environment = var.environment
    component   = "session-store"
    tier        = "ha"
  }
}

# Redis Instance for Application Cache (Basic tier)
resource "google_redis_instance" "app_cache" {
  name           = "cache-redis-${var.environment}"
  memory_size_gb = 10

    environment = var.environment

  region         = var.region
  location_id    = "${var.region}-a"
```

## CLI Examples

```bash
# Create Redis instance (Basic tier)
gcloud redis instances create my-redis \
  --size=2 \
  --region=us-central1 \
  --zone=us-central1-a \
  --redis-version=redis_7_0 \
  --tier=BASIC

# Create Redis instance (Standard HA)
gcloud redis instances create my-redis-ha \
  --size=5 \
  --region=us-central1 \
  --tier=STANDARD_HA \
  --redis-version=redis_7_0 \
  --authorized-network=my-vpc \
  --redis-version=redis_7_0 \
  --auth-enabled

gcloud redis instances create my-redis-enterprise \
  --size=32 \
  --region=us-central1 \
  --tier=ENTERPRISE_STANDALONE \
  --replica-count=2 \
  --redis-version=redis_7_0

# Create Memcached cluster
gcloud memcache instances create my-memcached \
  --size=5 \
  --node-count=3 \
  --region=us-central1 \
  --memcached-version=1.6 \
  --authorized-network=my-vpc

# List Redis instances
gcloud redis instances list --region=us-central1

# Describe Redis instance
gcloud redis instances describe my-redis --region=us-central1

# Get Redis connection info
gcloud redis instances describe my-redis \
  --region=us-central1 \
  --format="yaml(host,port,tier,memorySizeGb)"

# Update Redis instance
gcloud redis instances update my-redis \
  --region=us-central1 \
  --size=10

# Set maintenance window
gcloud redis instances update my-redis \
  --region=us-central1 \
  --maintenance-window-day=sunday \
  --maintenance-window-hour=2

# Import data into Redis
gcloud redis instances import my-redis \
  --region=us-central1 \
  --source=gs://my-bucket/dump.rdb

# Export data from Redis
gcloud redis instances export my-redis \
  --region=us-central1 \
  --destination=gs://my-bucket/dump.rdb

# Connect via Redis CLI (via a GCE instance in the VPC)
gcloud compute ssh my-bastion --zone=us-central1-a -- \
  "redis-cli -h \$(gcloud redis instances describe my-redis --region=us-central1 --format='value(host)') -a \$AUTH_PASSWORD PING"

# Failover for Standard/Enterprise tier
gcloud redis instances failover my-redis-ha --region=us-central1
```

## Monitoring

### Key Metrics
- **Memory usage**: Used/peak memory vs allocated
- **CPU utilization**: Instance CPU load
- **Client connections**: Number of connected clients
- **Commands/second**: Operation throughput
- **Cache hit ratio**: Effectiveness of caching
- **Replication lag**: For Standard/Enterprise tiers
- **Network bytes**: Data transfer rate

### Cloud Monitoring

```bash
# List Memorystore metrics
gcloud monitoring metrics list --filter="resource.type=redis_instance"

# Create alert for memory usage > 80%
gcloud monitoring policies create \
  --display-name="Memorystore High Memory" \
  --condition-display-name="Memory > 80%" \
  --condition-filter='resource.type="redis_instance" AND metric.type="redis.googleapis.com/memory/usage_ratio"' \
  --condition-threshold-value=0.8 \
  --notification-channels=CHANNEL_ID

  "redis.googleapis.com/stats/hits" \
  --filter='resource.instance_id="my-instance"'
```

## Troubleshooting

### Connection Issues
```bash
# Verify instance is running
gcloud redis instances describe my-redis --region=us-central1

# Check connectivity from VPC
gcloud compute ssh bastion --zone=us-central1-a -- \
  "nc -zv REDIS_HOST 6379"

# Test Redis connection
redis-cli -h REDIS_HOST -a PASSWORD PING

# Check if AUTH is enabled
gcloud redis instances describe my-redis --region=us-central1 \
  --format="yaml(authEnabled)"
```

### High Memory Usage
```bash
# Check memory details
redis-cli -h HOST -a PASSWORD INFO memory

# Find large keys
redis-cli -h HOST -a PASSWORD --bigkeys

# Monitor key expiration
redis-cli -h HOST -a PASSWORD INFO keyspace

# Set maxmemory policy if needed
redis-cli -h HOST -a PASSWORD CONFIG SET maxmemory-policy allkeys-lru
```

### Cache Inefficiency
```bash
# Check hit/miss ratio
redis-cli -h HOST -a PASSWORD INFO stats | grep keyspace

# Typical targets:
# Hit ratio > 90% is good for production
# If low, increase cache size or review cache key strategy

# Monitor eviction rate
redis-cli -h HOST -a PASSWORD INFO stats | grep evicted
```

### Replication Lag
```bash
# For Standard/Enterprise tiers
# Check replication lag
redis-cli -h HOST -a PASSWORD INFO replication

# lag: 0 means in sync
# Positive value means replica is behind (in bytes)

# Monitor via Cloud Monitoring
gcloud monitoring metrics read \
  "redis.googleapis.com/replication/offset_diff_bytes"
```

### Failover Issues
```bash
# Manual failover
gcloud redis instances failover my-redis --region=us-central1

# After failover, verify
redis-cli -h NEW_PRIMARY_HOST -a PASSWORD INFO replication

# Check application reconnection logic
# Applications should handle connection drops gracefully
```

## Interview Questions

**Q: What is the difference between Memorystore Basic and Standard tiers?**
A: Basic tier is a single-node instance with no replication. Standard tier adds automatic failover with a primary-replica setup - if the primary fails, the replica promotes automatically. Standard costs approximately 2x Basic.

**Q: How does Memorystore Redis handle high availability?**
A: Standard and Enterprise tiers use a primary-replica configuration with automatic failover. The replica continuously syncs from the primary. If the primary becomes unavailable, Memorystore automatically promotes the replica and updates DNS within seconds.

**Q: When should you use Memcached over Redis in Memorystore?**
A: Choose Memcached for simple key-value caching where you need horizontal scaling with consistent hashing and auto-discovery. Choose Redis for complex data types (lists, sets, sorted sets), pub/sub, transactions, scripting, and features like TTL on individual items.

**Q: What are the Memorystore connection limits?**
A: Redis supports up to 40,000 concurrent connections. Memcached supports up to 30,000 connections per node. Use connection pooling in applications to efficiently manage connections.

**Q: How does Memorystore pricing work?**
A: Memorystore pricing is based on memory size and tier: Basic ($0.045/GB/hour), Standard ($0.098/GB/hour), Enterprise ($0.17/GB/hour). Regional data transfer also adds cost. Reserved instances provide discounts for committed usage.

**Q: Can you use Memorystore from outside GCP?**
A: Memorystore is VPC-native and only accessible from within the VPC. For external access, you need to set up Cloud VPN or Cloud Interconnect, or use a GCE instance as a bastion to tunnel connections.
