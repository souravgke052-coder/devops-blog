# Artifact Registry

## Overview

Google Cloud Artifact Registry is a fully managed artifact repository service that stores, manages, and secures Docker images, Maven packages, npm packages, and more. It provides vulnerability scanning, fine-grained access control, and regional/multi-region deployment options for efficient artifact distribution.

## When To Use

- Storing and managing Docker container images
- Hosting private Maven/npm packages
- CI/CD pipeline artifact storage
- Multi-cloud artifact management
- Helm chart repositories
- Python package hosting

## When NOT To Use

- Public open-source package hosting (use Docker Hub, npm registry)
- Binary artifacts that don't fit supported formats
- Large binary blobs (use Cloud Storage directly)

```
│  └── Multi-region (us, eu, asia)                           │
└─────────────────────────────────────────────────────────────┘
```

## Core Concepts

### Repository
Container for artifacts with format-specific configuration.

### Artifact
Docker image, Maven package, npm package, etc.

### Virtual Repository
Aggregates multiple repositories under one endpoint.

### Remote Repository
Proxy cache for upstream repositories (e.g., Docker Hub).

### Vulnerability Scanning
Automatic scanning for known CVEs in container images.

## CLI Examples

```bash
# Create Docker repository
gcloud artifacts repositories create my-repo \
  --repository-format=docker \
  --location=us-central1 \
  --description="My Docker repository"

# Create with cleanup policy
gcloud artifacts repositories create my-repo \
  --repository-format=docker \
  --location=us-central1 \
  --docker-cleanup-policy=delete-unmatched

# List repositories
gcloud artifacts repositories list --location=us-central1

# Describe repository
gcloud artifacts repositories describe my-repo --location=us-central1

# Configure Docker authentication
gcloud auth configure-docker us-central1-docker.pkg.dev

# Push Docker image
docker push us-central1-docker.pkg.dev/my-project/my-repo/my-image:tag

# Pull Docker image
docker pull us-central1-docker.pkg.dev/my-project/my-repo/my-image:tag

# List images
gcloud artifacts docker list us-central1-docker.pkg.dev/my-project/my-repo

# List versions of an image
gcloud artifacts versions list us-central1-docker.pkg.dev/my-project/my-repo/my-image

# Delete image version
gcloud artifacts versions delete us-central1-docker.pkg.dev/my-project/my-repo/my-image:tag

# Create Maven repository
gcloud artifacts repositories create maven-repo \
  --repository-format=maven \
  --location=us-central1 \
  --description="Maven repository"

# Create npm repository
gcloud artifacts repositories create npm-repo \
  --repository-format=npm \
  --location=us-central1 \
  --description="NPM repository"

# Set IAM policy
gcloud artifacts repositories add-iam-policy-binding my-repo \
  --location=us-central1 \
  --member=serviceAccount:sa@project.iam.gserviceaccount.com \
  --role=roles/artifactregistry.writer

# Docker cleanup policies
gcloud artifacts repositories update my-repo \
  --location=us-central1 \
  --docker-cleanup-policy=delete-unmatched

# Update repository description
gcloud artifacts repositories update my-repo \
  --location=us-central1 \
  --description="Updated description"
```

## Interview Questions

**Q: What is the difference between Artifact Registry and Container Registry?**
A: Container Registry (gcr.io) is the legacy service. Artifact Registry is the modern replacement with additional features: multiple format support (Docker, Maven, npm), vulnerability scanning, fine-grained IAM, remote repositories for caching, and better scalability. Migration from Container Registry to Artifact Registry is recommended.

**Q: How does vulnerability scanning work in Artifact Registry?**
A: When you push a Docker image, Artifact Registry automatically scans it for known vulnerabilities using the Container Analysis service. You can view the vulnerability report via CLI or Cloud Console. Severity levels include CRITICAL, HIGH, MEDIUM, and LOW.

**Q: What are remote repositories in Artifact Registry?**
A: Remote repositories act as proxy caches for upstream repositories (like Docker Hub or npmjs.org). They cache fetched artifacts, reducing external network calls and providing faster subsequent pulls. The cache is populated on-demand.

**Q: How do you clean up old image versions?**
A: Use the cleanup policy on the repository: `delete-unmatched` removes manifests not tagged after a specified time. Alternatively, manually delete versions using `gcloud artifacts versions delete`. Lifecycle policies can automate cleanup.

**Q: What Docker cleanup policies are available?**
A: `delete-unmatched` removes untagged manifests that haven't been accessed for a configurable period. `never` (default) retains all manifests. You can also manually manage cleanup via lifecycle scripts.
