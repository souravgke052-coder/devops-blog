# Deployment Pipelines

## Overview

Google Cloud provides multiple services for orchestrating deployment pipelines: Cloud Deploy, GKE Deployment Manager, Config Connector, and Terraform. This page covers strategies and implementation patterns for reliable, automated deployment to GCP infrastructure.

## When To Use

- Progressive delivery to GKE/Cloud Run
- Multi-environment deployments (dev/staging/prod)
- GitOps-based deployments
- Infrastructure as Code
- Blue-green and canary deployments
- Policy-compliant deployments

## When NOT To Use

- One-time manual deployments
- Very simple single-service deployments
- Development experimentation

## Core Concepts

### Skaffold
Local development and CI/CD tool for Kubernetes applications.

### Cloud Deploy
Managed progressive delivery service for GKE.

### Config Sync
GitOps-based configuration synchronization for GKE.

### Config Controller
Kubernetes-style deployment using Config Connector.

### Progressive Delivery
Gradual rollout with canary/blue-green strategies.

## Real Production Example

**GitOps Deployment with Config Sync**

```hcl
# Terraform configuration for GitOps deployment infrastructure

# GKE Cluster
resource "google_container_cluster" "primary" {
  name               = "primary-cluster"
  location           = "us-central1"
  initial_node_count = 3
  
  network    = google_compute_network.vpc.id
  subnetwork = google_compute_subnetwork.primary.id
  
  addons_config {
    config_connector_config {
      enabled = true
    }
  }
  
  workload_identity_config {
    workload_pool = "${var.project}.svc.id.goog"
  }
  
  labels = {
    environment = "production"
  }
}

# Config Sync for GitOps
resource "google_gke_hub_feature" "config_sync" {
  name = "configsync"
  spec {
    configsync {
      enabled = true
    }
  }
}

resource "google_gke_hub_feature_membership" "config_sync_member" {
  location = "global"
  feature  = google_gke_hub_feature.config_sync.name
  configsync {
    git {
      remote   = "https://github.com/org/config-repo"
      branch   = "main"
      dir      = "deployments/prod"
      secret_type = "git"
    }
    sync_branch = "main"
    policy_dir  = "deployments/prod"
    source_format = "hierarchical"
  }
}

# Cloud Deploy for progressive delivery
resource "google_clouddeploy_delivery_pipeline" "production" {
  name        = "production-pipeline"
  location    = "us-central1"
  description = "Production deployment pipeline"
  
  serial_pipeline {
    stages {
      target_id = "staging"
      profiles = ["staging"]
    }
    stages {
      target_id = "production"
      profiles = ["production"]
    }
  }
  
  annotations = {
    environment = "production"
  }
}

# Staging target
resource "google_clouddeploy_target" "staging" {
  location                     = "us-central1"
  name                         = "staging"
  deploy_parameters = {
    "SERVICE_TYPE" = "staging"
    "REPLICAS"     = "2"
    "CLOUD_RUN_MEMORY" = "512Mi"
    "CLOUD_RUN_CPU" = "1"
  }
  
  gke {
    cluster = "projects/${var.project}/locations/us-central1/clusters/${google_container_cluster.primary.name}"
  }
}

# Production target
resource "google_clouddeploy_target" "production" {
  location                     = "us-central1"
  name                         = "production"
  require_approval             = true
  
  deploy_parameters = {
    "SERVICE_TYPE" = "production"
    "REPLICAS"     = "5"
    "CLOUD_RUN_MEMORY" = "1Gi"
    "CLOUD_RUN_CPU" = "2"
  }
  
  gke {
    cluster = "projects/${var.project}/locations/us-central1/clusters/${google_container_cluster.primary.name}"
  }
}

# Skaffold configuration (skaffold.yaml in repo)
# This would be in your source repository
```

### Skaffold Configuration (skaffold.yaml)

```yaml
# Skaffold configuration for multi-service deployment
apiVersion: skaffold/v4beta7
kind: Config

metadata:
  name: production-app

profiles:
  # Development profile
  - name: dev
    build:
      artifacts:
        - image: us-central1-docker.pkg.dev/my-project/app/backend
          docker:
            dockerfile: Dockerfile
        - image: us-central1-docker.pkg.dev/my-project/app/frontend
          docker:
            dockerfile: frontend/Dockerfile
    deploy:
      kubectl:
        manifests:
          - k8s/dev/*
    portForward:
      - resourceType: deployment
        resourceName: backend
        port: 8080
        localPort: 8080

  # Staging profile
  - name: staging
    build:
      artifacts:
        - image: us-central1-docker.pkg.dev/my-project/app/backend
          docker:
            dockerfile: Dockerfile
        - image: us-central1-docker.pkg.dev/my-project/app/frontend
          docker:
            dockerfile: frontend/Dockerfile
    deploy:
      kubectl:
        manifests:
          - k8s/staging/*
      helm:
        releases:
          - name: app
            chartPath: charts/app
            setValues:
              image.repository: us-central1-docker.pkg.dev/my-project/app/backend
              replicaCount: "2"
              environment: staging

  # Production profile
  - name: production
    build:
      artifacts:
        - image: us-central1-docker.pkg.dev/my-project/app/backend
          docker:
            dockerfile: Dockerfile
            cache:
              path: /cache
        - image: us-central1-docker.pkg.dev/my-project/app/frontend
          docker:
            dockerfile: frontend/Dockerfile
    deploy:
      kubectl:
        manifests:
          - k8s/production/*
      helm:
        releases:
          - name: app
            chartPath: charts/app
            setValues:
              image.repository: us-central1-docker.pkg.dev/my-project/app/backend
              replicaCount: "5"
              environment: production
              autoscaling:
                enabled: true
                minReplicas: 3
                maxReplicas: 10

# Default profile (development)
build:
  artifacts:
    - image: us-central1-docker.pkg.dev/my-project/app/backend
      docker:
        dockerfile: Dockerfile
```

### Kubernetes Manifests (k8s/production/)

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
  namespace: production
  labels:
    app: backend
    environment: production
spec:
  replicas: 5
  selector:
    matchLabels:
      app: backend
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: backend
        environment: production
    spec:
      serviceAccountName: backend
      containers:
        - name: backend
          image: us-central1-docker.pkg.dev/my-project/app/backend:latest
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 500m
              memory: 512Mi
            limits:
              cpu: 2000m
              memory: 1Gi
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
          env:
            - name: ENVIRONMENT
              value: production
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: backend-secrets
                  key: database-url
```

## Interview Questions

**Q: What is GitOps and how does Config Sync implement it?**
A: GitOps uses Git as the source of truth for infrastructure and configuration. Config Sync continuously monitors a Git repository and syncs desired state to GKE. Any drift from Git-defined state is automatically corrected.

**Q: How does Cloud Deploy handle progressive delivery?**
A: Cloud Deploy supports canary and rolling update strategies for GKE deployments. It manages the full release lifecycle including promotion between targets and rollback capabilities.

**Q: What is the difference between Cloud Deploy and manual kubectl?**
A: Cloud Deploy provides managed progressive delivery, approval gates, audit logging, and integration with Cloud Monitoring. kubectl is manual with no built-in delivery guarantees.

**Q: How do you handle secrets in GitOps?**
A: Use Secret Manager with external secrets operator, or use Sealed Secrets for encrypted-at-rest secrets in Git. Never commit raw secrets to Git.

**Q: What is Skaffold's role in the deployment pipeline?**
A: Skaffold automates the build-deploy-test cycle for Kubernetes applications during development. It supports local development, CI/CD integration, and multi-environment deployment.
