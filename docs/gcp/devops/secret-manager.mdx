# Secret Manager

## Overview

Google Cloud Secret Manager provides secure storage and management of sensitive data like API keys, passwords, certificates, and other secrets. It offers versioning, access control, automatic rotation support, and integration with GCP services. Secrets are encrypted at rest using Google-managed or customer-managed keys.

## When To Use

- Storing API keys and credentials
- Managing SSL/TLS certificates
- Database connection strings
- Encryption keys for custom applications
- Any sensitive configuration data
- Secrets requiring rotation

## When NOT To Use

- Non-sensitive configuration (use Cloud Configuration)
- Binary large objects (use Cloud Storage)
- Secrets requiring sub-second access (consider memory-only caching)
- Public certificates without private keys

```
│  Replication:                                              │

│  ├── Regional                                              │
│  └── Custom (user-managed)                                 │
└─────────────────────────────────────────────────────────────┘
```

## Core Concepts

### Secret
Logical container for secret versions with metadata and labels.

### Version
Point-in-time snapshot of secret data. Immutable after creation.

### Replication Policy
Defines where secret data is stored geographically.

### Secret Access
Rate-limited API access with audit logging.

## CLI Examples

```bash
# Create secret
gcloud secrets create my-secret --data-file=- <<< "my-secret-value"

# Create secret from file
gcloud secrets create api-key --data-file=./api.key

# List secrets
gcloud secrets list

# Describe secret
gcloud secrets describe my-secret

# Add version
gcloud secrets versions add my-secret --data-file=- <<< "new-value"

# Access secret value
gcloud secrets versions access latest --secret=my-secret
gcloud secrets versions access 1 --secret=my-secret

# List versions
gcloud secrets versions list my-secret

# Delete version
gcloud secrets versions destroy 1 --secret=my-secret

# Delete secret (all versions)
gcloud secrets delete my-secret --quiet

# Enable/disable version
gcloud secrets versions disable 1 --secret=my-secret
gcloud secrets versions enable 1 --secret=my-secret

# Update secret labels
gcloud secrets update my-secret --update-labels=env=prod,team=platform

# Create regional secret
gcloud secrets create my-regional-secret \
  --locations=us-central1,us-east1 \
  --data-file=- <<< "value"

# Replication policy update
gcloud secrets update my-secret \
  --replication-policy=automatic

# Add IAM policy
gcloud secrets add-iam-policy-binding my-secret \
  --member=serviceAccount:sa@project.iam.gserviceaccount.com \
  --role=roles/secretmanager.secretAccessor

# Get IAM policy
gcloud secrets get-iam-policy my-secret

# Backup secret
gcloud secrets versions list my-secret
gcloud secrets versions access 1 --secret=my-secret > backup.txt

# Rotate secret (add new version)
gcloud secrets versions add my-secret --data-file=- <<< "rotated-value"
```

## Security

### Encryption
- At rest: Google-managed encryption (AES-256)
- CMEK: Not supported (uses Google-managed keys only)
- In transit: TLS 1.2+

### IAM Best Practices
```bash
# Principle of least privilege
gcloud secrets add-iam-policy-binding my-secret \
  --member=serviceAccount:app@project.iam.gserviceaccount.com \
  --role=roles/secretmanager.secretAccessor  # Read-only

# For rotation automation
gcloud secrets add-iam-policy-binding my-secret \
  --member=serviceAccount:rotation@project.iam.gserviceaccount.com \
  --role=roles/secretmanager.admin
```

## Interview Questions

**Q: How does Secret Manager integrate with Cloud Run?**
A: Cloud Run uses Workload Identity to access secrets. The runtime service account needs `roles/secretmanager.secretAccessor`. Secrets are accessed via the Secret Manager API or automatically injected into environment variables via Kubernetes-style references.

**Q: What is the difference between disabling and destroying a secret version?**
A: Disabling a version prevents access but retains the data for rollback. Destroying permanently removes the data and cannot be undone. Use disable during rotation, destroy only when certain the version is no longer needed.

**Q: How do you implement secret rotation?**
A: Create a new version with `gcloud secrets versions add`, update the application to read the new version, verify the application works, then disable the old version. For automatic rotation, use Cloud Scheduler with Cloud Functions or a rotation service.

**Q: What are the replication options in Secret Manager?**
A: Automatic replication lets Google choose optimal locations. Regional replication stores secrets in a specific region. User-managed replication allows specifying exact replica locations for compliance requirements.

**Q: How does Secret Manager pricing work?**
A: Secrets cost $0.06/month. First 6 versions are free; subsequent versions cost $0.03/month each. Access operations cost $0.03 per 10,000 accesses. Disable old versions to avoid version storage charges.

**Q: Can you use CMEK with Secret Manager?**
A: No, Secret Manager does not support customer-managed encryption keys. All secrets are encrypted with Google-managed keys. If you need CMEK, you must encrypt the secret value yourself before storing it.
