# Workflows

## Overview

Google Cloud Workflows is a fully managed orchestration service for automating GCP and HTTP-based API workflows. It provides sequencing of services, support for human approval steps, error handling with retries, and built-in retry logic. Workflows scale automatically and can connect to any HTTP-based API, including GCP services, external APIs, and other workflows.

## When To Use

- Orchestrating multi-step business processes
- Connecting microservices into cohesive flows
- Automating DevOps pipelines
- Implementing human approval workflows
- Complex data processing pipelines
- Scheduled job orchestration
- Long-running background tasks

## When NOT To Use

- Simple single-step invocations (use Cloud Functions)
- Real-time request-response (use direct API calls)
- High-frequency micro-orchestrations
- Stream processing (use Dataflow)

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                      Workflows                               │
├─────────────────────────────────────────────────────────────┤
│  Components:                                                │
│  ├── Workflow Definition (YAML/JSON)                       │
│  ├── Execution (run instance)                               │
│  ├── Connector (pre-built integrations)                    │
│  └── Callback (human approval endpoint)                    │
├─────────────────────────────────────────────────────────────┤
│  Steps:                                                     │
│  ├── Call (HTTP request)                                   │
│  ├── Assign (variable assignment)                          │
│  ├── Switch (conditional branching)                        │
│  ├── ForEach (iteration)                                   │
│  ├── Try/Catch (error handling)                            │
│  └── Return (exit workflow)                                │
├─────────────────────────────────────────────────────────────┤
│  Features:                                                 │
│  ├── Built-in OAuth 2.0 support                            │
│  ├── Service connectors for GCP                           │
│  ├── Subworkflows (call another workflow)                  │
│  └── Parallel execution                                    │
└─────────────────────────────────────────────────────────────┘
```

## Core Concepts

### Workflow
A definition of steps and logic in YAML or JSON format.

### Execution
An instance of a running workflow.

### Step
Individual action within a workflow (call, assign, switch, etc.).

### Connector
Pre-built integration for GCP services (BigQuery, Cloud Storage, etc.).

### Callback
Endpoint for pausing workflow and waiting for external input.

### Transition
Connection between steps defining execution flow.

### Workflow Definition (order_processing.yaml)

```yaml
# Order Processing Workflow
- init:
    assign:
      - project_id: "${sys.get_env("GOOGLE_CLOUD_PROJECT_NUMBER")}"
      - region: "us-central1"
      - notification_email: "orders@company.com"

- fetch_pending_orders:
    call: googleapis.bigquery.v2.jobs.query
    args:
      projectId: ${project_id}
      query: |
        SELECT order_id, customer_id, total_amount, created_at
        FROM orders.pending
        WHERE status = 'pending'
        ORDER BY created_at
        LIMIT 100
      location: ${region}
      useLegacySql: false
    result: orders

- process_order:
    for-each: order in ${orders.rows}:
      steps:
        - validate_inventory:
            call: httpget
            args:
              url: ${"https://inventory.api.example.com/check/" + order.order_id}
            result: inventory_check

        - check_inventory:
            switch:
              - condition: ${inventory_check.available == true}
                steps:
                  - reserve_inventory:
                      call: httppost
                      args:
                        url: ${"https://inventory.api.example.com/reserve/" + order.order_id}
                        body:
                          quantity: ${order.quantity}
                        headers:
                          Authorization: ${"Bearer " + sys.get_env("INVENTORY_TOKEN")}
                      result: reservation

                  - process_payment:
                      try:
                        call: httppost
                        args:
                          url: ${"https://payment.api.example.com/charge/" + order.customer_id}
                          body:
                            amount: ${order.total_amount}
                            order_id: ${order.order_id}
                          headers:
                            Authorization: ${"Bearer " + sys.get_env("PAYMENT_TOKEN")}
                        result: payment_result
                      except:
                        steps:
                          - payment_failed:
                              call: httppost
                              args:
                                url: ${"https://notification.api.example.com/alert"}
                                body:
                                  type: "PAYMENT_FAILED"
                                  order_id: ${order.order_id}
                                  error: ${context.error.message}
                          
                          - mark_order_failed:
                              call: googleapis.bigquery.v2.jobs.query
                              args:
                                projectId: ${project_id}
                                query: ${"UPDATE orders.pending SET status = 'failed' WHERE order_id = '" + order.order_id + "'"}
                                location: ${region}
                                useLegacySql: false
                              
                              raise: ${"Payment failed: " + context.error.message}

                  - update_inventory:
                      call: googleapis.bigquery.v2.jobs.query
                      args:
                        projectId: ${project_id}
                        query: ${"UPDATE orders.pending SET status = 'confirmed', updated_at = CURRENT_TIMESTAMP() WHERE order_id = '" + order.order_id + "'"}
                        location: ${region}
                        useLegacySql: false

                  - send_confirmation:
                      call: httppost
                      args:
                        url: ${"https://notification.api.example.com/email/" + order.customer_id}
                        body:
                          template: "order_confirmation"
                          order_id: ${order.order_id}
                          total: ${order.total_amount}
                        headers:
                          Authorization: ${"Bearer " + sys.get_env("NOTIFICATION_TOKEN")}
                
              - condition: true  # Default: inventory not available
                steps:
                  - handle_out_of_stock:
                      call: googleapis.bigquery.v2.jobs.query
                      args:
                        projectId: ${project_id}
                        query: ${"UPDATE orders.pending SET status = 'out_of_stock' WHERE order_id = '" + order.order_id + "'"}
                        location: ${region}
                        useLegacySql: false
                      
                      raise: ${"Item out of stock for order: " + order.order_id}

- send_summary:
    call: httppost
    args:
      url: ${"https://notification.api.example.com/webhook"}
      body:
        workflow: "order_processing"
        processed: ${len(orders.rows)}
        timestamp: ${sys.now()}

- return:
    return:
      status: "completed"
      processed_count: ${len(orders.rows)}
      timestamp: ${sys.now()}
```

## CLI Examples

```bash
# Create workflow
gcloud workflows create my-workflow \
  --location=us-central1 \
  --description="My workflow"

# Update workflow with YAML file
gcloud workflows update my-workflow \
  --location=us-central1 \
  --source=workflow.yaml

  --location=us-central1 \

  --location=us-central1 \
```

## Terraform Example

```hcl
terraform {
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
  }
}

# Workflow with YAML definition
resource "google_workflows_workflow" "example" {
  name        = "example-workflow"
  region      = "us-central1"
  description = "Example workflow"
  
  service_account = google_service_account.workflow_sa.email
  
  workflow_contents = <<-EOF
  # Main routine
  - init:
      assign:
        - version: "1.0"

  - getTime:
      call: httpget
      args:
        url: https://worldtimeapi.org/api/timezone/Etc/UTC
      result: timeResponse

  - parseTime:
      call: sys.jsfn.parse
      args:
        script: "JSON.parse(content).datetime"
        content: ${timeResponse.body}
      result: parsedTime

  - return:
      return:
        version: ${version}
        currentTime: ${parsedTime}

  # Callback example
  - awaitApproval:
      call: httppost
      args:
        url: ${sys.get_env("CALLBACK_URL")}
        body:
          execution_id: ${sys.get_var("sys.execution.id")}
          status: "pending_approval"
      result: callback_response

  - checkApproval:
      switch:
        - condition: ${callback_response.body.approved == true}
          steps:
            - proceedWithApproval
        - condition: true
          steps:
            - handleRejection

  - proceedWithApproval:
      return:
        status: "approved"

  - handleRejection:
      raise: "Approval denied"
  EOF
  
  labels = {
    environment = "production"
  }
}

# Workflow with external definition file
data "local_file" "workflow_definition" {
  filename = "path/to/workflow.yaml"

resource "google_workflows_workflow" "from_file" {
```

## Interview Questions

**Q: How does Workflows handle errors and retries?**
A: Workflows supports try/catch blocks for error handling. Use `try` and `except` steps to define error handling logic. Retries can be configured with exponential backoff using the `retry` section within a step. Each step can have its own retry policy.

**Q: What is the difference between steps and subworkflows?**
A: Steps are individual actions within a workflow (call, assign, switch, etc.). Subworkflows are separate workflow definitions that can be called from within a parent workflow, enabling code reuse and modularity. Use `call: workflow_name` to invoke a subworkflow.

**Q: How do you implement human approval in Workflows?**
A: Use the callback pattern: pause the workflow with an HTTP call to an external approval service, which returns a callback URL. The workflow waits at that step until the callback is invoked with approval/rejection. Store the execution ID to resume later.

**Q: How does Workflows scale?**
A: Workflows is fully managed and scales automatically. It handles concurrent executions without configuration. The service manages infrastructure, scaling, and availability transparently.

**Q: What connectors are available in Workflows?**
A: Workflows provides connectors for many GCP services: BigQuery, Cloud Spanner, Cloud SQL, Cloud Storage, Pub/Sub, Firestore, Secret Manager, and more. These connectors simplify authentication and provide pre-built request/response handling.

## Cheat Sheet

```yaml
# Basic workflow structure
- init:
    assign:
      - var: value

- stepName:
    call: service.method
    args:
      param: value
    result: resultVar

- conditional:
    switch:
      - condition: ${var == "value"}
        steps:
          - doThis
      - condition: true
        steps:
          - doThat

- iterate:
    for-each: item in ${array}:
      steps:
        - process: ${item}

- errorHandle:
    try:
      steps:
        - riskyStep
    except:
      steps:
        - handleError

- return:
    return:
      result: ${resultVar}
```
