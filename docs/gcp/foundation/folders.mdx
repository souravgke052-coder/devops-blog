# Folders

## Overview

Folders are optional organizational containers in GCP that sit between the Organization node and Projects in the resource hierarchy. They provide a way to group projects and apply policies at a collective level. Folders enable logical grouping by team, product, environment, or any business structure that doesn't map directly to individual projects.

Folders support nesting up to 10 levels deep, allowing complex organizational structures. A project can belong to exactly one folder (or directly to the organization), and all resources within that project inherit folder-level IAM policies and organization policies. This inheritance makes folders powerful for applying consistent governance across many projects.

Common use cases include grouping projects by environment (production, staging, development), by team (platform, data, frontend), or by application (ecommerce, crm, analytics). The folder structure should mirror your organization's reporting structure and governance needs.

## When To Use

- **Environment separation**: Group projects by environment (prod/staging/dev)
- **Team organization**: Organize projects by team or department
- **Policy inheritance**: Apply consistent IAM or org policies to groups of projects
- **Billing aggregation**: Track costs by folder using billing exports
- **Access control**: Grant folder-level permissions to team leads
- **Multi-tenant isolation**: Separate tenants in different folders
- **Application grouping**: Group related microservices into logical applications

## When NOT To Use

- **Only one or two projects**: Folders add complexity without benefit for small deployments
- **Flat organization**: If all projects have the same governance needs, organization-level policies suffice
- **Over-nesting**: Deep folder hierarchies become hard to manage and audit
- **Replacing projects**: Folders don't replace projects—they group them

## Architecture

**Folder Hierarchy:**
```
Organization (root)
├── Folder: Environments
│   ├── Folder: Production
│   │   ├── Project: app-prod
│   │   └── Project: api-prod
│   ├── Folder: Staging
│   │   └── Project: app-staging
│   └── Folder: Development
│       └── Project: app-dev
└── Folder: Teams
    ├── Folder: Platform
    │   └── Project: platform-services
    └── Folder: Data
        └── Project: data-warehouse
```

**Folder Properties:**
- **Folder ID**: System-assigned unique identifier (`folders/123456789012`)
- **Display Name**: Human-readable name (can be changed)
- **Parent**: Organization or parent folder
- **IAM Policy**: Who can access this folder and its children
- **Org Policies**: Constraints applied to this folder and children

**Inheritance Behavior:**
- Policies cascade down (parent to child)
- Child policies can be more restrictive but not more permissive
- Folder-level policies apply to all descendant projects and resources

## Core Concepts

### Folder Operations

```bash
# Create folder
gcloud resource-manager folders create "folders/production" \
    --organization=123456789012 \
    --display-name="Production"

# Create nested folder
gcloud resource-manager folders create "folders/production/apps" \
    --organization=123456789012 \
    --display-name="Production Applications"

# List folders
gcloud resource-manager folders list --organization=123456789012

# Describe folder
gcloud resource-manager folders describe folders/123456789012

# Update folder
gcloud resource-manager folders update folders/123456789012 \
    --display-name="Updated Name"

# Move folder
gcloud resource-manager folders move folders/CHILD_ID \
    --folder=folders/PARENT_ID

# Delete folder (must be empty)
gcloud resource-manager folders delete folders/123456789012
```

### Moving Projects Between Folders

```bash
# Move project to folder
gcloud projects move my-project --folder=folders/123456789012

# Move project to organization root
gcloud projects move my-project --organization=123456789012

# Move project out of any folder
gcloud projects move my-project --folder=""
```

### Folder IAM

```bash
# Get folder IAM policy
gcloud resource-manager folders get-iam-policy folders/123456789012

# Set folder IAM policy
gcloud resource-manager folders set-iam-policy folders/123456789012 policy.json

# Add IAM binding
gcloud resource-manager folders add-iam-policy-binding folders/123456789012 \
    --member="group:team@company.com" \
    --role="roles/viewer"

# Remove IAM binding
gcloud resource-manager folders remove-iam-policy-binding folders/123456789012 \
    --member="group:team@company.com" \
    --role="roles/viewer"
```

### Folder Org Policies

```bash
# Set org policy at folder level
gcloud org-policies set-policy policy.yaml \
    --folder=folders/123456789012

# Reset org policy to inherited default
gcloud org-policies reset compute.disableSerialPortAccess \
    --folder=folders/123456789012
```

### Quotas and Limits

| Resource | Limit |
|----------|-------|
| Folders per organization | 1,000 |
| Folder nesting depth | 10 levels |
| Projects per folder | Unlimited |
| IAM bindings per folder | 1,500 |

### Key Gotchas

- **Folders can only be created by users with `resourcemanager.folderAdmin`** role
- **Deleting a folder requires it to be empty**—move or delete all projects first
- **Moving folders is slow**—changes can take 30+ minutes to propagate
- **Folder IAM inherits but doesn't replace project IAM**—both apply (union of permissions)
- **Org policies at folder level can only restrict further**, not expand Organization-level permissions
- **Folder display names can be duplicated** but folder IDs are unique
- **Moving projects between folders can affect IAM** if using conditional bindings with folder references

## Real Production Example

Production folder structure setup:

```bash
# 1. Create environment folders
gcloud resource-manager folders create "folders/environments" \
    --organization=123456789012 \
    --display-name="Environments"

gcloud resource-manager folders create "folders/environments/production" \
    --organization=123456789012 \
    --display-name="Production"

gcloud resource-manager folders create "folders/environments/staging" \
    --organization=123456789012 \
    --display-name="Staging"

gcloud resource-manager folders create "folders/environments/development" \
    --organization=123456789012 \
    --display-name="Development"

# 2. Create team folders
gcloud resource-manager folders create "folders/teams" \
    --organization=123456789012 \
    --display-name="Teams"

gcloud resource-manager folders create "folders/teams/platform" \
    --organization=123456789012 \
    --display-name="Platform Team"

gcloud resource-manager folders create "folders/teams/data" \
    --organization=123456789012 \
    --display-name="Data Team"

# 3. Apply folder-level IAM
# Grant platform team admin on production folders
gcloud resource-manager folders add-iam-policy-binding folders/PROD_ID \
    --member="group:platform-team@company.com" \
    --role="roles/resourcemanager.folderAdmin"

# Grant dev team viewer on all environments
gcloud resource-manager folders add-iam-policy-binding folders/ENVIRONMENTS_ID \
    --member="group:developers@company.com" \
    --role="roles/viewer"

# 4. Apply org policies at folder level
# More restrictive policy on production
cat > prod-restrictions.yaml << 'EOF'
name: folders/PROD_ID/policies/compute.vmExternalIp
spec:
  rules:
  - enforce: false
EOF
gcloud org-policies set-policy prod-restrictions.yaml \
    --folder=folders/PROD_ID

# 5. Create projects under folders
gcloud projects create my-app-prod \
    --folder=folders/PROD_ID \
    --name="My App Production"

gcloud projects create my-app-staging \
    --folder=folders/STAGING_ID \
    --name="My App Staging"

# 6. Set billing at folder level (via organization)
# Projects inherit billing from billing account
```

## Monitoring

**Folder-Level Monitoring:**

1. **Cost Monitoring by Folder**
   ```sql
   -- Cost breakdown by folder
   SELECT
     folder.id as folder_id,
     folder.display_name,
     SUM(cost) as total_cost
   FROM billing_export.gcp_billing_export_v1
   WHERE folder.id IS NOT NULL
   GROUP BY folder.id, folder.display_name
   ORDER BY total_cost DESC
   ```

2. **Resource Inventory by Folder**
   ```bash
   # List all projects in a folder
   gcloud projects list --filter="parent.id=FOLDER_ID"
   
   # List resources across all projects in folder
   gcloud compute instances list \
       --filter="project:my-proj-1 OR project:my-proj-2"
   ```

3. **IAM Audit at Folder Level**
   ```bash
   # Log IAM changes at folder level
   gcloud logging read 'resource.type="folder" AND protoPayload.methodName="SetIAMPolicy"' \
       --folder=FOLDER_ID
   
   # Alert on folder admin role grants
   gcloud logging read 'resource.type="folder" AND protoPayload.authorization.grantedRole:"admin"'
   ```

**Recommended Alerts:**
- New projects created outside approved folders
- Folder policy changes
- Excessive IAM grants at folder level

## Security

**Folder Security Best Practices:**

1. **Use folders for least-privilege access**
   ```bash
   # Grant folder admin only to team leads
   gcloud resource-manager folders add-iam-policy-binding folders/TEAM_FOLDER \
       --member="group:team-leads@company.com" \
       --role="roles/resourcemanager.folderAdmin"
   ```

2. **Apply restrictive org policies at folder level**
   ```bash
   # Strict production policies
   gcloud org-policies set-policy prod-strict.yaml --folder=folders/PROD_ID
   ```

3. **Use folder-level service accounts for cross-project access**
   ```bash
   # Create service account in folder-level project
   gcloud iam service-accounts create folder-sa \
       --display-name="Folder Service Account"
   
   # Grant cross-project access from folder project
   gcloud projects add-iam-policy-binding target-project \
       --member="serviceAccount:folder-sa@folders-project.iam.gserviceaccount.com" \
       --role="roles/bigquery.dataViewer"
   ```

4. **Enable Data Access audit logs at folder level**
   ```bash
   # Enable audit logs for folder
   gcloud logging write audit-log 'IAM policy change in folder' \
       --folder=FOLDER_ID
   ```

## Troubleshooting

**Common Issues:**

1. **Can't create folder**
   ```bash
   # Check permissions (need resourcemanager.folderAdmin)
   gcloud organizations get-iam-policy 123456789012
   
   # Check organization exists
   gcloud organizations list
   
   # Check quota (max 1000 folders per org)
   ```

2. **Can't delete folder**
   ```bash
   # Folder must be empty
   gcloud projects list --filter="parent.id=FOLDER_ID"
   
   # Move projects out first
   gcloud projects move PROJECT --folder=NEW_FOLDER_ID
   
   # Then delete empty folder
   gcloud resource-manager folders delete folders/FOLDER_ID
   ```

3. **Folder move failing**
   ```bash
   # Check folder depth limit (10 levels)
   # Check permissions (need folder admin on both source and target)
   # Verify target is not a child of source (can't move folder under its child)
   ```

4. **Org policies not applying to folder children**
   ```bash
   # Check policy is set correctly
   gcloud org-policies describe CONSTRAINT --folder=FOLDER_ID
   
   # Check inheritance (folder can only restrict, not expand)
   # Some policies only affect new resources
   ```

5. **IAM not working on folder**
   ```bash
   # Check effective policy
   gcloud resource-manager folders get-iam-policy folders/FOLDER_ID
   
   # IAM can take up to 7 minutes to propagate
   # Verify user is not in excluded group
   ```
