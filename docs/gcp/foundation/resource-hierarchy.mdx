# Resource Hierarchy

## Overview

GCP's Resource Hierarchy is the logical structure that organizes all cloud resources from the Organization node down to individual resources. Understanding this hierarchy is fundamental to GCP architecture because it determines how IAM policies, organization policies, and billing are inherited and applied. The hierarchy follows a strict parent-child relationship where resources at higher levels can impose policies on resources below them, but not vice versa.

The hierarchy has four main levels: Organization, Folders, Projects, and Resources. Each GCP resource must belong to exactly one parent at the level immediately above it—resources belong to projects, projects belong to folders (or directly to the organization), and folders can belong to the organization or other folders. This structure mirrors how organizations are typically structured and enables effective governance at scale.

The Resource Hierarchy is the foundation for Google Cloud's "context-aware" access control model. Rather than managing permissions on every individual resource, you can apply policies at higher levels and have them cascade down. This reduces administrative overhead while ensuring consistent security postures across your entire GCP environment.

## When To Use

- **Planning GCP organization structure**: Before creating projects and resources
- **Designing IAM strategy**: Understanding how policies inherit
- **Applying organization policies**: Knowing where to set constraints
- **Planning billing structure**: Understanding cost allocation through the hierarchy
- **Auditing access**: Understanding who has access to what and where
- **Designing Shared VPC**: Planning host and service project relationships

## When NOT To Use

- **Single-project simple deployments**: The hierarchy exists but you don't need to leverage it fully
- **Personal projects**: Personal projects don't need organizational hierarchy
- **Learning/experimentation**: Focus on individual service features first

## Architecture

**Hierarchy Diagram:**
```
Google Cloud Organization (organization)
├── Organization Policies (org-wide constraints)
├── Folders (logical grouping containers)
│   ├── Production Folder
│   │   ├── Project: App Production
│   │   │   ├── GKE Cluster (resource)
│   │   │   ├── Cloud SQL (resource)
│   │   │   └── Cloud Storage Bucket (resource)
│   │   └── Project: API Production
│   │       └── Compute Instances (resource)
│   ├── Staging Folder
│   │   └── Project: App Staging
│   └── Development Folder
│       └── Project: App Dev
└── Direct Projects (optional, not under folders)
```

**Hierarchy Levels:**

| Level | Purpose | Examples |
|-------|---------|----------|
| Organization | Root entity representing company | google.com org |
| Folders | Logical grouping by team/environment | Production, Teams |
| Projects | Billing/IAM boundary, resource container | app-prod, api-staging |
| Resources | Actual compute/storage/network resources | VMs, buckets, databases |

**Policy Inheritance Flow:**

```
Organization Policy: Deny serial ports everywhere
        ↓ (inherits)
Folder Policy: Allow serial ports in Development
        ↓ (inherits to children)
Project: Inherits from Folder (allows serial ports)
        ↓ (applies to)
Resources: Can use serial ports
```

**Key Principles:**
1. **Top-down inheritance**: Policies flow from parent to child
2. **Child restriction only**: Lower levels can only restrict further, not expand
3. **Union of permissions**: Effective permissions are the union of all granting policies
4. **Explicit deny wins**: Deny policies override allow policies
5. **Least privilege**: Grant minimum necessary permissions at appropriate level

### Quotas in Hierarchy

| Level | Quota Scope | Examples |
|-------|-------------|----------|
| Organization | Global, shared across org | Max folders, custom roles |
| Folder | Aggregated from children | Folder admin access |
| Project | Per-project limits | CPUs, IPs, instances |
| Resource | Per-resource limits | Instance sizes, disk sizes |

### Key Gotchas

- **Organization node requires Google Workspace or Cloud Identity**—not available for personal Gmail accounts
- **IAM policies are a union, not intersection**—if any policy grants access, access is allowed (unless denied)
- **Org policies cascade down** but can only restrict, not expand permissions
- **Deleting a parent resource requires deleting/moving all children first**
- **Project can't be moved to folder in different organization**—organizations are top-level
- **Folder moves affect ALL children**—can take 30+ minutes for large hierarchies
- **Default VPC exists at project level** but subnets are regional

## Real Production Example

Complete resource hierarchy setup:

```bash
# 1. Create organization structure
# Organization already exists; get the ID
ORG_ID=$(gcloud organizations list --format="value(name)" | head -1)
echo "Organization ID: $ORG_ID"

# 2. Create folder hierarchy
# Environment folders
gcloud resource-manager folders create "folders/environments" \
    --organization=$ORG_ID --display-name="Environments"

ENV_ID=$(gcloud resource-manager folders list \
    --organization=$ORG_ID --format="value(name)" | grep environments | head -1)

gcloud resource-manager folders create "folders/environments/production" \
    --organization=$ORG_ID --display-name="Production"

gcloud resource-manager folders create "folders/environments/staging" \
    --organization=$ORG_ID --display-name="Staging"

gcloud resource-manager folders create "folders/environments/development" \
    --organization=$ORG_ID --display-name="Development"

# Team folders
gcloud resource-manager folders create "folders/teams" \
    --organization=$ORG_ID --display-name="Teams"

TEAMS_ID=$(gcloud resource-manager folders list \
    --organization=$ORG_ID --format="value(name)" | grep teams | head -1)

gcloud resource-manager folders create "folders/teams/platform" \
    --organization=$ORG_ID --display-name="Platform Team"

gcloud resource-manager folders create "folders/teams/data" \
    --organization=$ORG_ID --display-name="Data Team"

# 3. Apply organization-level policies
# Disable serial port access org-wide
cat > disable-serial-port.yaml << 'EOF'
name: organizations/$ORG_ID/policies/compute.disableSerialPortAccess
spec:
  rules:
  - enforce: false
EOF
gcloud org-policies set-policy disable-serial-port.yaml \
    --organization=$ORG_ID

# Require OS Login
cat > require-os-login.yaml << 'EOF'
name: organizations/$ORG_ID/policies/compute.requireOsLogin
spec:
  rules:
  - enforce: true
EOF
gcloud org-policies set-policy require-os-login.yaml \
    --organization=$ORG_ID

# 4. Set folder-level IAM
PLATFORM_ID=$(gcloud resource-manager folders list \
    --organization=$ORG_ID --format="value(name)" | grep platform | head -1)

gcloud resource-manager folders add-iam-policy-binding $PLATFORM_ID \
    --member="group:platform@company.com" \
    --role="roles/resourcemanager.folderAdmin"

# 5. Create projects in folders
PROD_ID=$(gcloud resource-manager folders list \
    --organization=$ORG_ID --format="value(name)" | grep "environments/production\"" | head -1)

gcloud projects create my-app-prod \
    --name="My App Production" \
    --folder=$PROD_ID

gcloud projects create my-api-prod \
    --name="API Production" \
    --folder=$PROD_ID

# 6. Apply project-level IAM
gcloud projects add-iam-policy-binding my-app-prod \
    --member="group:app-team@company.com" \
    --role="roles/editor"

gcloud projects add-iam-policy-binding my-app-prod \
    --member="serviceAccount:ci@my-project.iam.gserviceaccount.com" \
    --role="roles/container.developer"
```

**Hierarchy Monitoring:**

1. **Track hierarchy changes**
   ```bash
   # Log all hierarchy changes
   gcloud logging read 'protoPayload.methodName=~"CreateFolder\|DeleteFolder\|MoveFolder\|CreateProject\|MoveProject"' \
       --organization=ORG_ID
   
   # Alert on hierarchy changes
   gcloud alpha monitoring policies create \
       --display-name="Hierarchy Change Alert" \
       --condition-display-name="Folder or Project change"
   ```

2. **Resource inventory**
   ```bash
   # List all resources in organization

       --scope=organizations/ORG_ID \

       --dest=gs://my-bucket/asset-inventory.json \

   ```

## Security

**Hierarchy Security Best Practices:**

1. **Use least privilege at each level**
   ```bash
   # Org level: security team only
   gcloud organizations add-iam-policy-binding ORG_ID \
       --member="group:security@company.com" \
       --role="roles/organization.admin"
   
   # Folder level: team leads
   gcloud resource-manager folders add-iam-policy-binding FOLDER_ID \
       --member="group:team-lead@company.com" \
       --role="roles/resourcemanager.folderAdmin"
   ```

2. **Apply restrictive org policies at top level**
   - Require OS Login for SSH
   - Disable serial port access
   - Restrict allowed VM disk images
   - Require secure boot for Shielded VMs

3. **Separate production from non-production at folder level**
   ```bash
   # More restrictive policies for production
   gcloud org-policies set-policy prod-restrictions.yaml --folder=folders/PROD_ID
   ```

4. **Use folders for sensitive data isolation**
   ```bash
   # Create compliance folder with strict policies
   gcloud org-policies set-policy compliance-policy.yaml --folder=folders/COMPLIANCE_ID
   ```

5. **Audit hierarchy access regularly**
   ```bash
   # Find all users with admin access
   gcloud asset search-all-iam-policies \
       --scope=organizations/ORG_ID \
       --query="policy:admin"
   ```

## Cheat Sheet

```
# Hierarchy Levels (top to bottom)
Organization (org-123456789012)
├── Folders (folders/123456789012)
│   └── Folders (nested, max 10 levels)
│       └── Projects (my-project)
│           └── Resources (instances, buckets, etc.)

# Commands by Level
gcloud organizations list/get-iam-policy/set-iam-policy
gcloud resource-manager folders create/list/describe/get-iam-policy
gcloud projects list/create/move/describe/get-iam-policy
gcloud org-policies list/set-policy/reset

# IAM Inheritance
- Organization → Folders → Projects → Resources
- Child policies can RESTRICT but not EXPAND parent policies
- All IAM grants are additive (union)

# Policy Evaluation
1. Check Organization policies
2. Check Folder policies (may add restrictions)
3. Check Project policies
4. Check Resource-level policies (may add restrictions)
5. Deny wins over allow
```
