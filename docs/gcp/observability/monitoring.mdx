# Cloud Monitoring

## Overview

Google Cloud Monitoring provides visibility into the performance, uptime, and health of your GCP infrastructure and applications. It offers metrics, dashboards, alerts, and alerting policies with integrations across all GCP services. The free tier includes 60 metrics and 6 dashboards.

## When To Use

- Infrastructure monitoring and alerting
- Application performance monitoring
- Service level objective (SLO) tracking
- Custom metrics collection
- Incident management
- Resource utilization analysis

## When NOT To Use

- Log aggregation (use Cloud Logging)
- Distributed tracing (use Cloud Trace)
- Real-time streaming analytics
- Deep application profiling

## Core Concepts

### Metric
Time-series data point with resource type, metric type, and value.

### Resource
GCP resource emitting metrics (VM, container, service, etc.).

### Monitoring Query Language (MQL)
SQL-like language for metrics manipulation.

### Alerting Policy
Defines conditions that trigger notifications.

### Notification Channel
Email, SMS, PagerDuty, Slack, webhook destinations.

### Uptime Check
Synthetic request testing from global locations.

  display_name = "Web Application Uptime"

    type = "uptime_url"
    labels = {
      host = "myapp.example.com"
    }
  }
  
  http_check {
    path         = "/health"
    port         = 443
    use_ssl      = true
    validate_ssl = true
  }
  
  period     = "60s"
  timeout    = "10s"
  
  checker_region = "USA"
  
  monitored_resources {
    type = "uptime_url"
    labels = {
      host = "myapp.example.com"
    }
  }
}

    type = "uptime_url"

  }

# Alerting policy for CPU utilization
resource "google_monitoring_alert_policy" "high_cpu" {
  display_name = "High CPU Alert"
  
  conditions {
    display_name = "CPU > 80%"
    
    condition_threshold {
      filter          = "resource.type=\"gce_instance\" AND metric.type=\"compute.googleapis.com/instance/cpu/utilization\""
      duration        = "300s"
      comparison      = "COMPARISON_GT"
      threshold_value = 0.8
      
      aggregations {
        alignment_period     = "60s"
        per_series_aligner   = "ALIGN_MEAN"
      }
    }
  }
  
  combiner = "OR"
  
  conditions {
    display_name = "GKE Container CPU > 80%"
    
    condition_threshold {
      filter          = "resource.type=\"k8s_container\" AND metric.type=\"kubernetes.io/container/cpu/core_usage_time\""
      duration        = "300s"
      comparison      = "COMPARISON_GT"
      threshold_value = 0.8
    }
  }
  
  notification_channels = [
    google_monitoring_notification_channel.email.name,
    google_monitoring_notification_channel.slack.name
  ]
  
  documentation {
    content = "CPU utilization is above 80% for 5 minutes. Investigate high CPU usage and consider scaling."
    mime_type = "text/markdown"
  }
}

# Alerting policy for high error rate
resource "google_monitoring_alert_policy" "high_error_rate" {
  display_name = "High Error Rate"
  
  conditions {
    display_name = "HTTP 5xx Error Rate > 1%"

## CLI Examples

```bash
# List metrics
gcloud monitoring metrics list

# Create alerting policy
gcloud monitoring policies create \
  --display-name="CPU Alert" \
  --condition-display-name="CPU > 70%" \
  --condition-filter='resource.type="gce_instance" AND metric.type="compute.googleapis.com/instance/cpu/utilization"' \
  --condition-threshold-value=0.7 \
  --notification-channels=CHANNEL_ID

# List alerting policies
gcloud monitoring policies list

# Describe policy
gcloud monitoring policies describe POLICY_ID

# Create uptime check
gcloud monitoring uptime-check-configs create \
  --display-name="Web Uptime" \
  --http-check-path="/health" \
  --monitored-resource-type=uptime_url \
  --monitored-resource-labels=host=myapp.example.com

# List uptime checks
gcloud monitoring uptime-check-configs list

# Create notification channel
gcloud monitoring channels create \
  --display-name="Email" \
  --type=email \
  --channel-labels=email_address=alerts@example.com

# List channels
gcloud monitoring channels list

# Query metrics
gcloud monitoring metrics read \
  "compute.googleapis.com/instance/cpu/utilization" \
  --filter='resource.instance_id="instance-id"'

# Create dashboard from JSON
gcloud monitoring dashboards create --config-from-file=dashboard.json

# List dashboards
gcloud monitoring dashboards list

# Create SLO
gcloud monitoring slos create \
  --service=my-service \
  --goal=0.99 \
  --latency-target=500ms
```

# Create custom metric via API or SDK
# Python example:
from google.cloud import monitoring_v3

client = monitoring_v3.MetricServiceClient()
project_name = f"projects/{project_id}"

descriptor = monitoring_v3.MetricDescriptor()
descriptor.type = f"custom.googleapis.com/my_metric"

## Interview Questions

**Q: How do you create an alerting policy with multiple conditions?**
A: Define multiple `conditions` blocks in the policy. Use `combiner = "OR"` or `combiner = "AND"` to determine if any or all conditions trigger the alert.

**Q: What is the difference between notification channels and alerting channels?**
A: In Terraform, `notification_channel` is the Terraform resource. Alerting policies reference notification channel IDs. Create channels separately, then reference their names in alerting policies.

**Q: How does MQL differ from basic metric filtering?**
A: MQL (Monitoring Query Language) provides SQL-like capabilities for metrics: aggregations, transformations, joins, and calculations. Basic filtering only selects based on resource and metric labels.

**Q: What is a Service Level Objective (SLO)?**
A: SLO defines a target for service reliability (e.g., 99.9% availability). Cloud Monitoring SLOs track burn rate and create alerts when SLO is at risk of being violated.

**Q: How do uptime checks work?**
A: Uptime checks periodically send synthetic requests from Google's global network to your endpoint. They test availability, response time, and content matching. Results determine uptime percentage.
