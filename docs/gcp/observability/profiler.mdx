# Cloud Profiler

## Overview

Google Cloud Profiler provides CPU and memory profiling data for applications running on GCP. It continuously collects profiling data with minimal overhead, helping identify performance optimization opportunities. Profiler supports Go, Java, Node.js, and Python applications.

## When To Use

- CPU usage optimization
- Memory leak detection
- Allocation hotspot analysis
- Performance tuning
- Resource efficiency improvements

## When NOT To Use

- Production overhead concerns (low overhead ~1-5%)
- Non-performance issues (use logging/tracing)
- Real-time request tracing (use Cloud Trace)

## Core Concepts

### Profile
Captured performance data snapshot showing function call patterns.

### Flat View
Top-level functions sorted by resource usage.

### Flame Graph
Visual representation of call stack with width proportional to resource use.

### Heap Profile
Memory allocation analysis showing where memory is allocated.

### Go Application

```go
package main

import (
    "contrib.go.opencensus.io/profiler"
    cloudprofiler "github.com/GoogleCloudPlatform/opentelemetry-operations-go/profiler"
)

// Initialize profiler
func initProfiler(service, version string) error {
    return cloudprofiler.Start(cloudprofiler.Config{
        Service: service,
        ServiceVersion: version,
        // Optional: ProjectID if not using ADC
        // ProjectID: "my-project",
    })
}

// Or using OpenTelemetry with Cloud Profiler exporter
import (
    "go.opentelemetry.io/otel/sdk/trace"
    texporter "github.com/GoogleCloudPlatform/opentelemetry-go-contrib/resourcemonitor"
)
```

### Node.js Application

```javascript
// npm install @google-cloud/profiler

// Option 1: Require before other modules
const { Profiler } = require('@google-cloud/profiler');

// Option 2: Using ES modules or top-level await
const profiler = new Profiler({
    projectId: 'my-project',
    serviceContext: {
        service: 'my-service',
        version: '1.0.0',
        // Optional: minor version for deployment tracking
    }
});

profiler.start();

// Your application code
const express = require('express');
const app = express();

// ...

// Graceful shutdown
process.on('SIGTERM', () => {
    profiler.stop();
    process.exit(0);
});
```

## Interview Questions

**Q: What is the difference between CPU time and wall time profiling?**
A: CPU time measures actual CPU time spent in a function (excluding wait time). Wall time measures total elapsed time. Use CPU time for CPU-bound code, wall time for I/O-bound or waiting code.

**Q: How does heap profiling work?**
A: Heap profiling records memory allocations, not deallocations. It shows where objects are created, helping identify memory leaks and excessive allocations. The profile shows allocation counts and sizes.

**Q: What is the overhead of Cloud Profiler?**
A: Typically 1-5% CPU overhead for most applications. The agent collects data in background threads and buffers it for periodic upload.

**Q: When should you use Profiler vs Trace?**
A: Use Profiler for understanding where CPU time and memory are spent. Use Trace for understanding request flow and latency across services.

**Q: How do you interpret a flame graph?**
A: Flame graphs show call stacks with width proportional to resource usage. The bottom is the entry point, top shows leaf functions. Wide segments indicate functions consuming significant resources.
