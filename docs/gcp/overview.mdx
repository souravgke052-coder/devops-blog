# Google Cloud Platform Overview

## Overview

Google Cloud Platform (GCP) is a suite of cloud computing services that runs on the same infrastructure that Google uses internally for its end-user products like Search, YouTube, and Gmail. Launched in 2008, GCP has grown to become one of the three major cloud providers alongside AWS and Azure, offering over 200 products across compute, storage, networking, big data, machine learning, and more.

GCP's global infrastructure consists of regions and zones strategically distributed worldwide. Each region is a geographic location containing multiple zones (data centers), providing high availability and low-latency access to services. As of 2024, GCP operates in 40 regions with 121 zones, making it ideal for globally distributed applications requiring strict data residency and disaster recovery requirements.

Engineering teams choose GCP for its deep integration with Kubernetes (GKE originated from Borg), strong data analytics and ML capabilities (BigQuery, TensorFlow), and networking features like Premium Tier routing that leverage Google's private global network. The platform's pay-as-you-go pricing with sustained-use discounts makes it cost-effective for production workloads.

## When To Use

- **Containerized applications**: GKE provides one of the most mature Kubernetes implementations with Autopilot mode for serverless-like experience
- **Data analytics and ML workloads**: BigQuery's serverless architecture handles petabytes of data without infrastructure management
- **High-performance global applications**: Premium Tier network provides consistent low-latency routing across Google's backbone
- **Microservices architecture**: Cloud Run offers a fully managed container runtime ideal for event-driven services
- **Hybrid cloud scenarios**: Anthos enables consistent Kubernetes across GCP, AWS, Azure, and on-premises
- **Compliance-heavy industries**: GCP's certifications (HIPAA, SOC 2, ISO 27001, FedRAMP) and data residency options suit healthcare and government

## When NOT To Use

- **Windows-centric workloads**: AWS has better Windows ecosystem support; consider AWS for .NET-heavy applications
- **Simple static websites**: Cloudflare Pages or Netlify offer simpler deployment with global CDN built-in
- **Legacy monolithic apps that can't be containerized**: Consider containerizing first or using App Engine flex
- **Very small projects with minimal budget tolerance**: AWS has more mature free tier and beginner-friendly documentation
- **Teams without cloud expertise**: GCP'sIAM and networking can be complex; consider managed services or training first

## Architecture

GCP's architecture follows a hierarchical resource model:

```
Organization (root)
└── Folders (optional nesting)
    └── Projects (billing boundary, IAM boundary)
        └── Resources (Compute instances, Cloud Storage buckets, etc.)
```

**Control Plane**: Every resource belongs to a project, which belongs to a folder and/or organization. IAM policies cascade down this hierarchy—the organization node can enforce policies on all child folders and projects, while more restrictive policies can be applied at lower levels.

**Data Plane**: Resources communicate over GCP's global network or the public internet. VPC networks span regions and can be shared across projects via VPC Peering or Shared VPC. Google's Premium Tier routing routes traffic over Google's backbone, while Standard Tier routes traffic over the public internet.

**Service Layer**: GCP services are either global (Cloud Storage, BigQuery), regional (GKE, Cloud Run), or zonal (Compute Engine instances). Regional services replicate data within a region automatically; zonal services are single-zone but can be made highly available through regional abstractions like Managed Instance Groups.

## Core Concepts

### Regions and Zones
- **Region**: Geographic area (e.g., us-central1, europe-west1)
- **Zone**: Physical data center within a region (e.g., us-central1-a)
- Resources in different zones within same region have ~1-2ms latency
- Cross-region communication adds ~30-100ms depending on distance

### Projects
- The fundamental organizational unit for GCP resources
- Each project has a unique ID, name, and number
- Projects are isolated by default; cross-project access requires explicit IAM grants
- Default quotas apply per project (can be increased via support ticket)

### Service Accounts
- Non-human identities for services and applications
- Managed by IAM, not passwords
- Best practice: Use Workload Identity Federation for external identities

### APIs and Services
- Most GCP services require enabling APIs at project level
- All API activity is logged to Cloud Audit Logs
- Quotas prevent runaway billing from misconfigured applications

### Quotas and Limits
| Resource | Default Limit |
|----------|---------------|
| Projects per organization | 30 (soft), expandable |
| CPUs per region | 8,000 (regular), 24,000 (trusted testers) |
| Static external IP addresses | 7 per region |
| Firewall rules per project | 100 |
| VPCs per project | 5 (can expand to 15) |
| Routes per VPC | 200 |
| Cloud SQL instances per region | 10 |

### Key Gotchas
- **Project IDs are globally unique and immutable** - choose carefully at creation
- **Deleting a project takes 30 days** - resources continue billing during this period
- **IAM changes can take up to 7 minutes** to propagate (usually &lt;2 minutes)
- **Default VPC cannot be deleted** - only modified
- **Cloud Storage objects are immutable** - versioning must be enabled for update semantics
- **GKE Autopilot clusters cannot be accessed with kubectl directly** - requires additional configuration

## Real Production Example

A typical production multi-tier web application on GCP:

```bash
# 1. Create project with organization
gcloud projects create my-prod-app --organization=123456789012

# 2. Enable required APIs
gcloud services enable compute.googleapis.com \
    container.googleapis.com \
    sqladmin.googleapis.com \
    secretmanager.googleapis.com \
    cloudkms.googleapis.com

# 3. Create custom VPC with private Google access
gcloud compute networks create prod-vpc --subnet-mode=custom

gcloud compute networks subnets create web-subnet \
    --network=prod-vpc \
    --region=us-central1 \
    --range=10.0.1.0/24 \
    --enable-private-ip-google-access \
    --secondary-ranges=gke-pods=10.1.0.0/16,gke-services=10.2.0.0/20

# 4. Create Cloud SQL with private IP
gcloud sql instances create prod-db \
    --database-version=MYSQL_8_0 \
    --tier=db-n1-standard-4 \
    --region=us-central1 \
    --network=prod-vpc \
    --no-assign-ip \
    --enable-bin-log \
    --backup-start-time=03:00

# 5. Create GKE cluster with private nodes
gcloud container clusters create prod-cluster \
    --region=us-central1 \
    --network=prod-vpc \
    --subnetwork=web-subnet \
    --cluster-secondary-range-name=gke-pods \
    --services-secondary-range-name=gke-services \
    --enable-private-nodes \
    --master-ipv4-cidr=172.16.0.0/28 \
    --enable-ip-alias \
    --workload-pool=${PROJECT_ID}.svc.id.goog

# 6. Configure firewall rules
gcloud compute firewall-rules create allow-internal \
    --network=prod-vpc \
    --allow=tcp,udp,icmp \
    --source-ranges=10.0.0.0/8

gcloud compute firewall-rules create allow-health-checks \
    --network=prod-vpc \
    --allow=tcp:80,tcp:443 \
    --source-ranges=130.211.0.0/22,35.191.0.0/16 \
    --target-tags=web-server
```

```hcl
# Terraform production setup
provider "google" {
  project = "my-prod-app"
  region  = "us-central1"
}

# VPC Network
resource "google_compute_network" "prod_vpc" {
  name                    = "prod-vpc"
  auto_create_subnetworks = false
  routing_mode            = "REGIONAL"
}

# Subnet with private Google access
resource "google_compute_subnetwork" "web_subnet" {
  name                             = "web-subnet"
  network                          = google_compute_network.prod_vpc.id
  ip_cidr_range                    = "10.0.1.0/24"
  region                           = "us-central1"
  private_ip_google_access         = true
  secondary_ip_range {
    range_name    = "gke-pods"
    ip_cidr_range = "10.1.0.0/16"
  }
  secondary_ip_range {
    range_name    = "gke-services"
    ip_cidr_range = "10.2.0.0/20"
  }
}

# Cloud SQL
resource "google_sql_database_instance" "prod_db" {
  name             = "prod-db"
  database_version = "MYSQL_8_0"
  region           = "us-central1"

  settings {
    tier              = "db-n1-standard-4"
    availability_type = "REGIONAL"
    backup_configuration {
      enabled                        = true
      start_time                     = "03:00"
      backup_retention_settings_exists = true
      retained_backups               = 7
      retained_intervals             = ["WEEKLY"]
    }
    ip_configuration {
      ipv4_enabled    = false
      private_network = google_compute_network.prod_vpc.id
      require_ssl     = true
    }
  }
}

# GKE Cluster
resource "google_container_cluster" "prod_cluster" {
  name     = "prod-cluster"
  location = "us-central1"

  network    = google_compute_network.prod_vpc.name
  subnetwork = google_compute_subnetwork.web_subnet.name

  ip_allocation_policy {
    cluster_secondary_range_name  = "gke-pods"
    services_secondary_range_name = "gke-services"
  }

  private_cluster_config {
    enable_private_nodes   = true
    enable_private_endpoint = true
    master_ipv4_cidr_block = "172.16.0.0/28"
  }

  workload_identity_config {
    workload_pool = "${var.project_id}.svc.id.goog"
  }

  master_authorized_networks_config {
    cidr_blocks {
      cidr_block   = "10.0.0.0/8"
      display_name = "Internal"
    }
  }
}
```

## Monitoring

**Key Metrics to Track:**
- **API Quota Usage**: Monitor via Cloud Monitoring > Resources > Quotas
- **Billing**: Set up budget alerts at 50%, 80%, 100% thresholds
- **Resource Utilization**: CPU, memory, disk I/O per service
- **Network Traffic**: Egress bytes, VPC flow logs analysis

**Alerting Setup:**
```bash
# Create alerting policy for quota usage
gcloud alpha monitoring policies create \
    --display-name="High Quota Usage Alert" \
    --condition-display-name="CPU Quota > 80%" \
    --notification-channels=CHANNEL_ID
```

**Integration with Cloud Monitoring:**
- All GCP services automatically emit metrics to Cloud Monitoring
- Use Metrics Explorer for custom dashboards
- Export metrics to BigQuery or Pub/Sub for advanced analysis
- Cloud Trace for distributed tracing, Cloud Profiler for performance analysis

## Security

**IAM Best Practices:**
- Apply principle of least privilege; use predefined roles over basic roles
- Use groups for human access, service accounts for workloads
- Enable_org_policy to restrict resource locations and API enablement
- Use Conditional Role Bindings for time-limited access
- Enable Security Command Center for threat detection

**Organization Policies:**
```bash
# Disable serial port access organization-wide
gcloud org-policies set-policy org-policy-disable-serial-port.yaml --project=my-project

# Restrict resource locations
gcloud org-policies set-policy org-policy-resource-locations.yaml
```

**Encryption:**
- All data encrypted at rest by default using Google-managed keys
- Customer-managed encryption keys (CMEK) via Cloud KMS
-叶 Compute Engine: persistent disks can use CMEK
- Cloud Storage: CMEK with Cloud KMS
- SQL: CMEK for MySQL and PostgreSQL

**Audit Logging:**
- Cloud Audit Logs enabled by default for all GCP services
- Admin Activity logs cannot be disabled
- Data Access logs (GCS, BigQuery, Cloud SQL) must be explicitly enabled
- Route logs to Cloud Storage or BigQuery for long-term retention

## Cost Optimization

**Pricing Model:**
- Pay-as-you-go with no upfront costs
- Sustained-use discounts (1-30% for consistent usage)
- Committed-use discounts (up to 70% for 1-3 year commitments)
- Resource-based pricing (CPU, memory, storage, network)

**Key Cost Drivers:**
- Compute Engine: CPU and memory usage
- Cloud Storage: Storage + operations + egress
- Network: Egress (cross-region, internet), load balancing
- BigQuery: Storage + queries (on-demand vs flat-rate)

**Optimization Strategies:**
1. **Use preemptible/spot VMs** for batch workloads (70% savings)
2. **Right-size resources** using Cloud Monitoring recommendations
3. **Enable committed-use discounts** for baseline workloads
4. **Use Standard Tier networking** for non-critical traffic
5. **Delete unused resources**: stale snapshots, unused IPs, unattached disks
6. **Schedule GKE node pools** to scale to zero during off-hours
7. **Use Cloud Storage lifecycle policies** to auto-archive or delete old data

**Cost Alerts:**
```bash
# Set budget alert at $10,000 with 50%, 80%, 100% thresholds
gcloud billing budgets create --billing-account=XXXXXX \
    --display-name="Production Budget" \
    --amount=10000USD \
    --threshold-rules=percent=0.5,percent=0.8,percent=1.0
```

## Troubleshooting

**Common Issues:**

1. **Permission Denied despite IAM role**
   - Check if resource is in different project (cross-project IAM required)
   - Verify IAM propagated (can take up to 7 minutes)
   - Check for org policies blocking access
   ```bash
   gcloud projects get-iam-policy my-project
   gcloud org-policies list-policies --project=my-project
   ```

2. **Quota exceeded errors**
   ```bash
   # View current quota usage
   gcloud compute regions describe us-central1 --format="yaml(quotas)"
   
   # Request quota increase
   gcloud compute regions describe us-central1 > quotas.yaml
   # Edit quotas.yaml with new limits, then:
   gcloud compute regions set-quota quotas.yaml --region=us-central1
   ```

3. **API not enabled**
   ```bash
   gcloud services list --available | grep SERVICE_NAME
   gcloud services enable SERVICE.googleapis.com --project=my-project
   ```

4. **VPC connectivity issues**
   ```bash
   # Check firewall rules
   gcloud compute firewall-rules list --filter="network:vpc-name"
   
   # Test connectivity
   gcloud compute ssh instance --zone=us-central1-a
   
   # Check routing tables
   gcloud compute routes list --filter="network:vpc-name"
   ```

5. **Billing surprises**
   ```bash
   # Check current spend
   gcloud beta billing accounts list
   # View in Console: Billing > Cost Table
   
   # Check for resources running
   gcloud compute instances list
   gcloud compute disks list
   ```
