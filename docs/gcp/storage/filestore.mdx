# Filestore

## Overview

Google Cloud Filestore is a managed Network File System (NFS) service that provides high-performance file storage for workloads requiring a file system interface and shared file access. Filestore offers fully managed NFS shares that can be mounted on GCE instances and GKE pods, making it ideal for enterprise applications, media rendering, and CI/CD build agents.

## When To Use

- Applications requiring POSIX-compliant file system access
- Multiple VM instances reading/writing to shared storage
- Media rendering and video processing workflows
- CI/CD build agents needing shared workspace
- Database backup storage requiring file-based access
- Home directories and shared application data
- Content management systems with file-based assets

## When NOT To Use

- Object storage needs (use Cloud Storage)
- Block storage for databases (use Persistent Disk)
- Temporary scratch space (use local SSD)
- Web-scale key-value workloads (use Memorystore)
- Archival/cold storage (use Nearline/Coldline Storage)
- High-frequency small file operations (evaluate alternatives)

## Core Concepts

### Filestore Tiers

| Tier | Storage Type | Min Capacity | Max Capacity | Throughput |
|------|-------------|--------------|--------------|------------|
| Basic | HDD | 2.5 TB | 10 TB | 320 MB/s |
| High Scale | HDD | 10 TB | 100 TB | 1.2 GB/s |
| Enterprise | SSD | 2.5 TB | 100 TB | 1.2 GB/s |
| Premium | SSD | 2.5 TB | 100 TB | 2.4 GB/s |

### Filestore Instance
A Filestore instance is a managed NFS server. Each instance provides one or more file shares with configurable capacity and performance.

### File Share
The NFS export/folder accessible to clients. Named with an instance-specific name and has a specific容量.

### VPC-Native
Filestore instances are deployed within a VPC network, accessible only from within that VPC or peered VPCs.

### Backup
Point-in-time snapshots of Filestore instances stored in Cloud Storage. Available for Enterprise and Premium tiers.

## Real Production Example

**Media Rendering Farm with Shared Filestore**

```hcl
# Terraform configuration for media rendering infrastructure

# Create VPC for Filestore connectivity
resource "google_compute_network" "filestore_vpc" {
  name                    = "filestore-vpc-${var.environment}"
  auto_create_subnetworks = false
  
  mtu = 1460  # Optimized for NFS
}

resource "google_compute_subnetwork" "filestore_subnet" {
  name          = "filestore-subnet-${var.environment}"
  network       = google_compute_network.filestore_vpc.id
  ip_cidr_range = "10.0.0.0/24"
  region        = var.region
  
  private_ip_google_access = true
  
  secondary_ip_range {
    range_name    = "render-workers"
    ip_cidr_range = "10.1.0.0/16"
  }
}

# Create Filestore instance for rendering
resource "google_filestore_instance" "render_storage" {
  name     = "render-storage-${var.environment}"
  location = var.region
  tier     = "PREMIUM"
  
  file_shares {
    capacity_gb = 2560  # 2.5 TB
    name        = "render_assets"
  }
  
  networks {
    network       = google_compute_network.filestore_vpc.id
    modes         = ["MODE_IPV4"]
    reserved_ip_range = "10.0.0.0/29"
  }
  
  labels = {
    environment = var.environment
    workload    = "media-rendering"
  }
  
  timeouts {
    create = "15m"
    update = "15m"
  }
}

# Create Filestore backup schedule
resource "google_filestore_backup" "render_backup" {
  name        = "render-backup-${var.environment}"
  location    = var.region
  source_file_share_name = "render_assets"
  source_instance = google_filestore_instance.render_storage.id
  
  description = "Daily backup of render assets"
  
  labels = {
    environment = var.environment
    backup_type = "daily"
  }
}

# GKE nodepool with Filestore access
resource "google_container_node_pool" "render_workers" {
  name       = "render-workers-pool"
  location   = var.region
  cluster    = google_container_cluster.render_cluster.id
  node_count = 4
  
  node_config {
    machine_type = "n2-standard-32"
    preemptible  = false
    
    boot_disk_kms_key = google_kms_crypto_key.filestore_kms_key.id
    
    service_account = google_service_account.render_worker.email
    
    scopes = ["https://www.googleapis.com/auth/cloud-platform"]
  }
  
  node_network_config {
    node_subnetwork = google_compute_subnetwork.filestore_subnet.id
    
    create_secondary_ranges = false
    secondary_range_blocks = {
      render-workers = google_compute_subnetwork.filestore_subnet.secondary_ip_range[0].range_name
    }
  }
}

# Kubernetes StorageClass for Filestore
resource "kubernetes_storage_class" "filestore_premium" {
  metadata {
    name = "filestore-premium"
  }
  
  storage_provisioner = "filestore.csi.storage.gke.io"
  
  parameters = {
    tier = "PREMIUM"
    network = google_compute_network.filestore_vpc.name
  }
  
  mount_options = ["rw", "relatime", "vers=3"]
  
  reclaim_policy = "Retain"
}
```

## CLI Examples

```bash
# Create a Filestore instance
gcloud filestore instances create my-filestore \
  --location=us-central1 \
  --tier=PREMIUM \
  --file-share=name=render_assets,capacity=2.5TB \
  --network=name=filestore-vpc,reserved-ip-range=10.0.0.0/29

# Create with Active Directory (Enterprise tier)
gcloud filestore instances create my-filestore-ad \
  --location=us-central1 \
  --tier=ENTERPRISE \
  --file-share=name=ad_data,capacity=2.5TB \
  --network=name=filestore-vpc \
  --active-directory-domain=example.com \
  --description="Filestore with AD integration"

# List Filestore instances
gcloud filestore instances list --location=us-central1

# Describe an instance
gcloud filestore instances describe my-filestore \
  --location=us-central1

# Update instance capacity
gcloud filestore instances update my-filestore \
  --location=us-central1 \
  --file-share=capacity=5TB

# Delete instance
gcloud filestore instances delete my-filestore \
  --location=us-central1 \
  --quiet

# Create backup (Enterprise/Premium only)
gcloud filestore backups create my-backup \
  --location=us-central1 \
  --instance=my-filestore \
  --file-share=render_assets

# List backups
gcloud filestore backups list --location=us-central1

# Restore from backup
gcloud filestore instances restore \
  --source-backup=projects/my-project/locations/us-central1/backups/my-backup \
  --location=us-central1 \
  --name=restored-filestore \
  --tier=PREMIUM \
  --file-share=name=restored_assets,capacity=2.5TB

# Mount Filestore on GCE instance
gcloud compute ssh my-instance --zone=us-central1-a -- \
  "sudo mkdir -p /mnt/render_assets && \
   sudo mount 10.0.0.2:/render_assets /mnt/render_assets && \
   df -h /mnt/render_assets"
```

# Kubernetes PersistentVolumeClaim using Filestore
resource "kubernetes_persistent_volume_claim" "filestore_pvc" {
  metadata {
    name = "filestore-pvc"
  }
  spec {
    access_modes       = ["ReadWriteMany"]
    storage_class_name = "filestore-premium"
    resources {
      requests = {
        storage = "100Gi"
      }
    }
  }
}

# Backup configuration
resource "google_filestore_backup" "daily_backup" {
  count                = 7  # Weekly retention
  name                 = "backup-${format("day-%d", count.index)}"
  location             = var.region
  source_file_share_name = google_filestore_instance.premium_kubernetes.file_shares[0].name
  source_instance      = google_filestore_instance.premium_kubernetes.id
  
  labels = {
    retention_day = count.index
  }
}

## Troubleshooting

### Mount Failures
```bash
# Check if instance is running
gcloud filestore instances describe my-filestore --location=us-central1

# Verify network connectivity
gcloud compute ssh my-instance --zone=us-central1-a -- \
  "ping -c 3 10.0.0.2"

# Check firewall rules
gcloud compute firewall-rules list --filter="network:filestore-vpc"

# Verify NFS port is open
gcloud compute ssh my-instance --zone=us-central1-a -- \
  "nc -zv 10.0.0.2 2049"

# Common mount syntax
sudo mount -t nfs -o vers=3 10.0.0.2:/render_assets /mnt/render_assets
```

### Performance Issues
```bash
# Check network bandwidth
gcloud compute ssh my-instance --zone=us-central1-a -- \
  "iperf3 -c 10.0.0.2"

# Monitor NFS operations
gcloud compute ssh my-instance --zone=us-central1-a -- \
  "nfsiostat 1 10"

# Check for network congestion
gcloud compute ssh my-instance --zone=us-central1-a -- \
  "netstat -i | grep eth0"

# Increase NFS read/write buffer sizes
sudo mount -t nfs -o vers=3,rsize=1048576,wsize=1048576 \

```

## Interview Questions

**Q: What is the difference between Filestore and Persistent Disk?**
A: Persistent Disk provides block storage that attaches to a single VM. Filestore provides NFS-based file storage accessible from multiple VMs simultaneously. Filestore is POSIX-compliant while Persistent Disk requires filesystem formatting within the VM.

**Q: When would you choose Enterprise over Premium Filestore?**
A: Enterprise includes Active Directory integration for Windows workloads and cross-region backup capability. Premium offers higher throughput (2.4 GB/s vs 1.2 GB/s). Choose Enterprise for AD-managed environments or cross-region DR; Premium for maximum Linux-based performance.

**Q: How does Filestore handle high availability?**
A: Enterprise and Premium tiers use regional deployment with synchronous replication within the region. Basic and High Scale are single-zone. Enterprise supports cross-region backups for disaster recovery. There's no automatic failover between zones.

**Q: What are the NFS version support limitations?**
A: Filestore supports NFSv3 only. NFSv4.1 is not supported. This affects some advanced features like parallel NFS (pNFS).

**Q: How do you connect GKE pods to Filestore?**
A: Use the Filestore CSI driver (`filestore.csi.storage.gke.io`). Create a StorageClass with the tier and network parameters, then use PersistentVolumeClaim with `ReadWriteMany` access mode.

**Q: What is the minimum capacity for each Filestore tier?**
A: Basic: 2.5 TB, High Scale: 10 TB, Enterprise: 2.5 TB, Premium: 2.5 TB. All tiers support up to 100 TB maximum.

**Q: Can Filestore data be encrypted with customer-managed keys?**
A: Filestore uses Google-managed encryption by default. Direct CMEK support is not available, but you can use GKE's CSI driver with GKE-managed CMEK for workloads running in Kubernetes.
