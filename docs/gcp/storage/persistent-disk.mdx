# Persistent Disk

## Overview

Google Cloud Persistent Disk provides durable, high-performance block storage for VM instances. It offers network-attached storage with automatic encryption at rest and supports both HDD and SSD options. Persistent Disk persists data independently from VM lifecycle and can be snapshotted for backups and migration.

## When To Use

- Primary storage for VM-based workloads (databases, apps, logs)
- State persistence for containers running on GKE
- Boot disks for VM instances
- Need for consistent, low-latency I/O
- Data requiring durability guarantees
- Scenarios needing disk snapshots and backups

## When NOT To Use

- File share requirements between multiple VMs (use Filestore)
- Object storage needs (use Cloud Storage)
- In-memory caching (use Memorystore)
- Serverless workloads (no persistent disk attachment)
- Temporary scratch space (use local SSDs)

## Core Concepts

### Disk Types

| Type | Use Case | Max IOPS | Max Throughput |
|------|----------|----------|----------------|
| pd-standard | Cold storage, sequential R/W | 1,500/3,000 | 180 MB/s |
| pd-balanced | General purpose, cost-effective | 6,000/12,000 | 240 MB/s |
| pd-ssd | High-performance databases | 60,000/100,000 | 480 MB/s |
| pd-extreme | Ultra-high performance | 120,000/400,000 | 2,000 MB/s |

### Regional vs Zonal Disks
- **Zonal (default)**: Attached to VM in same zone, lower latency
- **Regional**: Replicated across two zones for HA, higher latency, 2x cost

### Disk Performance
- IOPS scales with disk size (except pd-extreme)
- Throughput scales with disk size
- Provisioned IOPS (pd-extreme) allows independent performance settings

### Encryption
- Default: Google-managed encryption
- CMEK: Customer-managed via Cloud KMS
- PIE: Customer-supplied encryption (deprecated)

### Attach Modes
- **Read/write (default)**: Exclusive access, can attach to one VM
- **Read-only**: Can attach to multiple VMs simultaneously

### Resize vs Extend
- Resize: Increase disk size (online, no reboot for Linux)
- Extend: Filesystem/filesystem resizing after resize

## Real Production Example

**PostgreSQL Database on GKE with Persistent Disks**

```hcl
# Terraform configuration for production PostgreSQL storage

# Create Persistent Disk for PostgreSQL data
resource "google_compute_disk" "postgres_data" {
  name  = "postgres-data-disk-${var.environment}"
  type  = "pd-ssd"
  zone  = var.zone
  size  = 500
  
  physical_block_size_bytes = 4096
  
  encryption_settings {
    kms_key_self_link = google_kms_crypto_key.pg_disk_key.id
  }
  
  labels = {
    environment = var.environment
    component   = "database"
    team        = "platform"
  }
}

# Create Persistent Disk for PostgreSQL logs (separate for performance)
resource "google_compute_disk" "postgres_logs" {
  name  = "postgres-logs-disk-${var.environment}"
  type  = "pd-balanced"
  zone  = var.zone
  size  = 100
  
  labels = {
    environment = var.environment
    component   = "database-logs"
  }
}

# Regional Persistent Disk for high availability
resource "google_compute_disk" "postgres_ha_data" {
  name  = "postgres-ha-data-${var.environment}"
  type  = "pd-ssd"
  zone  = element(split("-", var.region), 0)
  size  = 500
  
  replica_zones = [
    "${var.region}-a",
    "${var.region}-b",
  ]
  
  labels = {
    environment = var.environment
    ha          = "enabled"
  }
}

# Disk snapshot for point-in-time recovery
resource "google_compute_snapshot" "postgres_daily_backup" {
  name        = "postgres-backup-${formatdate("YYYY-MM-DD", timestamp())}"
  source_disk = google_compute_disk.postgres_data.name
  zone        = var.zone
  
  snapshot_encryption_key {
    kms_key_self_link = google_kms_crypto_key.snapshot_key.id
  }
  
  labels = {
    environment = var.environment
    backup_type = "daily"
  }
}

# Scheduled snapshot via Cloud Scheduler
resource "google_cloud_scheduler_job" "postgres_snapshot" {
  name        = "postgres-daily-snapshot"
  schedule    = "0 2 * * *"  # 2 AM daily
  time_zone   = "UTC"
  region      = var.region
  
  http_target {
    http_method = "POST"
    uri         = "https://compute.googleapis.com/compute/v1/projects/${var.project}/zones/${var.zone}/disks/${google_compute_disk.postgres_data.name}/createSnapshot"
    headers = {
      "Content-Type" = "application/json"
    }
    oauth_token {
      service_account_email = google_service_account.snapshot_sa.email
    }
  }
}
```

## CLI Examples

```bash
# Create a new persistent disk
gcloud compute disks create my-disk \
  --size=500GB \
  --type=pd-ssd \
  --zone=us-central1-a

# Create disk from image
gcloud compute disks create my-boot-disk \
  --image-family=debian-12 \
  --image-project=debian-cloud \
  --zone=us-central1-a

# Create disk with custom encryption
gcloud compute disks create my-encrypted-disk \
  --size=100GB \
  --type=pd-ssd \
  --zone=us-central1-a \
  --kms-key=projects/my-project/locations/us-central1/keyRings/my-ring/cryptoKeys/my-key

# Resize an existing disk (no reboot needed for Linux)
gcloud compute disks resize my-disk \
  --size=1TB \
  --zone=us-central1-a

# Create snapshot
gcloud compute disks snapshot my-disk \
  --snapshot-name=my-daily-snapshot \
  --zone=us-central1-a

# Create disk from snapshot
gcloud compute disks create my-restored-disk \
  --source-snapshot=projects/my-project/global/snapshots/my-daily-snapshot \
  --zone=us-central1-a

# Attach disk to instance
gcloud compute instances attach-disk my-instance \
  --disk=my-disk \
  --zone=us-central1-a

# Attach disk read-only (for sharing data)
gcloud compute instances attach-disk my-instance \
  --disk=my-shared-data \
  --mode=ro \
  --zone=us-central1-a

# Detach disk
gcloud compute instances detach-disk my-instance \
  --disk=my-disk \
  --zone=us-central1-a

# List disks
gcloud compute disks list --filter="zone:us-central1-a"

# Describe disk details
gcloud compute disks describe my-disk --zone=us-central1-a

# Move disk to different zone
gcloud compute disks move my-disk \
  --destination-zone=us-east1-b \
  --source-zone=us-central1-a

# Regional disk creation
gcloud compute disks create my-regional-disk \
  --region=us-central1 \
  --replica-zones=us-central1-a,us-central1-b \
  --size=500GB \
  --type=pd-ssd
```

## Cost Optimization

### Disk Type Selection
Choose the right disk type based on workload:
- **pd-standard**: Dev/test, log storage, sequential workloads
- **pd-balanced**: General workloads, moderate cost/performance
- **pd-ssd**: Databases, critical applications
- **pd-extreme**: Ultra-high performance, provisioned IOPS

### Resize Strategically
```bash
# Monitor disk usage
gcloud monitoring metrics read \
  "compute.googleapis.com/disk/disk_utilization" \
  --filter='resource.disk_id="my-disk"'

# Resize only when needed (no downsizing)
gcloud compute disks resize my-disk --size=new-size --zone=ZONE
```

### Snapshot Cost Management
```bash
# Delete old snapshots
gcloud compute snapshots delete old-snapshot --quiet

# Use snapshot schedules with retention
gcloud compute resource-policies create-snapshot-schedule my-schedule \
  --region=us-central1 \
  --schedule=0 2 * * * \
  --retention-days=30 \
  --max-retention-days=90 \
  --snapshot-locations=us-central1
```

### Regional vs Zonal Cost
Regional disks cost 2x zonal disks but provide HA. Use for:
- Production databases
- Critical application data
- State that cannot be easily reproduced

```bash
gcloud compute instances start instance --zone=ZONE
```

## Interview Questions

**Q: What's the difference between zonal and regional Persistent Disks?**
A: Zonal disks exist in a single zone with lower latency and cost. Regional disks synchronously replicate data across two zones in the same region, providing high availability with automatic failover. Regional disks cost 2x but are essential for HA deployments.

**Q: How does Persistent Disk performance scale?**
A: For pd-standard, pd-balanced, and pd-ssd, IOPS and throughput scale linearly with disk size up to their maximums. Pd-extreme allows independent provisioning of IOPS separate from size, giving predictable performance.

**Q: Can you shrink a Persistent Disk?**
A: No, Persistent Disks can only be resized (increased), not shrunk. You must create a smaller disk, migrate data, then delete the old disk.

**Q: How do Persistent Disk snapshots work?**
A: Snapshots are incremental—they only store changes since the last snapshot. This makes them storage-efficient and fast to create. They support cross-regional copying for disaster recovery.

**Q: What are the encryption options for Persistent Disk?**
A: Three options: Google-managed keys (default, free), Customer-Managed Encryption Keys (CMEK) via Cloud KMS ($0.03/month per key), and Customer-Supplied Encryption Keys (deprecated).

**Q: How do you migrate a Persistent Disk to a different zone?**
A: Use `gcloud compute disks move` for quick relocation, or create a snapshot, then create a new disk in the target zone from that snapshot.

**Q: What's the maximum Persistent Disk size?**
A: 65,536 GB (64 TB) for all disk types. Note that some OS filesystems have their own limits (e.g., ext4 max is 1 EB but in practice much less).
