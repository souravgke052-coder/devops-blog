# Architecture Interview Questions

## Overview

System design interviews test your ability to think through complex, ambiguous problems at scale. Unlike implementation questions where there's usually a clear answer, architecture questions have multiple valid solutions вҖ” the goal is to evaluate your reasoning, trade-off analysis, and communication.

These questions cover distributed systems fundamentals, database design, caching, messaging patterns, and real-world architectural decisions.

## Question Categories

### Conceptual Questions

**Q: Explain the CAP theorem and PACELC. What are practical implications for system design?**

A: **CAP theorem**:

```
         P
        / \
       /   \
      /     \
    C в”Җв”Җв”Җв”Җв”Җв”Җ A
     Two of three
```

**CAP states**: A distributed system can provide at most two of three guarantees:
- **Consistency**: Every read receives the most recent write
- **Availability**: Every request receives a response
- **Partition tolerance**: System continues despite network partitions

**The reality**: Partitions happen. You must choose between CP (give up availability) or AP (give up consistency during partition).

**Practical interpretation**:
| System Type | Trade-off | Example |
|------------|-----------|---------|
| CP | Unavailable during partition | etcd, Zookeeper, HBase |
| AP | May return stale data during partition | Cassandra, DynamoDB, CouchDB |

**PACELC** (more nuanced):
> In case of network Partition (P), choose between Availability (A) and Consistency (C). Else (E), when there is no partition, choose between Latency (L) and Consistency (C).

**PACELC implications**:
| System | During Partition | Normal Operation |
|--------|-----------------|-------------------|
| DynamoDB | Available + eventual consistency | Low latency |
| Cassandra | Available + eventual consistency | Low latency |
| HBase | Unavailable | Strong consistency |
| MongoDB | Unavailable | Strong consistency |
| etcd | Unavailable | Strong consistency |

**Design implications**:
```yaml
# If you need strong consistency during partition:
# Choose: etcd, Zookeeper, HBase, Spanner

# If you need availability during partition:
# Choose: DynamoDB, Cassandra, CouchDB
# Handle inconsistency in application logic

# Question to ask:
# "Can users tolerate slightly stale data?"
# If yes: choose AP
# If no: choose CP
```

**Q: Compare database sharding strategies. How do you choose the right one?**

A: **Sharding approaches**:

**1. Range-based sharding**:
```sql
-- Users 1-1000 on shard A, 1001-2000 on shard B
CREATE TABLE users (
  user_id BIGINT,
  ...
  SHARD KEY (user_id)
);

-- Problem: Hot spots on boundaries
-- Solution: Use large ranges, monitor distribution
```

**2. Hash-based sharding**:
```python
shard_id = hash(user_id) % num_shards

# Problem: Can't do range queries efficiently
# Problem: Resharding is expensive
# Solution: Consistent hashing
```

**Choosing a sharding strategy**:

| Strategy | Best For | Avoid For |
|----------|----------|-----------|
| Range | Time-series, append-only | Hot spots at boundaries |
| Hash | Even distribution | Range queries |
| Consistent | Dynamic scaling | Complex queries |
| Directory | Multiple access patterns | Simplicity |

**Sharding challenges**:
```yaml

# Solution: Don't do it!
# Use sagas or accept eventual consistency

# 4. Join operations
# Solution: Data locality
# Keep related data on same shard (e.g., user + orders)
```

**Q: How does distributed consensus work? Explain Raft vs Paxos.**

A: **Why consensus matters**:

In distributed systems, nodes must agree on:
- Who is leader?
- What is the current state?
- Is this write committed?

**Paxos** (Leslie Lamport, 1989):
```
Proposer: "I propose value 42. Phase 1: Is anyone listening?"
Acceptor: "I'm listening. What's your proposal number?"
Proposer: "OK, proposal #101, value 42"
Acceptor: "OK, I've accepted #101 with value 42"
Learner: "I heard value 42 was chosen!"
```

**Paxos phases**:
1. **Prepare**: Proposer gets majority to accept its proposal number
2. **Accept**: Proposer sends value, majority accepts
3. **Learn**: Learners learn the committed value

**Raft** (simplified Paxos, 2014):
```
Raft separates into:
- Leader election (using term numbers)
- Log replication (append-only)
- Safety (follows Raft rules)
```

**Raft leader election**:
```python
# Node states: Follower, Candidate, Leader

# Election timeout (randomized 150-300ms)
# If no heartbeat from leader within timeout:
#   вҶ’ Become Candidate
#   вҶ’ Increment term
#   вҶ’ Vote for self
#   вҶ’ Request votes from others

# If majority votes:
#   вҶ’ Become Leader
#   вҶ’ Send heartbeats

# If higher term received:
#   вҶ’ Become Follower
```

**Raft log replication**:
```
Leader: "Here's entry #42: set x=42"
Follower1: "OK, I've got entry #42"
Follower2: "OK, I've got entry #42"
(When majority have entry)
Leader: "Entry #42 is committed"
```

**Paxos vs Raft comparison**:

| Aspect | Paxos | Raft |
|--------|-------|------|
| Complexity | Very complex | More understandable |
| Implementation | Hard | More implementable |
| Leader | Optional | Required |
| Partition tolerance | Yes | Yes |
| Performance | Similar | Similar |
| Clients | Must handle leader | Must handle leader |

**Real-world usage**:
```yaml
# etcd uses Raft
# Good: Simple leader-follower
# Problem: Leader is bottleneck

# Zookeeper uses Zab (Zookeeper Atomic Broadcast)
# Similar to Raft

# Spanner uses Paxos
# With TrueTime for global consistency
```

**Q: Explain CQRS (Command Query Responsibility Segregation). When should you use it?**

A: **CQRS principle**:

Separate read and write models for the same data:
- **Command**: Writes (create, update, delete)
- **Query**: Reads (select, search)

**Traditional approach**:
```

в”Ӯ  (writes)   в”Ӯ     в”Ӯ             в”Ӯ     в”Ӯ   (reads)   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ     в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ     в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**CQRS approach**:
```

      в”Ӯ                             в–І
      в–ј                             в”Ӯ
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ              в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”җ
```

**When to use CQRS**:

| Indicator | Use CQRS? |
|-----------|-----------|
| Read/write ratio > 10:1 | Yes |
| Different read/write patterns | Yes |

**Q: What caching strategies exist? When would you use each?**

A: **Cache strategies**:

**1. Cache-aside (lazy loading)**:
```python
def get_user(user_id):
    # Try cache first
    user = cache.get(f"user:{user_id}")
    if user:
        return user
    
    # Cache miss, load from DB
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    
    # Populate cache
    cache.set(f"user:{user_id}", user, ttl=3600)
    return user
# PRO: Only cache what's needed
# CON: First request is slow (cache miss)
```

**2. Write-through**:
```python
def update_user(user_id, data):
    # Write to DB and cache together
    db.update("UPDATE users SET ... WHERE id = ?", user_id, data)
    cache.set(f"user:{user_id}", data)
# PRO: Cache always fresh
# CON: Write latency includes cache write
```

**3. Write-behind (write-back)**:
```python
def update_user(user_id, data):
    # Write to cache, return immediately
    cache.set(f"user:{user_id}", data)
    # Async write to DB
    queue.enqueue("write_user", user_id, data)
# PRO: Fastest writes
# CON: Risk of data loss if cache fails before DB write
```

**4. Refresh-ahead**:
```python
# Background refresh of frequently accessed items
# Before TTL expires
# PRO: Users never see cache miss
# CON: Wasted refresh if item not accessed
```

**Cache invalidation**:
```yaml
# TTL-based: Simple but not precise
ttl: 300  # 5 minutes

# Event-based: Precise but complex
# On User updated вҶ’ invalidate user:{id} cache

# Version-based: Simple versioning
cache_key: "user:123:v2"  # Invalidate by bumping version
```

**Multi-level caching**:
```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
```

**Q: Explain message queue patterns. When would you use Kafka vs RabbitMQ vs SQS?**

A: **Message queue patterns**:

**Point-to-point** (SQS, RabbitMQ direct):
```
Producer в”Җв”Җв–¶ Queue в”Җв”Җв–¶ Consumer
                   (only one reads)
```

**Pub/Sub** (Kafka, RabbitMQ exchange):
```
Producer в”Җв”Җв–¶ Topic в”Җв”Җв–¶ Subscriber A
                    в”Җв”Җв–¶ Subscriber B
                    в”Җв”Җв–¶ Subscriber C
```

**Kafka vs RabbitMQ vs SQS**:

| Feature | Kafka | RabbitMQ | SQS |
|---------|-------|----------|-----|
| Order | Per partition | Per queue | No guarantee |
| Replay | Yes (retain messages) | No (ack = delete) | No |
| Throughput | Very high | Medium | Medium |
| Latency | Low | Very low | Low |
| Durability | Excellent | Good | Excellent |
| Scaling | Partition count | Queue count | Auto |
| Operations | Complex | Medium | Managed |

**When to use each**:

| Use Case | Choice |
|----------|--------|
| Event streaming, audit log | Kafka |
| Task queue, async jobs | RabbitMQ |
| Decoupling microservices | SQS/SNS |
| Real-time notifications | RabbitMQ |
| Data pipeline | Kafka |
| Fire-and-forget | SQS |

**At-least-once vs exactly-once**:
```yaml
# At-least-once: Message may be delivered multiple times
# Consumer must handle duplicates (idempotency)

# Exactly-once: Message delivered exactly once
# Implementation:
# Kafka: transactionId + deduplication
# SQS: MessageDeduplicationId
# RabbitMQ: Publisher confirms + consumer deduplication

# Recommendation:
# Most cases: at-least-once + idempotent consumers
# Exactly-once is expensive and rarely necessary
```

**Saga pattern for distributed transactions**:
```yaml
# Instead of ACID transactions across services:
# Use choreography or orchestration

# Choreography-based saga:
OrderService в”Җв”Җв–¶ PaymentService в”Җв”Җв–¶ InventoryService
      в–І                в–І                  в”Ӯ
      в”Ӯ                в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
      в”Ӯ              (events on failure: compensate)

      в”Ӯ
      в”ңв”Җв”Җв–¶ InventoryService

      в””в”Җв”Җв–¶ ShippingService
```

```python
# Option 1: Hash of long URL (deterministic)
import hashlib
def short_url(long_url):
    hash = hashlib.md5(long_url.encode()).hexdigest()[:7]

# Option 2: Random (more common)
import random
```

**Real-time delivery**:
```python
class WebSocketServer:
    def __init__(self):
        self.clients = {}  # user_id -> set of connections
    
    async def connect(self, user_id, websocket):
        self.clients.setdefault(user_id, set()).add(websocket)
    
    async def send_notification(self, user_id, notification):
        # Fan out to all user's connections
        for conn in self.clients.get(user_id, []):
            await conn.send_json(notification)
```

## Common Mistakes Candidates Make

1. **Not asking clarifying questions**: Never jump into design without understanding requirements and constraints.

2. **Over-engineering**: Start simple, scale as needed. Don't design for 10x if 2x is the requirement.

3. **Ignoring operational concerns**: How will you monitor, debug, and maintain this?

4. **Forgetting about failures**: Assume components will fail. Design for it.

5. **Skipping trade-off discussion**: Always explain why you chose A over B.

## How to Structure Your Answers

For **design questions**: Requirements вҶ’ High-level design вҶ’ Data model/API вҶ’ Scaling вҶ’ Trade-offs.

For **trade-off questions**: Explain both sides, then justify your choice based on requirements.

For **scaling questions**: Vertical вҶ’ Caching вҶ’ Read replicas вҶ’ Sharding вҶ’ New services.
