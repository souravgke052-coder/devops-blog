# Terraform Interview Questions

## Overview

Terraform has become the de facto standard for infrastructure as code in cloud environments. Senior engineers must understand not just the HCL syntax but the deeper concepts around state management, provider architecture, module design, and the trade-offs between different approaches.

These questions probe your understanding of Terraform's operational model, your ability to debug infrastructure issues, and your knowledge of production-hardened patterns.

## Question Categories

### Conceptual Questions

**Q: Explain Terraform state. Why is it critical and what are the risks of improper state management?**

A: **Terraform state purpose**:

State is how Terraform tracks the mapping between your configuration files and real-world infrastructure. Without state, Terraform wouldn't know which resources it created or whether they need to be created, updated, or destroyed.

**State file structure**:
```json
{
  "version": 4,
  "terraform_version": "1.5.0",
  "serial": 1,
  "lineage": "abc-123",
  "outputs": {},
  "resources": [
    {
      "mode": "managed",
      "type": "aws_instance",
      "name": "web",
      "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]",
      "instances": [
        {
          "attributes": {
            "id": "i-1234567890abcdef0",
            "ami": "ami-12345678",
            "instance_type": "t3.micro",
            "private_ip": "10.0.1.100"
          }
        }
      ]
    }
  ]
}
```

**State stores remote reality**:
- Resource IDs and attributes not in config
- Computed values (like generated IDs)
- Metadata for ordering and relationships

**Critical risks**:

1. **State exposure**: State contains sensitive data (passwords, keys, private IPs). In plaintext state, these are visible.

2. **State locking**: Without locking, concurrent applies can corrupt state:
   ```
   terraform apply # T1: reads state v1
   terraform apply # T2: reads state v1
   terraform apply # T1: writes state v2
   terraform apply # T2: writes state v3 (overwrites T1's changes)
   ```

3. **State file loss**: Without backups, losing state means infrastructure becomes unmanageable. Must manually import resources or use `terraform import`.

**Best practices**:
```hcl
# Remote state with locking
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"
  }
}
```

```bash
# Enable versioning on S3 for state history
aws s3api put-bucket-versioning \
  --bucket my-terraform-state \
  --versioning-configuration Status=Enabled
```

**Q: What is the difference between Terraform modules and workspaces? When would you use each?**

A: **Modules vs Workspaces**:

**Modules** are reusable components — encapsulation of infrastructure patterns:
```
modules/
  networking/
    main.tf
    variables.tf
    outputs.tf
  ecs-cluster/
    main.tf
    variables.tf
    outputs.tf
```

**Workspaces** are isolated state for the same configuration:
```
terraform.workspace = "production"  → state: terraform.prod.tfstate
terraform.workspace = "staging"     → state: terraform.staging.tfstate
```

**When to use modules**:

| Scenario | Approach |
|----------|----------|
| Reusable infrastructure pattern | Module |
| Different environments (prod vs staging) | Workspaces OR separate configs |
| Team isolation | Either (depends on use case) |
| Product isolation | Separate configs (sometimes modules) |

**Module design patterns**:

```hcl
# Simple module invocation
module "vpc" {
  source  = "./modules/networking"
  version = "1.0.0"
  
  cidr           = var.vpc_cidr
  environment    = var.environment
  tags           = var.tags
}
```

```hcl
# Module with for_each (create multiple)
module "security_groups" {
  source = "./modules/security-group"
  
  for_each = toset(["web", "api", "database"])
  
  name        = each.value
  vpc_id      = module.vpc.id
  environment = var.environment
}
```

**When to use workspaces**:

```bash
# Good for: dev/staging/prod using same code
terraform workspace new production
terraform workspace new staging

# In Terraform config:
locals {
  instance_type = terraform.workspace == "production" ? "t3.large" : "t3.micro"
}
```

**When NOT to use workspaces**:
- When you need different variables files anyway (just use separate configs)
- When workspaces would create resource name conflicts
- When compliance requires completely separate state (some auditors want this)

**Q: Explain the Terraform provider architecture. How do providers work internally?**

A: **Provider architecture**:

```
┌─────────────────────────────────────────────────────────┐
│                    Terraform Core                       │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              Configuration Parser                │   │
│  │           (HCL → Internal Graph)                 │   │
│  └─────────────────────┬───────────────────────────┘   │
│                        │                                │
│  ┌─────────────────────▼───────────────────────────┐   │
│  │              Resource Graph                      │   │
│  │     (Dependency resolution, ordering)            │   │
│  └─────────────────────┬───────────────────────────┘   │
│                        │                                │
│  ┌─────────────────────▼───────────────────────────┐   │
│  │              Provider Plugin Protocol            │   │
│  │    (gRPC: Plan/Apply/Read/Delete)               │   │
│  └─────────────────────┬───────────────────────────┘   │
└────────────────────────┼────────────────────────────────┘
                         │
          ┌──────────────┼──────────────┐
          │              │              │
          ▼              ▼              ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │   AWS    │   │   GCP    │   │  Azure   │
    │ Provider │   │ Provider │   │ Provider │
    └──────────┘   └──────────┘   └──────────┘
```

**Provider responsibilities**:
- Convert Terraform resource types to cloud API calls
- Handle authentication (AWS credentials, GCP tokens, Azure AD)
- Implement resource CRUD operations
- Handle pagination, rate limiting, retries

**Provider configuration**:
```hcl
# Required_providers in terraform block
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# Provider configuration
provider "aws" {
  region = "us-east-1"
  
  # Can also use environment variables:
  # AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY
  # AWS_PROFILE
  # Or IAM roles (for EC2, Lambda)
  
  default_tags {
    tags = {
      Environment = "production"
      ManagedBy   = "terraform"
    }
  }
}
```

**Plugin discovery**:
```bash
# Terraform downloads providers from registry
# ~/.terraform.d/plugins/
# Or mirrors for air-gapped environments
terraform {
  plugin_cache_dir = "/opt/terraform/plugin-cache"
  provider_meta "aws" {
    module = "terraform-aws-modules/vpc/aws"
  }
}
```

**Q: What are Terraform backends? Compare the major backends and their use cases.**

A: **Backend types**:

A backend defines where and how Terraform operations are executed and where state is stored.

**Local backend** (default for development):
```hcl
terraform {
  backend "local" {
    path = "terraform.tfstate"
  }
}
```
- Simple, no infrastructure needed
- No state locking (concurrent operations dangerous)
- No state versioning
- Not suitable for teams

**Remote backends**:

| Backend | State Storage | Locking | Use Case |
|---------|--------------|---------|----------|
| S3 + DynamoDB | S3 | DynamoDB | AWS teams, production |
| GCS | GCS | GCS native | GCP teams |
| Azure Blob | Azure Storage | Azure native | Azure teams |
| Terraform Cloud | Managed | Managed | Teams wanting SaaS |
| Consul | Consul KV | Consul | Existing Consul users |
| Kubernetes | K8s ConfigMap | K8s native | K8s-first workflows |

**S3 backend with DynamoDB locking**:
```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/vpc/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"
    
    # For state versioning (recovery)
    # Enable in bucket settings, not in backend config
  }
}
```

**GCS backend**:
```hcl
terraform {
  backend "gcs" {
    bucket  = "my-terraform-state"
    prefix  = "prod/vpc"
    
    # GCS supports object versioning automatically
  }
}
```

**Terraform Cloud backend**:
```hcl
terraform {
  backend "remote" {
    organization = "my-org"
    
    workspaces {
      name = "prod-vpc"
      # or: project = "my-project" (for VCS-driven)
    }
  }
}
```

**Q: Explain Terraform's dependency graph and how it determines resource creation order.**

A: **Dependency graph internals**:

Terraform builds a directed acyclic graph (DAG) from your configuration:
```
Module A ──┐
           ├──▶ Resource C ──▶ Resource D
Module B ──┘         ▲
                     │
               (implicit dependency)
```

**Dependency types**:

1. **Implicit dependencies** (automatic from resource references):
```hcl
resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t3.micro"
}

resource "aws_eip" "web_ip" {
  instance = aws_instance.web.id  # Implicit dependency on aws_instance.web
}
```

2. **Explicit dependencies** (depends_on):
```hcl
resource "aws_s3_bucket" "app" {
  bucket = "my-app"
}

resource "aws_s3_bucket_policy" "app_policy" {
  bucket = aws_s3_bucket.app.id
  policy = data.aws_iam_policy_document.app.json
  
  depends_on = [aws_s3_bucket.app]  # Explicit (when not inferrable)
}
```

**Graph resolution rules**:
1. Resources without dependencies run first (parallel)
2. Resources with all dependencies satisfied can run
3. `count`/`for_each` creates multiple nodes in graph
4. Provider constraints limit parallelism

**Practical implications**:

```hcl
# Module with explicit dependencies (good practice)
module "database" {
  source = "./modules/rds"
  
  # Ensure VPC is created before RDS
  vpc_id = module.vpc.vpc_id
  
  # Terraform auto-detects this
}

# But for resources Terraform can't infer:
resource "null_resource" "wait_for_vpc" {
  triggers = {
    vpc_id = module.vpc.vpc_id
  }
  
  provisioner "local-exec" {
    command = "echo VPC ${module.vpc.vpc_id} is ready"
  }
}
```

**Visualizing the graph**:
```bash
terraform graph | dot -Tpng > graph.png
# Or use terraform graph | grep -v "^digraph"
```

### Practical / Hands-On Questions

**Q: How do you handle secrets in Terraform without exposing them in state or logs?**

A: **Secret management strategies**:

1. **Environment variables** (never in config):
```bash
export TF_VAR_db_password="secret123"
# In .tfvars:
db_password = var.db_password
# var.db_password from env
```

2. **AWS Secrets Manager / Systems Manager Parameter Store**:
```hcl
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "prod/db-password"
}

resource "aws_db_instance" "app" {
  password = data.aws_secretsmanager_secret_version.db_password.secret_string
}
```

3. **GCP Secret Manager**:
```hcl
data "google_secret_manager_secret_version" "db_password" {
  secret_id = "prod-db-password"
}

resource "google_sql_database_instance" "app" {
  password = data.google_secret_manager_secret_version.db_password.secret_data
}
```

4. **HashiCorp Vault**:
```hcl
data "vault_kv_secret_v2" "db_creds" {
  mount = "secret"
  name  = "prod/db"
}

resource "aws_db_instance" "app" {
  password = data.vault_kv_secret_v2.db_creds.data["password"]
}
```

**S3 backend with encryption**:
```hcl
terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "prod/terraform.tfstate"
    region = "us-east-1"
    encrypt = true  # State encrypted at rest
  }
}

# But: Terraform state IN MEMORY still contains the decrypted secrets
# This is unavoidable
```

**State encryption considerations**:
```bash
# State contains plaintext sensitive values
# S3 encryption protects state at rest
# But anyone with S3 access can read state
# Solution: Use KMS for state encryption with controlled key access
```

**Q: How do you manage Terraform when state gets corrupted or out of sync with reality?**

A: **Recovery strategies**:

**1. State manipulation (dangerous but sometimes necessary)**:

```bash
# Show current state
terraform state list

# Show specific resource state
terraform state show aws_instance.web

# Move resource to new name (when you renamed in config)
terraform state mv aws_instance.web aws_instance.api

# Remove resource from state (when manually deleted)
terraform state rm aws_instance.old

# Push local state to backend (recovery)
terraform state push terraform.tfstate.backup
```

**2. Import existing infrastructure**:

```bash
# When reality exists but state doesn't
terraform import aws_instance.existing i-1234567890abcdef0
```

```hcl
# More complex imports with configuration
resource "aws_db_instance" "existing" {
  # Define the resource that matches existing infrastructure
  # Terraform will import current state
}
```

**3. State version recovery**:
```bash
# If using S3 with versioning:
aws s3api list-object-versions \
  --bucket my-terraform-state \
  --prefix prod/terraform.tfstate

# Get specific version
aws s3api get-object \
  --bucket my-terraform-state \
  --key prod/terraform.tfstate \
  --version-id VERSION_ID \
  terraform.tfstate.backup

# Push to Terraform
terraform state push terraform.tfstate.backup
```

**4. Destroy and recreate (last resort)**:
```bash
# If infrastructure is broken:
terraform plan  # See what would be destroyed
terraform destroy
terraform apply  # Recreate cleanly
```

**Prevention**:
```bash
# Always enable state versioning in S3
# Always use state locking
# Always run terraform plan before apply (never auto-apply without review)
```

**Q: How do you test Terraform code? What testing frameworks and approaches exist?**

A: **Testing pyramid for Terraform**:

```
        ┌─────────────┐
        │  Terratest  │  E2E: spin up real resources
        │   (Go)      │  Most realistic, slowest
        └──────┬──────┘
               │
        ┌──────▼──────┐
        │ terraform-  │  Integration: validate outputs
        │  validate   │  Fast
        │  fmt check  │
        └──────┬──────┘
               │
        ┌──────▼──────┐
        │   Checkov   │  Static analysis
        │  TFSec       │  Policy as code
        └─────────────┘
```

**Static analysis (pre-apply)**:
```bash
# terraform validate (built-in)
terraform validate

# Checkov (CIS benchmarks)
checkov -d .

# tfsec
tfsec .
```

**Integration testing**:
```hcl
# Using Terratest (Go)
package test

import (
    "testing"
    "github.com/gruntwork-io/terratest/modules/terraform"
    "github.com/stretchr/testify/assert"
)

func TestTerraformEC2(t *testing.T) {
    terraformOptions := &terraform.Options{
        TerraformDir: "../examples/ec2",
        Vars: map[string]interface{}{
            "instance_type": "t3.micro",
        },
    }
    
    defer terraform.Destroy(t, terraformOptions)
    terraform.InitAndApply(t, terraformOptions)
    
    instanceID := terraform.Output(t, terraformOptions, "instance_id")
    assert.NotEmpty(t, instanceID)
}
```

**Module testing with test directory**:
```bash
# Terraform 0.15+ has built-in test
# tests/test.tf
variable "example" {}

test "validate_instance_type" {
  command = plan
  
  assert {
    condition     = length(regexall("t[23]", var.instance_type)) > 0
    error_message = "Instance type must be t2 or t3"
  }
}
```

### Troubleshooting Questions

**Q: Terraform plan shows changes that shouldn't happen. What do you do?**

A: **Troubleshooting unexpected changes**:

**1. Check for drift in actual infrastructure**:
```bash
# Compare state to reality
terraform plan -refresh-only
# This updates state to match reality without making changes
```

**2. Common causes of unexpected changes**:

a) **Manual changes to infrastructure** (drift):
```bash
# Someone changed an EC2 instance manually
# terraform plan will try to "fix" it back

# Options:
# 1. Accept drift by importing manual changes: terraform import
# 2. Accept Terraform's fix by applying
# 3. Modify Terraform config to match reality (if manual is correct)
```

b) **Provider version differences**:
```bash
# Pin exact provider version
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "=5.0.0"  # Exact version
    }
  }
}

# Check current provider version
terraform version
```

c) **Computed values changing**:
```hcl
# If a resource generates a random ID or timestamp
resource "random_id" "app" {
  byte_length = 8
}

# Every terraform plan will differ if this is recreated
# Fix: use keepers to only regenerate when needed
resource "random_id" "app" {
  byte_length = 8
  keepers = {
    # Only regenerate when environment changes
    environment = var.environment
  }
}
```

d) **Lifecycle ignores**:
```hcl
resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t3.micro"
  
  lifecycle {
    # Ignore these attribute changes
    ignore_changes = [
      ami,
      tags,
    ]
  }
}
```

**3. Debugging**:
```bash
# Verbose logging
TF_LOG=TRACE terraform plan 2>&1 | less
TF_LOG=DEBUG terraform plan 2>&1 | grep -E "aws_|resource" | head -100
```

**Q: A terraform apply partially fails. How do you handle state consistency?**

A: **Partial failure handling**:

**Understanding Terraform's atomicity**:
- Terraform tries to be atomic per resource
- But a provider failure mid-apply can leave inconsistent state
- State file is updated incrementally

**Recovery steps**:

1. **Check what was created**:
```bash
# See current state
terraform state list

# See what plan shows as existing
terraform plan
```

2. **Handle specific failures**:

```bash
# If a resource failed to create but was added to state:
terraform state rm <resource_that_failed>

# Then fix the configuration and apply again

# If a resource partially created (e.g., RDS instance but not security groups):
# You'll need manual cleanup
aws rds delete-db-instance --db-instance-identifier my-db --skip-final-snapshot

# Then terraform apply will recreate correctly
```

3. **Taint approach** (forcing recreation):
```bash
# Mark resource as tainted (corrupted)
terraform taint aws_instance.corrupted

# Next apply will destroy and recreate
terraform plan  # Shows destroy/recreate
terraform apply
```

4. **Use targeted apply to fix specific resources**:
```bash
# Only apply specific resources
terraform apply -target=aws_instance.web
terraform apply -target=aws_security_group.app

# This avoids re-applying everything
```

5. **For complex multi-resource failures**:
```bash
# Import failed resources to state as-is
# Then manually fix what you can
# Or use terraform import to bring manual resources under management
```

**Prevention**:
```hcl
# Use create_before_destroy for critical resources
resource "aws_db_instance" "app" {
  identifier = "my-db"
  
  lifecycle {
    create_before_destroy = true  # Creates new before destroying old
  }
}

# Use ignore_changes for volatile computed attributes
lifecycle {
  ignore_changes = [last_updated, arn]
}
```

### Architecture / Design Questions

**Q: Design a Terraform project structure for a multi-team, multi-environment organization.**

A: **Recommended structure**:

```
infrastructure/
├── README.md
├── .gitignore
├── .terraform-version
├── Makefile
│
├── modules/                    # Reusable modules
│   ├── networking/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   └── versions.tf
│   ├── ecs-cluster/
│   │   └── ...
│   └── common-tags/
│       └── ...
│
├── global/                     # Global resources (S3, IAM, etc.)
│   ├── backend.tf
│   ├── main.tf
│   └── versions.tf
│
├── environments/
│   ├── production/
│   │   ├── backend.tf
│   │   ├── main.tf           # Uses modules
│   │   ├── variables.tf
│   │   └── terraform.tfvars
│   │
│   ├── staging/
│   │   └── ...
│   │
│   └── development/
│       └── ...
│
└── scripts/
    ├── plan.sh
    ├── apply.sh
    └── cost-estimate.sh
```

**Backend configuration per environment**:
```hcl
# environments/production/backend.tf
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-prod"
    key            = "production/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks-prod"
  }
}
```

**Environment configuration**:
```hcl
# environments/production/terraform.tfvars
environment = "production"
instance_types = {
  web     = "t3.medium"
  api     = "t3.large"
  worker  = "t3.micro"
}
replicas = 3
enable_deletion_protection = true
```

**Root module usage**:
```hcl
# environments/production/main.tf
module "networking" {
  source = "../../modules/networking"
  
  environment = var.environment
  cidr_block  = "10.1.0.0/16"
}

module "ecs" {
  source = "../../modules/ecs-cluster"
  
  environment      = var.environment
  vpc_id           = module.networking.vpc_id
  instance_types   = var.instance_types
  desired_capacity = var.replicas
  
  # Pass in tags from environment
  common_tags = local.common_tags
}
```

**Makefile for CI/CD**:
```makefile
.PHONY: plan apply destroy

PLAN = terraform plan -var-file=terraform.tfvars
APPLY = terraform apply -var-file=terraform.tfvars -auto-approve

plan:
	cd environments/$(ENV) && $(PLAN)

apply:
	cd environments/$(ENV) && $(APPLY)

destroy:
	cd environments/$(ENV) && terraform destroy -var-file=terraform.tfvars -auto-approve

init:
	cd environments/$(ENV) && terraform init -upgrade
```

### Security Questions

**Q: How do you implement a security review process for Terraform changes?**

A: **Security review pipeline**:

1. **Static analysis in CI**:
```yaml
# .github/workflows/terraform.yml
name: Terraform Security Scan
on: [pull_request]

jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Checkov Security Scan
        uses: bridgecrewio/checkov-action@master
        with:
          directory: .
          framework: terraform
          output_format: sarif
          
      - name: tfsec
        uses: aquasecurity/tfsec-action@v1
        with:
          sarif_file: tfsec.sarif
          
      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v2
```

2. **Policies with OPA/Conftest**:
```rego
# policy.rego
deny[msg] {
  input.resource[_].type == "aws_db_instance"
  input.resource[_].properties.storage_encrypted != true
  msg = "Database must have encryption at rest enabled"
}

deny[msg] {
  input.resource[_].type == "aws_s3_bucket"
  not input.resource[_].properties.acl == "private"
  msg = "S3 buckets must be private"
}
```

3. **Pre-commit hooks**:
```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/antonbabenko/pre-commit-terraform
    rev: v1.77.0
    hooks:
      - id: terraform_fmt
      - id: terraform_validate
      - id: checkov
      - id: tfsec
      - id: infracost_breakdown
```

4. **Sensitive variable handling**:
```hcl
# sensitive variables marked as such
variable "db_password" {
  description = "Database password (sensitive)"
  type        = string
  sensitive   = true  # Won't show in output
}
```

## Common Mistakes Candidates Make

1. **Not using remote state**: Local state in a team environment leads to state conflicts and corruption.

2. **Not understanding create_before_destroy**: Resource replacement can fail if you don't plan for it.

3. **Using too many provisioners**: Provisioners should be last resort, not primary configuration method.

4. **Not pinning provider versions**: Updates can break things silently. Always use version constraints.

5. **Confusing terraform workspaces with environments**: For production, separate configs or modules usually make more sense.

## How to Structure Your Answers

For **state management questions**: Explain the mechanism first, then the risks, then the solutions.

For **troubleshooting questions**: Follow the diagnostic order — check configuration, check state, check provider behavior.

For **design questions**: Start with requirements, then discuss component selection, then address failure modes.
