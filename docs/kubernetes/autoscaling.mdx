# Kubernetes Autoscaling

## Overview

Kubernetes provides multiple layers of autoscaling to match workload demand with resource capacity. **Horizontal Pod Autoscaler (HPA)** scales pods based on metrics, **Vertical Pod Autoscaler (VPA)** adjusts container resource requests, and **Cluster Autoscaler** adjusts the number of nodes in a cluster. Together they provide comprehensive autoscaling for cloud-native applications.

## When to Use

- **Variable traffic patterns** - web applications with fluctuating users
- **CPU-intensive workloads** - batch processing, ML inference
- **Memory-intensive applications** - caching, data processing
- **Cost optimization** - scale down during low traffic
- **High availability** - ensure capacity during traffic spikes
- **Resource optimization** - right-size pod resource requests

## When NOT to Use

- **Stable workloads** - autoscaling adds overhead
- **Very short-lived tasks** - scaling takes time
- **Stateful applications with fixed scaling** - use alternative patterns
- **Predictable batch jobs** - schedule capacity in advance

## Core Concepts

### Autoscaling Stack

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Kubernetes Autoscaling Stack                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Cluster Autoscaler (Node Level)                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Adds/removes nodes based on pending pods                            │   │
│  │  Responds to: Pods pending due to resource constraints              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              │                                               │
│  Horizontal Pod Autoscaler (Pod Level)                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Scales pod replicas based on CPU, memory, custom metrics           │   │
│  │  Adjusts: Replica count                                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              │                                               │
│  Vertical Pod Autoscaler (Container Level)                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Adjusts container resource requests based on usage                  │   │
│  │  Adjusts: CPU/memory requests                                        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### HPA Behavior

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         HPA Decision Flow                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Metric Collection (every 15 seconds)                                        │
│      │                                                                        │
│      ▼                                                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Calculate: current_replicas × (current_metric / target_metric)    │   │
│  │  Example: 5 × (80 / 60) = 6.67 → round up = 7                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              │                                               │
│                              ▼                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Stabilization Window (default 5 min)                                │   │
│  │  - Prevents rapid scaling                                           │   │
│  │  - Configurable via behavior section                                 │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              │                                               │
│                              ▼                                               │
│  Scale Action (if needed)                                                   │
│      │                                                                        │
│      ▼                                                                        │
│  kubectl scale deployment → new replica count                                │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Real Production Example

### Production API with Full Autoscaling Stack

```yaml
# HorizontalPodAutoscaler
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-service-hpa
  namespace: production
  labels:
    app: api-service
spec:
  # Scale target
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-service
  
  # Replica bounds
  minReplicas: 3
  maxReplicas: 50
  
  # Metrics
  metrics:
  # CPU-based scaling
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  
  # Memory-based scaling
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  
  # Custom metric from Prometheus
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  
  # Scaling behavior
  behavior:
    # Scale down behavior
    scaleDown:
      # Stabilization window
      stabilizationWindowSeconds: 300
      # Policies evaluated in order
      policies:
      # Scale down by 10% every minute
      - type: Percent
        value: 10
        periodSeconds: 60
      # Scale down by max 4 pods per minute
      - type: Pods
        value: 4
        periodSeconds: 60
      # Use most conservative policy (smallest scale)
      selectPolicy: Min
    
    # Scale up behavior
    scaleUp:
      stabilizationWindowSeconds: 0  # Immediate scale up
      policies:
      # Double pods every 15 seconds
      - type: Percent
        value: 100
        periodSeconds: 15
      # Add max 10 pods every 15 seconds
      - type: Pods
        value: 10
        periodSeconds: 15

---
# VerticalPodAutoscaler for right-sizing
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: api-service-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-service
  updatePolicy:
    # Auto mode: automatically update pod resources
    # "Off" mode just provides recommendations
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: api
      # Constraints on resource adjustments
      minAllowed:
        cpu: 100m
        memory: 128Mi
      maxAllowed:
        cpu: 4
        memory: 8Gi
      # Controlled resource mutation
      controlledResources:
      - cpu
      - memory
      # Pod disruption budget during updates
      controlledValues: RequestsAndLimits
      # Mode for resource updates
      mode: "Auto"

---
# Deployment with resource requests/limits for HPA
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-service
  template:
    metadata:
      labels:
        app: api-service
    spec:
      containers:
      - name: api
        image: gcr.io/myproject/api-service:v2.3.1
        ports:
        - containerPort: 8080
        
        # Required for HPA - requests must be set
        resources:
          requests:
            cpu: 500m
            memory: 512Mi
          limits:
            cpu: 2000m
            memory: 2Gi
        
        livenessProbe:
          httpGet:
            path: /healthz/live
            port: 8080
        readinessProbe:
          httpGet:
            path: /healthz/ready
            port: 8080
```

## CLI Examples (kubectl)

### Basic HPA Operations

```bash
# Create HPA
kubectl apply -f hpa.yaml

# Auto-create HPA from deployment
kubectl autoscale deployment myapp --cpu-percent=70 --min=2 --max=10

# List HPAs
kubectl get hpa
kubectl get hpa -o wide

# Describe HPA
kubectl describe hpa myapp-hpa

# Get HPA YAML
kubectl get hpa myapp-hpa -o yaml

# Get HPA status
kubectl get hpa myapp-hpa -o jsonpath='{.status}'

# Delete HPA
kubectl delete hpa myapp-hpa

# Watch HPA
kubectl get hpa -w
```

### HPA Operations

```bash
# Edit HPA
kubectl edit hpa myapp-hpa

# Patch HPA
kubectl patch hpa myapp-hpa -p '{"spec":{"minReplicas":5}}'

# Set replicas directly (pause autoscaling)
kubectl scale deployment myapp --replicas=10

# Resume autoscaling
kubectl rollout undo deployment myapp

# View HPA recommendations
kubectl get hpa myapp-hpa -o jsonpath='{.status}' | jq
```

### VPA Operations

```bash
# Create VPA (recommendation only)
kubectl apply -f vpa.yaml

# Get VPA recommendations
kubectl get vpa myapp-vpa -o yaml | grep -A 20 "recommendation"

# Update VPA mode
kubectl patch vpa myapp-vpa -p '{"spec":{"updatePolicy":{"updateMode":"Auto"}}}'

# Delete VPA
kubectl delete vpa myapp-vpa
```

## Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| HPA not scaling | No resource requests | Add requests to containers |
| HPA stuck | Deployment conflicts | Check deployment spec |
| VPA not applying | Wrong updateMode | Set updateMode: Auto |
| Cluster not scaling | Quota limit | Check cloud quotas |
| Over-scaling | Metric too aggressive | Adjust target metrics |

## Interview Questions

### Basic Questions

**Q: What is the Horizontal Pod Autoscaler (HPA)?**
A: HPA automatically scales the number of pods in a Deployment or ReplicaSet based on observed CPU utilization, memory usage, or custom metrics.

**Q: How does HPA calculate the number of replicas?**
A: HPA uses the formula: desiredReplicas = ceil(currentReplicas * (currentMetricValue / targetMetricValue))

**Q: What is the difference between HPA and VPA?**
A: HPA scales the number of replicas horizontally. VPA adjusts the CPU and memory resource requests of existing pods vertically.

### Intermediate Questions

**Q: What metrics does HPA support?**
A: HPA supports Resource metrics (CPU, memory), Pod metrics (custom per-pod), Object metrics (external objects), and External metrics (from external systems like Prometheus).

**Q: How do you prevent HPA from scaling down too quickly?**
A: Configure scaleDown stabilizationWindowSeconds in the behavior section. This adds a delay before scaling down, preventing rapid oscillation.

### Advanced Questions

**Q: How does HPA work with Cluster Autoscaler?**
A: HPA scales pods based on metrics. If pods can't be scheduled due to resource constraints, Cluster Autoscaler adds nodes. Together they provide complete autoscaling coverage.

**Q: How would you implement scale-to-zero?**
A: Scale-to-zero requires specific workload types (Knative, KEDA). Standard Kubernetes HPA has minReplicas >= 1. For true scale-to-zero, use Knative Serving or KEDA with specific scalers.

### Autoscaling Comparison

| Type | What Changes | Use Case |
|------|--------------|----------|
| HPA | Replica count | Variable traffic, parallel workloads |
| VPA | Resource requests | Right-sizing, memory optimization |
| Cluster Autoscaler | Node count | Node capacity, cloud costs |
| KEDA | Replica count (event-driven) | Queue-based, metric-driven |
