# Kubernetes Deployments

## Overview

A **Deployment** is a Kubernetes resource that provides declarative updates for Pods and ReplicaSets. It allows you to describe an application's desired state in a manifest, and the Deployment controller will systematically change the actual state to the desired state at a controlled rate. Deployments are the primary mechanism for managing application lifecycle in production environments.

## When To Use

- **Application deployments** requiring rolling updates with zero downtime
- **When you need rollback capabilities** to previous versions
- **When you need to pause and resume** deployments
- **Scaling applications** up or down
- **Blue-green or canary deployments** requiring controlled traffic shifting
- **Any production workload** that needs self-healing and high availability

## When NOT To Use

- **Stateful applications requiring ordered deployment** - use StatefulSets
- **Daemon sets running one pod per node** - use DaemonSets
- **Jobs or batch workloads** - use Jobs or CronJobs
- **When you need exactly-once execution** - use Jobs

## Architecture / How It Works

### Deployment Controller Reconciliation

```
┌──────────────────────────────────────────────────────────────┐
│                   Deployment Controller                        │
├──────────────────────────────────────────────────────────────┤
│  1. Watch Deployment spec for changes                         │
│  2. Generate new ReplicaSet spec (if image/version changed)  │
│  3. Create new ReplicaSet                                     │
│  4. Scale up new ReplicaSet based on strategy                 │
│  5. Scale down old ReplicaSet                                 │
│  6. Record revision in history                                │
│  7. Report status                                             │
└──────────────────────────────────────────────────────────────┘
```

### Strategy Types

| Strategy | Description | Use Case |
|----------|-------------|----------|
| RollingUpdate | Gradually replace pods | Default, zero-downtime updates |
| Recreate | Kill all pods, then create new | Breaking changes, no concurrent runs |
| BlueGreen | Run both versions, switch traffic | Instant rollback capability |

## Real Production Example

### Production API Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
  namespace: production
  labels:
    app: api-service
    team: platform
    cost-center: "12345"
  annotations:
    kubernetes.io/change-cause: "Update to v2.3.1 - Performance improvements"
spec:
  # Replica count
  replicas: 5
  
  # Rolling update configuration
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2           # Can have 2 more than desired during update
      maxUnavailable: 0     # Never have fewer than desired ready
  
  # Selector (immutable)
  selector:
    matchLabels:
      app: api-service
  
  # Minimum time before pod is considered ready
  minReadySeconds: 15
  
  # Revision history for rollback
  revisionHistoryLimit: 10
  
  # Pause deployment
  paused: false
  
  # Pod template
  template:
    metadata:
      labels:
        app: api-service
        version: v2.3.1
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
    spec:
      # Service account
      serviceAccountName: api-service-account
      
      # Termination
      terminationGracePeriodSeconds: 60
      
      # Security context
      securityContext:
        runAsNonRoot: true
        runAsUser: 10000
        runAsGroup: 10000
        fsGroup: 10000
        seccompProfile:
          type: RuntimeDefault
      
      # Topology spread for HA
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: api-service
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: ScheduleAnyway
        labelSelector:
          matchLabels:
            app: api-service
      
      # Anti-affinity to spread across nodes
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - api-service
            topologyKey: kubernetes.io/hostname
      
      # Pod disruption budget
      podDisruptionBudget:
        minAvailable: 3
      
      # Init containers
      initContainers:
      - name: migrations
        image: gcr.io/myproject/api-service:v2.3.1
        command: ["./run-migrations.sh"]
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: api-secrets
              key: database-url
        resources:
          requests:
            cpu: 100m
            memory: 64Mi
          limits:
            cpu: 500m
            memory: 256Mi
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities:
            drop:
            - ALL
        volumeMounts:
        - name: tmp
          mountPath: /tmp
      
      # Main container
      containers:
      - name: api
        image: gcr.io/myproject/api-service:v2.3.1
        imagePullPolicy: IfNotPresent
        ports:
        - name: http
          containerPort: 8080
          protocol: TCP
        - name: grpc
          containerPort: 9090
          protocol: TCP
        
        env:
        - name: NODE_ENV
          value: "production"
        - name: LOG_LEVEL
          value: "info"
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: api-secrets
              key: database-url
        - name: REDIS_HOST
          value: "redis.production.svc.cluster.local"
        - name: METRICS_USERNAME
          valueFrom:
            secretKeyRef:
              name: api-secrets
              key: metrics-user
        - name: METRICS_PASSWORD
          valueFrom:
            secretKeyRef:
              name: api-secrets
              key: metrics-password
        
        resources:
          requests:
            cpu: 500m
            memory: 512Mi
          limits:
            cpu: 2000m
            memory: 2Gi
        
        # Health checks
        livenessProbe:
          httpGet:
            path: /healthz/live
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
          timeoutSeconds: 5
          failureThreshold: 3
        
        readinessProbe:
          httpGet:
            path: /healthz/ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5
          timeoutSeconds: 3
          failureThreshold: 3
          successThreshold: 1
        
        startupProbe:
          httpGet:
            path: /healthz
            port: 8080
          failureThreshold: 30
          periodSeconds: 10
        
        # Lifecycle hooks
        lifecycle:
          preStop:
            exec:
              command: ["/bin/sh", "-c", "sleep 10"]
          postStart:
            exec:
              command: ["/bin/sh", "-c", "echo 'Container started'"]
        
        # Security
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities:
            drop:
            - ALL
          seccompProfile:
            type: RuntimeDefault
        
        # Volume mounts
        volumeMounts:
        - name: tmp
          mountPath: /tmp
        - name: ca-certificates
          mountPath: /etc/ssl/certs
          readOnly: true
        - name: ssl-certs
          mountPath: /etc/ssl/private
          readOnly: true
      
      volumes:
      - name: tmp
        emptyDir:
          medium: Memory
          sizeLimit: 100Mi
      - name: ca-certificates
        configMap:
          name: ca-certificates
      - name: ssl-certs
        secret:
          secretName: api-tls
```

### Canary Deployment with Multiple Deployments

```yaml
# Primary deployment (95% traffic)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
  namespace: production
spec:
  replicas: 19
  selector:
    matchLabels:
      app: api-service
      track: stable
  template:
    metadata:
      labels:
        app: api-service
        track: stable
        version: v2.3.0
    spec:
      containers:
      - name: api
        image: gcr.io/myproject/api-service:v2.3.0

---
# Canary deployment (5% traffic)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service-canary
  namespace: production
spec:
  replicas: 1
  selector:
    matchLabels:
      app: api-service
      track: canary
  template:
    metadata:
      labels:
        app: api-service
        track: canary
        version: v2.4.0
    spec:
      containers:
      - name: api
        image: gcr.io/myproject/api-service:v2.4.0

---
# Service with canary weight (using nginx-ingress annotation)
apiVersion: v1
kind: Service
metadata:
  name: api-service
  namespace: production
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "5"
spec:
  selector:
    app: api-service
  ports:
  - port: 80
    targetPort: 8080
```

```bash

kubectl rollout status deployment/api-service

kubectl rollout restart deployment/api-service
```

## Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Deployment stuck | Pod scheduling failure, image pull | Check events, describe pods |
| Rolling update not progressing | Liveness probe failing | Adjust probes, check app health |
| High restart count | OOMKilled, crash loops | Check resource limits, logs |
| Old ReplicaSet not deleted | revisionHistoryLimit too high | Reduce limit, check rollback policy |

## Interview Questions

### Basic Questions

**Q: What is a Kubernetes Deployment?**
A: A Deployment is a resource that provides declarative updates for Pods and ReplicaSets. It allows you to describe an application's desired state and will manage the rollout and rollback of changes systematically.

**Q: What is the difference between RollingUpdate and Recreate strategies?**
A: RollingUpdate gradually replaces old pods with new ones, maintaining availability throughout. Recreate terminates all existing pods before creating new ones, which causes downtime but ensures no version overlap.

**Q: What does maxSurge and maxUnavailable control?**
A: maxSurge is the number of extra pods allowed during updates. maxUnavailable is the number of pods that can be unavailable. Together they control the speed vs. availability tradeoff during rolling updates.

### Intermediate Questions

**Q: How do you perform a rolling update with zero downtime?**
A: Set maxUnavailable: 0 and maxSurge: 1. This ensures all old pods stay up until new pods are ready. Configure proper readiness probes so new pods only receive traffic when healthy.

**Q: How does the deployment controller track revisions?**
A: Each time a deployment is updated, a new ReplicaSet is created and the old one is kept (up to revisionHistoryLimit). The controller stores the revision annotation on the ReplicaSet for tracking.

**Q: What is the purpose of minReadySeconds?**
A: minReadySeconds specifies the minimum time a pod must be running without issues before it's considered ready. This prevents moving to the next pod too quickly in a rolling update.

### Advanced Questions

**Q: How would you implement a canary deployment?**
A: Run two Deployments: stable (majority of replicas) and canary (few replicas). Use Service with label selectors to route most traffic to stable. Use weighted routing (via Ingress or service mesh) for gradual traffic shifting.

**Q: What happens during a Deployment update with HPA enabled?**
A: The HPA and Deployment controller coordinate: HPA may scale up during the update (due to maxSurge), and the rollout proceeds. After the update, HPA adjusts replicas based on actual metrics.

**Q: How do you debug a Deployment that's not progressing?**
A: Check: 1) kubectl rollout status for errors, 2) kubectl describe pods for scheduling/image issues, 3) pod logs for application errors, 4) events for resource issues, 5) readiness/liveness probe configuration.

| Strategy | maxSurge | maxUnavailable | Zero Downtime | Resource Usage |
|----------|----------|----------------|---------------|----------------|
