# Kubernetes Ingress

## Overview

An **Ingress** is a Kubernetes resource that provides HTTP/HTTPS routing to Services based on hostnames and URL paths. It acts as a smart router or entry point for external traffic, enabling features like load balancing, SSL termination, and name-based virtual hosting. Ingress controllers (like nginx-ingress, Traefik, or cloud load balancers) are responsible for fulfilling Ingress resources.

## When To Use

- **HTTP/HTTPS routing** based on hostnames and paths
- **SSL/TLS termination** at the edge
- **Name-based virtual hosting** for multiple applications
- **URL path-based routing** to different services
- **Load balancing** with advanced algorithms
- **Canary deployments** and traffic splitting

## When NOT To Use

- **TCP/UDP traffic** - use Service with LoadBalancer type
- **Non-HTTP protocols** - use port forwarding or LoadBalancer
- **Simple internal routing** - use ClusterIP Services
- **When service mesh** provides better control (use Istio Gateway)

## Architecture / How It Works

### Ingress Controller Types

| Controller | Implementation | SSL/TLS | Features |
|------------|----------------|---------|----------|
| nginx-ingress | NGINX | Terminate/Passthrough | Rate limiting, auth, rewrites |
| GCE L7 | GCP Load Balancer | Managed certs | Cloud-native integration |
| Contour | Envoy proxy | Terminate | Dynamic updates, rate limiting |
| Traefik | Traefik | Terminate | Auto-discovery, letsencrypt |
| Istio Gateway | Envoy proxy | Terminate | Full mesh integration |

### TLS Termination Flow

```
Client (HTTPS)
      │
      │ TLS handshake with certificate
      ▼
┌──────────────────┐
│ Ingress          │
│ Controller       │
│ (Terminates TLS) │
└────────┬─────────┘
         │ HTTP
         ▼
┌──────────────────┐
│ Services → Pods  │
└──────────────────┘
```

## Real Production Example

### Production API Ingress

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  namespace: production
  labels:
    app: api-service
  annotations:
    # nginx-ingress annotations
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"
    nginx.ingress.kubernetes.io/rate-limit: "100"
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "300"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "300"
    nginx.ingress.kubernetes.io/upstream-hash-by: "$remote_addr"
    nginx.ingress.kubernetes.io/affinity: "cookie"
    nginx.ingress.kubernetes.io/affinity-mode: "persistent"
    nginx.ingress.kubernetes.io/session-cookie-name: "route"
    nginx.ingress.kubernetes.io/session-cookie-expires: "172800"
    nginx.ingress.kubernetes.io/session-cookie-change-on-failure: "true"
    
    # GKE specific
    kubernetes.io/ingress.global-static-ip-name: "api-ip"
    networking.gke.io/managed-certificates: "api-cert"
    kubernetes.io/ingress.class: "gce"
spec:
  ingressClassName: nginx
  
  # Default backend (fallback)
  defaultBackend:
    service:
      name: default-backend
      port:
        number: 80
  
  # TLS configuration
  tls:
  - hosts:
    - api.example.com
    - "*.api.example.com"
    secretName: api-tls-secret
  
  # Routing rules
  rules:
  - host: api.example.com
    http:
      paths:
      # API v1
      - path: /v1/users
        pathType: Prefix
        backend:
          service:
            name: users-service
            port:
              number: 8080
      - path: /v1/products
        pathType: Prefix
        backend:
          service:
            name: products-service
            port:
              number: 8080
      - path: /v1/orders
        pathType: Prefix
        backend:
          service:
            name: orders-service
            port:
              number: 8080
      
      # Health check endpoint
      - path: /health
        pathType: Exact
        backend:
          service:
            name: health-service
            port:
              number: 8080
      
      # API docs
      - path: /docs
        pathType: Prefix
        backend:
          service:
            name: docs-service
            port:
              number: 8080
      
      # Default: API service
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080

---
# Default backend service
apiVersion: v1
kind: Service
metadata:
  name: default-backend
  namespace: production
spec:
  selector:
    app: default-backend
  ports:
  - port: 80
    targetPort: 8080

---
# TLS Secret
apiVersion: v1
kind: Secret
metadata:
  name: api-tls-secret
  namespace: production
type: kubernetes.io/tls
data:
  # Base64 encoded cert and key
  tls.crt: LS0tLS1...
  tls.key: LS0tLS1...
```

### Canary Deployment with Ingress

```yaml
# Production ingress (90% traffic)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  namespace: production
  annotations:
    nginx.ingress.kubernetes.io/canary: "false"
spec:
  ingressClassName: nginx
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service-v1
            port:
              number: 8080

---
# Canary ingress (10% traffic)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress-canary
  namespace: production
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"
    nginx.ingress.kubernetes.io/canary-by-header: "X-Canary"
    nginx.ingress.kubernetes.io/canary-by-header-value: "always"
spec:
  ingressClassName: nginx
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service-v2
            port:
              number: 8080
```

## Interview Questions

### Basic Questions

**Q: What is an Ingress in Kubernetes?**
A: An Ingress is a Kubernetes resource that provides HTTP/HTTPS routing to Services based on hostnames and URL paths. It requires an Ingress controller to fulfill the Ingress rules.

**Q: What is an Ingress Controller?**
A: An Ingress controller is a pod that watches the API server for Ingress resources and configures a web server (like NGINX, Traefik, or Envoy) to route traffic accordingly.

**Q: What is the difference between Ingress and Service?**
A: A Service provides layer-4 load balancing within the cluster. An Ingress provides layer-7 HTTP/HTTPS routing based on hostnames and paths, with SSL termination capabilities.

### Intermediate Questions

**Q: How do you configure TLS in an Ingress?**
A: Create a TLS Secret with the certificate and key, then reference it in the Ingress spec's tls section with the hosts. The Ingress controller terminates SSL.

**Q: What is the purpose of pathType in Ingress?**
A: pathType determines how the path is matched: ImplementationSpecific (controller-dependent), Exact (exact match), or Prefix (URL prefix match).

### Advanced Questions

**Q: How would you implement canary deployments with Ingress?**
A: Create two Ingress resources: the main one and a canary one with canary annotations (canary-weight, canary-by-header). Route a percentage of traffic to the canary version.
