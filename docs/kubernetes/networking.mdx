# Kubernetes Networking

## Overview

Kubernetes networking defines how pods communicate with each other, with services, and with external resources. It encompasses the Container Network Interface (CNI), service mesh, DNS resolution, Ingress controllers, and NetworkPolicies for security. Understanding networking is essential for debugging connectivity issues and designing scalable architectures.

## When to Use

- **Designing multi-tier applications** - frontend, backend, database communication
- **Debugging connectivity issues** - pod-to-pod, service-to-service
- **Implementing network security** - isolating workloads with NetworkPolicies
- **Configuring load balancing** - internal and external traffic
- **Setting up service discovery** - DNS-based naming

## When NOT to Use

- **Simple single-node setups** - networking is complex, start simple
- **When managed solutions exist** - GKE handles much networking automatically

## Architecture / How It Works

### CNI Plugin Architecture

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        CNI Plugin Flow                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Pod Creation:                                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  1. kubelet requests CNI plugin for network setup                    │   │
│  │  2. CNI plugin creates network namespace for pod                     │   │
│  │  3. CNI plugin creates veth pair (virtual ethernet)                  │   │
│  │  4. One end goes to pod, other to bridge                             │   │
│  │  5. CNI plugin assigns IP address to pod                             │   │
│  │  6. CNI plugin configures routes                                     │   │
│  │  7. CNI plugin configures DNS via /etc/resolv.conf                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  Common CNI Plugins:                                                         │
│  - Calico: BGP, network policies, overlay                                   │
│  - Cilium: eBPF-based, L7 filtering                                         │
│  - Flannel: simple overlay network                                          │
│  - Weave: overlay with encryption                                           │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### NetworkPolicies for Production

```yaml
# Frontend namespace
apiVersion: v1
kind: Namespace
metadata:
  name: frontend
  labels:
    tier: frontend

---
# Frontend to API communication
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: frontend-egress
  namespace: frontend
spec:
  podSelector:
    matchLabels:
      app: frontend
  policyTypes:
  - Egress
  egress:
  # Allow DNS
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
  # Allow traffic to API service in backend namespace
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: backend
      podSelector:
        matchLabels:
          app: api
    ports:
    - protocol: TCP
      port: 8080

---
# Backend namespace
apiVersion: v1
kind: Namespace
metadata:
  name: backend
  labels:
    tier: backend

---
# Backend default deny
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-default-deny
  namespace: backend
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

---
# API ingress rules
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-ingress
  namespace: backend
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  ingress:
  # Allow from frontend namespace
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  # Allow DNS
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - protocol: UDP
      port: 53
  # Allow to database
  - to:
    - podSelector:
        matchLabels:
          app: postgres
    ports:
    - protocol: TCP
      port: 5432
  # Allow to Redis
  - to:
    - podSelector:
        matchLabels:
          app: redis
    ports:
    - protocol: TCP
      port: 6379
```

### Internal Load Balancer (GKE)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-service-internal
  namespace: production
  annotations:
    # GCP Internal Load Balancer
    networking.gke.io/load-balancer-type: "Internal"
spec:
  type: LoadBalancer
  selector:
    app: api-service
  ports:
  - name: http
    port: 80
    targetPort: 8080
    protocol: TCP
  # Preserve client source IP
  externalTrafficPolicy: Local
  healthCheckNodePort: 30000
```

## Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Pod can't reach service | Wrong DNS, missing endpoints | Check kube-proxy, service selector |
| Service has no endpoints | Selector doesn't match pods | Check pod labels |
| External traffic blocked | Firewall rules, NodePort | Check cloud firewall |
| DNS not resolving | CoreDNS issues | Check CoreDNS pods |
| Network policy blocking | Policy too restrictive | Add appropriate rules |

## Interview Questions

### Basic Questions

**Q: How does Kubernetes networking work?**
A: Every pod gets a unique IP address. Pods on the same node communicate via a bridge. Pods on different nodes communicate via an overlay network managed by the CNI plugin. Services provide stable virtual IPs that load-balance to pods.

**Q: What is the difference between ClusterIP, NodePort, and LoadBalancer?**
A: ClusterIP is internal-only. NodePort exposes on each node's IP at a static port. LoadBalancer provisions an external load balancer (cloud-specific).

**Q: What does kube-proxy do?**
A: kube-proxy maintains network rules on each node that forward traffic to service endpoints. It uses iptables or IPVS for load balancing.

### Intermediate Questions

**Q: How do NetworkPolicies work?**
A: NetworkPolicies are namespace-scoped. They filter ingress and egress traffic based on pod selectors, namespace selectors, and ports. The CNI plugin enforces these rules.

**Q: How does DNS work in Kubernetes?**
A: CoreDNS watches the API server for Services and creates DNS records. Pod DNS: pod-ip.namespace.pod.cluster.local. Service DNS: service-name.namespace.svc.cluster.local.

**Q: What is a Headless Service?**
A: A Headless Service has clusterIP: None. DNS returns pod IPs directly instead of a service IP. Useful for stateful applications needing direct pod access.

### Advanced Questions

**Q: How would you troubleshoot pod-to-pod connectivity issues?**
A: 1) Check pod IPs, 2) Verify CNI plugin is working, 3) Check network policies, 4) Test with busybox pod, 5) Check kube-proxy, 6) Verify routing tables.

**Q: How does IPVS mode differ from iptables mode in kube-proxy?**
A: IPVS uses the Linux Virtual Server for O(1) load balancing regardless of service count. iptables uses probability rules with O(n) lookup. IPVS supports more scheduling algorithms.
