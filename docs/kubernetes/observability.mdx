# Kubernetes Observability

## Overview

Kubernetes observability encompasses the collection, analysis, and visualization of metrics, logs, and traces to understand cluster and application behavior. A comprehensive observability stack includes Prometheus for metrics, Loki or Elasticsearch for logs, and Jaeger or Tempo for distributed tracing.

## When to Use

- **Production debugging** - understanding failures and performance issues
- **Capacity planning** - right-sizing resources
- **Incident response** - rapid root cause analysis
- **Performance optimization** - identifying bottlenecks
- **SLO/SLA monitoring** - tracking service levels

## When NOT to Use

- **Development clusters** - focus on functionality first
- **Simple applications** - may not need full stack

## Real Production Example

### Monitoring Stack Setup (Prometheus Operator)

```yaml
# Prometheus Operator CRDs - typically installed via OperatorHub
# This is an example of the configuration

# ServiceMonitor for applications
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: api-service-monitor
  namespace: production
  labels:
    app: api-service
    release: prometheus
spec:
  selector:
    matchLabels:
      app: api-service
  namespaceSelector:
    matchNames:
    - production
  endpoints:
  - port: metrics
    path: /metrics
    interval: 15s
    scrapeTimeout: 10s
  - port: grpc-metrics
    path: /grpc/metrics
    interval: 30s

---
# PodMonitor for pod metrics
apiVersion: monitoring.coreos.com/v1
kind: PodMonitor
metadata:
  name: api-service-pod-monitor
  namespace: production
  labels:
    app: api-service
spec:
  selector:
    matchLabels:
      app: api-service
  podMetricsEndpoints:
  - port: metrics
    path: /metrics
    interval: 15s

---
# PrometheusRule for alerts
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: api-service-alerts
  namespace: production
  labels:
    app: api-service
    release: prometheus
spec:
  groups:
  - name: api-service
    rules:
    # High error rate
    - alert: APIHighErrorRate
      expr: |
        sum(rate(http_requests_total{service="api-service", status=~"5.."}[5m])) 
        / 
        sum(rate(http_requests_total{service="api-service"}[5m])) > 0.05
      for: 5m
      labels:
        severity: critical
      annotations:
        summary: "API service high error rate"
        description: "API service error rate is above 5%"
    
    # High latency
    - alert: APIHighLatency
      expr: |
        histogram_quantile(0.99, 
          sum(rate(http_request_duration_seconds_bucket{service="api-service"}[5m])) 
          by (le)
        ) > 2
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "API service high latency"
        description: "API service p99 latency is above 2 seconds"
    
    # Pod not ready
    - alert: APIPodNotReady
      expr: |
        kube_pod_status_ready{namespace="production", pod=~"api-service-.*", condition="true"} == 0
      for: 10m
      labels:
        severity: warning
      annotations:
        summary: "API pods not ready"
        description: "API service pods have been not ready for more than 10 minutes"

---

# AlertManager configuration
apiVersion: monitoring.coreos.com/v1
kind: AlertmanagerConfig
metadata:
  name: api-service-alerts
  namespace: monitoring
spec:
  receivers:
  - name: default
    slackConfigs:
    - channel: "#alerts-api"
      apiUrl:
        name: slack-webhook
        key: url
      sendResolved: true
      title: |
        {{ if eq .Status "firing" }}:fire: API Service Alert{{ else }}:white_check_mark: API Service Resolved{{ end }}
      text: |
        {{ range .Alerts }}
        *Alert:* {{ .Labels.alertname }}
        *Severity:* {{ .Labels.severity }}
        *Description:* {{ .Annotations.description }}
        *Details:* {{ range .Labels.SortedPairs }} • {{ .Name }}: {{ .Value }}{{ end }}
        {{ end }}
  
  route:
    receiver: default
    groupBy: ['alertname', 'severity']
    groupWait: 30s
    groupInterval: 5m
    repeatInterval: 4h
    matchers:
    - severity = critical
```

### Prometheus Query Examples

```promql
# Pod CPU usage
sum(rate(container_cpu_usage_seconds_total{pod=~"api-.*"}[5m])) by (pod)

# Pod memory usage
sum(container_memory_working_set_bytes{pod=~"api-.*"}) by (pod)

# Request rate by status
sum(rate(http_requests_total{service="api"}[5m])) by (status)

# Error rate
sum(rate(http_requests_total{service="api",status=~"5.."}[5m])) / sum(rate(http_requests_total{service="api"}[5m]))

# Latency percentiles
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket{service="api"}[5m])) by (le))
```

## Interview Questions

### Basic Questions

**Q: What are the three pillars of observability?**
A: Metrics (quantitative measurements), Logs (event records), and Traces (request flow tracking). Together they provide comprehensive visibility into system behavior.

**Q: How does Prometheus work with Kubernetes?**
A: Prometheus uses Kubernetes service discovery to find targets (pods, services). It scrapes metrics endpoints based on ServiceMonitor or pod annotations. Data is stored in TSDB and queried via PromQL.

**Q: What is the difference between Pushgateway and Prometheus scraping?**
A: Prometheus typically pulls metrics from targets. Pushgateway is used for short-lived jobs that can't be scraped, allowing them to push metrics.

### Intermediate Questions

**Q: How would you set up alerting for a Kubernetes application?**
A: Define PrometheusRules with alert expressions, route alerts to AlertManager, configure receivers (Slack, PagerDuty, Email), and create Grafana dashboards for visualization.

**Q: What is the purpose of ServiceMonitor in Prometheus Operator?**
A: ServiceMonitor defines which services to scrape and how. The Prometheus Operator automatically configures Prometheus to scrape the defined endpoints.

### Advanced Questions

**Q: How do you handle high cardinality metrics?**
A: Use appropriate label values, avoid labels with unbounded cardinality (user IDs, request IDs), aggregate before scraping, and use recording rules.

**Q: How would you implement distributed tracing in Kubernetes?**
A: Deploy a tracing backend (Jaeger, Tempo), instrument applications with OpenTelemetry or similar, propagate trace context via headers, and create traces for service-to-service calls.
