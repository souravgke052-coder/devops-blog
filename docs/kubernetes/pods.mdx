# Kubernetes Pods

## Overview

A **Pod** is the smallest deployable unit in Kubernetes. It represents a single instance of a running process in the cluster and can contain one or more containers that share network, storage, and process namespace. Pods are ephemeral by design - they are created, destroyed, and recreated based on workload requirements.

## When To Use

- **Running a single container** that needs to be managed by Kubernetes
- **Running multiple containers** that must work together and share resources (sidecars, adapters, init containers)
- **Jobs and batch processing** where each item is processed by a pod
- **Stateful applications** requiring stable network identity and persistent storage
- **Any workload** that needs Kubernetes scheduling, scaling, or self-healing

## When NOT To Use

- **Running multiple unrelated containers** - use separate pods
- **Replacing a full container orchestration** - Pods don't handle cross-pod coordination directly
- **Long-running services without replication** - use Deployments for self-healing

## Real Production Example

### Production API Pod with All Features

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: api-service-pod
  namespace: production
  labels:
    app: api-service
    version: v2.3.1
    team: platform
    environment: production
  annotations:
    prometheus.io/scrape: "true"
    prometheus.io/port: "8080"
    prometheus.io/path: "/metrics"
spec:
  # Service account for RBAC
  serviceAccountName: api-service-account
  
  # TerminationgracePeriodSeconds for graceful shutdown
  terminationGracePeriodSeconds: 60
  
  # Restart policy
  restartPolicy: Always
  
  # Security context at pod level
  securityContext:
    runAsNonRoot: true
    runAsUser: 10000
    runAsGroup: 10000
    fsGroup: 10000
    seccompProfile:
      type: RuntimeDefault
  
  # Init containers
  initContainers:
  - name: migrations
    image: gcr.io/myproject/api-service:v2.3.1
    command: ["./run-migrations.sh"]
    env:
    - name: DATABASE_URL
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: database-url
    resources:
      requests:
        cpu: 100m
        memory: 64Mi
      limits:
        cpu: 500m
        memory: 256Mi
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL
    volumeMounts:
    - name: tmp
      mountPath: /tmp
  
  # Main containers
  containers:
  # Sidecar: log aggregator
  - name: log-aggregator
    image: fluent/fluent-bit:2.1.0
    resources:
      requests:
        cpu: 50m
        memory: 32Mi
      limits:
        cpu: 100m
        memory: 64Mi
    env:
    - name: FLUENTD_POS_FILE
      value: /var/log/fluentd-pos.txt
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log
    - name: fluentd-config
      mountPath: /fluentd/etc
  
  # Main application container
  - name: api
    image: gcr.io/myproject/api-service:v2.3.1
    imagePullPolicy: IfNotPresent
    ports:
    - name: http
      containerPort: 8080
      protocol: TCP
    - name: grpc
      containerPort: 9090
      protocol: TCP
    
    env:
    - name: NODE_ENV
      value: "production"
    - name: LOG_LEVEL
      value: "info"
    - name: DATABASE_URL
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: database-url
    - name: REDIS_HOST
      value: "redis.production.svc.cluster.local"
    - name: REDIS_PASSWORD
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: redis-password
    - name: API_KEY
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: api-key
    - name: AWS_ACCESS_KEY_ID
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: aws-access-key
    - name: AWS_SECRET_ACCESS_KEY
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: aws-secret-key
    - name: METRICS_USERNAME
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: metrics-user
    - name: METRICS_PASSWORD
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: metrics-password
    
    resources:
      requests:
        cpu: 500m
        memory: 512Mi
      limits:
        cpu: 2000m
        memory: 2Gi
    
    livenessProbe:
      httpGet:
        path: /healthz/live
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
      timeoutSeconds: 5
      failureThreshold: 3
    
    readinessProbe:
      httpGet:
        path: /healthz/ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      timeoutSeconds: 3
      failureThreshold: 3
      successThreshold: 1
    
    startupProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 0
      periodSeconds: 5
      timeoutSeconds: 3
      failureThreshold: 30

      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 10 && nginx -s quit"]
      postStart:
        exec:
          command: ["/bin/sh", "-c", "echo 'Container started'"]
    
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL
      seccompProfile:
        type: RuntimeDefault
    
    volumeMounts:
    - name: tmp
      mountPath: /tmp
    - name: shared-logs
      mountPath: /var/log/api
    - name: ca-certificates
      mountPath: /etc/ssl/certs

  topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels:
        app: api-service
  - maxSkew: 1
    topologyKey: kubernetes.io/hostname
    whenUnsatisfiable: ScheduleAnyway
    labelSelector:
      matchLabels:
        app: api-service
  
  # Anti-affinity for spreading
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - api-service
        topologyKey: kubernetes.io/hostname
  
  volumes:
  - name: tmp
    emptyDir:
      medium: Memory
      sizeLimit: 100Mi
  - name: shared-logs
    emptyDir: {}
  - name: fluentd-config
    configMap:
      name: fluentd-config
  - name: ca-certificates
    configMap:
      name: ca-certificates
```

### QoS Classes

| QoS Class | Criteria | Behavior |
|-----------|----------|----------|
| Guaranteed | All containers have requests AND limits, and they match | Last to be evicted |
| Burstable | At least one container has requests OR limits | Evicted after Guaranteed |
| BestEffort | No resource requests or limits | First to be evicted |

```yaml
# Guaranteed QoS (requests == limits for all containers)
resources:
  requests:
    memory: "128Mi"
    cpu: "100m"
  limits:
    memory: "128Mi"
    cpu: "100m"

# Burstable QoS (requests != limits or partial)
resources:
  requests:
    memory: "64Mi"
    cpu: "50m"
  limits:
    memory: "128Mi"
    cpu: "200m"

# BestEffort (no resources specified)
# Not recommended for production
```

## Troubleshooting

### Common Pod Issues

| Issue | Symptoms | Cause | Solution |
|-------|----------|-------|----------|
| ImagePullBackOff | Pod stuck | Image not found, pull secret missing | Check image, credentials |
| CrashLoopBackOff | Pod restarting repeatedly | Application error, bad health check | Check logs, probe settings |
| OOMKilled | Pod killed | Memory limit exceeded | Increase limit, optimize app |
| Terminated | Pod completed | Job finished, init failure | Check logs |
| Pending forever | Pod not scheduled | Resource shortage, affinity | Check resources, relax constraints |

## Interview Questions

### Basic Questions

**Q: What is a Pod in Kubernetes?**
A: A Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process and can contain one or more containers that share network, storage, and process namespace.

**Q: What are the different phases of a Pod?**
A: Pod phases are: Pending (scheduler processing), Running (containers running), Succeeded (completed successfully, not restarted), Failed (all containers terminated with failure), and Unknown (node communication lost).

**Q: What is the difference between init containers and regular containers?**
A: Init containers run before regular containers and must complete successfully before regular containers start. They are useful for setup tasks like database migrations. Regular containers run concurrently and represent the main workload.

### Intermediate Questions

**Q: What is the purpose of liveness, readiness, and startup probes?**
A: Liveness probe determines if a container is alive (restart if failing). Readiness probe determines if a container can serve traffic (remove from Service endpoints if failing). Startup probe allows slow-starting containers time to initialize before other probes run.

**Q: How do Pods communicate within the same node?**
A: Pods on the same node communicate via the loopback interface (localhost) if they share a network namespace. Inter-node communication goes through the node's network interface and the CNI plugin.

**Q: What happens when a Pod is terminated?**
A: When terminated: SIGTERM is sent to containers (grace period starts), application has time to shut down gracefully, SIGKILL is sent after grace period, kubelet reports termination to API server, and Pod is removed from endpoints.

### Advanced Questions

**Q: How does Kubernetes handle Pod scheduling with resource constraints?**
A: The scheduler evaluates resource requests (not limits) when placing pods. It ensures requested resources are available on the target node. If insufficient resources exist, pods remain Pending. Limits are enforced at runtime by the kubelet.

**Q: Explain Pod priority and preemption.**
A: Pods can have priority classes that determine their scheduling order. Higher priority pods are scheduled first. If a node lacks resources, Kubernetes can preempt (evict) lower priority pods to make room for higher priority ones.

**Q: How do you handle graceful shutdown of applications in Pods?**
A: Configure terminationGracePeriodSeconds appropriately. Use preStop hook for non-graceful shutdown needs. Implement SIGTERM handling in application. For Kubernetes 1.21+, endpoints are removed immediately on termination signal, stopping new traffic before container stop.

### Pod YAML Template

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-name
  namespace: default
spec:
  # Service account
  serviceAccountName: default
  
  # Restart policy: Always, OnFailure, Never
  restartPolicy: Always
  
  # Security context
  securityContext:
    runAsNonRoot: true
    runAsUser: 10000
  
  # Init containers
  initContainers:
  - name: init
    image: busybox:latest
    command: ['sh', '-c', 'echo init']
  
  # Main containers
  containers:
  - name: main
    image: nginx:latest
    imagePullPolicy: IfNotPresent
    
    ports:
    - containerPort: 80
      name: http
      protocol: TCP
    
    env:
    - name: ENV
      value: "production"
    
    resources:
      requests:
        memory: "64Mi"
        cpu: "100m"
      limits:
        memory: "128Mi"
        cpu: "200m"
    
    livenessProbe:
      httpGet:
        path: /healthz
        port: 80
      initialDelaySeconds: 30
      periodSeconds: 10
    
    readinessProbe:
      httpGet:
        path: /ready
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 5
    
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
    
    volumeMounts:
    - name: data
      mountPath: /data
  
  volumes:
  - name: data
    emptyDir: {}
```

### Probe Configuration

| Probe | Purpose | Failure Action |
|-------|---------|----------------|
| livenessProbe | Is container alive? | Restart container |
| readinessProbe | Can container serve traffic? | Remove from Service |
| startupProbe | Has container started? | Disable other probes |
