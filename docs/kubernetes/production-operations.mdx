# Kubernetes Production Operations

## Overview

Production Kubernetes operations encompass cluster lifecycle management, workload reliability, disaster recovery, cost optimization, and incident response. These practices ensure high availability, performance, and security in production environments.

## When to Use

- **Production cluster management** - upgrades, scaling, maintenance
- **Disaster recovery planning** - backups, failovers
- **Cost optimization** - right-sizing, scheduling
- **Incident response** - debugging, mitigation
- **Change management** - deployments, rollbacks

## When NOT to Use

- **Development clusters** - full operational suite overkill
- **Simple applications** - basic monitoring sufficient

### Cluster Lifecycle Management

```yaml
# GKE Cluster upgrade configuration
apiVersion: container.cnrm.cloud.google.com/v1beta1
kind: ContainerCluster
metadata:
  name: production-cluster
  namespace: config-management
spec:
  # Network configuration
  networkRef:
    name: production-vpc
  subnetworkRef:
    name: production-subnet
  
  # Release channel (Stable, Regular, Rapid)
  releaseChannel:
    channel: Stable
  
  # Node management
  nodePools:
  - name: system-pool
    locations:
    - us-central1-a
    - us-central1-b
    - us-central1-c
    config:
      machineType: n2-standard-4
      diskSizeGb: 100
      diskType: pd-standard
      preemptible: false
      shieldedInstanceConfig:
        enableSecureBoot: true
        enableIntegrityMonitoring: true
    management:
      autoUpgrade: true
      autoRepair: true
      upgradeSettings:
        maxSurge: 1
        maxUnavailable: 0
    version: 1.28
  
  - name: workload-pool
    locations:
    - us-central1-a
    - us-central1-b
    - us-central1-c
    config:
      machineType: n2-standard-8
      diskSizeGb: 100
      diskType: pd-standard
      preemptible: false
      spot: false
    management:
      autoUpgrade: true
      autoRepair: true
      upgradeSettings:
        maxSurge: 2
        maxUnavailable: 0
    version: 1.28
   Autoscaling:
      autoscalingProfile: BALANCED
  
  - name: spot-pool
    config:
      machineType: n2-standard-8
      preemptible: true
      spot: true
      nodeLocations:
      - us-central1-a
      - us-central1-b
    Autoscaling:
      enableAutoscaling: true
      minNodeCount: 0
      maxNodeCount: 20
  
  # Security
  enablePrivateNodes: true
  masterAuth:
    clusterCaCertificate: <base64-encoded-cert>
    privateEndpoint: <private-endpoint>
    publicEndpoint: <public-endpoint>
  
  # Network policy
  networkPolicy:
    enabled: true
    provider: CALICO
  
  # Workload identity
  workloadIdentity:
    enabled: true
  
  # Vertical Pod Autoscaling
  verticalPodAutoscaling:
    enabled: true
  
  # Maintenance window
  maintenanceWindow:
    startTime: "2024-01-01T02:00:00Z"
    recurrence: FREQ=WEEKLY;BYDAY=SU
  
  # Binary authorization
  binaryAuthorization:
    enabled: true
    defaultAdmissionRule:
      enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG

---
# Node pool upgrade automation
apiVersion: batch/v1
kind: CronJob
metadata:
  name: node-pool-upgrade
  namespace: production
spec:
  schedule: "0 2 * * 0"  # Weekly Sunday 2 AM
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: node-upgrade-sa
          containers:
          - name: upgrade
            image: gcr.io/google.com/cloudsdktool/cloud-sdk:slim
            command:
            - bash
            - -c
            - |
              # Get current versions
              CURRENT_VERSION=$(gcloud container clusters describe production-cluster --region=us-central1 --format='value(currentMasterVersion)')
              
              # Check for available upgrades
              AVAILABLE=$(gcloud container get-server-config --region=us-central1 --format='value(validMasterVersions[0])')
              
              echo "Current: $CURRENT_VERSION, Latest: $AVAILABLE"
              
              # Only upgrade if there are security patches
              if [ "$CURRENT_VERSION" != "$AVAILABLE" ]; then
                echo "Upgrading cluster to $AVAILABLE"
                # In production, this would be a controlled upgrade process
                # gcloud container clusters upgrade production-cluster --region=us-central1
              fi
            resources:
              requests:
                cpu: 100m
                memory: 128Mi
          restartPolicy: OnFailure

---
# PodDisruptionBudget for critical services
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-service-pdb
  namespace: production
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: api-service
      tier: backend

---
# Backup schedule for etcd
apiVersion: batch/v1
kind: CronJob
metadata:
  name: etcd-backup
  namespace: kube-system
spec:
  schedule: "0 */4 * * *"  # Every 4 hours
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: etcd-backup-sa
          containers:
          - name: backup
            image: k8s.gcr.io/etcd:3.5.9
            command:
            - /bin/sh
            - -c
            - |
              ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot-$(date +%Y%m%d-%H%M%S).db \
                --endpoints=https://127.0.0.1:2379 \
                --cacert=/etc/kubernetes/pki/etcd/ca.crt \
                --cert=/etc/kubernetes/pki/etcd/server.crt \
                --key=/etc/kubernetes/pki/etcd/server.key
              
              # Upload to GCS
              gsutil cp /backup/etcd-snapshot-*.db gs://my-bucket/etcd-backups/
              
              # Clean old backups (keep last 7 days)
              find /backup -name "etcd-snapshot-*.db" -mtime +7 -delete
            volumeMounts:
            - name: etcd-data
              mountPath: /var/lib/etcd
            - name: backup
              mountPath: /backup
          volumes:
          - name: etcd-data
            hostPath:
              path: /var/lib/etcd
          - name: backup
            emptyDir: {}
          hostNetwork: true
          securityContext:
            runAsUser: 0
          tolerations:
          - key: node-role.kubernetes.io/control-plane
            operator: Exists
          restartPolicy: OnFailure
```

### Disaster Recovery Runbook

```yaml
# Application backup cronjob
apiVersion: batch/v1
kind: CronJob
metadata:
  name: app-backup
  namespace: production
spec:
  schedule: "0 3 * * *"  # Daily at 3 AM
  jobTemplate:
    spec:
      ttlSecondsAfterFinished: 3600
      template:
        spec:
          serviceAccountName: backup-sa
          containers:
          - name: backup
            image: gcr.io/myproject/backup-tool:v1.0
            env:
            - name: BACKUP_DEST
              value: gs://my-bucket/app-backups
            - name: DB_HOST
              value: postgres.production.svc.cluster.local
            - name: DB_NAME
              valueFrom:
                configMapKeyRef:
                  name: backup-config
                  key: db-name
            command:
            - /bin/sh
            - -c
            - |
              # Backup database
              PGPASSWORD=$DB_PASSWORD pg_dump -h $DB_HOST -U $DB_USER -d $DB_NAME > /backup/db-$(date +%Y%m%d-%H%M%S).sql
              
              # Upload to GCS
              gsutil rsync -r /backup gs://$BACKUP_DEST/
              
              # Upload to Cloud Storage with lifecycle
              gsutil defstorageclass set NEARLINE gs://$BACKUP_DEST/
            resources:
              requests:
                cpu: 500m
                memory: 512Mi
              limits:
                cpu: 1000m
                memory: 1Gi
          restartPolicy: OnFailure
```

### Cost Optimization Automation

```yaml
# Resource usage optimization cronjob
apiVersion: batch/v1
kind: CronJob
metadata:
  name: resource-optimizer
  namespace: production
spec:
  schedule: "0 6 * * 1"  # Weekly Monday 6 AM
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: cost-optimizer-sa
          containers:
          - name: optimizer
            image: bitnami/kubectl:latest
            command:
            - /bin/sh
            - -c
            - |
              # Get pods without resource requests
              echo "=== Pods without resource requests ==="
              kubectl get pods -A -o json | jq -r '.items[] | select(.spec.containers[].resources.requests == null) | "\(.metadata.namespace)/\(.metadata.name)"'
              
              # Get oversized PVCs (> 80% unused)
              echo "=== Oversized PVCs ==="
              kubectl get pvc -A -o json | jq -r '.items[] | select(.status.capacity != null) | {name: .metadata.name, requested: .spec.resources.requests.storage, used: .status.capacity}'
              
              # Check for idle deployments
              echo "=== Idle deployments (0 replicas) ==="
              kubectl get deployments -A -o json | jq -r '.items[] | select(.spec.replicas == 0) | "\(.metadata.namespace)/\(.metadata.name)"'
          restartPolicy: OnFailure
```

## Troubleshooting

### Common Production Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Node NotReady | Kubelet issue, resource exhaustion | Check kubelet, SSH node |
| Pod stuck terminating | Finalizers, volume mounts | Check mounts, force delete |
| API server slow | etcd issues, resource limits | Check etcd, scale up |
| High pod restart | OOMKilled, crash loops | Check logs, limits |
| Service unreachable | Endpoint issues | Check pods, selectors |

## Interview Questions

### Basic Questions

**Q: How do you upgrade a Kubernetes cluster?**
A: Control plane first (managed service handles this for GKE/EKS), then node pools. For node upgrades, cordon/drain nodes, upgrade, then uncordon. Use rolling update strategy with PDBs.

**Q: What is a PodDisruptionBudget?**
A: A PDB ensures minimum number of pods available during voluntary disruptions (drains, upgrades). It prevents too many pods being evicted simultaneously.

**Q: How do you handle etcd backup?**
A: Use etcdctl snapshot save regularly (every few hours). Store in multiple locations. Test restore periodically. In GKE, managed etcd handles backups.

### Intermediate Questions

**Q: How would you implement zero-downtime deployment?**
A: Use RollingUpdate with maxUnavailable: 0 and maxSurge: 1. Ensure proper readiness probes. Use PDBs for critical services. Test with canary before full rollout.

**Q: What is the difference between on-demand and maintenance windows?**
A: On-demand upgrades happen immediately. Maintenance windows schedule upgrades during low-traffic periods. Use maintenance windows for production clusters.

### Advanced Questions

**Q: How would you recover from a failed cluster upgrade?**
A: Roll back to previous version (if available), restore etcd from backup, redeploy workloads from manifests in git, verify state.
