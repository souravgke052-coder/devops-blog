# Kubernetes RBAC

## Overview

**RBAC (Role-Based Access Control)** is a Kubernetes mechanism for regulating access to cluster resources based on the roles and permissions assigned to users, groups, and service accounts. RBAC uses four key concepts: Subjects (who), Roles/ClusterRoles (what permissions), RoleBindings/ClusterRoleBindings (how they're connected), and Resources (what they affect).

## When to Use

- **Controlling access** to Kubernetes resources
- **Implementing least privilege** for users and services
- **Managing multi-tenant clusters** with namespace isolation
- **API access control** for CI/CD pipelines
- **Separation of duties** between teams
- **Audit compliance** requirements

## When NOT to Use

- **Simple local clusters** with single user
- **Testing environments** where quick access is needed
- **When other auth methods** are more appropriate (OAuth, OIDC)

## Core Concepts

### RBAC Model

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           RBAC Architecture                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Subjects (WHO)                                                             │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐                   │
│  │    User       │  │    Group      │  │ServiceAccount │                   │
│  │  (external)   │  │  (external)   │  │  (internal)   │                   │
│  └───────────────┘  └───────────────┘  └───────────────┘                   │
│                                                                              │
│  Roles/ClusterRoles (WHAT)                                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Permissions: get, list, watch, create, update, patch, delete       │   │
│  │  Resources: pods, services, deployments, configmaps, secrets        │   │
│  │  API Groups: "" (core), apps, batch, networking.k8s.io              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  RoleBindings/ClusterRoleBindings (HOW)                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Connects Subject → Role                                            │   │
│  │  RoleBinding: Namespace-scoped                                      │   │
│  │  ClusterRoleBinding: Cluster-wide                                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Role vs ClusterRole

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Role vs ClusterRole                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Role (Namespace-scoped)                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  apiVersion: rbac.authorization.k8s.io/v1                           │   │
│  │  kind: Role                                                          │   │
│  │  metadata:                                                           │   │
│  │    namespace: production                                            │   │
│  │  rules:                                                              │   │
│  │  - apiGroups: [""]                                                   │   │
│  │    resources: ["pods"]                                               │   │
│  │    verbs: ["get", "list"]                                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│  → Grants access within a specific namespace                                │
│                                                                              │
│  ClusterRole (Cluster-scoped)                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  apiVersion: rbac.authorization.k8s.io/v1                           │   │
│  │  kind: ClusterRole                                                   │   │
│  │  rules:                                                              │   │
│  │  - apiGroups: [""]                                                   │   │
│  │    resources: ["nodes"]                                              │   │
│  │    verbs: ["get", "list"]                                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│  → Can grant access to:                                                     │
│    - Cluster-scoped resources (nodes, PVs, namespaces)                     │
│    - Namespaced resources across all namespaces                            │
│    - Non-resource URLs (/api, /healthz)                                    │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Architecture / How It Works

### RBAC Resolution Flow

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        RBAC Authorization Flow                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  User makes API request                                                      │
│      │                                                                        │
│      ▼                                                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  kube-apiserver receives request                                     │   │
│  │  - Authenticates user identity (cert, token, OIDC)                  │   │
│  │  - Extracts user, groups, and attributes                            │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│      │                                                                        │
│      ▼                                                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  RBAC Authorization                                                   │   │
│  │  1. Find RoleBindings in namespace (or ClusterRoleBindings)          │   │
│  │  2. Check if user/group matches any subject                          │   │
│  │  3. Check if Role/ClusterRole grants requested permission            │   │
│  │  4. Allow or deny based on match                                     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  Result: ALLOWED or DENIED                                                   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Aggregation

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      ClusterRole Aggregation                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Built-in ClusterRole: view                                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  aggregationRule:                                                   │   │
│  │    clusterRoleSelectors:                                            │   │
│  │    - matchLabels:                                                   │   │
│  │        rbac.example.com/aggregate-to-view: "true"                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  Custom ClusterRole with label:                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  metadata:                                                          │   │
│  │    labels:                                                          │   │
│  │      rbac.example.com/aggregate-to-view: "true"                     │   │
│  │  rules:                                                             │   │
│  │  - apiGroups: ["custom.example.com"]                                │   │
│  │    resources: ["customresources"]                                   │   │
│  │    verbs: ["get", "list"]                                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  Result: "view" ClusterRole automatically includes custom ClusterRole rules │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Real Production Example

### Production RBAC Configuration

```yaml
# Developer Role - read-only access to production
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer-read
  namespace: production
rules:
# Read all resources
- apiGroups: ["", "apps", "batch", "networking.k8s.io"]
  resources: ["*"]
  verbs: ["get", "list", "watch"]
# Read logs
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get", "list"]
# Describe resources (for debugging)
- apiGroups: [""]
  resources: ["pods", "services", "configmaps", "secrets"]
  verbs: ["describe"]
# Get events
- apiGroups: [""]
  resources: ["events"]
  verbs: ["get"]

---
# Developer RoleBinding - bind developers to read role
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developer-read-binding
  namespace: production
subjects:
# Users
- kind: User
  name: john.doe@example.com
  apiGroup: rbac.authorization.k8s.io
- kind: User
  name: jane.smith@example.com
  apiGroup: rbac.authorization.k8s.io
# Groups
- kind: Group
  name: engineering
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer-read
  apiGroup: rbac.authorization.k8s.io

---
# CI/CD Service Account with deployment access
apiVersion: v1
kind: ServiceAccount
metadata:
  name: ci-pipeline
  namespace: production
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: ci-pipeline
  namespace: production
rules:
# Manage deployments
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch", "update", "patch"]
# Manage services
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# Manage configmaps and secrets
- apiGroups: [""]
  resources: ["configmaps", "secrets"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# Manage jobs
- apiGroups: ["batch"]
  resources: ["jobs", "cronjobs"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# Read pods and logs
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
# Create pods/exec
- apiGroups: [""]
  resources: ["pods/exec"]
  verbs: ["create"]
# Events
- apiGroups: [""]
  resources: ["events"]
  verbs: ["get", "list", "watch", "create"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: ci-pipeline-binding
  namespace: production
subjects:
- kind: ServiceAccount
  name: ci-pipeline
  namespace: production
roleRef:
  kind: Role
  name: ci-pipeline
  apiGroup: rbac.authorization.k8s.io

---
# Platform Admin ClusterRole - cluster-wide admin access
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: platform-admin
rules:
# Full access to all namespaced resources in all namespaces
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]
# Read cluster-scoped resources
- apiGroups: [""]
  resources: ["nodes", "namespaces", "persistentvolumes"]
  verbs: ["get", "list", "watch"]
# Manage cluster roles
- apiGroups: ["rbac.authorization.k8s.io"]
  resources: ["clusterroles", "clusterrolebindings"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# Access to non-resource URLs
- nonResourceURLs: ["/api", "/apis", "/metrics", "/healthz", "/version"]
  verbs: ["get"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: platform-admin-binding
subjects:
- kind: Group
  name: platform-team
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: platform-admin
  apiGroup: rbac.authorization.k8s.io

---
# Read-only ClusterRole for cluster monitoring
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-reader
rules:
# Read nodes
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "list", "watch"]
# Read namespaces
- apiGroups: [""]
  resources: ["namespaces"]
  verbs: ["get", "list", "watch"]
# Read persistent volumes
- apiGroups: [""]
  resources: ["persistentvolumes"]
  verbs: ["get", "list", "watch"]
# Read storage classes
- apiGroups: ["storage.k8s.io"]
  resources: ["storageclasses"]
  verbs: ["get", "list", "watch"]
# Read cluster roles and bindings
- apiGroups: ["rbac.authorization.k8s.io"]
  resources: ["clusterroles", "clusterrolebindings"]
  verbs: ["get", "list", "watch"]
# Read all resources (aggregated to this ClusterRole)
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["get", "list", "watch"]
# Access to metrics endpoint
- nonResourceURLs: ["/metrics", "/metrics/prometheus"]
  verbs: ["get"]
```

### Monitoring Role (for Prometheus)

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: prometheus
rules:
# Read pods for service discovery
- apiGroups: [""]
  resources: ["pods", "services", "endpoints", "nodes"]
  verbs: ["get", "list", "watch"]
# Read node metrics via kubelet
- apiGroups: [""]
  resources: ["nodes/metrics", "nodes/proxy"]
  verbs: ["get"]
# Read PVCs
- apiGroups: [""]
  resources: ["persistentvolumeclaims"]
  verbs: ["get", "list"]
# Read ingress
- apiGroups: ["networking.k8s.io"]
  resources: ["ingresses"]
  verbs: ["get", "list", "watch"]
# Read HPA
- apiGroups: ["autoscaling"]
  resources: ["horizontalpodautoscalers"]
  verbs: ["get", "list", "watch"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: prometheus-binding
subjects:
- kind: ServiceAccount
  name: prometheus
  namespace: monitoring
roleRef:
  kind: ClusterRole
  name: prometheus
  apiGroup: rbac.authorization.k8s.io
```

### Pre-defined ClusterRoles

| ClusterRole | Purpose |
|-------------|---------|
| admin | Full access to namespace |
| edit | Read/write except RBAC |
| view | Read-only access |
| cluster-admin | Superuser (cluster-wide) |

### RBAC Recommendations

```bash
# Check for overly permissive roles
kubectl get clusterroles -o json | jq '.items[] | select(.rules[].resources | if type=="array" then . else [.] end | any == "*") | .metadata.name'

# Avoid these patterns:
# - resources: ["*"] with verbs: ["*"] unless necessary
# - apiGroups: ["*"] with resources: ["*"]
```

## Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| 403 Forbidden | No RBAC permission | Add RoleBinding |
| RoleBinding not working | Wrong namespace | Check subject namespace |
| ServiceAccount can't access | Missing binding | Create RoleBinding |
| Cluster-wide access needed | Namespace scope insufficient | Use ClusterRoleBinding |

## Interview Questions

### Basic Questions

**Q: What is Kubernetes RBAC?**
A: Role-Based Access Control in Kubernetes regulates access to resources using Roles, ClusterRoles, RoleBindings, and ClusterRoleBindings. It controls who (Subjects) can perform what actions (Verbs) on which resources (Resources).

**Q: What is the difference between Role and ClusterRole?**
A: Role is namespace-scoped and grants access to namespaced resources. ClusterRole can grant access to both cluster-scoped resources and namespaced resources across all namespaces.

**Q: What is a RoleBinding vs ClusterRoleBinding?**
A: RoleBinding grants permissions defined in a Role or ClusterRole to a user/group within a specific namespace. ClusterRoleBinding grants permissions cluster-wide or to multiple namespaces.

### Intermediate Questions

**Q: What are the available RBAC verbs?**
A: get, list, watch, create, update, patch, delete, deletecollection. Special: bind, escalate for roles.

**Q: How do you grant access to all namespaces?**
A: Use ClusterRoleBinding to bind a ClusterRole to subjects across the cluster, or bind to a Group that exists in all namespaces.

**Q: What is the purpose of aggregation in ClusterRoles?**
A: Aggregation allows custom ClusterRoles to automatically include rules from other ClusterRoles based on labels. This enables composition and reuse of permissions.

### Advanced Questions

**Q: How would you implement least privilege for a CI/CD pipeline?**
A: Create a dedicated ServiceAccount, create a Role with minimal required permissions (only what's needed for deployment), and bind it to the ServiceAccount. Avoid cluster-admin.

**Q: What is the difference between Role and ClusterRole for namespaced resources?**
A: Both can reference namespaced resources. Role grants access within one namespace. ClusterRole grants access across ALL namespaces, which may be more permissive than intended.

**Q: How do you audit RBAC configuration?**
A: Review ClusterRoleBindings for cluster-admin usage, check for overly broad permissions (* on resources/verbs), monitor API server audit logs for authorization failures, and use tools like Polaris or KubeLinter.
