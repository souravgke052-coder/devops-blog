# Kubernetes Secrets

## Overview

A **Secret** is a Kubernetes resource designed to store small amounts of sensitive data such as passwords, OAuth tokens, SSH keys, and TLS certificates. Unlike ConfigMaps, Secrets are intended to hold confidential information and provide additional security measures including encryption at rest and proper RBAC controls.

## When to Use

- **Database credentials** - usernames and passwords
- **API keys and tokens** - authentication credentials
- **TLS certificates** - SSL/TLS termination
- **SSH keys** - git credentials, remote access
- **OAuth tokens** - third-party authentication
- **Docker registry credentials** - image pull secrets

## When NOT to Use

- **Large files or binary data** - use volumes or external storage
- **Non-sensitive configuration** - use ConfigMaps
- **Database connection strings** - consider splitting credentials
- **Configuration data over 1MB** - Secrets are limited to 1MB

## Architecture / How It Works

### Secret Encryption

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Secret Encryption                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  etcd (encrypted at rest)                                                    │
│      │                                                                        │
│      ▼                                                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Secret data (stored as base64)                                      │   │
│  │  Data is NOT encrypted in memory in etcd by default                  │   │
│  │  Encryption at rest via API server encryption configuration          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│      │                                                                        │
│      ▼                                                                        │
│  API Server                                                                  │
│      │                                                                        │
│      ▼                                                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Decrypted data sent to kubelet                                      │   │
│  │  Data in transit: TLS encrypted                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Real Production Example

### Production Database and API Secrets

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: api-secrets
  namespace: production
  labels:
    app: api-service
    environment: production
  annotations:
    description: "Secrets for API service"
type: Opaque
data:
  # Base64 encoded values
  database-url: cG9zdGdyZXM6Ly91c2VyOnBhc3N3b3JkQGRhdGFiYXNlLnN2Yy5jbHVzdGVyLmxvY2Fs
  database-password: cGFzc3dvcmQxMjM=
  redis-password: cmVkaXNwYXNzd29yZA==
  api-key: YXBpa2V5MTIzNDU2Nzg5MA==
  jwt-secret: c2VjcmV0a2V5Zm9yand0dG9rZW5nZW5lcmF0aW9u
  metrics-username: bWV0cmljc3VzZXI=
  metrics-password: bWV0cmljc3Bhc3N3b3Jk

# To decode: echo "base64value" | base64 -d
```

### TLS Secret

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: api-tls-secret
  namespace: production
type: kubernetes.io/tls
data:
  # Base64 encoded certificate and key
  # Generate with: openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes
  # Encode with: cat cert.pem | base64 -w 0
  tls.crt: LS0tLS1CRUdJTiBDRVJUSUZJQ0FURS0tLS0tCk1JSURYVENDQWtXZ0F3SUJBZ0lVRXd...
  tls.key: LS0tLS1CRUdJTiBQUklWQVRFIEtFWS0tLS0tCk1JSUV2UUlCQURBTkJna3Foa2lHOXcwQkFR...

---
# Alternative: Create TLS secret from files
kubectl create secret tls api-tls \
  --cert=path/to/cert.pem \
  --key=path/to/key.pem \
  -n production
```

### Docker Registry Secret

```yaml
apiVersion: v1
kind: Secret
```

## CLI Examples (kubectl)

### Basic Operations

```bash
# Create secret from literal
kubectl create secret generic my-secret \
  --from-literal=username=admin \
  --from-literal=password=secret123

# Create secret from file
kubectl create secret generic my-secret \
  --from-file=username=./username.txt \
  --from-file=password=./password.txt

# Create TLS secret
kubectl create secret tls api-tls \
  --cert=path/to/cert.pem \
  --key=path/to/key.pem

# Create docker config secret
kubectl create secret docker-registry my-registry \
  --docker-username=user \
  --docker-password=token \
  --docker-server=gcr.io

# Create from env file
kubectl create secret generic my-secret \
  --from-env-file=./secrets.env

# Apply from YAML
kubectl apply -f secret.yaml

# List secrets
kubectl get secrets
kubectl get secret
kubectl get secret -n production

# Describe secret (values hidden)
kubectl describe secret my-secret -n production

# Get secret YAML
kubectl get secret my-secret -n production -o yaml

# Delete secret
kubectl delete secret my-secret -n production
```

### Inspecting Secrets

```bash
# Get specific value (decode)
kubectl get secret my-secret -n production -o jsonpath='{.data.password}' | base64 -d

# Get all values
kubectl get secret my-secret -n production -o json

# Check which pods use a secret
kubectl get pods -n production -o json | jq '.items[] | select(.spec.containers[].env[] | select(.valueFrom.secretKeyRef.name=="my-secret")) | .metadata.name'

# List all secrets with types
kubectl get secrets -n production -o custom-columns='NAME:{.metadata.name},TYPE:{.type},AGE:{.metadata.creationTimestamp}'
```

### Updating Secrets

```bash
# Edit secret (will overwrite)
kubectl edit secret my-secret -n production

# Patch secret
kubectl patch secret my-secret -n production -p '{"stringData":{"NEW_KEY":"NEW_VALUE"}}'

# Replace secret
kubectl replace -f updated-secret.yaml
```

## Security

### Secret Encryption at Rest

```yaml
# Encryption configuration (kube-apiserver)
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
- resources:
  - secrets
  - configmaps
  providers:
  # AES-GCM (best)
  - aescbc:
      keys:
      - name: key1
        secret: <base64-encoded-32-byte-key>
  # KMS (recommended for production)
  - kms:
      name: gcp-kms
      endpoint: unix:///var/run/vaultrpc
      cachesize: 10
      timeout: 3s
  # identity (fallback, not encrypted)
  - identity: {}
```

### RBAC for Secrets

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-manager
  namespace: production
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
- apiGroups: [""]
  resources: ["secrets/status"]
  verbs: ["get"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: secret-reader-binding
subjects:
- kind: ServiceAccount
  name: api-service-account
  namespace: production
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
```

### Pod Security with Secrets

```yaml
# Pod with secret volume and limited access
spec:
  serviceAccountName: api-service-account
  securityContext:
    runAsUser: 10000
    runAsGroup: 10000
  
  containers:
  - name: app
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true
    env:
    - name: API_KEY
      valueFrom:
        secretKeyRef:
          name: api-secrets
          key: api-key
  
  volumes:
  - name: secrets
    secret:
      secretName: api-secrets
      defaultMode: 0600
      optional: false  # Pod fails if secret doesn't exist
```

### Security Best Practices

```bash
# 1. Enable encryption at rest
# In kube-apiserver:
--encryption-provider-config=/etc/kubernetes/encryption-config.yaml

# 2. Use RBAC to limit secret access
# Create roles with minimal required permissions

# 3. Use short-lived tokens
# Enable TokenRequest API for ServiceAccount tokens

# 4. Don't commit secrets to git
# Use Sealed Secrets, Vault, or external secret operators

# 5. Rotate secrets regularly
# Automate rotation with operators

# 6. Use separate secrets per environment
# Don't share secrets across environments
```

## Cost Optimization

Secrets have minimal cost impact as they are stored in etcd. Focus on:
- Using fewer, larger secrets vs many small ones (reduces etcd overhead)
- Setting TTL on ServiceAccount tokens
- Cleaning up unused secrets

## Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Secret value is empty | Base64 decode error | Verify encoding |
| Pod can't read secret | Secret doesn't exist | Check secret name |
| ImagePullBackOff | Wrong registry credentials | Verify docker config |
| Invalid certificate | TLS secret misconfigured | Regenerate certificate |
| ServiceAccount token missing | Missing secret annotation | Recreate ServiceAccount |

## Interview Questions

### Basic Questions

**Q: What is a Kubernetes Secret?**
A: A Secret is a Kubernetes resource for storing sensitive data like passwords, tokens, and keys. Secrets provide better security than ConfigMaps through RBAC controls and optional encryption at rest.

**Q: How are Secrets stored in Kubernetes?**
A: Secrets are stored in etcd as base64-encoded strings. They can be encrypted at rest using encryption providers. In transit, data is protected by TLS.

**Q: What are the types of Secrets?**
A: Types include Opaque (generic), TLS, Docker registry, SSH auth, Basic auth, and ServiceAccount tokens.

### Intermediate Questions

**Q: How do you consume Secrets in a Pod?**
A: Secrets can be consumed as environment variables (env or envFrom), volume mounts (files), or imagePullSecrets for container registries.

**Q: How do you secure Secrets?**
A: Enable encryption at rest in etcd, use RBAC to restrict access, use short-lived ServiceAccount tokens, rotate secrets regularly, and use external secret management (Vault, Sealed Secrets).

**Q: What is the difference between stringData and data in a Secret?**
A: data contains base64-encoded values. stringData accepts plain text which is automatically base64-encoded. stringData is write-only in the API (reading returns base64).

### Advanced Questions

**Q: How would you implement secret rotation?**
A: Use external secret management like Vault with the CSI Secret Store driver, or implement a custom operator that watches for secret updates and triggers application reloads.

**Q: How does encryption at rest work for Secrets?**
A: The API server uses an EncryptionConfiguration to determine how to encrypt Secrets before storing in etcd. Providers include AES-CBC, AES-GCM, and KMS plugins.
