# Kubernetes Security

## Overview

Kubernetes security is a multi-layered approach spanning authentication, authorization, network policies, Pod Security Standards, Secrets management, and admission control. A comprehensive security strategy protects the control plane, worker nodes, workloads, and data through defense in depth.

## When to Use

- **Production clusters** - always implement security hardening
- **Multi-tenant environments** - isolate workloads
- **Compliance requirements** - PCI-DSS, SOC2, HIPAA
- **Protecting sensitive data** - secrets, credentials
- **Network segmentation** - isolate workloads

## When NOT to Use

- **Development clusters** - security may impede rapid iteration
- **Testing with containers** - some restrictions may not be needed

### Pod Security Standards

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Pod Security Standards (PSS)                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Privileged (unrestricted)                                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  - No restrictions                                                   │   │
│  │  - Use for infrastructure components only                            │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  Baseline (minimum restrictions)                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  - Disallow privileged pods                                          │   │
│  │  - Restrict hostPath, hostNetwork, hostPID                          │   │
│  │  - Require secure capabilities (drop ALL)                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  Restricted (strictest)                                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  - All baseline + additional restrictions                           │   │
│  │  - Require non-root user                                            │   │
│  │  - Require read-only root filesystem                                 │   │
│  │  - Require seccomp profile                                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Real Production Example

### Production Security Configuration

```yaml
# Namespace with PodSecurity Standards
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    # Enforce baseline, audit restricted, warn restricted
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
---
# Network Policy - Default Deny
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
---
# Network Policy - Allow DNS
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
---
# ServiceAccount with restricted permissions
apiVersion: v1
kind: ServiceAccount
metadata:
  name: api-service-account
  namespace: production
---
# Production Deployment with security hardening
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-service
  template:
    metadata:
      labels:
        app: api-service
    spec:
      serviceAccountName: api-service-account
      
      # Security context at pod level
      securityContext:
        runAsNonRoot: true
        runAsUser: 10000
        runAsGroup: 10000
        fsGroup: 10000
        seccompProfile:
          type: RuntimeDefault
      
      # Topology spread for availability
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: api-service
      
      # Pod disruption budget
      podDisruptionBudget:
        minAvailable: 2
      
      containers:
      - name: api
        image: gcr.io/myproject/api-service:v2.3.1
        imagePullPolicy: Always
        
        ports:
        - containerPort: 8080
          name: http
          protocol: TCP
        - containerPort: 9090
          name: grpc
          protocol: TCP
        
        # Environment variables from secrets
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: api-secrets
              key: database-url
              optional: false
        - name: API_KEY
          valueFrom:
            secretKeyRef:
              name: api-secrets
              key: api-key
        
        # Resource limits (required for security)
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 512Mi
        
        # Security context at container level
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          runAsUser: 10000
          runAsGroup: 10000
          capabilities:
            drop:
            - ALL
          seccompProfile:
            type: RuntimeDefault
        
        # Health checks
        livenessProbe:
          httpGet:
            path: /healthz/live
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
          timeoutSeconds: 5
          failureThreshold: 3
        
        readinessProbe:
          httpGet:
            path: /healthz/ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5
          timeoutSeconds: 3
          failureThreshold: 3
        
        # Volume mounts
        volumeMounts:
        - name: tmp
          mountPath: /tmp
        - name: ca-certificates
          mountPath: /etc/ssl/certs
          readOnly: true
      
      # Init containers
      initContainers:
      - name: check-permissions
        image: busybox:latest
        command: ["sh", "-c", "echo 'Checking permissions'"]
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities:
            drop:
            - ALL
      
      # Ephemeral volumes
      volumes:
      - name: tmp
        emptyDir:
          medium: Memory
          sizeLimit: 100Mi
      - name: ca-certificates
        configMap:
          name: ca-certificates
      
      # Tolerations for tainted nodes
      tolerations:
      - key: "node-role.kubernetes.io/control-plane"
        operator: "Exists"
        effect: "NoSchedule"

---
# RBAC - Role for the service account
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: api-service-role
  namespace: production
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list", "watch"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: api-service-role-binding
  namespace: production
subjects:
- kind: ServiceAccount
  name: api-service-account
  namespace: production
roleRef:
  kind: Role
  name: api-service-role
  apiGroup: rbac.authorization.k8s.io
```

## Interview Questions

### Basic Questions

**Q: What is the principle of least privilege in Kubernetes?**
A: Grant only the minimum permissions required for a user, service, or process to perform its function. Use RBAC to limit access to specific resources and verbs.

**Q: What are Pod Security Standards?**
A: PSS defines three policy levels (Privileged, Baseline, Restricted) that control what security settings pods must or should use. They replace the deprecated PodSecurityPolicy.

**Q: How do NetworkPolicies work?**
A: NetworkPolicies are namespace-scoped resources that filter network traffic using pod selectors, namespace selectors, and port specifications. By default, all traffic is allowed; NetworkPolicies restrict traffic.

### Intermediate Questions

**Q: What is the difference between PodSecurityPolicy and PodSecurity Standards?**
A: PSP was an admission controller (removed in 1.25). PSS uses labels on namespaces to enforce security standards with built-in profiles.

**Q: How do you secure Secrets in Kubernetes?**
A: Enable encryption at rest in etcd, use RBAC to restrict access, set short TTLs on ServiceAccount tokens, use external secrets management (Vault), and rotate secrets regularly.

### Advanced Questions

**Q: How would you implement zero-trust networking in Kubernetes?**
A: Implement default-deny NetworkPolicies, use mutual TLS (mTLS) via service mesh, enforce encryption in transit, apply strict RBAC, and use admission controllers to enforce security policies.
