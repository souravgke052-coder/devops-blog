# CPU Management

## Overview

CPU management encompasses scheduling, affinity, and resource limiting. Linux uses the Completely Fair Scheduler (CFS) for normal processes and real-time scheduling for special cases. Understanding load average, CPU usage metrics, and scheduler behavior is essential for capacity planning and debugging performance issues.

Load average represents the average number of runnable processes over 1, 5, and 15 minutes. A load of 4.0 on a 4-core system means all cores are busy. Load > cores indicates queues. Understanding the difference between CPU utilization (percentage of time executing) and system load (runnable processes) is critical.

## When To Use

- Debugging high load situations
- CPU affinity for performance-sensitive applications
- Setting CPU limits in containers
- Real-time application tuning
- Capacity planning

## When NOT To Use

- When auto-scaling handles capacity (cloud)
- When application handles its own CPU management

## Core Concepts

### Load Average

```bash
# View load
uptime
# 09:23:45 up 45 days, 14:22,  2 users,  load average: 2.45, 1.89, 1.52

# What it means:
# 2.45 = average processes waiting/running in last 1 minute
# 1.89 = last 5 minutes
# 1.52 = last 15 minutes

# Context: 4 cores means:
# 2.45 / 4 = ~61% utilization
```

### Scheduler

```bash
# CFS bandwidth control
# Minimum slice = 1ms
cat /proc/sys/kernel/sched_min_granularity_ns
# 10000000 = 10ms

# Latency target
cat /proc/sys/kernel/sched_latency_ns
# 10000000 = 10ms

# Check scheduler for block device
cat /sys/block/sda/queue/scheduler
# [mq-deadline] kyber bfq none

# Change I/O scheduler
echo mq-deadline | sudo tee /sys/block/sda/queue/scheduler
```

### CPU Affinity

```bash
# View CPU affinity for process
taskset -cp <pid>
# pid <pid>'s current affinity list: 0-7

# Set CPU affinity (pin process to cores)
taskset -cp 0,1 <pid>       # Pin to cores 0 and 1
taskset -c 0-3 <command>    # Run on cores 0-3

# Set specific core
taskset -cp 2 <pid>

# View all processes with affinity
ps -eo pid,psr,comm | sort -k2 -n
# PSR = last CPU the process ran on
```

### CPU Cgroups

```bash
# cgroup v2 CPU limiting (systemd)
# In unit file:
CPUAffinity=0-3             # Pin to cores 0-3
CPUQuota=200%               # Limit to 2 cores (200%)
CPUWeight=512               # Relative weight (default 100)

# Or runtime:
systemctl set-property nginx CPUAffinity=0-3
systemctl set-property nginx CPUQuota=200%

# cgroup v2 files
cat /sys/fs/cgroup/systemd/<cgroup>/cpu.max
# 200000 100000 = 200ms per 100ms = 200% = 2 cores

cat /sys/fs/cgroup/systemd/<cgroup>/cpu.weight
# 100 (default)
```

### Real-Time Scheduling

```bash
# SCHED_FIFO, SCHED_RR, SCHED_BATCH, SCHED_IDLE
chrt -f 50 <command>        # FIFO, priority 50
chrt -r 50 <command>        # Round-robin, priority 50
chrt -p <pid>               # Show current scheduler

# SCHED_FIFO: runs until done or yields
# SCHED_RR: like FIFO but time-quantum based
# Only root can set real-time policies
```

## Real Production Example

Performance tuning a web server application:

```bash
#!/bin/bash
# tune-webserver.sh

set -euo pipefail

# Pin nginx workers to specific cores (NUMA-aware)
# Assuming 8 cores, dedicate 4 to nginx, 2 to PHP-FPM, 1 to system

cat > /etc/nginx/nginx.conf <<'EOF'
worker_processes 4;
worker_cpu_affinity 0001 0010 0100 1000;
worker_rlimit_nofile 65536;

events {
    worker_connections 8192;
    use epoll;
    multi_accept on;
}

thread_pool default threads=16 max_queue=65536;
EOF

# PHP-FPM config
cat > /etc/php-fpm.d/www.conf <<'EOF'
[www]
pm = static
pm.max_children = 50
pm.start_servers = 10
pm.min_spare_servers = 5
pm.max_spare_servers = 15
; Pin to cores 4-5
php_admin_value[cpuset] = 4-5
EOF

# System tuning
cat >> /etc/sysctl.conf <<'EOF'
# Scheduler tuning
kernel.sched_autogroup_enabled = 0    # Disable for server workloads
kernel.sched_migration_cost_ns = 5000000

# CPU isolated from kernel
# Add to grub: isolcpus=2,3,4,5,6,7
EOF

# Set CPU affinity via systemd override
mkdir -p /etc/systemd/system/php-fpm.service.d
cat > /etc/systemd/system/php-fpm.service.d/cpu.conf <<'EOF'
[Service]
CPUAffinity=4-7
CPUQuota=400%
EOF

systemctl daemon-reload
systemctl restart nginx php-fpm
```

```bash
# Basic CPU
uptime
top (press 1 for per-core)

# Per-core usage
mpstat -P ALL 1

# Load average by core

cpupower frequency-info

dmesg | grep -i throttle

```

## Security

**CPU vulnerability mitigations:**

```bash
# Check Meltdown/Spectre mitigation status
cat /sys/devices/system/cpu/vulnerabilities/*

# Disable SMT (Simultaneous Multi-Threading) if needed
# Edit GRUB: isolcpus=0,2,4,6 (disable hyperthreads)
# Or BIOS setting

# Limit CPU for containers
docker run --cpus=2 container    # Limit to 2 cores
docker run --cpuset-cpus=0,1 container  # Pin to cores

# cgroup v2
docker run --cpus=2 container
kubectl set resources container-name --limits=cpu=2
```

## Troubleshooting

**High load but low CPU:**

```bash
# Usually I/O bound
iostat -x 1
# Check for:
# - Disk I/O saturation
# - Network I/O wait
# - Swapping (vmstat 1)

# Check wait I/O
vmstat 1
# wa column = waiting for I/O
```

**One process using all CPU:**

```bash
top   # Press P to sort by CPU
# Identify the process

# strace (careful, verbose)
strace -c -p <pid>

#perf
perf top -p <pid>
```

**CPU throttling (thermal):**

```bash
# Check dmesg
dmesg | grep -i thermal

# Check CPU frequency
watch -n 1 'cat /proc/cpuinfo | grep MHz'

# Check thermal zone
cat /sys/class/thermal/thermal_zone*/temp
```

## Interview Questions

1. **What does load average represent and how should you interpret it?**

Load average is the average number of processes in a runnable or running state over the specified period. On a 4-core system, load of 4.0 means all cores are busy (100% utilization). Load of 8.0 means 8 processes waiting for 4 cores (100% queue). A load above the core count indicates processes are queuing. Compare load to `nproc` for true utilization percentage.

2. **What's the difference between CPU utilization and system load?**

CPU utilization measures how busy the CPU is (percentage of time executing). Load measures how many processes are waiting to run (runnable queue). You can have high load with low CPU (I/O bound processes waiting) or high CPU with low load (compute-bound single process maxing one core).

3. **How does the CFS (Completely Fair Scheduler) work?**

CFS uses a red-black tree to track process virtual runtime (vruntime). Each process gets CPU time proportional to its weight (nice value). Lower nice = higher weight = more CPU. CFS targets latency by ensuring no process waits more than `latency_target / N` where N is process count. With many processes, timeslice shrinks to maintain fairness.

4. **When would you use CPU affinity (taskset)?**

CPU affinity pins processes to specific cores. Use cases: reducing context switching overhead for latency-sensitive apps, keeping processes on the same NUMA node, isolating noisy processes from critical ones, meeting licensing requirements (some software licenses per physical core).

5. **What's CPUQuota in systemd and how does it differ from CPUAffinity?**

CPUQuota sets a hard limit on CPU usage as a percentage. CPUQuota=200% on a system with 100% per core means 2 full cores. CPUAffinity just pins to specific cores without limiting. CPUQuota is a throttle (enforced via cfs quota); a process at CPUQuota=100% won't use more even if other cores are idle. CPUAffinity allows a process to use all pinned cores.
