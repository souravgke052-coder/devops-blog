# Linux Logging

## Overview

Linux logging is the foundation of production observability — every service, kernel event, security event, and application behavior is recorded somewhere in the log ecosystem. Understanding how journald, rsyslog, and logrotate work together, and how to ship logs to a central location, is essential for any DevOps engineer running production infrastructure.

The Linux logging stack has three layers: the kernel writes to the ring buffer (dmesg), systemd-journald collects structured logs from services and the kernel, and rsyslog/syslog-ng forwards logs from both journald and traditional syslog sources to files, databases, or remote collectors. logrotate prevents logs from consuming all disk space. Understanding the interaction between these components — especially journald's binary format vs rsyslog's text format — determines whether you can actually debug production incidents.

The shift from pure syslog to journald happened gradually. RHEL 7+ and Ubuntu 16.04+ ship with systemd and journald enabled by default. journald is structured (JSON over the wire), queryable, and more performant than raw text files. But rsyslog still runs alongside it for compatibility and forwarding to legacy systems.

## When To Use

- Incident response: determining what caused a service failure
- Security auditing: tracking login attempts, privilege escalations, file access
- Performance analysis: correlating load spikes with application log events
- Compliance: retaining logs for PCI-DSS, SOC2, HIPAA, or GDPR
- Capacity planning: understanding disk usage trends from log volume

## When NOT To Use

- Real-time metrics: logs show events, not rates — use Prometheus for gauges/counters
- Distributed tracing: use OpenTelemetry/traces for request-level visibility across services
- When logs are too verbose: structured logging + sampling is better than dumping everything
- Application state: logs aren't a database — don't query them like one

## Core Concepts

### journald Architecture

systemd-journald is a system service that collects and stores structured log data. It reads from:
- Kernel ring buffer (`/dev/kmsg`)
- System log socket (`/dev/log`)
- STDOUT/STDERR of system services via systemd

Journal files are binary and stored in `/var/log/journal/` (or `/run/log/journal/` if volatile). Each entry has structured fields:

| Field | Description |
|-------|-------------|
| `__REALTIME_TIMESTAMP` | Wall-clock time when entry was received |
| `_SYSTEMD_UNIT` | The systemd unit that generated the entry |
| `_MACHINE_ID` | Machine identifier |
| `PRIORITY` | 0=emergency to 7=debug |
| `SYSLOG_FACILITY` | Syslog facility code |
| `_UID` | UID of the process |
| `_PID` | PID of the process |

Configuration is in `/etc/systemd/journald.conf`:

```bash
# /etc/systemd/journald.conf
[Journal]
Storage=persistent       # persistent=disk, volatile=memory, auto=auto (dir exists)
Compress=yes             # Compress old entries
Seal=yes                 # Append-only MAC to detect tampering
MaxRetentionSec=30day    # Maximum log retention
MaxFileSize=100M         # Maximum file size before rotation
SystemMaxFiles=10        # Maximum number of journal files
RateLimitInterval=30s    # Per-UID/service rate limit interval
RateLimitBurst=1000      # Max messages per interval before rate limiting
```

### rsyslog Architecture

rsyslog is the traditional syslog daemon. It loads input modules (imuxsock for local, imtcp/imudp for network) and processes rules that route messages to outputs (files, databases, remote hosts).

Key modules:
- `imuxsock`: local syslog socket input
- `imtcp`/`imudp`: network input
- `omfile`: write to files
- `omfwd`: forward to another syslog server
- `mmjsonparse`: parse JSON-structured messages

rsyslog config is in `/etc/rsyslog.conf` and `/etc/rsyslog.d/`:

```bash
# /etc/rsyslog.d/01-server.conf
# Enable TCP input
module(load="imtcp" streams流="1514")

# Create ruleset for remote logging
ruleset(name="remote") {
  action(type="omfile" file="/var/log/remote/$HOSTNAME/$PROGRAMNAME.log")
  stop
}

# Route remote messages to ruleset
if $fromhost-ip != '127.0.0.1' then {
  call remote
  stop
}

# Local logging with timestamps
$ActionFileDefaultTemplate RSYSLOG_TraditionalFileFormat
$FileOwner syslog
$FileGroup adm
$FileCreateMode 0640
$DirCreateMode 0755
$Umask 0064
```

### logrotate

logrotate runs daily via cron and rotates log files based on size or time. It compresses old logs, optionally mails them, and can run post-rotation commands.

```bash
# /etc/logrotate.d/rsyslog
/var/log/syslog
/var/log/messages
/var/log/kern.log
/var/log/auth.log
/var/log/user.log
{
    daily              # Rotate daily
    rotate 14          # Keep 14 days
    missingok          # Don't error if file is missing
    notifempty         # Don't rotate empty files
    compress           # Compress with gzip
    delaycompress      # Compress the previous file (not the current one)
    sharedscripts      # Run postrotate script once for all files
    postrotate
        systemctl reload rsyslog > /dev/null 2>&1 || true
    endscript
}

# For high-volume application logs
/var/log/myapp/*.log {
    daily
    rotate 7
    missingok
    notifempty
    compress
    size 100M          # Rotate if file > 100M
    maxsize 200M       # Don't keep if already > 200M
    create 0640 appuser appuser  # Mode/owner/grp of new file
    sharedscripts
    postrotate
        systemctl restart myapp || true
    endscript
}
```

### Syslog Facility and Priority

Traditional syslog uses facility (source) and priority (severity) levels:

| Priority | Name | Description |
|----------|------|-------------|
| 0 | emerg | System unusable |
| 1 | alert | Must be fixed immediately |
| 2 | crit | Critical conditions |
| 3 | err | Error conditions |
| 4 | warning | Warning conditions |
| 5 | notice | Normal but significant |
| 6 | info | Informational |
| 7 | debug | Debug-level messages |

| Facility | Description |
|----------|-------------|
| 0 (kern) | Kernel messages |
| 1 (user) | User-level messages |
| 2 (mail) | Mail system |
| 3 (daemon) | System daemons |
| 4 (auth) | Authentication/security |
| 10 (local0-7) | Local use (applications) |

## Real Production Example

Setting up centralized logging for a 3-tier web application (nginx → app → postgres) with TLS-encrypted forwarding:

```bash
# === CLIENT SIDE (all web/app servers) ===

# /etc/systemd/journald.conf
# [Journal]
# Storage=persistent
# Compress=yes
# Seal=yes
# MaxFileSize=200M
# SystemMaxFiles=5
# ForwardToSyslog=yes          # Forward to rsyslog
# SystemMaxUse=2G

# /etc/rsyslog.d/60-remote.conf
# Send all logs to central log server via TLS
global(workDirectory="/var/spool/rsyslog")
global(netstreamDriver="gtls")
global(netstreamDriverCaFile="/etc/rsyslog.d/keys/ca.pem")
global(netstreamDriverCertFile="/etc/rsyslog.d/keys/cert.pem")
global(netstreamDriverKeyFile="/etc/rsyslog.d/keys/key.pem")

module(load="omfwd"）

# Template for structured remote logging
template(name="RemoteForward" type="string"
  string="%timestamp% %hostname% %syslogtag% %msg%\n")

# Forward to central server
action(type="omfwd"
  Target="logs.internal.example.com"
  Port="6514"
  Protocol="tcp"
  TCP_Framing="octet-counted"
  StreamDriver="gtls"
  StreamDriverMode="1"
  StreamDriverAuthMode="validcert"
  Template="RemoteForward"
  Queue.Type="LinkedList"
  Queue.Size="10000"
  Queue.SaveOnShutdown="on"
)

# === SERVER SIDE (central log server) ===

# /etc/rsyslog.d/01-inputs.conf
# Accept TLS-protected input from clients
global(workDirectory="/var/spool/rsyslog")
global(netstreamDriver="gtls")
global(netstreamDriverCaFile="/etc/rsyslog.d/keys/ca.pem")

module(load="imtcp"
  streams流="6514")

# Validate client certs
input(type="imtcp"
  port="6514"
  StreamDriver="gtls"
  StreamDriverMode="1"
  StreamDriverAuthMode="x509/name"
  PermittedPeer=["web-01.internal.example.com","web-02.internal.example.com","app-01.internal.example.com"])

# Template for stored logs
template(name="RemoteLogs" type="string"
  string="/var/log/remote/%HOSTNAME%/%PROGRAMNAME%.log"
  type="string")

# Store with rotation handled by logrotate
if $fromhost-ip != '127.0.0.1' then {
  action(type="omfile"
    dynaFile="RemoteLogs"
    template="RSYSLOG_TraditionalFileFormat"
    createOwner="syslog"
    createGroup="adm"
    createMode="0640"
  )
}

# Parse JSON from application logs
module(load="mmjsonparse")
if $msg contains '={' then {
  action(type="mmjsonparse")
}
```

```bash
# Verify log forwarding
# On client:
logger -n logs.internal.example.com -P 6514 -T "Test from $(hostname)"
# On server:
tail -f /var/log/remote/*/user.log

# Check rsyslog connection status
rsyslogd -v  # version info
systemctl status rsyslog
journalctl -u rsyslog -n 50 --no-pager
```

## CLI Examples

```bash
# journalctl — core tool for systemd logs
journalctl -b                  # Logs since last boot
journalctl -b -1               # Logs from previous boot
journalctl --since "1 hour ago"
journalctl --since "2024-01-01" --until "2024-01-02"
journalctl -u nginx.service    # Specific unit
journalctl -u nginx.service -u php-fpm.service  # Multiple units
journalctl -p err              # Priority err and above
journalctl -p warning -p err   # Multiple priorities
journalctl -f                  # Follow (like tail -f)
journalctl --disk-usage        # Show journal disk usage
journalctl --vacuum-size=500M  # Limit disk usage
journalctl --vacuum-time=30d   # Keep 30 days

# Complex journal queries
journalctl -u nginx.service --since "1 hour ago" | grep -i error
journalctl _UID=1000 --since today  # Logs from specific user
journalctl _SYSTEMD_UNIT=sshd.service PRIORITY=3  # SSH errors
journalctl -o json-pretty | jq '.MESSAGE'  # JSON output
journalctl -o verbose          # Full field names

# rsyslog debugging
rsyslogd -d                   # Debug mode (foreground)
rsyslogd -N 1                 # Check config syntax

# logrotate
logrotate -d /etc/logrotate.conf  # Dry run
logrotate -f /etc/logrotate.conf  # Force rotation
logrotate -s /var/lib/logrotate/logstatus  # Use status file

# Traditional syslog
tail -f /var/log/syslog
tail -f /var/log/messages
grep "FAILED\|ERROR" /var/log/auth.log | tail -50

# Log analysis patterns
# Count errors by hour
journalctl -p err --since "24 hours ago" --output=short-iso | cut -d'T' -f1 | sort | uniq -c

# Find top error messages
journalctl -p err -b | awk '{print $5}' | sort | uniq -c | sort -rn | head -10

# Watch logs for specific pattern in real-time
journalctl -f | grep --line-buffered "TIMEOUT\|CONNREFUSED"

# Extract IPs from nginx access logs for abuse detection
awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20
```

## Security

**Log injection prevention:**
rsyslog can interpret some characters in messages — never let untrusted input write directly to syslog. Use template escaping:

```bash
template(name="ForwardNoInject" type="string" string="%msg:R,BERNSTEIN--end%")
```

**Log integrity:**
- Enable journald `Seal=yes` (FSB - forward-secure integrity)
- Use AIDE to detect log tampering: `aide --check /var/log`
- Mount `/var/log` as read-only when possible (especially in containers)
- Set immutable flags on critical log files: `chattr +i /var/log/auth.log`

**What NOT to log:**
- Passwords, API keys, tokens: sanitize before logging
- PII without consent: GDPR compliance
- Full request/response bodies with sensitive data

**Compliance requirements:**
- PCI-DSS: 1-year log retention, log protection, regular review
- SOC2: audit trails, tamper-evident logs
- HIPAA: access logging for PHI
- ISO 27001: event logging, protected logs

```bash
# Immutable log file (even root can't modify/delete without removing the flag)
chattr +i /var/log/secure
chattr +a /var/log/messages   # append-only

# auditd rules for file integrity
auditctl -w /etc/shadow -p wa -k shadow_changes
auditctl -w /etc/sudoers -p wa -k sudoers_changes
```

## Troubleshooting

**Journal file corruption:**
```
Failed to open /var/log/journal/.../system.journal: Corrupted
```
Fix: Remove the corrupted file — journald will create a new one:
```bash
sudo rm /var/log/journal/USERID/system.journal
sudo systemctl restart systemd-journald
```

**Disk full from logs:**
```bash
# Immediate: vacuum old entries
journalctl --vacuum-size=500M
journalctl --vacuum-time=7d

# Emergency: disable logging temporarily (not recommended)
# echo "Storage=volatile" > /etc/systemd/journald.conf.d/emergency.conf
# systemctl restart systemd-journald

# Prevent future: set SizeLimit in unit file
```

**rsyslog not receiving remote logs:**
```bash
# 1. Check firewall
sudo iptables -L -n | grep 514

# 2. Check rsyslog is listening
ss -tulpn | grep rsyslog
# or: netstat -tulpn | grep 514

# 3. Check rsyslog config
rsyslogd -N 1

# 4. Test connectivity
# On client:
nc -zv logserver 514
# On server (with TCP):
# add module(load="imtcp") to rsyslog.conf

# 5. Check SELinux (RHEL)
semanage port -l | grep syslog
# If needed: semanage port -a -t syslogd_port_t -p tcp 514
```

**Log timestamps are wrong (timezone):**
```bash
# Check system timezone
timedatectl

# journald uses local time by default
# To use UTC:
# echo "RateLimitInterval=30s" >> /etc/systemd/journald.conf
# To force UTC timestamps in syslog:
# $ActionFileDefaultTemplate RSYSLOG_FileFormat
# (includes timezone in RFC 3164 format)

# If rsyslog receives messages with wrong timestamps:
# Use template to include timezone:
template(name="WithTZ" type="string" string="%timestamp% %hostname% %msg%\n")
```

**Log lost after reboot (volatile journal):**
```bash
# journald defaults to volatile on systems without /var/log/journal/
# Make persistent:
sudo mkdir -p /var/log/journal
sudo systemd-tmpfiles --create --prefix=/var/log/journal
sudo systemctl restart systemd-journald
# Verify:
ls -la /var/log/journal/
```

## Interview Questions

1. **What happens when a log file grows to fill the entire disk? How do you prevent this?**

When `/var/log` fills the disk, applications that write logs will fail with ENOSPC, systemd services will crash, and the system can become unbootable if critical logging is blocked. The kernel may kill processes via OOM killer. Prevention: use `journald.conf` SizeLimit/MaxFiles, logrotate with size limits, `du -f` monitoring alerts, separate `/var/log` into its own partition with `nospace` reserved blocks (`mke2fs -r 1`), and enable log compression.

2. **How does journald differ from traditional syslog, and when would you still use rsyslog?**

journald stores structured binary logs, can be queried with `journalctl`, has built-in indexing and rate limiting, and persists in `/var/log/journal/`. Traditional syslog is text-based, simpler, and has a wider ecosystem. You still need rsyslog when forwarding to external systems (SIEM, Splunk, Elasticsearch), when legacy applications only know syslog, or when you need complex filtering/parsing rules that journald doesn't support. rsyslog + journald work together — journald forwards to rsyslog via its `ForwardToSyslog=yes` option.

3. **What is log injection, and how do you prevent it?**

Log injection is when an attacker crafts log entries that spoof legitimate sources or break log parsers. For example, newlines in an HTTP request header can create fake log entries. Prevention: input validation to strip control characters, using journald's structured logging (which doesn't interpret newlines as delimiters), escaping message content in rsyslog templates (`%msg:R,BERNSTEIN--end%`), and never letting untrusted input write directly to log files.

4. **How would you design log retention for a PCI-DSS compliant environment?**

PCI-DSS requires 1-year log retention, immediately accessible for 90 days, and restorable from archive. Use a centralized log server with tiered storage: hot (local SSD, 7 days), warm (HDD, 90 days), cold (object storage/S3, 1 year). Enable write-once, tamper-evident storage (WORM). Ensure logs from all cardholder data environment systems are collected. Rotate and archive daily, with integrity checking (checksums). Audit log access — who reads logs should itself be logged.

5. **What is the difference between journald's `Storage=persistent` and `Storage=auto`?**

`persistent` stores logs on disk in `/var/log/journal/` — survives reboots, grows until SizeLimit. `auto` stores in `/var/log/journal/` if that directory exists, otherwise falls back to `/run/log/journal/` (RAM-backed, volatile). `volatile` stores only in `/run/log/journal/` — lost on reboot. The key difference is persistence across reboots and whether disk space is consumed permanently.
