# HTTP/HTTPS

## Overview

HTTP (Hypertext Transfer Protocol) is the foundation of the web. Despite being around since 1990, HTTP continues to evolve—HTTP/1.1 remains ubiquitous, HTTP/2 is widely deployed, and HTTP/3 (built on QUIC) is rapidly gaining adoption. Understanding HTTP at a deep level matters for DevOps because: debugging API issues requires reading HTTP headers, optimizing performance means understanding connection reuse and compression, and securing applications means implementing TLS correctly.

HTTP/2 introduced multiplexing (multiple requests over a single connection), header compression (HPACK), and server push. HTTP/3 goes further with QUIC's 0-RTT connection establishment and stream independence. Yet understanding HTTP/1.1 remains essential—it's what you'll debug when inspecting raw traffic, and many internal services still use it.

This page covers HTTP from a DevOps perspective: the wire protocol, headers that matter for operations, connection management, caching, and security considerations.

## When To Use

- **API development**: REST, GraphQL, and gRPC all use HTTP as transport
- **Load balancer configuration**: L7 load balancing requires understanding headers and routing
- **Performance optimization**: Connection reuse, compression, caching strategies
- **Security hardening**: HSTS, CSP, CORS, content type sniffing
- **Debugging production issues**: Reading access logs, decoding headers, replaying requests

## When NOT To Use

- **Binary protocols**: gRPC uses HTTP/2 but has its own protobuf encoding
- **WebSocket**: Different upgrade mechanism, though uses HTTP for handshake
- **Non-HTTP services**: Databases, SSH, DNS don't use HTTP (different transport)

### Important HTTP Headers

**Request Headers**:
- `Host`: Required for HTTP/1.1, identifies target (enables virtual hosting)
- `Authorization`: Credentials (Bearer, Basic, Digest)
- `Accept`: Media types client accepts
- `Accept-Encoding`: Supported compression (gzip, br, deflate)
- `Accept-Language`: Preferred languages
- `User-Agent`: Client identification
- `Cookie`: Session cookies from server
- `Cache-Control`: Cache directives
- `If-None-Match`: Conditional request with ETag
- `If-Modified-Since`: Conditional request with date

**Response Headers**:
- `Content-Type`: Media type (application/json, text/html)
- `Content-Length`: Size in bytes
- `Content-Encoding`: Compression (gzip, br)
- `Cache-Control`: Caching directives
- `ETag`: Version identifier for caching
- `Set-Cookie`: Cookies to store
- `Location`: Redirect target (3xx)
- `WWW-Authenticate`: Authentication challenge (401)
- `Strict-Transport-Security`: HSTS (force HTTPS)
- `Access-Control-Allow-Origin`: CORS (4+ hours)

### HTTP/2 Improvements

HTTP/2 addresses HTTP/1.1's head-of-line blocking by multiplexing multiple streams over a single TCP connection:

**Key features**:
- **Multiplexing**: Multiple requests/responses simultaneously (no request 2 waiting for request 1)
- **Header compression**: HPACK reduces header overhead by ~85%
- **Server push**: Server can push resources before client requests
- **Stream prioritization**: Streams have weights and dependencies
- **Single connection**: Eliminates TCP handshake overhead

```
HTTP/1.1: 6 connections for parallel resources, serialized headers
          Connection 1: GET /index.html
          Connection 2: GET /style.css
          Connection 3: GET /app.js
          ... (browsers limit concurrent connections per host)

HTTP/2: Single connection, multiplexed streams
          Stream 1: GET /index.html
          Stream 2: GET /style.css
          Stream 3: GET /app.js
          (all over single TCP connection)
```

**HTTP/2 frame structure**:
```
+-----------------------------------------------+
| Length (24 bits) | Type (8) | Flags (8) | R | |
|              Stream ID (31 bits)              |
+-----------------------------------------------+
|               Frame Payload                   |
+-----------------------------------------------+
```

### HTTP/3 and QUIC

HTTP/3 runs over QUIC (UDP-based) instead of TCP:

**Advantages**:
- **0-RTT**: Can send data immediately on reconnect (resumption)
- **No head-of-line blocking**: Stream loss only affects that stream
- **Connection migration**: Switch networks without reconnecting
- **Built-in TLS 1.3**: Encryption integrated into transport

**Disadvantages**:
- QUIC still has CPU overhead (userspace stack)
- Not all networks allow UDP 443
- Newer, less deployed

## Real Production Example

### NGINX Configuration for HTTP/2 and TLS

```nginx
# /etc/nginx/nginx.conf
http {
    # Enable HTTP/2
    server {
        listen 443 ssl http2;
        server_name api.example.com;

        # SSL/TLS configuration
        ssl_certificate /etc/ssl/certs/api.example.com.bundle.crt;
        ssl_certificate_key /etc/ssl/private/api.example.com.key;
        
        # Modern TLS configuration
        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
        ssl_prefer_server_ciphers off;
        
        # OCSP stapling
        ssl_stapling on;
        ssl_stapling_verify on;
        resolver 8.8.8.8 1.1.1.1 valid=300s;
        
        # Security headers
        add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
        add_header X-Frame-Options DENY always;
        add_header X-Content-Type-Options nosniff always;
        add_header X-XSS-Protection "1; mode=block" always;
        
        # Gzip compression
        gzip on;
        gzip_types application/json application/javascript text/css text/html;
        gzip_min_length 1000;
        
        # Connection management
        keepalive_timeout 65;
        keepalive_requests 1000;
        
        # Proxy to upstream
        location /api/ {
            proxy_pass http://backend;
            proxy_http_version 1.1;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
    
    upstream backend {
        server 10.0.0.5:8080;
        server 10.0.0.6:8080;
        keepalive 32;  # Connection pooling
    }
}
```

```nginx

# Permissions Policy
```

### HTTP Optimization for Cost

**Compression reduces bandwidth**:
```bash
# Gzip typically reduces text by 70-80%
# Brotli (br) is 15-20% better than gzip

# nginx: enable both
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 6;
gzip_types application/json application/javascript text/css text/html;

# brotli (requires module)

```

**Connection reuse**:
```bash
# Keep-alive reduces connection overhead
# HTTP/1.1 keeps connection open by default

# HTTP/2 multiplexes over single connection
listen 443 ssl http2;

# Keep connection pool to upstream
upstream backend {
    server 10.0.0.5:8080;
    keepalive 32;  # 32 idle connections
}

# For API clients: use HTTP/2 or connection pooling
```

## Interview Questions

**Q1: What's the difference between HTTP/1.1, HTTP/2, and HTTP/3?**

A: HTTP/1.1 (1999) introduced persistent connections (keep-alive), chunked transfer encoding, and virtual hosting. Its limitation: head-of-line blocking—only one request/response can be in flight per connection, and browsers use 6 connections per domain to parallelize. HTTP/2 (2015) adds multiplexing (multiple streams over one connection), HPACK header compression, server push, and stream prioritization. Fixes head-of-line blocking, but TCP head-of-line blocking remains—if one packet is lost, all streams stall until retransmission. HTTP/3 (2022) runs over QUIC (UDP), eliminating TCP head-of-line blocking—stream loss only affects that stream. Also offers 0-RTT resumption and connection migration. HTTP/2 and HTTP/3 negotiate via ALPN during TLS handshake; HTTP/2 can also use prior knowledge.

**Q2: What is CORS and how do you debug CORS issues?**

A: CORS (Cross-Origin Resource Sharing) restricts which origins (scheme + host + port) can access resources. Browsers enforce CORS; servers send `Access-Control-Allow-Origin` headers. Preflight requests (OPTIONS) check allowed methods/headers before actual request. Common CORS issues: (1) Missing `Access-Control-Allow-Origin` header—server must explicitly allow origin, (2) Wrong origin in header—must match exactly, (3) Credentials with wildcard origin—cannot use `Access-Control-Allow-Origin: *` with `Access-Control-Allow-Credentials: true`, (4) Missing preflight handling—server must handle OPTIONS for non-simple methods/headers. Debug with curl: `curl -H "Origin: https://app.example.com" -I https://api.example.com` to see headers returned.

**Q3: How does HTTP caching work, and what are the tradeoffs?**

A: HTTP caching uses headers: `Cache-Control` (directives like max-age, no-cache, no-store, private), `ETag` (opaque version token), `Last-Modified` (timestamp). Cacheable responses are stored by browsers, CDNs, proxies. On subsequent requests, client checks: (1) Freshness—within max-age? Use cached. (2) Stale-while-revalidate—use cached, update in background. (3) Must-revalidate—revalidate if stale. Conditional requests use `If-None-Match` (ETag) or `If-Modified-Since`. Server returns 304 Not Modified if valid, saving bandwidth. Tradeoffs: longer max-age = faster response, but stale content during deploys; shorter max-age = more revalidation requests; `no-store` = never cached, always fetch.

**Q4: What is a 502 error and how do you debug it?**

A: 502 Bad Gateway means the gateway/proxy received an invalid response from upstream. Common causes: (1) Upstream server down or unreachable (check upstream health), (2) Upstream returning malformed response (check upstream logs), (3) Timeout (upstream slow, increase timeouts), (4) Upstream overloaded (scale or queue requests), (5) DNS resolution failure (upstream hostname doesn't resolve). Debug: check NGINX/HAProxy logs for upstream errors, `curl -v` directly to upstream IP to isolate, check upstream response format, check connection pool exhaustion (`upstream sent too big header`).

**Q5: How would you implement request tracing across microservices?**

A: Distributed tracing requires: (1) Trace ID generation at entry (load balancer or first service), propagated via `X-Request-ID` or `X-B3-TraceId` header, (2) Propagate trace context in all outgoing HTTP calls, (3) Log correlation—include trace ID in all logs, (4) Collect traces (Jaeger, Zipkin, AWS X-Ray) via sidecar or agent. Implementation: NGINX can generate/forward trace ID with `$request_id` variable (unique 32-char hex). Services extract from header, add to logs, pass in outgoing requests. Format: `X-Request-ID: abc123def456` or B3 format: `X-B3-TraceId: 4bf92f3577b34da6a3ce929d0e0e4736`. Sample rate: don't trace 100%—use header or config to sample 1-10%.
