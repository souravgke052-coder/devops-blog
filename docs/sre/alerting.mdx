# Alerting

## Overview

Alerting is the practice of sending notifications when monitoring data indicates a potential problem requiring human attention. Effective alerting is not about detecting every anomaly — it's about detecting the right events at the right time with the right urgency. Alert fatigue (too many irrelevant pages) kills response effectiveness just as surely as missing critical alerts.

The philosophy of SRE alerting is that **every alert must be actionable**. If an engineer cannot look at an alert and immediately know what to do, it's not a good alert. Alerts that fire but don't require action erode trust in the alerting system and lead to ignored pages.

Modern alerting systems support multi-level alerts (warning vs critical), routing to different channels (SMS vs Slack vs PagerDuty), and suppression during maintenance windows. The goal is ensuring the right person sees the right alert at the right time with enough context to respond.

## When To Use

- **SLO breach risk** — When error budget is depleting faster than expected
- **User-impacting issues** — When errors or latency affect users
- **Capacity limits** — When systems approach saturation
- **Dependency failures** — When downstream services are unavailable
- **Security events** — When suspicious activity is detected
- **Scheduled validations** — When periodic health checks fail

## When NOT To Use

- **For every metric** — Not every deviation requires action
- **During maintenance windows** — Suppress expected changes
- **Without runbooks** — Alerts without documentation lead to delayed response
- **Based on single measurements** — Use for/window to confirm sustained issues

## Core Concepts

### Alert Severity Levels

```yaml
severity_levels:
  P1_Critical:
    response_time: "5 minutes"
    channel: "Phone call (auto-escalate)"
    description: "Complete service outage or severe user impact"
    examples:
      - "Site returns 5xx for all requests"
      - "Payment processing completely down"
      - "Database unreachable"
      
  P2_High:
    response_time: "15 minutes"
    channel: "SMS + Slack"
    description: "Major feature unavailable or significant degradation"
    examples:
      - "Login fails for >50% of users"
      - "Checkout error rate > 5%"
      - "Latency p99 > 10 seconds"
      
  P3_Medium:
    response_time: "1 hour"
    channel: "Slack"
    description: "Degraded performance or minor feature issues"
    examples:
      - "Non-critical service errors elevated"
      - "Latency p99 > 2 seconds"
      - "Memory usage > 80%"
      
  P4_Low:
    response_time: "4 hours or next business day"
    channel: "Email/Ticket"
    description: "Minor issues not currently impacting users"
    examples:
      - "Disk usage > 70%"
      - "Certificate expiring in 30 days"
      - "Intermittent 4xx errors"
```

### Alert Quality Checklist

An alert should have:
- [ ] Clear, specific title indicating the problem
- [ ] Severity level matching actual impact
- [ ] Link to runbook with response steps
- [ ] Context (which service, how many users affected)
- [ ] Metrics showing the issue
- [ ] Duration threshold avoiding flapping

### Alert Burn Rate Philosophy

For SLO-based alerting, use multi-window burn rate alerts:

| Window | Burn Rate | Meaning | Action |
|--------|-----------|---------|--------|
| 1 hour | > 360x | 50% budget in 1 hour | Critical |
| 6 hours | > 60x | 50% budget in 6 hours | Warning |
| Multi-window | Both > 14.4x | Sustained burn | Critical |

## Real Production Example

### SLO-Based Alert Rules

```yaml
# prometheus/slo-alerts.yaml
groups:
  - name: slo-error-budget
    interval: 60s
    rules:
      # Fast burn: 50% of budget in 1 hour
      # For 30-day SLO (99.9%), this is burn rate > 360x
      - alert: SLOErrorBudgetFastBurn
        expr: |
          (
            1 - (
              sum(rate(http_requests_total{job="api",status!~"5.."}[1h]))
              /
              sum(rate(http_requests_total{job="api"}[1h]))
            )
          )
          /
          (1 - 0.999)
          /
          (1 / (30 * 24 * 60))
          > 50
        for: 0m
        labels:
          severity: critical
          slo: api-availability
        annotations:
          summary: "Critical: Error budget burning fast"
          description: |
            50% of monthly error budget burned in 1 hour.
            Burn rate: {{ $value | printf "%.0f" }}x normal.
            Halt non-critical deployments immediately.
          runbook_url: "https://runbooks.example.com/slo-fast-burn"

      # Slow burn: 50% of budget in 6 hours
      # Burn rate > 60x
      - alert: SLOErrorBudgetSlowBurn
        expr: |
          (
            1 - (
              sum(rate(http_requests_total{job="api",status!~"5.."}[6h]))
              /
              sum(rate(http_requests_total{job="api"}[6h]))
            )
          )
          /
          (1 - 0.999)
          /
          (6 / (30 * 24 * 60))
          > 50
        for: 0m
        labels:
          severity: warning
          slo: api-availability
        annotations:
          summary: "Warning: Sustained error budget burn"
          description: |
            50% of monthly error budget burned in 6 hours.
            Investigate root cause.
          runbook_url: "https://runbooks.example.com/slo-slow-burn"

      # Multi-window burn: both 1h and 6h exceeding threshold
      - alert: SLOErrorBudgetMultiWindowBurn
        expr: |
          (
            (
              1 - (
                sum(rate(http_requests_total{job="api",status!~"5.."}[1h]))
                /
                sum(rate(http_requests_total{job="api"}[1h]))
              )
            )
            /
            (1 - 0.999)
            /
            (1 / (30 * 24 * 60))
            > 14.4
          )
          and
          (
            (
              1 - (
                sum(rate(http_requests_total{job="api",status!~"5.."}[6h]))
                /
                sum(rate(http_requests_total{job="api"}[6h]))
              )
            )
            /
            (1 - 0.999)
            /
            (6 / (30 * 24 * 60))
            > 14.4
          )
        for: 5m
        labels:
          severity: critical
          slo: api-availability
        annotations:
          summary: "Critical: Sustained error budget burn across windows"
          description: |
            Both 1h and 6h burn rates exceed 14.4x.
            Page on-call immediately.
```

## Metrics / Formulas / Frameworks

### Burn Rate Alert Calculations

```python
def calculate_burn_rate_alert_thresholds(slo_target, window_days):
    """
    Calculate burn rate thresholds for multi-window alerting.
    
    Returns thresholds for:
    - 1h window (fast burn)
    - 6h window (slow burn)
    - Multi-window (both elevated)
    """
    window_hours = window_days * 24
    
    # To burn 50% of budget in window X hours:
    # burn_rate = (50% / budget%) * (window_hours / X)
    
    budget_percent = (1 - slo_target) * 100  # e.g., 0.1 for 99.9%
    
    # 1 hour: burn 50% budget in 1 hour
    fast_burn_1h = (50 / budget_percent) * (window_hours / 1)
    
    # 6 hours: burn 50% budget in 6 hours
    slow_burn_6h = (50 / budget_percent) * (window_hours / 6)
    
    # Multi-window: burn 50% budget in 6h AND 1h
    multi_window_1h = (50 / budget_percent) * (window_hours / 1) / 25  # ~14.4x
    multi_window_6h = (50 / budget_percent) * (window_hours / 6) / 25  # ~14.4x
    
    return {
        'fast_burn_1h': fast_burn_1h,
        'slow_burn_6h': slow_burn_6h,
        'multi_window_1h': multi_window_1h,
        'multi_window_6h': multi_window_6h
    }

# Example for 99.9% SLO (30-day window)
thresholds = calculate_burn_rate_alert_thresholds(0.999, 30)
print(f"99.9% SLO burn rate thresholds:")
print(f"  Fast burn (50% in 1h): {thresholds['fast_burn_1h']:.0f}x")
print(f"  Slow burn (50% in 6h): {thresholds['slow_burn_6h']:.0f}x")
print(f"  Multi-window 1h: {thresholds['multi_window_1h']:.1f}x")
print(f"  Multi-window 6h: {thresholds['multi_window_6h']:.1f}x")

# Output:
# 99.9% SLO burn rate thresholds:
#   Fast burn (50% in 1h): 360x
#   Slow burn (50% in 6h): 60x
#   Multi-window 1h: 14.4x
#   Multi-window 6h: 14.4x
```

## Troubleshooting / Common Mistakes

### Common Alerting Mistakes

| Mistake | Impact | Solution |
|---------|--------|----------|
| Too many alerts | Alert fatigue | Only alert on SLO impacts |
| No runbook | Slow response | Document response steps |
| Alert storms | Missed critical | Use grouping, rate limiting |
| No severity tiers | Everything is critical | Define severity clearly |
| Flapping alerts | Trust erosion | Use for/window properly |

### Alert Troubleshooting

**Symptom**: Alert firing but nothing seems wrong

**Diagnosis**:
```bash
# Check the actual metric value
promtool query instant http://localhost:9090 'error_rate{job="api"}'

# Check historical values
promtool query range http://localhost:9090 'error_rate{job="api"}[1h]'

# Check if query is correct
# Common issues:
# - Label mismatch (e.g., job="api" vs job="api-server")
# - Metric name wrong
# - Aggregation missing
```

## Interview Questions

### Q1: What makes a good alert?

**A**: A good alert has these properties:

1. **Actionable** — The responder knows exactly what to do
2. **Specific** — Clear what system and what's wrong
3. **Timely** — Fires fast enough to prevent user impact
4. **Prioritized** — Severity matches actual impact
5. **Documented** — Has a runbook with response steps
6. **No noise** — Doesn't fire for non-issues

Example of a good alert:
```
CRITICAL: API Error Rate > 5%
Metric: 7.3% of requests returning 5xx
Duration: Sustained for 5 minutes
Impact: ~5,000 users affected
Runbook: https://runbooks.example.com/api-errors
Action: Check deployment logs, consider rollback
```

### Q2: How do you prevent alert fatigue?

**A**: Alert fatigue prevention:

1. **Only page for user-visible issues** — Don't page for internal metrics
2. **Use severity correctly** — Not everything is P1
3. **Require sustained issues** — Use `for:` to filter spikes
4. **Multi-window alerts** — Don't alert on transient issues
5. **Regular alert review** — Monthly review of all alerts
6. **Track false positive rate** — Address alerts with >50% false positive

Key principle: If you're not willing to wake someone up at 3 AM for it, it shouldn't be a page.

### Q3: What's the difference between PagerDuty and simple webhook alerts?

**A**: Key differences:

| Aspect | Webhooks | PagerDuty |
|--------|----------|-----------|
| Escalation | Manual | Automatic |
| On-call rotation | Not built-in | Built-in |
| Acknowledgment | Not tracked | Tracked |
| Mobile app | Not included | Included |
| Duty scheduling | Manual | Automatic |
| Alert grouping | Manual | Automatic |
| Analytics | Limited | Built-in |

PagerDuty (or similar) is essential for production services because it handles:
- On-call scheduling and rotations
- Automatic escalation if not acknowledged
- Mobile push notifications with critical importance
- Alert grouping and deduplication
- Response analytics

### Q4: How do you handle alert routing for different services?

**A**: I use hierarchical routing:

```yaml
route:
  # Primary group by service/team
  group_by: [team, service]
  
  routes:
    # Infrastructure alerts → platform team
    - match:
        type: infrastructure
      receiver: platform-oncall
      routes:
        - match:
            component: database
          receiver: dba-oncall  # Override for DB issues
            
    # Application alerts → service team
    - match:
        type: application
      receiver: service-oncall
      routes:
        - match:
            severity: critical
          receiver: service-critical-oncall  # Second on-call for critical
            
    # Security → security team
    - match:
        category: security
      receiver: security-oncall
```

### Q5: How do you test alerting before going to production?

**A**: Testing strategy:

1. **Unit test rules** — Use `promtool test rules` to validate PromQL
2. **Dry run** — Deploy rules in dry-run mode first
3. **Local testing** — Use AlertManager webhook to test routing
4. **Canary** — Deploy to single cluster first, monitor for noise
5. **Gradual rollout** — Start with warning-only, add critical later
6. **Post-deployment monitoring** — Watch for false positives
