# Capacity Planning

## Overview

Capacity planning is the practice of ensuring a system has enough resources to handle current and projected future load. It encompasses understanding current utilization, forecasting growth, planning capacity additions, and implementing autoscaling to handle variability.

Poor capacity planning leads to two failure modes: over-provisioning (wasting money on unused resources) and under-provisioning (service degradation or outages during peak load). The goal is right-sizing — having just enough capacity to meet SLAs at acceptable cost.

SRE teams own capacity planning because it directly impacts reliability. A system without enough capacity will fail under load, regardless of how well-engineered it is. Capacity planning ensures reliability targets can be met as load grows.

## When To Use

- **Resource allocation** — Planning infrastructure spend and procurement
- **Scaling decisions** — When to add capacity (horizontal or vertical)
- **SLA planning** — Ensuring capacity meets reliability targets
- **Cost optimization** — Right-sizing to reduce waste
- **Architecture decisions** — Designing for expected scale
- **Budget planning** — Quarterly/annual capacity roadmaps

## When NOT to Use

- **For static, rarely-changing systems** — Less frequent planning needed
- **Without usage data** — Forecasting without data is guesswork
- **As one-time activity** — Continuous monitoring and adjustment needed

## Metrics / Formulas / Frameworks

### Capacity Forecasting

```python
import numpy as np
from datetime import datetime, timedelta

def linear_forecast(values, periods_ahead):
    """
    Simple linear regression forecast.
    
    Args:
        values: List of historical values (e.g., daily peak usage)
        periods_ahead: Number of future periods to forecast
        
    Returns:
        List of forecasted values
    """
    n = len(values)
    x = np.arange(n)
    y = np.array(values)
    
    # Linear regression: y = mx + b
    m, b = np.polyfit(x, y, 1)
    
    # Forecast future periods
    future_x = np.arange(n, n + periods_ahead)
    forecasts = m * future_x + b
    
    return forecasts.tolist(), m, b  # forecasts, slope, intercept

def calculate_runway(current_capacity, current_usage, growth_rate_per_week):
    """
    Calculate how many weeks until capacity is exhausted.
    
    Args:
        current_capacity: Total capacity (e.g., 100 instances)
        current_usage: Current usage (e.g., 60 instances)
        growth_rate_per_week: Weekly growth rate (e.g., 0.05 for 5%)
        
    Returns:
        Number of weeks until capacity reached
    """
    weeks = 0
    usage = current_usage
    
    while usage < current_capacity and weeks < 104:  # Max 2 years
        usage *= (1 + growth_rate_per_week)
        weeks += 1
        
    return weeks

# Example
capacity = 100  # 100 instances max
usage = 65      # Currently using 65
growth = 0.03   # 3% weekly growth

runway = calculate_runway(capacity, usage, growth)
print(f"Capacity runway: {runway} weeks ({runway/4:.1f} months)")

# Forecast usage in 12 weeks
forecast, _, _ = linear_forecast([65, 67, 69, 71, 73], 12)
print(f"Expected usage in 12 weeks: {forecast[-1]:.0f} instances")
```

### Right-Sizing Calculations

```python
def calculate_right_sized_instance(current_utilization_avg, 
                                    current_utilization_peak,
                                    instance_options):
    """
    Calculate right-sized instance based on utilization.
    
    Args:
        current_utilization_avg: Average CPU utilization %
        current_utilization_peak: Peak CPU utilization %
        instance_options: List of (name, vCPU, memory) tuples
        
    Returns:
        Recommended instance type
    """
    # Target: ~70% utilization at peak (some headroom)
    target_utilization = 0.70
    required_capacity = current_utilization_peak / target_utilization
    
    # Find smallest instance that provides enough headroom
    for name, vcpu, mem in sorted(instance_options, key=lambda x: x[1]):
        if vcpu >= required_capacity:
            return {
                'instance': name,
                'vCPU': vcpu,
                'memory_gb': mem,
                'projected_peak_util': current_utilization_peak / (vcpu / required_capacity),
                'cost_savings_vs_current': f"{calculate_savings(name)}"
            }
    
    return None  # Needs horizontal scaling

def calculate_autoscaling_metrics(current_rps, target_latency_ms, 
                                   avg_request_duration_ms, max_instances):
    """
    Calculate autoscaling parameters.
    """
    # Target RPS per instance
    rps_per_instance = 1000 / avg_request_duration_ms * 1000  # rough estimate
    
    # Minimum instances needed for target latency
    min_instances = int(current_rps / rps_per_instance) + 1
    
    # Scale threshold (70% CPU target)
    scale_threshold = 0.70
    
    return {
        'min_instances': min(min_instances, max_instances),
        'max_instances': max_instances,
        'scale_threshold_pct': scale_threshold * 100,
        'cooldown_seconds': 300,
        'target_rps_per_instance': rps_per_instance
    }
```

### Queue Depth and Saturation

```python
def calculate_queue_impact(arrival_rate, service_rate, num_servers):
    """
    Calculate queue metrics using M/M/c queuing model.
    
    Args:
        arrival_rate: Requests per second
        service_rate: Service rate per server
        num_servers: Number of servers
        
    Returns:
        Dictionary with queue metrics
    """
    # Traffic intensity per server
    rho = arrival_rate / (num_servers * service_rate)
    
    # Must be < 1 for stable system
    if rho >= 1:
        return {'stable': False, 'message': 'Overloaded - need more servers'}
    
    # Utilization
    utilization = rho * 100
    
    # Average queue length (approximation)
    # Using Kingman's formula approximation
    import math
    queue_length = (rho ** 2) / (1 - rho) / num_servers
    
    # Average wait time
    avg_wait_time = queue_length / arrival_rate if arrival_rate > 0 else 0
    
    return {
        'stable': True,
        'utilization_pct': utilization,
        'queue_length': queue_length,
        'avg_wait_time_ms': avg_wait_time * 1000,
        'recommendation': 'scale_up' if utilization > 80 else 'adequate'
    }
```

    """
    Calculate potential savings from right-sizing.
    """
    # Current: 100 instances, 40% avg utilization (over-provisioned)

## Troubleshooting / Common Mistakes

### Common Capacity Planning Mistakes

| Mistake | Impact | Solution |
|---------|--------|----------|
| No historical data | Poor forecasts | Start collecting metrics NOW |
| Ignoring seasonality | Under-provisioning during peaks | Include seasonal patterns |
| Planning only for peak | Wasted resources | Use autoscaling |
| No headroom | Frequent scaling events | Plan 30-50% headroom |
| Ignoring growth | Capacity surprises | Include growth rate in plans |
| No cost modeling | Budget overruns | Model costs before scaling |

## Interview Questions

### Q1: How do you forecast capacity needs?

**A**: I use multiple methods:

1. **Historical trend analysis**:
   - Collect 3-6 months of usage data
   - Calculate weekly/monthly growth rates
   - Project forward

2. **Seasonal patterns**:
   - Identify recurring patterns (daily, weekly, seasonal)
   - Plan for known peaks (Black Friday, product launches)
   - Include buffer for unknown peaks

3. **SLO-based calculation**:
   - What's the maximum latency we can accept?
   - How many requests can each instance handle at that latency?
   - How many instances do we need for peak + headroom?

4. **Business projections**:
   - Marketing's growth projections
   - Product roadmap (new features = new usage)
   - Customer commitments

Key formula:
```
Needed capacity = Peak demand × (1 + growth rate) ^ time_period × headroom_factor
```

### Q2: What's the difference between reactive and predictive autoscaling?

**A**:

| Aspect | Reactive | Predictive |
|--------|----------|------------|
| **Timing** | Responds after threshold exceeded | Scales before thresholds exceeded |
| **Basis** | Current metrics | Forecasted demand |
| **Pros** | Simple, proven | No latency from scaling wait |
| **Cons** | Some requests served slowly during scale-up | Can over-provision |
| **Tools** | HPA, ASG | KEDA, AWS Predictive Scaling |

Reactive is the foundation — always implement that first. Predictive adds value for known predictable patterns (daily peaks, scheduled events).

### Q3: How do you handle capacity planning for databases?

**A**: Database capacity is more complex because:

1. **Vertical scaling often required**:
   - Sharding is complex
   - Connection pooling helps but has limits
   - Read replicas can offload read traffic

2. **Key metrics**:
   - CPU (compute bound)
   - Memory (buffer pool hits)
   - Disk I/O (query performance)
   - Connections (pool exhaustion)
   - Replication lag (for read replicas)

3. **Planning approach**:
   - Monitor all metrics
   - Right-size based on data volume and query patterns
   - Plan vertical scaling in advance (downtime often needed)
   - Use read replicas for read scaling
   - Consider sharding for extreme scale (careful!)

Key difference from stateless services: databases often need planned maintenance windows for major scaling operations.

### Q4: How do you calculate autoscaling thresholds?

**A**: Thresholds should balance responsiveness with stability:

1. **CPU threshold**:
   ```
   Target: 70% average utilization at peak
   Reason: Some headroom needed, avoid thrashing
   ```

2. **Scale up**:
   - Be aggressive (respond quickly to demand)
   - No cooldown or minimal cooldown (30-60 seconds)
   - Allow significant scale per cycle (100% doubling OK)

3. **Scale down**:
   - Be conservative (avoid oscillation)
   - Long cooldown (5-10 minutes)
   - Small scale-down per cycle (10-25%)

4. **Multiple metrics**:
   - Use composite of CPU + memory + custom metrics
   - Scale when ANY threshold exceeded
   - Avoid over-scaling due to one noisy metric

Example HPA behavior:
```yaml
scaleUp:
  stabilizationWindowSeconds: 0  # Immediate scale up
scaleDown:
  stabilizationWindowSeconds: 300  # 5 minute delay
```

### Q5: How do you plan capacity for a new service?

**A**: Without historical data, I use:

1. **Similar service benchmarks**:
   - What's the utilization of similar existing services?
   - Adjust for expected differences

2. **Load testing**:
   - Test with expected peak load
   - Measure what breaks first
   - Size based on limits

3. **Industry benchmarks**:
   - Typical RPS per instance for similar workloads
   - Memory per concurrent user
   - Database queries per request

4. **Safety factors**:
   - Start with 2-3x expected capacity
   - Use autoscaling to adjust
   - Monitor and right-size after seeing real usage

5. **Phased rollout**:
   - Start with baseline capacity
   - Monitor closely during launch
   - Adjust based on actual usage patterns

## Cheat Sheet

### Capacity Planning Quick Reference

```
CPU Scaling Formula:
  min_replicas = ceil(peak_rps × avg_duration / target_utilization)

Memory Scaling Formula:
  min_replicas = ceil(peak_concurrent_users × memory_per_request / instance_memory)

Database Connection Sizing:
  max_connections = 100 × vCPU_count
  pool_size = max_connections / num_app_instances

Autoscaling Parameters:
  CPU threshold: 70%
  Scale up stabilization: 0-30s
  Scale down stabilization: 5-10 minutes
  Min instances: 2-3 (for availability)
  Max instances: Based on cost/architecture limits
```

### Right-Sizing Checklist

```
[ ] Collect 30+ days of baseline metrics
[ ] Identify peak usage patterns

[ ] Identify growth rate (weekly/monthly)
[ ] Calculate runway (weeks until capacity exhausted)

[ ] Set capacity alerts at 70%, 80%, 90%
[ ] Review monthly and adjust
```

### Growth Rate Calculation

```python
# Weekly growth rate
growth_rate = (current_usage / previous_usage) - 1

# Monthly growth rate  
monthly_growth = ((current_usage / usage_30_days_ago) ** (30/days_between)) - 1

# Runway calculation
runway_weeks = ln(max_capacity/current_usage) / ln(1 + growth_rate)
```
