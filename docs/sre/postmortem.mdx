# Postmortem

## Overview

A postmortem is a structured review conducted after an incident to understand what happened, why it happened, and what can be done to prevent recurrence. The SRE philosophy of "blameless postmortems" means focusing on systemic failures rather than individual mistakes — the goal is learning, not punishment.

Blameless postmortems recognize that systems fail, but the way organizations respond to failures determines whether they improve. When people fear blame, they hide mistakes. When they feel safe, they surface root causes. The most reliable systems are built by organizations that learn from failures openly and systematically.

A good postmortem is not about finding someone to blame — it's about finding the conditions that allowed a failure to occur and improving those conditions. Individual actions are less important than the system design that made those actions possible or necessary.

## When To Use

- **After any SEV1 or SEV2 incident** — Required for major incidents
- **After recurring issues** — If the same problem happens multiple times
- **Near-misses** — Situations where failure was narrowly avoided
- **Customer-impacting outages** — Any user-visible issue
- **Security incidents** — Especially for breach or data exposure

## When NOT to Use

- **Every minor issue** — Don't postmortem everything; focus on meaningful learning
- **During active incident** — Document timeline but defer review until resolved
- **Without enough information** — Wait for complete data before writing
- **When no action can prevent recurrence** — Some incidents are pure bad luck

## Core Concepts

### Blameless Culture

Blameless does not mean "blame-free" — it means the focus is on the system, not the individual. The question is not "who made a mistake?" but "what conditions led to this?"

```python
# Examples of blame vs. blameless analysis

# BLAME-FOCUSED (wrong):
"Engineer deployed without proper testing"
"Operations team didn't follow the runbook"

# BLAMELESS (right):
"The deployment pipeline allowed production deployment without staging validation"
"The runbook was incomplete for the edge case encountered"
```

### Postmortem Timing

| Incident Severity | Postmortem Due | Author | Review |
|-------------------|----------------|--------|--------|
| SEV1 | 48 hours | Senior engineer + IC | All-hands review |
| SEV2 | 72 hours | On-call engineer | Team review |
| SEV3 | 1 week | Team | Async |
| SEV4 | Optional | Optional | Optional |

### Action Item Principles

Every postmortem must result in concrete action items:

1. **Actionable** — Someone can actually do it, not just "improve monitoring"
2. **Specific** — Not "improve reliability" but "add alert when error rate > 1%"
3. **Verifiable** — We can check if it was completed
4. **Prioritized** — Most impactful changes first
5. **Owned** — Single person responsible

## Root Cause Analysis

### What Happened

The data migration job (newly deployed at 14:00) opened database connections 
but did not release them properly. The migration code:

```python
# PROBLEMATIC CODE (simplified)
def migrate_orders():
    db = get_db_connection()  # Opens connection
    for batch in get_batches():
        process_batch(batch)
    # Missing: db.close()
    # Missing: Using context manager
```

Without proper cleanup, each batch processing kept connections open. With 100 
connections in the pool and the migration consuming ~5 connections per batch, 
the pool was exhausted after processing ~20 batches (~25 minutes).

### Contributing Factors

1. **Code Review Gap**: PR reviewer focused on business logic, didn't catch 
   missing connection cleanup. No linter rule for this pattern.

2. **Insufficient Testing**: Migration was tested with 1,000 records but 
   production had 500,000. Connection duration scaled with record count.

3. **No Pool Monitoring**: There was no alert on connection pool utilization. 
   Alerts existed only for application error rates.

4. **Missing Context Usage**: Python database code should use `with` context 
   manager or explicit cleanup. Migration used neither pattern.

### 5 Whys

```
Why: Checkout service errors?
→ Why: Database connection pool exhausted
→ Why: Migration job consumed all connections
→ Why: Migration didn't close connections
→ Why: Developer used try/except without finally cleanup
→ Why: Code review didn't catch pattern, no linting rule for this

Root Cause: Missing connection cleanup in migration code, not caught in review.
```

- High error rate alert fired when error rate exceeded 5% for 3 minutes
- 7 minutes passed between connection pool exhaustion and first user errors

**What could have detected earlier:**
- Connection pool monitoring (would have caught at 80% threshold, ~10 min earlier)
- Load testing with production data volumes (would have caught during QA)

## Metrics / Formulas / Frameworks

### Postmortem Metrics

```python
def calculate_postmortem_effectiveness(incidents):
    """
    Measure how effectively postmortems are preventing recurrence.
    """
    recurrence_rate = calculate_recurrence_rate(incidents)
    action_completion = calculate_action_item_completion(incidents)
    mttr_trend = calculate_mttr_trend(incidents)
    
    return {
        'recurrence_rate': recurrence_rate,  # Lower is better
        'action_item_completion': action_completion,  # Higher is better
        'mttr_trend': mttr_trend  # Lower is better (faster resolution)
    }

def calculate_recurrence_rate(incidents):
    """
    What percentage of incidents are recurrences of previous issues?
    """
    total = len(incidents)
    if total == 0:
        return 0
    
    recurring = sum(1 for i in incidents if i.is_recurrence)
    return (recurring / total) * 100
```

### Action Item Tracking

```python
def action_item_status_summary(action_items):
    """
    Summarize status of all action items from postmortems.
    """
    status_counts = {
        'done': 0,
        'in_progress': 0,
        'not_started': 0,
        'blocked': 0
    }
    
    for item in action_items:
        status_counts[item.status] += 1
    
    total = len(action_items)
    completion_rate = status_counts['done'] / total if total > 0 else 0
    
    return {
        'total': total,
        'by_status': status_counts,
        'completion_rate': completion_rate,
        'overdue': [i for i in action_items if i.is_overdue]
    }
```

## Security

### Postmortem Confidentiality

```markdown
# Handling sensitive information in postmortems

Allowed in postmortems:
- System behavior and failures
- Process and tooling gaps
- Metrics and timelines
- Technical root causes

Requires special handling:
- Security vulnerabilities (separate security postmortem)
- Customer PII (minimize exposure, aggregate)
- Credentials or keys (redact, don't log in postmortem)
- Legal/contractual details (review with legal first)

Distribution:
- Internal postmortems: Full detail within engineering
- Customer-facing: Sanitized version with impact summary
- Public postmortems: Generic description, no sensitive details
```

## Cost Optimization

### Postmortem ROI

```python
def postmortem_roi(incident, postmortem):
    """
    Estimate the value of a postmortem.
    """
    # Cost of postmortem
    postmortem_cost = (
        postmortem.author_hours * 100 +  # Engineering time
        postmortem.review_hours * 100 +
        postmortem.action_items_hours
    )
    
    # Estimated prevention value
    recurrence_prevented = postmortem.recurrence_prevented
    incident_cost = incident.duration_hours * incident.cost_per_hour
    
    prevention_value = recurrence_prevented * incident_cost
    
    # MTTR improvement
    mttr_improvement = postmortem.mttr_improvement_hours * incident.frequency
    
    return {
        'postmortem_cost': postmortem_cost,
        'prevention_value': prevention_value,
        'mttr_improvement_value': mttr_improvement,
        'net_value': prevention_value + mttr_improvement - postmortem_cost
    }
```

## Troubleshooting / Common Mistakes

### Common Postmortem Mistakes

| Mistake | Impact | Solution |
|---------|--------|----------|
| Blame language | No psychological safety | Use "the system" not "the engineer" |
| No action items | No improvement | Every postmortem needs concrete actions |
| Vague actions | Can't verify completion | SMART actions |
| No follow-up | Actions never get done | Track and verify monthly |
| Too long | Won't be read | Keep under 5 pages |
| Missing timeline | Hard to follow | Use table with timestamps |

### Blameless Anti-Patterns

```markdown
# These phrases indicate blame:
❌ "Developer forgot to add closing tag"
❌ "Operations team didn't check the logs"
❌ "The user made an error"
❌ "Someone deployed without testing"

# Better framing:
✅ "The deployment process allowed code without proper resource cleanup"
✅ "Logs weren't surfaced in the alerting channel"
✅ "The user interface didn't prevent invalid input"
✅ "The CI/CD pipeline didn't enforce testing gate"
```

## Interview Questions

### Q1: What's the difference between a postmortem and a RCA?

**A**: 

**RCA (Root Cause Analysis)** focuses on identifying the root cause — the "why" behind the incident. It's analytical, focused on causation.

**Postmortem** is broader — it covers what happened, how it was detected, how it was responded to, root cause, AND what we learned. It includes process and response evaluation, not just root cause.

Think of RCA as part of postmortem. Postmortem asks:
1. What happened?
2. How did we detect it?
3. How did we respond?
4. Why did it happen? (RCA)
5. What will we do differently?

### Q2: How do you write a blameless postmortem?

**A**:

1. **Start with facts** — What actually happened? Use timestamps and data.
2. **Focus on systems** — "The system allowed X" not "Person Y did X"
3. **Ask "why" multiple times** — Don't stop at first cause
4. **Identify contributing factors** — Not just root cause, but conditions
5. **Be specific about actions** — Not "improve monitoring" but "add alert X"
6. **Include what went well** — Balance, don't just focus on problems
7. **Keep it actionable** — Every problem should have an action item

Key phrase: "The system should have prevented X" vs "X should have done Y".

### Q3: How do you ensure action items are completed?

**A**:

1. **One owner per action** — Clear responsibility
2. **Due dates** — Time-bound, not "eventually"
3. **Follow-up process** — Monthly review of all open items
4. **Metrics** — Track action item completion rate
5. **Escalation** — Uncompleted items escalate to management
6. **Verification** — Confirm action actually prevents recurrence

### Q4: How do you handle a postmortem for a security incident?

**A**:

Security postmortems have additional considerations:

1. **Legal review** — May need legal involvement before writing
2. **Forensics preservation** — Don't destroy evidence during response
3. **Customer notification** — If PII was exposed, notification requirements
4. **Separate from regular postmortems** — Security has different distribution
5. **Focus on detection** — How did the attacker get in? What signals were missed?

Security postmortem template includes:
- Attack vector analysis
- Detection gap analysis  
- Access control review
- Logging/auditing gaps
- Customer notification timeline (if applicable)

### Q5: What makes a postmortem ineffective?

**A**:

1. **Blaming language** — Creates fear, suppresses disclosure
2. **No action items** — Postmortem without change is just documentation
3. **Vague actions** — "Improve monitoring" can't be verified
4. **Missing timeline** — Can't verify or learn without sequence
5. **Too long** — If it's 50 pages, no one reads it
6. **No follow-up** — Actions never get done

A good test: Could someone read this postmortem in 10 minutes and know exactly:
- What happened
- Why it happened
- What needs to change
- Who is responsible for each change

## Cheat Sheet

### Postmortem Template Structure

```markdown
1. Title & Metadata (1 section)
   - Date, Severity, Duration, Authors

2. Executive Summary (1 paragraph)
   - What happened, impact, key learnings

3. Impact (bullet points)
   - Users affected, revenue, duration, SLA

4. Timeline (table)
   - Timestamps and events

5. Root Cause Analysis
   - What happened
   - Why it happened (5 Whys)
   - Contributing factors

6. Detection Analysis
   - How detected
   - What could detect earlier

7. Response Analysis
   - What went well
   - What could improve

8. Action Items (table)
   - ID, Action, Owner, Due, Status

9. Lessons Learned (bullets)
   - Key takeaways
```

### Action Item Quality Checklist

```
[ ] Specific - Can someone verify completion?
[ ] Owned - Single person responsible
[ ] Dated - Has a due date
[ ] Tracked - In project management system
[ ] Verified - Completion has been checked
```

```

❌ "User error" → ✅ "The interface didn't prevent..."
❌ "Developer mistake" → ✅ "The code path wasn't covered by..."
```
