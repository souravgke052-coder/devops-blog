# Reliability

## Overview

Reliability is the measure of a system's ability to function correctly and consistently over time. In SRE, reliability is not about building perfect systems — that's impossible and expensive. Instead, reliability is about making explicit tradeoffs between the cost of improving reliability and the cost of failures, then systematically improving reliability where it matters most.

Reliability is a product feature. Users choose services based on reliability. Businesses succeed or fail based on reliability. But reliability must be balanced against features and velocity — an infinitely reliable system that ships nothing has no value. SRE's error budget framework provides a principled way to balance these competing priorities.

The goal is not 100% reliability — that's impossible and not cost-effective. The goal is "reliable enough" — defined by user expectations, business requirements, and competitive benchmarks. Once you know what "reliable enough" is (your SLO), you can systematically improve toward that target.

## When To Use

- **Designing new systems** — Build reliability in from the start
- **Improving existing systems** — Systematic reliability improvement programs
- **Setting SLOs** — Understanding what reliability level is needed
- **Capacity planning** — Knowing how much redundancy is needed
- **Incident review** — Understanding reliability impact of failures
- **Architecture decisions** — Evaluating reliability tradeoffs

## When NOT to Use

- **Perfect systems** — Impossible and not cost-effective
- **Before understanding user needs** — Reliability targets should match expectations
- **Without measurement** — You can't improve what you don't measure
- **At all costs** — Some reliability investments aren't worth the cost

## Core Concepts

### Availability vs Reliability

**Availability**: The fraction of time a system is operational. "99.9% uptime"

**Reliability**: How likely a system is to function correctly over time. Related but distinct:

```
A system can have high availability but low reliability:
- Uptime: 99.9% (only 8.76 hours downtime/year)
- But: When it works, it has bugs, data corruption, or errors

A system can have high reliability but lower availability:
- Never produces wrong results
- But: May have scheduled maintenance windows
```

Reliability is about correctness, availability is about uptime. Both matter.

## Real Production Example

### Availability Calculations

```python
def calculate_availability(mtbf_hours, mttr_hours):
    """
    Calculate availability percentage.
    
    Args:
        mtbf_hours: Mean time between failures in hours
        mttr_hours: Mean time to repair in hours
    """
    total_cycle = mtbf_hours + mttr_hours
    return (mtbf_hours / total_cycle) * 100 if total_cycle > 0 else 0

# Example: System fails every 720 hours (30 days), takes 1 hour to repair
mtbf = 720  # 30 days between failures
mttr = 1    # 1 hour to repair
availability = calculate_availability(mtbf, mttr)
print(f"Availability: {availability:.4f}%")  # 99.86%

# Example: System fails every 720 hours, takes 30 minutes to repair
mttr = 0.5  # 30 minutes
availability = calculate_availability(mtbf, mttr)
print(f"Availability: {availability:.4f}%")  # 99.93%
```

### Composite System Availability

```python
def composite_availability_serial(components):
    """
    Calculate availability for components in series.
    System fails if ANY component fails.
    
    Availability = A1 × A2 × A3 × ... × An
    
    Example: 99.9% API × 99.9% DB × 99.9% Cache = 99.7%
    """
    result = 1.0
    for avail in components:
        result *= avail / 100
    return result * 100

def composite_availability_parallel(components):
    """
    Calculate availability for components in parallel.
    System fails only if ALL components fail.
    
    Availability = 1 - (1-A1) × (1-A2) × ... × (1-An)
    
    Example: Two 99% available systems in parallel = 99.99%
    """
    result = 1.0
    for avail in components:
        result *= (100 - avail) / 100
    return (1 - result) * 100

# Example: API (99.9%), Database (99.99%), Cache (99.9%)
components = [99.9, 99.99, 99.9]
serial_avail = composite_availability_serial(components)
print(f"Overall availability (serial): {serial_avail:.4f}%")  
# 99.79%

# Example: Two API instances, each 99.9%, running in parallel
# If one fails, the other handles traffic
api_instances = [99.9, 99.9]
parallel_avail = composite_availability_parallel(api_instances)
print(f"API availability (2x parallel): {parallel_avail:.4f}%")
# 99.99%
```

## Metrics / Formulas / Frameworks

### Availability Math

```python
import math

def downtime_from_availability(target_pct, period_days):
    """
    Calculate allowed downtime for target availability.
    """
    allowed_downtime_pct = 100 - target_pct
    total_minutes = period_days * 24 * 60
    return total_minutes * (allowed_downtime_pct / 100)

def reliability_probability(failure_rate_per_hour, time_hours):
    """
    Calculate probability system works for given time.
    
    Based on exponential distribution.
    """
    return math.exp(-failure_rate_per_hour * time_hours)

def mtbf_from_failure_rate(failure_rate_per_hour):
    """Convert failure rate to MTBF."""
    return 1 / failure_rate_per_hour if failure_rate_per_hour > 0 else float('inf')

def failure_rate_from_mtbf(mtbf_hours):
    """Convert MTBF to failure rate."""
    return 1 / mtbf_hours if mtbf_hours > 0 else 0

# Examples
print(f"Downtime at 99.9% over 30 days: {downtime_from_availability(99.9, 30):.1f} minutes")
# 43.8 minutes

print(f"Downtime at 99.99% over 30 days: {downtime_from_availability(99.99, 30):.1f} minutes")
# 4.4 minutes

# Reliability after 24 hours (1 failure per 720 hours = 1 per month)
failure_rate = 1 / 720  # per hour
reliability = reliability_probability(failure_rate, 24)
print(f"Reliability over 24h: {reliability:.4f} ({reliability*100:.2f}%)")
# 96.72%
```

### Reliability Investment Analysis

```python
def reliability_investment_analysis(current_reliability, target_reliability, 
                                     cost_per_incident, incidents_per_year):
    """
    Analyze whether reliability investment is worth it.
    """
    # Calculate cost of current unreliability
    current_cost = cost_per_incident * incidents_per_year
    
    # Estimate improvement from investment
    # Doubling reliability roughly halves incident rate (rough approximation)
    improvement_factor = target_reliability / current_reliability
    improved_incidents = incidents_per_year / improvement_factor
    
    # Cost of improvement
    # These would be actual estimates
    investment_cost = estimate_investment_cost(current_reliability, target_reliability)
    improved_cost = cost_per_incident * improved_incidents
    
    roi = (current_cost - improved_cost - investment_cost) / investment_cost * 100
    
    return {
        'current_annual_cost': current_cost,
        'improved_annual_cost': improved_cost,
        'investment_cost': investment_cost,
        'roi_percentage': roi,
        'payback_months': investment_cost / ((current_cost - improved_cost) / 12)
    }

def reliability_cost_curve():
    """
    Show that reliability costs increase exponentially.
    """
    reliability_levels = [99, 99.5, 99.9, 99.95, 99.99, 99.999]
    relative_costs = [1, 1.5, 2.5, 4, 8, 20]
    
    return list(zip(reliability_levels, relative_costs))
# Output: [(99, 1), (99.5, 1.5), (99.9, 2.5), (99.95, 4), (99.99, 8), (99.999, 20)]
```

## Troubleshooting / Common Mistakes

### Common Reliability Mistakes

| Mistake | Impact | Solution |
|---------|--------|----------|
| Over-engineering | Wasted resources | Match reliability to actual needs |
| Under-monitoring | Slow detection | Monitor at all critical points |
| No error budget | Unmanaged reliability | Set SLOs, track budget |

| No rollback plan | Extended outages | Automate rollback, test regularly |

## Interview Questions

### Q1: How do you decide what reliability level to target?

**A**: I base reliability targets on multiple factors:

1. **User expectations**: What do users expect for this type of service?
   - Banking: 99.99%+ (users will leave if unreliable)
   - Social media: 99.9% (users tolerate brief issues)
   - Batch processing: 99% (runs in background, retries OK)

2. **Business impact**: What does downtime cost?
   - Calculate cost per hour of downtime
   - Compare to cost of improving reliability
   - ROI analysis

3. **Competitive benchmarks**: What do similar services offer?
   - Don't significantly underperform competitors
   - Don't over-engineer beyond market standards

4. **Technical feasibility**: What can we actually achieve?
   - Consider current architecture
   - Factor in dependency reliability
   - Plan improvement roadmap

The result is typically a range: "Between 99.9% and 99.99%" — then I set SLOs at the achievable end and improve from there.

### Q2: How do you design for reliability?

**A**: Reliability design principles:

1. **Design for failure**: Assume components WILL fail
   - Implement circuit breakers
   - Use bulkheads to isolate failures
   - Design graceful degradation

2. **Eliminate single points of failure**:
   - Redundancy at each layer
   - Understand shared failure modes
   - Test failover behavior

3. **Automate recovery**:
   - Self-healing infrastructure
   - Automated rollback
   - Auto-scaling

4. **Monitor deeply**:
   - Golden signals at minimum
   - Component-level health
   - Predictive metrics (trending issues)

5. **Reduce complexity**:
   - Simpler systems fail in simpler ways
   - Minimize dependencies
   - Prefer proven patterns

### Q3: What's the difference between horizontal and vertical scaling?

**A**:

| Aspect | Vertical (Scale Up) | Horizontal (Scale Out) |
|--------|--------------------|-----------------------|
| **How** | Add more CPU/RAM to existing machine | Add more machines |
| **Limitation** | Hardware limits | Application must support distributed |
| **Reliability** | Single point of failure | Built-in redundancy if done right |
| **Cost** | Discrete jumps (buy bigger machine) | Linear (pay per instance) |
| **Operational complexity** | Lower | Higher (distributed system challenges) |
| **Best for** | Databases, stateful services | Stateless web services, APIs |

Modern cloud-native architectures typically prefer horizontal scaling for reliability (easy to add/remove capacity, inherent redundancy), but some workloads (databases) still benefit from vertical scaling.

### Q4: How do you balance reliability with feature velocity?

**A**: The error budget framework provides a principled answer:

- **Budget healthy**: Ship features aggressively
- **Budget depleting**: Pause features, invest in reliability
- **Budget exhausted**: All-hands on reliability, no new features

Key practices:
1. **Set SLOs and track error budget** — Visible, objective measure
2. **Automate reliability work** — Prevents it from competing with features
3. **Build reliability into deployment** — Rollback, canary, feature flags
4. **Prioritize high-impact reliability work** — Not everything matters equally
5. **Measure toil** — Automated away repetitive work doesn't compete with features

### Q5: What is chaos engineering and when should you use it?

**A**: Chaos engineering is the practice of deliberately injecting failures into systems to test their resilience. Netflix pioneered this with Chaos Monkey.

**When to use**:
- Building confidence in complex distributed systems
- Validating that redundancy actually works
- Finding hidden failure modes before users do
- Testing disaster recovery procedures

**How to do it safely**:
1. Start with non-production experiments
2. Define steady-state hypothesis (what should work)
3. Inject small, controlled failures
4. Observe and learn
5. Gradually increase scope

**Tools**: Litmus (Kubernetes), Chaos Monkey (Netflix), Gremlin (commercial)

**Don't do**:
- Run chaos in production without mature monitoring
- Target critical systems without stakeholder buy-in
- Inject failures that will definitely cause incidents

### Reliability Formulas

```python
# Availability
availability = MTBF / (MTBF + MTTR)

# MTBF
MTBF = Total Operational Time / Number of Failures

# MTTR
MTTR = Total Repair Time / Number of Repairs

# Reliability (probability of success over time)
R(t) = e^(-λt)  where λ = failure rate = 1/MTBF

# Series system availability (both must work)
A_series = A1 × A2 × A3 × ...

# Parallel system availability (either can work)
A_parallel = 1 - (1-A1) × (1-A2) × ...
```

### Redundancy Availability

```
N systems at 99% each:
  Series: 99% (any failure fails the system)
  Parallel (any): 99.99% (all must fail to fail)

N=2 systems at 99.9% each:
  Parallel: 99.9999% (0.0001% * 0.0001% = both fail)

N=3 systems at 99% each:
  2-of-3 voting: 99.99%+
```
