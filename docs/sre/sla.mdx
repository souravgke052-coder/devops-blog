# SLA (Service Level Agreement)

## Overview

A Service Level Agreement (SLA) is a formal commitment between a service provider and a customer, defining the level of service expected. Unlike SLOs (which are internal targets) and SLIs (which are measurements), SLAs carry contractual obligations and often financial consequences for non-compliance.

SLAs typically have legal weight. They're found in contracts, terms of service, and service descriptions. Breaching an SLA can trigger service credits, refunds, penalties, or even contract termination. This makes SLAs more conservative than SLOs — you promise less than you aim for.

The relationship is hierarchical: **SLI ≤ SLO ≤ SLA**. Your SLI (actual measurement) should consistently exceed your SLO (internal target), which should exceed the SLA (customer commitment). If your SLA is 99.9%, your SLO should be 99.95% and your actual performance should be 99.98%.

## When To Use

- **B2B or enterprise services** — When you have contractual relationships with customers
- **Regulated industries** — Healthcare, finance, and other regulated sectors often require documented SLAs
- **Multi-tenant platforms** — SaaS products with free/paid tiers define SLAs per tier
- **Partnership agreements** — When other businesses depend on your service
- **Customer-facing commitments** — Any time you're promising specific reliability to users

## When NOT To Use

- **Internal services** — Use SLOs for internal systems, not SLAs
- **Development/staging** — SLA definitions are for production
- **Without legal review** — SLA contracts should be reviewed by legal
- **When you can't measure** — Don't promise what you can't track

## Core Concepts

### SLA Structure

A complete SLA document typically includes:

1. **Service scope** — What services are covered
2. **Availability target** — The specific uptime/downtime commitment
3. **Measurement methodology** — How availability is calculated
4. **Exclusions** — What doesn't count against SLA (planned maintenance, force majeure)
5. **Credits/remedies** — What happens when SLA is breached
6. **Escalation process** — How to report and resolve SLA breaches
7. **Review period** — How often SLA terms are revisited

### SLA Tiers

Many services offer tiered SLAs based on customer pricing:

### SLA vs SLO Differences

| Aspect | SLA | SLO |
|--------|-----|-----|
| **Audience** | External customers | Internal engineering |
| **Enforcement** | Contractual/legal | Operational policies |
| **Consequences** | Financial penalties, credits | Internal prioritization |
| **Flexibility** | Hard to change | Easy to adjust |
| **Precision** | Conservative (lower) | Aggressive (higher) |
| **Coverage** | User-facing services | All critical services |

### Measurement Windows

SLAs are typically measured monthly and reported:

```yaml
# SLA measurement example
measurement:
  window: Calendar month (UTC)
  calculation: |
    Uptime Percentage = 
      (Total Minutes in Month - Downtime Minutes) / Total Minutes in Month × 100%
  
  downtime_includes:
    - Unplanned outages
    - Degraded performance below threshold
    - Service errors
    
  exclusions:
    - Scheduled maintenance (with 48h notice)
    - Force majeure events
    - Customer-caused issues
    - Third-party service failures outside our control
```

## Metrics / Formulas / Frameworks

### SLA Calculation

```python
def calculate_sla_achievement(total_minutes, downtime_minutes, exclusions=0):
    """
    Calculate SLA achievement percentage.
    
    Args:
        total_minutes: Total minutes in measurement period
        downtime_minutes: Minutes of unplanned downtime
        exclusions: Minutes excluded from calculation
    """
    billable_minutes = total_minutes - exclusions
    available_minutes = billable_minutes - downtime_minutes
    return (available_minutes / billable_minutes) * 100

# Example: 30-day month, 45 minutes downtime
total = 30 * 24 * 60  # 43,200 minutes
downtime = 45  # minutes
achievement = calculate_sla_achievement(total, downtime)
print(f"SLA Achievement: {achievement:.4f}%")
# 99.896%
# Below 99.9% SLA threshold
```

### Service Credit Calculation

```python
def calculate_service_credit(monthly_fee, actual_sla, target_sla):
    """
    Calculate service credit based on SLA shortfall.
    
    Returns credit amount or 0 if SLA was met.
    """
    if actual_sla >= target_sla:
        return 0  # No credit needed
    
    shortfall = target_sla - actual_sla
    
    # Credit tiers (example)
    if shortfall < 0.1:  # < 0.1% below
        credit_percent = 10
    elif shortfall < 0.5:  # < 0.5% below
        credit_percent = 25
    else:  # >= 0.5% below
        credit_percent = 50
    
    credit_amount = monthly_fee * (credit_percent / 100)
    return credit_amount

# Example
monthly_fee = 5000  # $5,000/month
actual_sla = 99.85  # Actual achievement
target_sla = 99.9   # SLA target
credit = calculate_service_credit(monthly_fee, actual_sla, target_sla)
print(f"Service Credit: ${credit:.2f}")
# Service Credit: $250.00 (5% of $5,000)
```

### Multi-Component SLA

```python
def composite_sla(services):
    """
    Calculate composite SLA for multiple services.
    System is available only if ALL services are available.
    """
    composite = 1.0
    for sla in services:
        composite *= sla / 100
    return composite * 100

# Example: API + Database + Cache
services = [99.95, 99.99, 99.9]  # Each at their respective SLAs
composite = composite_sla(services)
print(f"Composite SLA: {composite:.4f}%")
# 99.84%
```

### RTO/RPO Calculation

```python
def calculate_downtime_risk(target_sla, target_rto_hours, incidents_per_year):
    """
    Calculate expected downtime and business risk.
    """
    # Annual downtime based on SLA
    annual_downtime_minutes = (100 - target_sla) / 100 * 365 * 24 * 60
    annual_downtime_hours = annual_downtime_minutes / 60
    
    # Risk exposure (potential data loss)
    potential_data_loss_hours = incidents_per_year * target_rto_hours
    
    return {
        "annual_downtime_hours": annual_downtime_hours,
        "annual_downtime_minutes": annual_downtime_minutes,
        "incidents_per_year": incidents_per_year,
        "potential_data_loss_hours": potential_data_loss_hours
    }

# Example
risk = calculate_downtime_risk(99.9, 4, 3)  # 99.9% SLA, 4hr RTO, 3 incidents/year
print(risk)
# {
#     'annual_downtime_hours': 8.76,
#     'annual_downtime_minutes': 525.6,
#     'incidents_per_year': 3,
#     'potential_data_loss_hours': 12
# }
```

## Troubleshooting / Common Mistakes

### Common SLA Mistakes

| Mistake | Impact | Solution |
|---------|--------|----------|
| SLA tighter than SLO | Constant breaches, credit liability | SLO should be stricter than SLA |
| Not tracking exclusions | Over-counting downtime | Define and monitor exclusions |
| No measurement methodology | Disputes with customers | Document calculation clearly |
| SLA credits too generous | Revenue loss | Model credit exposure before agreeing |
| Not reviewing SLA regularly | Stale targets | Annual SLA review with data |

### SLA Troubleshooting

**Symptom**: Consistent SLA misses despite good SLO performance

**Diagnosis**:
```bash
# 1. Check if SLO and SLA thresholds match
# SLO: 99.95%, SLA: 99.9% - should be fine
# But check calculation methodology differences

# 2. Check measurement window differences
# Maybe SLA measures calendar month while SLO uses rolling window

# 3. Check if exclusions aren't being applied
# Maintenance windows should reduce downtime count
```

## Interview Questions

### Q1: What's the difference between SLA, SLO, and SLI?

**A**: Think of them as a hierarchy:

- **SLI (Service Level Indicator)**: What you measure. The raw metric. "Our API returns 99.7% successful responses"
- **SLO (Service Level Objective)**: What you target. The internal goal. "We aim for 99.9% availability"  
- **SLA (Service Level Agreement)**: What you promise. The contractual commitment. "We guarantee 99.9% or issue credits"

The key distinction: SLAs have legal/financial consequences. You should always exceed your SLA internally (via SLOs), so you never accidentally breach the customer-facing commitment.

### Q2: How do you handle SLA breaches?

**A**: Breach handling has both technical and customer relations aspects:

**Technical**:
1. Acknowledge and diagnose root cause
2. Implement fix or workaround
3. Verify fix resolves issue
4. Update monitoring to prevent recurrence

**Customer Relations**:
1. Notify customer per SLA terms (often within 24 hours)
2. Provide incident timeline and impact
3. Issue service credits per SLA formula
4. Share postmortem if appropriate

**Prevention**:
- Proactive SLA monitoring (project before month end)
- SLO stricter than SLA with error budget alerts
- Regular SLA performance reviews

### Q3: How do you design SLA tiers?

**A**: Tiers balance customer value with operational cost:

tier_design:
  free_tier:
    availability: 99.0%  # Lower commitment
    support: Self-service
    price: $0
    
  standard_tier:
    availability: 99.5%  # Balanced

### Q4: How do you calculate SLA credits?

**A**: Credit calculation depends on your SLA terms:

```python
def calculate_sla_credit(monthly_revenue, actual_uptime, target_uptime):
    """
    Common credit calculation approach.
    """
    if actual_uptime >= target_uptime:
        return 0  # No breach, no credit
    
    # Calculate shortfall
    shortfall = target_uptime - actual_uptime  # e.g., 0.05% (99.95% - 99.9%)
    
    # Tiered credit structure
    if shortfall <= 0.05:
        credit_percent = 10  # 10% of monthly fee
    elif shortfall <= 0.1:
        credit_percent = 25
    elif shortfall <= 0.25:
        credit_percent = 50
    else:
        credit_percent = 100  # Full refund if severe breach
        
    return monthly_revenue * (credit_percent / 100)

# Example
credit = calculate_sla_credit(10000, 99.85, 99.9)
print(f"Credit due: ${credit}")  # $250 (2.5% shortfall × 10% rate)
```

### Q5: How do you handle SLA during deployments?

**A**: Deployments are a common cause of SLA risk:

```yaml
deployment_strategy:
  blue_green:
    description: "Maintain two environments, switch traffic"
    sla_impact: "Zero downtime deployment"
    requirements:
      - Load balancer with instant switchover
      - Data sync between environments
      
  rolling:
    description: "Gradually update instances"
    sla_impact: "Maintains partial availability"
    max_unavailable: 10%  # Never exceed 10% down
      
  canary:
    description: "Test with small % first"
    sla_impact: "Limited blast radius"
    rollout: "1% → 10% → 100% over hours"
    
  maintenance_window:
    description: "Scheduled downtime with notice"
    sla_impact: "Excluded from SLA if properly noticed"
    requirements:
      - 7+ days advance notice
      - Within excluded hours (e.g., 2am-6am)
      - Limited to 4 hours/month
```

Always apply SLA exclusions for planned maintenance, but ensure proper notice is given per contract terms.
