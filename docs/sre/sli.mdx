# SLI (Service Level Indicator)

## Overview

A Service Level Indicator (SLI) is a quantitative measure of service behavior. It is the actual metric you collect to determine whether your service is meeting its reliability targets. SLIs are the foundation upon which SLOs and error budgets are built — without accurate SLIs, you cannot reliably measure or improve service quality.

An SLI measures something that matters to users: latency, availability, throughput, or correctness. The key constraint is that SLIs must be measurable, unambiguous, and directly tied to user experience. A bad SLI measures something engineers care about but users don't; a good SLI measures something users feel directly.

SLIs typically aggregate over a time window. "99.9% availability over 30 days" means you sum up all successful requests and divide by total requests over that 30-day window. The window matters: a 99.9% SLO over 30 days allows 43.8 minutes of downtime, but the same 99.9% SLO over 1 day allows only 1.44 minutes.

## When To Use

- **Defining reliability targets** — SLIs are the raw measurements that inform SLO decisions
- **Building dashboards** — SLIs power real-time visibility into service health
- **Alerting** — Alert thresholds should be based on SLI values
- **Postmortem analysis** — SLI trends show whether reliability is improving or degrading
- **SLO achievement reporting** — Business and customer reports on service reliability
- **Comparing services** — Normalized SLIs allow apples-to-apples comparison across services

## When NOT To Use

- **Before you have users** — SLIs matter when you have real traffic to measure
- **For every internal metric** — Not every metric needs to be an SLI; some are just operational
- **Without clear measurement infrastructure** — Don't define SLIs you can't actually measure
- **As the only reliability measure** — SLIs measure what's measurable, not what's important

## Core Concepts

### SLI Types

**Availability**: The fraction of requests that succeed. Typically measured as `successful_requests / total_requests`.

```
SLI_availability = successful_requests / total_requests × 100
```

For a payments API, "successful" might mean HTTP 2xx response within 30 seconds. The definition of "successful" must be explicit.

**Latency**: How long requests take. Use percentiles, not averages — a single slow request can skew averages.

```
SLI_latency_p99 = 99th percentile of request latencies
```

Typical latency SLIs: p50 (median), p95, p99. p99 is often the SLO target because it captures what "almost all" users experience.

**Throughput**: How much work the system can handle. Measured as requests per second, transactions per second, or similar.

```
SLI_throughput = requests_per_second
```

**Error Rate**: The fraction of requests that fail. Distinguish between client errors (4xx, which may be user error) and server errors (5xx, which are definitely failures).

```
SLI_error_rate = (5xx_requests + failed_requests) / total_requests × 100
```

**Freshness** (for data services): How up-to-date is the data? Critical for dashboards, feeds, and streaming systems.

```
SLI_freshness = records_older_than_threshold / total_records × 100
```

### SLI Measurement Challenges

**Missing data**: If you can't measure 100% of requests, extrapolate carefully. Sampling can introduce bias.

**Multi-region**: A request might succeed in one region and fail in another. Count success based on user-facing outcome, not internal routing.

**Degraded success**: A request that returns 200 OK but with wrong data is worse than one that returns 500. Define "success" precisely.

**Dependency SLIs**: Your SLI depends on your dependencies. If downstream services are slow, your latency SLI degrades even if your code is fine.

### Good SLI Properties (SMART)

- **Specific**: Unambiguous definition of what is being measured
- **Measurable**: Can be collected automatically without manual intervention
- **Actionable**: Changes in SLI indicate something you can do something about
- **Relevant**: Directly tied to user experience
- **Timely**: Available within seconds to minutes, not hours or days

## Real Production Example

### Google Cloud SLI Definitions

Google Cloud defines SLIs for their managed services with precise formulas:

**Cloud Storage Availability**:
```
Availability = (1 - (FailedRequests / TotalRequests)) × 100%

Where:
- FailedRequests = requests returning 5xx or failing to complete
- TotalRequests = successful requests + failed requests
- Measured over 1-minute intervals, aggregated monthly
```

**Cloud BigQuery SLI**:
```
Success Rate = (Completed Queries / Total Queries) × 100%
- Query must complete within 100 seconds to count as success
- Timeout queries count as failures
```

## Metrics / Formulas / Frameworks

### Availability Calculations

```python
# Simple availability
def calculate_availability(successful, total):
    return (successful / total) * 100 if total > 0 else 0

# Availability with weighted windows
def weighted_availability(periods):
    """
    periods: list of (duration_hours, availability_percent)
    """
    total_weighted = sum(d * a for d, a in periods)
    total_duration = sum(d for d, _ in periods)
    return total_weighted / total_duration if total_duration > 0 else 0

# Example: 23 hours at 100%, 1 hour downtime at 0%
# weighted_availability([(23, 100), (1, 0)]) = 95.83%
```

### Latency Percentile Calculations

```python
import numpy as np

def calculate_latency_percentile(durations, percentile):
    """
    durations: list of request durations in seconds
    percentile: 0-100 (e.g., 99 for p99)
    """
    if not durations:
        return None
    return np.percentile(durations, percentile)

# Prometheus-style percentile from histogram buckets
def prometheus_percentile(buckets_with_values, percentile):
    """
    buckets_with_values: [(bucket_upper_bound, count), ...]
    Sorted by bucket_upper_bound ascending
    
    For Prometheus histograms:
    Each bucket contains counts for requests <= bucket_bound
    """
    total = sum(count for _, count in buckets_with_values)
    threshold = total * (percentile / 100)
    
    cumulative = 0
    for bound, count in buckets_with_values:
        cumulative += count
        if cumulative >= threshold:
            return bound
    return buckets_with_values[-1][0]  # Max bucket
```

### Composite SLI Formulas

For services with multiple components:

```python
def composite_availability(components):
    """
    Calculate overall availability for services in series.
    Each component failure causes system failure.
    
    System_Availability = A1 × A2 × A3 × ... × An
    
    Example: 99.9% API × 99.9% DB × 99.9% Cache = 99.7%
    """
    result = 1.0
    for name, availability in components:
        result *= availability / 100
    return result * 100

# Example
components = [
    ("API Gateway", 99.95),
    ("Application Servers", 99.9),
    ("Database", 99.99),
    ("Cache", 99.9),
]
print(f"Overall availability: {composite_availability(components):.4f}%")
# Output: 99.74%
```

### SLI Window Calculations

```python
def error_budget_remaining(sli_value, slo_target, window_days):
    """
    Calculate remaining error budget in minutes.
    
    Args:
        sli_value: Current SLI value as percentage (e.g., 99.95)
        slo_target: SLO target as percentage (e.g., 99.9)
        window_days: Measurement window in days
    """
    minutes_in_window = window_days * 24 * 60
    allowed_errors = (100 - slo_target) / 100 * minutes_in_window
    actual_errors = (100 - sli_value) / 100 * minutes_in_window
    remaining = allowed_errors - actual_errors
    return max(0, remaining)  # Don't go negative

def budget_burn_rate(sli_value, slo_target, window_hours, burn_window_hours):
    """
    Calculate how fast we're burning through error budget.
    
    burn_rate > 1 means we're burning faster than allowed
    burn_rate = 1 means we're on target
    burn_rate < 1 means we're better than target
    """
    minutes_in_window = window_hours * 60
    minutes_in_burn = burn_window_hours * 60
    
    budget_per_minute = (100 - slo_target) / 100 / minutes_in_window
    actual_burn_per_minute = max(0, (100 - sli_value) / 100) / minutes_in_burn
    
    return actual_burn_per_minute / budget_per_minute if budget_per_minute > 0 else 0

# Example
print(f"Error budget remaining: {error_budget_remaining(99.92, 99.9, 30):.2f} minutes")
# Output: Error budget remaining: 18.70 minutes

print(f"Burn rate: {budget_burn_rate(99.5, 99.9, 720, 1):.2f}x")
# Output: Burn rate: 8.00x (4 hours at 99.5% = burning 8x too fast)
```

### Data Freshness SLI

```python
def data_freshness_sli(records, max_age_seconds):
    """
    Calculate data freshness as percentage of records within age threshold.
    
    Args:
        records: list of (record_id, timestamp) tuples
        max_age_seconds: maximum acceptable age
    """
    import time
    now = time.time()
    fresh_records = sum(1 for _, ts in records if now - ts <= max_age_seconds)
    return (fresh_records / len(records) * 100) if records else 0

# Example: 1000 records, 10 are stale (> 5 minutes old)
# freshness_sli = 990 / 1000 = 99.0%
```

## Security

### SLI Measurement Security

**Data integrity**: Ensure SLI measurements aren't manipulated or falsified. Attackers might try to hide their activity by affecting metrics.

```yaml
# Ensure metrics instrumentation can't be disabled
# Use pushgateway with authentication
- job_name: 'pushgateway'
  static_configs:
    - targets: ['localhost:9091']
  basic_auth:
    username: 'prometheus'
    password_file: '/etc/prometheus/secure/password'
  # Or use mTLS
```

**Access control**: SLI data can reveal sensitive information about system vulnerabilities. Restrict access to production metrics.

```yaml
# Prometheus access control
groups:
  - name: security
    rules:
      # Alert if metrics endpoint returns fewer metrics than expected
      # Could indicate instrumentation tampering
      - alert: MetricsInstrumentationGap
        expr: |
          count({job="api"}) < 100  # Expected minimum metric count
        for: 5m
        labels:
          severity: warning
```

### Confidential SLIs

Some SLIs might reveal business-sensitive information (e.g., "orders per second" reveals revenue rate):

- Consider data classification for SLI dashboards
- Restrict access to executive/customer-facing SLI reports
- Aggregate to hide granular data where appropriate

## Cost Optimization

### SLI Measurement Costs

Measuring SLIs isn't free. Consider:

1. **Metrics storage**: High-resolution metrics (10-second intervals) cost more than 1-minute intervals
2. **Query complexity**: Complex PromQL queries consume more resources
3. **Cardinality**: Too many label combinations explode storage

```yaml
# Cost optimization: Use appropriate scrape intervals
# Critical SLIs: 15s interval
# Non-critical operational metrics: 60s interval
scrape_configs:
  - job_name: 'api-sli'
    scrape_interval: 15s  # For SLI-critical metrics
    metrics_path: '/metrics'
    
  - job_name: 'api-operational'
    scrape_interval: 60s  # For non-critical metrics
    metrics_path: '/metrics'
```

### Efficient SLI Queries

```promql
# Bad: Queries all time series, expensive
sum(rate(http_requests_total[5m])) by (job, instance, namespace, pod, endpoint, method, status)

# Good: Aggregate early, reduce cardinality
sum(rate(http_requests_total[5m])) by (job)

# Better: Use recording rules to pre-compute
# Recording rule: sli:request_throughput:rate5m
# Then just query: sli:request_throughput:rate5m
```

## Troubleshooting / Common Mistakes

### Common SLI Mistakes

| Mistake | Impact | Solution |
|---------|--------|----------|
| Measuring internal metrics instead of user-facing | SLO looks good while users suffer | Measure from user's perspective |
| Not defining "success" precisely | Ambiguous measurements | Define success criteria explicitly |
| Too many SLIs | Alert fatigue,分散 focus | Pick 3-5 most important SLIs |
| Not measuring all requests | Sample bias, inaccurate SLI | Ensure complete instrumentation |
| Ignoring dependency SLIs | False sense of reliability | Include dependency health |

### SLI Debugging

**Symptom**: SLI shows degradation but no user complaints

**Possible causes**:
```bash
# 1. Check if measurement includes all requests
sum(increase(http_requests_total[5m]))

# 2. Check if metric labels changed
count({job="api"}) by (__name__)

# 3. Check for aggregation issues
# Are you measuring the right thing?
sum by (region) (rate(http_requests_total[5m]))
# vs
sum(rate(http_requests_total[5m]))
```

**Symptom**: SLI shows good performance but users complain

**Diagnosis**:
```bash
# 1. Check if you're measuring the right percentiles
# Maybe p99 is fine but p99.9 is bad
histogram_quantile(0.999, 
  sum(rate(http_request_duration_seconds_bucket[5m])) by (le)
)

# 2. Check geographic distribution
sum by (region) (rate(http_requests_total{status=~"5.."}[5m]))
/
sum by (region) (rate(http_requests_total[5m]))

# 3. Check device/browser differences
sum by (client_type) (rate(http_requests_total[5m]))
```

## Interview Questions

### Q1: How do you define an SLI for a new service?

**A**: I follow the "user journey" approach:

1. **Identify user touchpoints**: What does a user do with this service?
2. **Define success criteria**: What constitutes a successful interaction?
3. **Choose measurement points**: Where can I measure this reliably?
4. **Set precision**: What aggregation window and percentiles make sense?

For a new API service, I might define:
- Availability: "HTTP request returns 2xx within 5 seconds"
- Latency: "p99 response time < 500ms"
- Throughput: "Sustained 1000 RPS"

The key is making the definition unambiguous — "successful" must have a precise technical meaning.

### Q2: What's the difference between measuring SLIs at the edge vs internally?

**A**: Measuring at the edge (load balancer, CDN) gives you the user's perspective but may miss internal failures that don't propagate to errors. Measuring internally (application logs, metrics) gives complete coverage but may include requests that failed before reaching the user.

Best practice: **Measure at both levels and reconcile**. Edge measurement for the SLO (user experience), internal measurement for debugging (root cause).

Example:
- Edge SLI shows 99.5% availability
- Internal shows 0.1% error rate
- Gap suggests responses are slow (timeout) rather than errors

### Q3: How do you handle SLIs for services with batch workloads?

**A**: Batch workloads don't have "requests" in the traditional sense. SLIs should measure:

1. **Throughput**: Records processed per second
2. **Latency**: Time from job start to completion
3. **Freshness**: How up-to-date is the output
4. **Reliability**: Jobs completed vs jobs failed

Example batch SLI:
```yaml
slis:
  - name: batch-completion-rate
    measurement: |
      sum(rate(batch_jobs_completed_total[1h]))
      /
      sum(rate(batch_jobs_started_total[1h]))
    target: "> 99.5%"
    
  - name: batch-freshness
    measurement: |
      # Percentage of records updated within 1 hour
      sum(rate(records_updated_timestamp{age_bucket="&lt;1h"}[5m]))
      /
      sum(rate(records_updated_timestamp[5m]))
    target: "> 99%"
```

### Q4: How do you avoid "metric vanity" in SLI definitions?

**A**: Metric vanity is defining SLIs that look good but don't reflect reality. Avoid by:

1. **Correlate with user feedback**: Do SLI improvements match user satisfaction?
2. **Compare edge vs internal**: Internal metrics should predict edge metrics
3. **Test assumptions**: Periodically verify that "99% success" means users got what they needed
4. **Add negative SLIs**: Also measure bad outcomes (abandonment, errors, latency spikes)

Key question: "If this SLI were 100%, would users be happy?" If not, the SLI definition needs work.

### Q5: How do SLIs differ from traditional APM metrics?

**A**: Traditional APM (Application Performance Monitoring) metrics focus on infrastructure and code performance: CPU, memory, GC pauses, thread counts. SLIs focus on user-visible behavior.

| Aspect | APM Metrics | SLIs |
|--------|-------------|------|
| Perspective | Infrastructure | User |
| Audience | Developers | SRE/Business |
| Aggregation | Various | Typically success rate, percentiles |
| Actionability | Debugging | SLO-based decisions |

Modern observability combines both: APM metrics for debugging, SLIs for SLO tracking. But don't confuse them — having low CPU doesn't mean users are happy.

## Cheat Sheet

### SLI Quick Reference

| Service Type | Key SLIs | Typical Targets |
|--------------|----------|-----------------|
| API Service | Availability, Latency p99, Error rate | 99.9%, &lt;200ms, &lt;0.1% |

### Availability Math

```python
# Convert availability % to downtime
def downtime_minutes(availability_pct, days=30):
    """Calculate allowed downtime in minutes for a given availability target."""
    return (100 - availability_pct) / 100 * days * 24 * 60

# Examples:
downtime_minutes(99.9, 30)    # 43.8 minutes
downtime_minutes(99.95, 30)   # 21.9 minutes
downtime_minutes(99.99, 30)   # 4.4 minutes

# Calculate composite availability
def composite_availability(components):
    """Availability for services in series."""
    result = 1.0
    for avail in components:
        result *= avail / 100
    return result * 100

# Example: 99.9% API + 99.9% DB = 99.8% combined
composite_availability([99.9, 99.9])  # 99.8001%
```
