# Terraform Basics

## Overview

Terraform is an Infrastructure as Code (IaC) tool developed by HashiCorp that enables you to define, provision, and manage cloud infrastructure using declarative configuration files. It supports multiple cloud providers including Google Cloud Platform (GCP), Amazon Web Services (AWS), Microsoft Azure, and hundreds of others.

## When To Use

- **Creating infrastructure from code**: Define all cloud resources in configuration files
- **Reproducible environments**: Deploy identical infrastructure across dev/staging/prod
- **Version-controlled infrastructure**: Track infrastructure changes in Git
- **Multi-cloud deployments**: Manage resources across multiple cloud providers
- **Infrastructure automation**: Automate provisioning via CI/CD pipelines

## When NOT To Use

- **Simple ad-hoc tasks**: For one-time commands, `gcloud`, `aws`, or `az` CLI may be faster
- **Container orchestration**: Use Kubernetes (GKE/EKS/AKS) for container workloads
- **Application deployment**: Use deployment tools (ArgoCD, Spinnaker) for app workloads
- **Real-time configuration management**: Use Ansible, Chef, or Puppet for config management

## Core Concepts

### Configuration File Structure

```
project/
├── main.tf           # Main configuration (resources)
├── providers.tf      # Provider configuration
├── variables.tf      # Input variables
├── outputs.tf        # Output values
├── terraform.tfvars  # Variable values
├── versions.tf       # Version constraints
└── backend.tf        # State backend (often in versions.tf)
```

### Basic Resource Block

```hcl
resource "RESOURCE_TYPE" "LOCAL_NAME" {
  # Configuration arguments
  name     = "example"
  location = "us-central1"
  
  # Resource-specific settings
  machine_type = "e2-medium"
  disk_size_gb = 50
}
```

### Provider Configuration

```hcl
provider "google" {
  project = "my-project-id"
  region  = "us-central1"
}
```

### Variable Definition

```hcl
variable "project_id" {
  description = "GCP project ID"
  type        = string
  default     = "my-project-id"
}
```

### Output Value

```hcl
output "instance_ip" {
  description = "The external IP of the instance"
  value       = google_compute_instance.my_server.network_interface[0].access_config[0].nat_ip
}
```

### main.tf

```hcl
locals {
  common_tags = {
    environment = var.environment
    managed_by  = "terraform"
  }
}

# VPC Network
resource "google_compute_network" "vpc" {
  name                    = "my-vpc"
  auto_create_subnetworks = false
  description             = "Custom VPC network"
}

# Subnet
resource "google_compute_subnetwork" "subnet" {
  name          = "my-subnet"
  network       = google_compute_network.vpc.id
  ip_cidr_range = "10.0.0.0/24"
  region        = var.region
  
  private_ip_google_access = true
  
  log_config {
    aggregation_interval = "INTERVAL_10_MIN"
    flow_sampling        = 0.5
    metadata             = "INCLUDE_ALL_METADATA"
  }
}

# Firewall rule
resource "google_compute_firewall" "allow_internal" {
  name        = "allow-internal"
  network     = google_compute_network.vpc.id
  direction   = "INGRESS"
  source_ranges = ["10.0.0.0/24"]
  
  allow {
    protocol = "tcp"
    ports    = ["0-65535"]
  }
  
  allow {
    protocol = "udp"
    ports    = ["0-65535"]
  }
  
  allow {
    protocol = "icmp"
  }
}

# Firewall for HTTP
resource "google_compute_firewall" "allow_http" {
  name        = "allow-http"
  network     = google_compute_network.vpc.id
  source_ranges = ["0.0.0.0/0"]
  
  target_tags = ["http-server"]
  
  allow {
    protocol = "tcp"
    ports    = ["80", "443"]
  }
}

# Instance template
resource "google_compute_instance_template" "web" {
  name        = "web-template"
  machine_type = var.machine_type
  
  disk {
    source_image = "debian-cloud/debian-11"
    auto_delete  = true
    boot         = true
    
    disk_size_gb = var.disk_size
    disk_type    = "pd-ssd"
  }
  
  network_interface {
    network    = google_compute_network.vpc.id
    subnetwork = google_compute_subnetwork.subnet.id
    
    access_config {
      network_tier = "PREMIUM"
    }
  }
  
  metadata = {
    startup-script = file("${path.module}/scripts/startup.sh")
  }
  
  labels = merge(local.common_tags, {
    component = "web"
  })
  
  tags = ["http-server", "https-server"]
  
  scheduling {
    on_host_maintenance = "MIGRATE"
    automatic_restart   = true
  }
  
  lifecycle {
    create_before_destroy = true
  }
}

# Managed Instance Group
resource "google_compute_region_instance_group_manager" "web_mig" {
  name        = "web-mig"
  base_instance_name = "web"
  region      = var.region
  
  version {
    instance_template = google_compute_instance_template.web.id
    name              = "v1"
  }
  
  auto_healing_policies {
    initial_delay_sec = 60
    health_check      = google_compute_health_check.web.id
  }
  
  update_policy {
    type                  = "ROLLING_TYPE_UPDATE"
    minimal_action        = "REPLACE"
    max_surge_fixed       = 1
    max_unavailable_fixed = 0
    min_ready_sec         = 60
  }
}

# Health check
resource "google_compute_health_check" "web" {
  name               = "web-health-check"
  check_interval_sec = 10
  timeout_sec        = 5
  healthy_threshold  = 2
  unhealthy_threshold = 3
  
  http_health_check {
    port         = 80
    request_path = "/healthz"
  }
}

# Load Balancer
resource "google_compute_forwarding_rule" "http" {
  name       = "http-lb"
  target     = google_compute_target_pool.web.self_link
  port_range = "80"
  
  ip_protocol = "TCP"
}

resource "google_compute_target_pool" "web" {
  name      = "web-target-pool"
  instances = google_compute_region_instance_group_manager.web_mig.instance_group
  
  health_checks = [google_compute_health_check.web.id]
}
```

### for_each and count

```hcl
# Create multiple instances using count
resource "google_compute_instance" "server" {
  count = 3
  
  name  = "server-${count.index}"
  zone  = "${var.region}-${count.index == 0 ? "a" : count.index == 1 ? "b" : "c"}"
  
  machine_type = var.machine_type
  # ...
}

# Create instances using for_each
resource "google_compute_instance" "server" {
  for_each = toset(["web", "api", "worker"])
  
  name  = each.key
  zone  = "${var.region}-a"
  
  machine_type = var.machine_type
  # ...
}
```

### Dependencies

```hcl
# Implicit dependency (automatic)
resource "google_compute_instance" "server" {
  network_interface {
    network = google_compute_network.vpc.id  # Depends on vpc
  }
}

# Explicit dependency
resource "google_sql_database_instance" "db" {
  name = "example-db"
  # ...
}

resource "google_compute_instance" "app" {
  # ...
  
  # Explicitly depend on database
  depends_on = [
    google_sql_database_instance.db
  ]
}
```

## CLI Examples

### Initialize and Plan

```bash
# Initialize Terraform (download providers, setup backend)
terraform init

# Initialize with backend configuration
terraform init \
  -backend-config="bucket=my-state-bucket" \
  -backend-config="prefix=prod"

# Create and show execution plan
terraform plan

# Save plan to file
terraform plan -out=tfplan

# Plan with variable file
terraform plan -var-file="prod.tfvars"

# Plan targeting specific resource
terraform plan -target=google_compute_instance.server
```

### Apply and Destroy

```bash
# Apply changes (with confirmation prompt)
terraform apply

# Apply without confirmation (CI/CD only)
terraform apply -auto-approve

# Apply saved plan
terraform apply tfplan

# Apply with variables
terraform apply -var="environment=prod" -var="project_id=my-project"

# Destroy resources (with confirmation)
terraform destroy

# Destroy without confirmation
terraform destroy -auto-approve

# Destroy specific resource
terraform destroy -target=google_compute_instance.server
```

### State Management

```bash
# Show current state
terraform show

# List resources in state
terraform state list

# Show specific resource
terraform state show google_compute_instance.server

# Move/rename resource in state
terraform state mv google_compute_instance.old google_compute_instance.new

# Remove resource from state (doesn't destroy infrastructure)
terraform state rm google_compute_instance.old

# Pull state from remote
terraform state pull > state.tfstate

# Push state to remote
terraform state push state.tfstate
```

### Import and Validate

```bash
# Import existing infrastructure
terraform import google_compute_instance.server \
  projects/my-project/zones/us-central1-a/instances/server-name

# Validate configuration
terraform validate

# Format configuration files
terraform fmt

# Check formatting without changes
terraform fmt -check

# Show providers
terraform providers

# Graph dependencies
terraform graph | dot -Tsvg > graph.svg
```

### Output

```bash
# List all outputs
terraform output

# Get specific output
terraform output load_balancer_ip

# Get raw output (no quotes)
terraform output -raw load_balancer_ip

# JSON output
terraform output -json
```

## Best Practices

### 1. Use Remote State

```hcl
# versions.tf
terraform {
  backend "gcs" {
    bucket = "my-terraform-state"
    prefix = "prod"
  }
}
```

### 2. Pin Provider Versions

```hcl
terraform {
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
  }
}
```

### 3. Use Variables for Configuration

```hcl
# Never hardcode values
# Good
resource "google_compute_instance" "server" {
  zone = var.zone
}

# Bad
zone = "us-central1-a"
```

### 4. Tag Resources

```hcl
resource "google_compute_instance" "server" {
  labels = {
    environment = var.environment
    team        = "platform"
    managed_by  = "terraform"
  }
}
```

### 5. Handle Sensitive Data

```hcl
# Mark sensitive variables
variable "db_password" {
  type      = string
  sensitive = true
}

# Don't output sensitive values
output "password" {
  value     = var.db_password
  sensitive = true
}
```

### 6. Use Modules for Reuse

```hcl
# Reuse infrastructure code
module "webserver" {
  source = "./modules/webserver"
  
  project_id  = var.project_id
  environment = var.environment
}
```

## Monitoring / State Management

### State File

Terraform stores infrastructure state in a state file. This file:
- Maps configuration to actual resources
- Tracks resource dependencies
- Enables Terraform to know what changed

### State Locking

Remote backends provide state locking to prevent concurrent modifications:

```hcl
# GCS backend (automatic locking)
terraform {
  backend "gcs" {
    bucket = "my-state-bucket"
    prefix = "prod"
  }
}
```

### State Recovery

```bash
# Enable versioning on GCS bucket
gsutil versioning set on gs://my-state-bucket

# List state versions
gsutil ls -a gs://my-state-bucket/prod/

# Restore previous version
gsutil cp gs://my-state-bucket/prod/default.2024-01-01T00:00:00.0~1 /tmp/state.tfstate
terraform state push /tmp/state.tfstate
```

## Troubleshooting

### Common Errors

#### State Lock

```
Error: Error acquiring the state lock
```
**Solution:**
```bash
# Wait for lock to be released
terraform apply

# Or force unlock (emergency only)
terraform force-unlock <lock-id>
```

#### Provider Not Initialized

```
Error: Provider not installed
```
**Solution:**
```bash
terraform init
```

#### Resource Already Exists

```
Error: Resource already exists
```
**Solution:**
```bash
# Import existing resource
terraform import google_compute_instance.server \
  projects/PROJECT/zones/ZONE/instances/NAME
```

#### Circular Dependency

```
Error: Cycle: resource references
```
**Solution:** Restructure resources to break the cycle. Common causes:
- Two resources depending on each other
- Using `depends_on` unnecessarily

## Interview Questions

### Q1: What is Terraform and how does it work?
**A:** Terraform is an IaC tool that uses declarative configuration files to provision and manage infrastructure. It compares desired state (config) with actual state (infrastructure) and creates an execution plan to reconcile differences.

### Q2: What is the difference between Terraform and cloud-specific IaC tools?
**A:** Terraform is provider-agnostic - one language (HCL) works across GCP, AWS, Azure, and 100+ providers. Cloud-specific tools (Deployment Manager, CloudFormation) are limited to their provider.

### Q3: What is a Terraform provider?
**A:** A provider is a plugin that interacts with cloud APIs. Official providers exist for major clouds, and custom providers can be developed for other services.

### Q4: What is Terraform state and why is it important?
**A:** State maps configuration to actual resources and tracks dependencies. It enables Terraform to know what changed and what to update. State should be stored remotely with locking for teams.

### Q5: What is the difference between `terraform apply` and `terraform plan`?
**A:** `terraform plan` shows what changes would be made without making them. `terraform apply` executes the changes to match configuration.

### Q6: What are Terraform modules?
**A:** Modules are reusable, packaged collections of Terraform resources. They enable code reuse, abstraction, and organization. Use `module` block to instantiate.

### Q7: How do you handle secrets in Terraform?
**A:** Never store secrets in config or state files. Use Secret Manager, environment variables (`TF_VAR_*`), or CI/CD secrets injection.

### Q8: What is the purpose of `terraform init`?
**A:** Initializes a Terraform working directory by downloading provider plugins, setting up the backend, and preparing the environment for `plan` and `apply`.

### Q9: How do you import existing infrastructure?
**A:** Use `terraform import RESOURCE_TYPE.NAME RESOURCE_ID`. The resource block must already exist in config with matching resource type and name.

### Q10: What is the difference between `count` and `for_each`?
**A:** `count` creates multiple instances using numeric index. `for_each` creates instances using a map or set of strings, allowing named instances and more flexibility.
