# Terraform Lifecycle

## Overview

The lifecycle block in Terraform controls how resources are created, updated, and destroyed. Understanding lifecycle management is essential for zero-downtime deployments, data safety, and handling complex infrastructure changes.

## When To Use

- **Zero-downtime deployments**: Use `create_before_destroy` for in-place updates
- **Prevent accidental deletion**: Protect critical resources with `prevent_destroy`
- **Ignore external changes**: Don't reconcile changes made outside Terraform
- **Custom update behavior**: Control how updates are applied

## When NOT To Use

- **Default behavior is acceptable**: Terraform's default create-before-destroy for most resources
- **Temporary resources**: Lifecycle policies add unnecessary complexity

## Core Concepts

### Lifecycle Block

```hcl
resource "google_compute_instance" "example" {
  # Resource configuration
  
  lifecycle {
    create_before_destroy = true
    prevent_destroy       = false
    ignore_changes        = []
  }
}
```

### Default Behavior

```
Terraform Default Lifecycle:
1. Create new resource
2. Update dependencies
3. Destroy old resource
```

### Custom Lifecycle

```
Custom Lifecycle (create_before_destroy = true):
1. Create new resource
2. Update dependencies to new resource
3. Destroy old resource
```

```hcl
resource "google_compute_region_instance_group_manager" "app_mig" {
  name                  = "app-mig-${var.environment}"
  base_instance_name    = "app"
  region                = var.region
  target_size           = var.target_size
  
  version {
    name            = "v1"

    initial_delay_sec = 60

    type                  = "ROLLING_TYPE_UPDATE"
    minimal_action        = "REPLACE"
    max_surge_fixed       = 3
    max_unavailable_fixed = 0
    min_ready_sec         = 60

    rolling_update_policy {
      type                   = "PROACTIVE"  # PROACTIVE or OPPORTUNISTIC
      minimal_action         = "REPLACE"
      max_surge_fixed        = 3
      max_unavailable_fixed  = 0
      replacement_method     = "RECREATE"  # or "SUBSTITUTE"
    }
  }
  
  # Stateful MIG configuration
  stateful_disks {
    device_name = "boot"
    source      = google_compute_instance_template.app.disks[0].source
    mode        = "READ_ONLY"  # or READ_WRITE
  }
  
  # Wait for instances to be created
  wait_for_instances = true
  wait_for_instances_stady = true
}
```

### Prevent Destroy

```hcl
# Critical resources that should never be accidentally deleted
resource "google_kms_key" "critical" {
  name = "critical-key"
  
  lifecycle {
    prevent_destroy = true
  }
}

# Set to false only during intentional migration
resource "google_kms_key" "critical" {
  name = "critical-key"
  
  lifecycle {
    prevent_destroy = false  # Set to allow destruction
  }

```

### Ignore Changes

```hcl
# Ignore changes made outside Terraform
resource "google_compute_instance" "app" {
  # ...
  
  lifecycle {
    ignore_changes = [
      # Ignore metadata changes (may be set by external systems)
      metadata,
      # Ignore tags modified by other tools
      labels,
    ]
  }
}

# Use with tags for GitOps scenarios
resource "google_compute_instance" "app" {
  name = "app-server"
  tags = ["app"]
  
  lifecycle {
    ignore_changes = [
      tags,  # GitOps tool adds tags
    ]
  }
}
```


## Best Practices

### 1. Use Create Before Destroy for Critical Services

```hcl
# For databases, use create_before_destroy for major version upgrades
resource "google_sql_database_instance" "postgres" {
  lifecycle {
    create_before_destroy = true
  }
}
```

### 2. Protect Critical Resources

```hcl
# Always protect production resources
resource "google_kms_key" "prod_key" {
  lifecycle {
    prevent_destroy = true
  }
}

resource "google_sql_database_instance" "prod_db" {
  deletion_protection = true
  
  lifecycle {
    prevent_destroy = true
  }
}
```

### 3. Use Ignore Changes for External Modifications

```hcl
# When using GitOps or external automation
resource "google_compute_instance" "app" {
  lifecycle {
    # Ignore tags added by GitOps system
    ignore_changes = [labels, metadata]
  }
}
```

### 4. Configure Timeouts

```hcl
# Large resources need longer timeouts
resource "google_container_cluster" "primary" {
  timeouts {
    create = "45m"
    update = "45m"
    delete = "45m"
  }
}
```

### 5. State Management for Stateful Resources

```hcl
# Stateful MIG keeps instance state
resource "google_compute_region_instance_group_manager" "stateful" {
  # Stateful disks preserved across instance recreation
  stateful_disks {
    device_name = "data-disk"
    mode        = "READ_WRITE"
  }
  
  lifecycle {
    create_before_destroy = true
  }
}
```

### 6. Canary Deployments

```hcl
# Deploy to subset of instances first
resource "google_compute_region_instance_group_manager" "mig" {
  version {
    instance_template = google_compute_instance_template.canary.id
    target_size       = 1  # Only 1 canary instance
  }
  
  # Then roll out to remaining
  update_policy {
    rolling_update_policy {
      type                   = "PROACTIVE"
      minimal_action         = "REPLACE"
      max_surge_fixed        = 3
      max_unavailable_fixed  = 0
    }
  }
}
```

### Lifecycle and State Lock

```bash

# Create before destroy: create new -> update dependencies -> delete old
# Without: create new -> delete old -> update dependencies (breaks dependencies)
```


## Troubleshooting

### Prevent Destroy Error

```bash
# Error: Cannot destroy resource
# Solution: Set prevent_destroy to false before destroy
# Edit resource:
lifecycle {
  prevent_destroy = false
}

terraform apply
terraform destroy
```

### Create Before Destroy Loop

```bash
# Error: Cyclic dependency
# Solution: Break the cycle
# Use depends_on to establish explicit ordering
resource "google_sql_database_instance" "db" {
  depends_on = [google_service_account.app]
  
  lifecycle {
    create_before_destroy = true
  }
}
```

### Ignore Changes Not Working

```bash
# If ignore_changes doesn't prevent updates:
# 1. Verify syntax is correct
# 2. Check Terraform version supports the attribute
# 3. Use precondition/postcondition instead

# Some attributes cannot be ignored
lifecycle {
  ignore_changes = [
    # Cannot ignore computed attributes
    id,
    self_link,
  ]
}
```

### Timeout Issues

```bash
# Error: Timeout during operation
# Solution: Increase timeouts
resource "google_container_cluster" "primary" {
  timeouts {
    create = "60m"
    update = "60m"
    delete = "60m"
  }
}
```

## Interview Questions

### Q1: What is `create_before_destroy` lifecycle policy?
**A:** Creates new resource before destroying old one. Enables zero-downtime updates but doubles resource count during updates.

### Q2: When should you use `prevent_destroy`?
**A:** On critical resources that should never be deleted - encryption keys, service accounts, production databases, VPCs.

### Q3: What does `ignore_changes` do?
**A:** Tells Terraform to ignore changes made to specific attributes outside of Terraform - useful for GitOps or external automation.

### Q4: Can you use `prevent_destroy` on instance templates?
**A:** No, instance templates don't have state so `prevent_destroy` has no effect. Use it on persistent resources like disks, databases, KMS keys.

### Q5: How do timeouts work in Terraform?
**A:** `timeouts` block configures duration for create/update/delete operations. If operation exceeds timeout, Terraform fails with timeout error.

### Q6: What's the difference between lifecycle and timeouts?
**A:** Lifecycle controls create/destroy ordering and behavior. Timeouts control how long operations can take before failing.

### Q7: How do you do canary deployments with Terraform?
**A:** Create new instance template with canary changes, deploy to small subset via target_size, then update policy to roll out gradually.

### Q8: What happens if prevent_destroy is true and you run terraform destroy?
**A:** Terraform will fail with error: "Cannot destroy resource." Must set to false first.

## Cheat Sheet

### Lifecycle Block
```hcl
lifecycle {
  create_before_destroy = true   # Create before destroy
  prevent_destroy       = true   # Block destruction
  ignore_changes        = [...]  # Ignore external changes
}
```

### Timeouts Block
```hcl
timeouts {
  create = "30m"
  update = "30m"
  delete = "30m"
  read   = "20m"
}
```

### Common Lifecycle Patterns
| Pattern | Use Case |
|---------|----------|
| `create_before_destroy` | Zero-downtime updates, databases |
| `prevent_destroy` | Production databases, KMS keys, VPCs |
| `ignore_changes` | Tags, labels, GitOps scenarios |
