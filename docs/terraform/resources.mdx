# Terraform Resources

## Overview

Resources are the fundamental building blocks in Terraform. Each resource block describes a specific infrastructure object (compute instance, network, storage bucket, etc.). Understanding resource configuration is critical for building production-grade infrastructure.

## When To Use

- **Creating infrastructure**: All cloud resources are defined as resources
- **Updating existing resources**: Modify resource attributes in-place
- **Destroying resources**: Terraform manages lifecycle via `terraform destroy`
- **Importing existing infrastructure**: Map existing resources to Terraform config

## When NOT To Use

- **Reading existing infrastructure**: Use `data` sources instead
- **Transient configuration**: Use local values or variables
- **Non-infrastructure configuration**: Use other Terraform constructs

## Core Concepts

### Resource Block Structure

```hcl
resource "TYPE" "NAME" {
  # Configuration
  
  # Meta-arguments
  depends_on = [...]
  count      = 3
  for_each   = var.instance_names
  lifecycle {
    create_before_destroy = true
  }
  provider = google.west
}
```

### Resource Dependencies

```hcl
# Implicit dependency (Terraform infers)
resource "google_compute_instance" "web" {
  network_interface {
    network = google_compute_network.vpc.id  # Depends on network
  }
}

# Explicit dependency
resource "google_compute_instance" "db" {
  # ...
  depends_on = [
    google_sql_database_instance.primary
  ]
}
```

```hcl
  }

  disk {
    source_image = "debian-cloud/debian-11"
    auto_delete  = true
    boot         = true
    
    disk_size_gb = 50
    disk_type    = "pd-ssd"
    
    encryption_key = google_kms加密密钥.crypto_key.id
  }
  
  # Network interface
  network_interface {
    network    = google_compute_network.vpc.id

    secondary_ip_range {
      ip_cidr_range = "${var.pod_cidr}.0/20"
      range_name    = "pods"
    }
    
    # Access config for public IP
    access_config {
      network_tier = "PREMIUM"
```

### Network Resources

```hcl
# VPC Network
resource "google_compute_network" "vpc" {
  name                    = "vpc-${var.environment}"
  auto_create_subnetworks = false
  description             = "Custom VPC for ${var.environment}"
  
  # Private Google Access
  private_ip_google_access = true
  
  mtu = 1460  # Jumbo frames
  
  routing_mode = "Regional"  # or GLOBAL for HA
  
  # BGP routing for Cloud Router
  bgp {
    asn = var.vpc_asn
  }
}

# Subnet
resource "google_compute_subnetwork" "private" {
  name          = "subnet-private-${var.region}-${var.environment}"
  ip_cidr_range = var.private_subnet_cidr
  network       = google_compute_network.vpc.id
  region        = var.region
  
  # Private Google Access
  private_ip_google_access = true
  
  # Private Google Access with Private NAT
  private_ipv6_google_access = "ENABLE_OUTBOUND_VM_PUCCESS_TO_GOOGLE"
  
  # Secondary ranges for pods/services
  secondary_ip_range = [
    {
      ip_cidr_range = "${var.pod_cidr}.0/20"
      range_name    = "pods"
    },
    {
      ip_cidr_range = "${var.service_cidr}.0/20"
      range_name    = "services"
    }
  ]
  
  # Flow logs for VPC
  log_config {
    aggregation_interval = "INTERVAL_10_MIN"
    flow_sampling        = 0.5
    metadata             = "INCLUDE_ALL_METADATA"
  }
  
  # Purpose for Shared VPC
  purpose = "PRIVATE"  # or REGIONAL_MANAGED |
}

# Firewall Rules
resource "google_compute_firewall" "allow_internal" {
  name        = "allow-internal-${var.environment}"
  network     = google_compute_network.vpc.id
  direction   = "INGRESS"
  priority    = 65534
  source_ranges = ["10.0.0.0/8"]
  
  allow {
    protocol = "tcp"
    ports    = ["0-65535"]
  }
  
  allow {
    protocol = "udp"
    ports    = ["0-65535"]
  }
  
  allow {
    protocol = "icmp"
  }
  
  # Exempt from logging for performance
  disabled = false
}

resource "google_compute_firewall" "allow_health_checks" {
  name        = "allow-health-checks-${var.environment}"
  network     = google_compute_network.vpc.id
  direction   = "INGRESS"
  priority    = 1000
  
  # GCP health check IP ranges
  source_ranges = [
    "35.191.0.0/16",  # Health check
    "130.211.0.0/22", # Load balancer
  ]
  
  target_tags = ["web", "api"]
  
  allow {
    protocol = "tcp"
    ports    = ["8080"]
  }
}
```

```hcl
resource "google_storage_bucket" "app_bucket" {
```

## CLI Examples

### Resource Operations

```bash
# Plan specific resource
terraform plan -target=google_compute_instance.web_server

# Apply specific resource
terraform apply -target=google_compute_instance.web_server

# Destroy specific resource
terraform destroy -target=google_compute_instance.web_server

# Import existing resource
terraform import google_compute_instance.web_server \
  projects/my-project/zones/us-central1-a/instances/web-server

# Import to module
terraform import module.vpc.google_compute_network.vpc \
  projects/my-project/global/networks/vpc-name

# List resources
terraform state list

# Show resource
terraform state show google_compute_instance.web_server
```

### Resource Graph

```bash
# Generate dependency graph
terraform graph | dot -Tsvg > graph.svg

# Graph with dependencies
terraform graph -type=plan | jq '.'

# Count resources by type
terraform state list | cut -d. -f1 | sort | uniq -c
```

## Common Commands

| Command | Description |
|---------|-------------|
| `terraform plan -target=RESOURCE` | Plan specific resource |
| `terraform apply -target=RESOURCE` | Apply specific resource |
| `terraform destroy -target=RESOURCE` | Destroy specific resource |
| `terraform import RESOURCE ID` | Import existing resource |
| `terraform state show RESOURCE` | Show resource details |
| `terraform state mv SRC DST` | Move/rename resource |
| `terraform state rm RESOURCE` | Remove from state |

## Best Practices

### 1. Use Lifecycle Policies

```hcl
resource "google_compute_instance" "server" {
  # ...
  
  lifecycle {
    create_before_destroy = true   # Zero-downtime deployments
    prevent_destroy       = false  # Set true for critical resources
    ignore_changes        = [tags] # Don't reconcile external changes
  }
}
```

### 2. Use count or for_each for Multiple Resources

```hcl
# For multiple similar resources
resource "google_compute_instance" "server" {
  count = 3
  name  = "server-${count.index}"
  # ...
}

# For dynamic resources based on map
resource "google_compute_instance" "server" {
  for_each = toset(["web", "api", "worker"])
  name     = each.key
  # ...
}
```

### 3. Use Sensitive Flag for Secrets

```hcl
variable "db_password" {
  type      = string
  sensitive = true
}

output "db_password" {
  value     = var.db_password
  sensitive = true
}
```

### 4. Add Descriptive Comments

```hcl
# Web server instances behind load balancer
resource "google_compute_instance" "web" {
  # ...
}
```

### 5. Use Remote State for Cross-Environment References

```hcl
data "terraform_remote_state" "network" {
  backend = "gcs"
  config = {
    bucket = "my-project-terraform-state"
    prefix = "network"
  }
}

resource "google_compute_instance" "app" {
  network_interface {
    subnetwork = data.terraform_remote_state.network.outputs.subnet_id
  }
}
```

## Troubleshooting

### Resource Already Exists

```bash
# Error: Resource already exists
# Solution: Import existing resource
terraform import google_compute_instance.web_server \
  projects/my-project/zones/us-central1-a/instances/web-server
```

### Resource Not Found

```bash
# Error: Resource not found during plan
# Solution: Resource was deleted outside Terraform
terraform plan  # Shows drift
terraform apply # Reconcile
```

### Circular Dependency

```bash
# Error: Cycle: resource references
# Solution: Break the cycle
# Use depends_on sparingly
resource "null_resource" "break_cycle" {
  depends_on = [google_sql_database_instance.db]
}
```

### Concurrent Modification

```bash
# Error: State lock
# Solution: Wait or force unlock
terraform apply  # Wait for lock
terraform force-unlock <lock-id>  # Emergency only
```

## Interview Questions

### Q1: What are meta-arguments in Terraform resources?
**A:** `depends_on`, `count`, `for_each`, `provider`, `lifecycle` - control resource behavior and dependencies.

### Q2: Difference between `count` and `for_each`?
**A:** `count` creates multiple instances of a resource using numeric index. `for_each` creates instances based on a map or set of strings, allowing named instances.

### Q3: What is `create_before_destroy` lifecycle policy?
**A:** Creates new resource before destroying old one, enabling zero-downtime deployments. Used with `lifecycle { create_before_destroy = true }`.

### Q4: When would you use `depends_on`?
**A:** When there's a隐性 dependency Terraform can't infer, or when ordering matters for non-resource dependencies.

### Q5: How do you import existing infrastructure?
**A:** Use `terraform import RESOURCE_TYPE.NAME RESOURCE_ID`. Resource configuration block should match actual infrastructure.

### Q6: What is resource targeting?
**A:** Using `-target=RESOURCE` flag to plan/apply specific resources without affecting others.

### Q7: How do you handle sensitive data in resources?
**A:** Mark variables as `sensitive = true`, use Secret Manager for passwords, avoid hardcoding secrets in configuration.

### Q8: What is drift in Terraform context?
**A:** When actual infrastructure differs from Terraform state due to out-of-band changes. Detected via `terraform plan`.

### Resource Syntax

### Common Meta-Arguments
| Meta-argument | Purpose |
|--------------|---------|
| `depends_on` | Explicit dependencies |

| `for_each` | Map/set of instances |
| `provider` | Non-default provider |
| `lifecycle` | Lifecycle policies |
