# Terraform Secrets Management

## Overview

Secrets management in Terraform is critical for protecting sensitive data like passwords, API keys, and certificates. Never store secrets in state files, configuration, or version control. Use secure patterns for handling secrets throughout the Terraform lifecycle.

## When To Use

- **Database credentials**: Passwords, connection strings
- **API keys**: External service authentication
- **Certificates**: SSL/TLS certificates and private keys
- **Service account keys**: Legacy authentication (prefer Workload Identity)
- **Encryption keys**: Customer-managed keys

## When NOT To Use

- **Version control**: Never commit secrets to Git
- **Plain text files**: Don't store secrets in tfvars unencrypted
- **State files**: Don't let secrets end up in state without protection

## Core Concepts

### Sensitive Variables

```hcl
variable "db_password" {
  type      = string
  sensitive = true
}

output "connection_string" {
  value     = "postgres://user:${var.db_password}@host:5432/db"
  sensitive = true
}
```

### Secret Sources

```
1. Secret Manager (Recommended for GCP)
2. Environment Variables
3. Encrypted Files
4. HashiCorp Vault
5. AWS Secrets Manager
6. Azure Key Vault
```

## Real Production Example

### GCP Secret Manager Integration

```hcl
# versions.tf
terraform {
  required_version = ">= 1.5.0"
  
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
    google-beta = {
      source  = "hashicorp/google-beta"
      version = "~> 5.0"
    }
  }
}

# =============================================================================
# Secret Manager Configuration
# =============================================================================

# Enable Secret Manager API
resource "google_project_service" "secret_manager" {
  project = var.project_id
  service = "secretmanager.googleapis.com"
}

# =============================================================================
# Secrets Creation
# =============================================================================

# Database password secret
resource "google_secret_manager_secret" "db_password" {
  secret_id = "${var.environment}-db-password"
  
  replication {
    auto {}
  }
  
  labels = {
    environment = var.environment
    managed_by  = "terraform"
  }
}

# Add secret version (initial password)
resource "google_secret_manager_secret_version" "db_password_v1" {
  secret = google_secret_manager_secret.db_password.id
  
  secret_data = var.db_password  # Set via variable or environment
}

# API key secret
resource "google_secret_manager_secret" "api_key" {
  secret_id = "${var.environment}-api-key"
  
  replication {
    auto {}
  }
  
  labels = {
    environment = var.environment
    managed_by  = "terraform"
    type        = "api-key"
  }
}

resource "google_secret_manager_secret_version" "api_key_v1" {
  secret = google_secret_manager_secret.api_key.id
  secret_data = var.api_key
}

# =============================================================================
# Service Account for Secrets Access
# =============================================================================

resource "google_service_account" "secrets_reader" {
  account_id   = "secrets-reader-${var.environment}"
  display_name = "Secrets Reader - ${var.environment}"
}

resource "google_project_iam_member" "secrets_reader" {
  project = var.project_id
  role    = "roles/secretmanager.secretAccessor"
  member  = "serviceAccount:${google_service_account.secrets_reader.email}"
}

# =============================================================================
# Database with Secret Manager Integration
# =============================================================================

data "google_secret_manager_secret_version" "db_password" {
  secret = "${var.environment}-db-password"
  # Or use resource ID: google_secret_manager_secret.db_password.id
}

resource "google_sql_database_instance" "postgres" {
  name             = "postgres-${var.environment}"
  database_version = "POSTGRES_15"
  region           = var.region
  
  deletion_protection = var.environment == "prod" ? true : false
  
  settings {
    tier              = var.db_tier
    availability_type = var.environment == "prod" ? "REGIONAL" : "ZONAL"
    
    ip_configuration {
      ipv4_enabled    = false
      private_network = google_compute_network.vpc.id
      require_ssl     = true
    }
    
    backup_configuration {
      enabled                        = true
      start_time                     = "03:00"
      point_in_time_recovery_enabled = true
    }
  }
  
  # Use secret from Secret Manager
  root_password = data.google_secret_manager_secret_version.db_password.secret_data
  
  depends_on = [
    google_project_service.secret_manager
  ]
}

# Database user with IAM authentication
resource "google_sql_user" "app_user" {
  name     = "app_user"
  instance = google_sql_database_instance.postgres.name
  
  # For IAM authentication (preferred)
  # password field omitted when using IAM auth
  type = "CLOUD_IAM_USER"
}

# Database user with password from Secret Manager
resource "google_sql_user" "app_user_password" {
  name     = "app_user_password"
  instance = google_sql_database_instance.postgres.name
  
  password = data.google_secret_manager_secret_version.db_password.secret_data
}

# =============================================================================
# Cloud SQL with Connection String Output
# =============================================================================

locals {
  db_host = google_sql_database_instance.postgres.private_ip_address
  db_name = "appdb"
}

output "database_connection_string" {
  description = "PostgreSQL connection string"
  value       = "postgresql://app_user:${data.google_secret_manager_secret_version.db_password.secret_data}@${local.db_host}:5432/${local.db_name}?sslmode=require"
  sensitive   = true
}

output "database_host" {
  description = "Database private IP"
  value       = local.db_host
  sensitive   = false
}

# =============================================================================
# Application Configuration
# =============================================================================

# Kubernetes Secret (via manifest)
resource "kubernetes_secret" "db_credentials" {
  metadata {
    name      = "db-credentials"
    namespace = var.namespace
  }
  
  data = {
    "password" = data.google_secret_manager_secret_version.db_password.secret_data
    "host"     = local.db_host
    "database" = local.db_name
  }
  
  type = "Opaque"
  
  depends_on = [
    google_container_cluster.primary
  ]
}

# Environment variables (for Cloud Run, Cloud Functions)
resource "google_cloud_run_service" "app" {
  name     = "app-${var.environment}"
  location = var.region
  
  template {
    spec {
      service_account_name = google_service_account.app.email
      
      containers {
        image = var.container_image
        
        env {
          name  = "DB_HOST"
          value = local.db_host
        }
        
        env {
          name  = "DB_NAME"
          value = local.db_name
        }
        
        env {
          name = "DB_PASSWORD"
          value_from {
            secret_key_ref {
              name = google_secret_manager_secret.db_password.secret_id
              key  = "latest"
            }
          }
        }
      }
    }
  }
}

# =============================================================================
# Secrets Rotation
# =============================================================================

# Cloud Scheduler for secret rotation
resource "google_cloud_scheduler_job" "rotate_db_password" {
  name        = "rotate-db-password-${var.environment}"
  description = "Rotate database password monthly"
  schedule    = "0 4 1 * *"  # First day of month at 4 AM
  time_zone   = "UTC"
  region      = var.region
  
  http_target {
    uri         = "https://cloudfunctions.googleapis.com/v2/projects/${var.project_id}/locations/${var.region}/functions/rotate-db-password"
    http_method = "POST"
    
    oidc_token {
      service_account_email = google_service_account.rotation_sa.email
    }
  }
}

resource "google_service_account" "rotation_sa" {
  account_id   = "rotation-sa-${var.environment}"
  display_name = "Secret Rotation Service Account"
}

resource "google_project_iam_member" "rotation_secrets" {
  project = var.project_id
  role    = "roles/secretmanager.admin"
  member  = "serviceAccount:${google_service_account.rotation_sa.email}"
}
```

### Variables for Secrets

```hcl
# variables.tf

# Database password - set via environment variable or Secret Manager
variable "db_password" {
  description = "Database master password"
  type        = string
  sensitive   = true
  
  validation {
    condition     = length(var.db_password) >= 16
    error_message = "Password must be at least 16 characters."
  }
}

# API key
variable "api_key" {
  description = "External API key"
  type        = string
  sensitive   = true
  
  validation {
    condition     = can(regex("^[A-Za-z0-9_-]{32,}$", var.api_key))
    error_message = "API key must be at least 32 characters alphanumeric."
  }
}

# SSL Certificate
variable "ssl_certificate" {
  description = "SSL certificate content"
  type        = string
  sensitive   = true
}

variable "ssl_private_key" {
  description = "SSL private key content"
  type        = string
  sensitive   = true
}
```

### CI/CD Secrets Injection

```yaml
# GitHub Actions
name: Terraform Production

on:
  push:
    branches: [main]

jobs:
  terraform:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Authenticate to GCP
        uses: google-github-actions/auth@v1
        with:
          credentials_json: ${{ secrets.GCP_SA_KEY }}
      
      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v2
        with:
          terraform_version: 1.5.0
      
      - name: Fetch Secrets
        run: |
          echo "DB_PASSWORD=$(gcloud secrets versions access latest --secret=prod-db-password)" >> $GITHUB_ENV
          echo "API_KEY=$(gcloud secrets versions access latest --secret=prod-api-key)" >> $GITHUB_ENV
      
      - name: Terraform Init
        run: terraform init -backend-config="bucket=${{ secrets.TF_STATE_BUCKET }}"
      
      - name: Terraform Plan
        env:
          TF_VAR_db_password: ${{ env.DB_PASSWORD }}
          TF_VAR_api_key: ${{ env.API_KEY }}
        run: terraform plan -out=tfplan
      
      - name: Terraform Apply
        if: github.ref == 'refs/heads/main'
        env:
          TF_VAR_db_password: ${{ env.DB_PASSWORD }}
          TF_VAR_api_key: ${{ env.API_KEY }}
        run: terraform apply -auto-approve tfplan
```

```bash
# Encrypt tfvars
gcloud kms encrypt \
  --location=global \
  --keyring=tf-keyring \
  --key=tf-key \
  --plaintext-file=prod.tfvars \
  --ciphertext-file=prod.tfvars.enc

# Decrypt and use
gcloud kms decrypt \
  --location=global \
  --keyring=tf-keyring \
  --key=tf-key \
  --ciphertext-file=prod.tfvars.enc \
  --plaintext-file=/dev/stdout | \
  terraform apply -var-file=/dev/stdin
```

## Troubleshooting

### Secret Not Found

```bash
# Error: Secret not found
# Solution: Create secret first
gcloud secrets create prod-db-password --data-file=/dev/stdin <<< "password123"

# Or import existing
gcloud secrets import prod-db-password --data-file=password.txt
```

### Permission Denied

```bash
# Error: Permission denied on secret
# Solution: Grant IAM access
gcloud secrets add-iam-policy-binding prod-db-password \
  --member="serviceAccount:terraform@project.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"
```

### Stale Secret Version

```bash
# Access specific version
data "google_secret_manager_secret_version" "db_password" {
  secret  = "prod-db-password"
  version = "5"  # Specific version
}

# Or always use latest
data "google_secret_manager_secret_version" "db_password" {
  secret = "prod-db-password"
  # version defaults to "latest"
}
```

### Sensitive Value in State

```bash
# If sensitive value appears in state:
# 1. Use Secret Manager (values fetched at runtime)
# 2. Or mark variable sensitive (masks in output)

  sensitive = true
}
```

## Interview Questions

### Q1: How do you handle secrets in Terraform?
**A:** Never store secrets in code or state. Use Secret Manager for storage, fetch at runtime via data source, mark variables sensitive, use environment variables or CI/CD secrets.

### Q2: What's wrong with storing secrets in tfvars files?
**A:** tfvars files can be accidentally committed to version control, exposed in logs, or visible in CI/CD pipelines. Use encrypted tfvars with KMS or Secret Manager.

### Q3: How does Secret Manager integrate with Terraform?
**A:** Using `google_secret_manager_secret` for creation and `google_secret_manager_secret_version` data source for retrieval.

### Q4: What's the difference between sensitive variables and Secret Manager?
**A:** `sensitive = true` only masks output. The value still exists in state. Secret Manager retrieves values at runtime without storing in state.

### Q5: How do you rotate secrets with Terraform?
**A:** Use Cloud Scheduler + Cloud Function to rotate, update Secret Manager, then trigger Terraform refresh or re-apply.

### Q6: Why use Workload Identity over service account keys?
**A:** Workload Identity eliminates key management, rotates credentials automatically, and follows principle of least privilege.

### Q7: How do you audit secret access?
**A:** Use Cloud Audit Logs for Secret Manager. Filter by methodName for access events.

### Q8: Can Terraform state contain secrets?
**A:** If secrets are passed via variables, they may appear in state. Use Secret Manager data sources to avoid this.
