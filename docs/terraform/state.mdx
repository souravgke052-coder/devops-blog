# Terraform State

## Overview

Terraform state is a critical component that maps your configuration to real-world infrastructure. It stores resource metadata, relationships, and current attribute values. Understanding state management is essential for production-grade Terraform deployments.

## When To Use

- **Every Terraform operation**: State is required for `plan`, `apply`, and `destroy`
- **Team collaboration**: Remote state enables concurrent operations without conflicts
- **Disaster recovery**: Versioned state allows rollback to previous configurations
- **Cross-environment references**: State sharing enables module dependencies

## When NOT To Use

- **Local state only for teams**: Never use local state in a team environment
- **Version control for state**: State files contain sensitive data and should never be committed to Git

## Core Concepts

### State File Structure

```json
{
  "version": 4,
  "terraform_version": "1.5.0",
  "serial": 1,
  "lineage": "xxxx-xxxx-xxxx-xxxx",
  "outputs": {},
  "resources": [
    {
      "mode": "managed",
      "type": "google_compute_instance",
      "name": "web_server",
      "provider": "provider[\"registry.terraform.io/hashicorp/google\"]",
      "instances": [
        {
          "schema_version": 6,
          "attributes": {
            "id": "projects/my-project/zones/us-central1-a/instances/web-server",
            "name": "web-server",
            "machine_type": "e2-medium",
            "zone": "us-central1-a"
          }
        }
      ]
    }
  ]
}
```

### State Locking

Prevents concurrent modifications that could corrupt state:

```bash

# Lock info stored alongside state file

# Acquires lock
# Performs operation
# Releases lock
```

## Real Production Example

### GCS Backend with Full Configuration

```hcl
terraform {
  backend "gcs" {
    bucket = "my-project-terraform-state-prod"
    prefix = "networking"
    
    # Encryption
    encryption_key = google_kms_key.terraform_state_key.id
    
    # State file versioning enabled on bucket
  }
}

# KMS Key for State Encryption
resource "google_kms_key_ring" "tf_keyring" {
  name     = "tf-keyring-${var.environment}"
  location = "global"
  
  lifecycle {
    prevent_destroy = true
  }
}

resource "google_kms_key" "terraform_state_key" {
  name         = "tf-state-key-${var.environment}"
  key_ring     = google_kms_key_ring.tf_keyring.key_ring
  purpose      = "ENCRYPT_DECRYPT"
  
  rotation_period = "7776000s"  # 90 days
  
  lifecycle {
    prevent_destroy = true
  }
}

# GCS Bucket for State
resource "google_storage_bucket" "terraform_state" {
  name          = "${var.project_id}-terraform-state-${var.environment}"
  location      = "US"
  storage_class = "STANDARD"
  
  uniform_bucket_level_access = true
  
  versioning {
    enabled = true  # Keep state history
  }
  
  encryption {
    default_kms_key_name = google_kms_key.terraform_state_key.id
  }
  
  lifecycle_rule {
    condition {
      age = 30  # Keep state versions for 30 days
    }
    action {
      type = "Delete"
    }
  }
  
  lifecycle {
    prevent_destroy = true
  }
}

# IAM for State Bucket
resource "google_storage_bucket_iam_member" "terraform_state" {
  bucket = google_storage_bucket.terraform_state.name
  role   = "roles/storage.objectAdmin"
  member = "serviceAccount:${var.terraform_service_account}"
}
```

### S3 Backend with DynamoDB Locking (AWS Alternative)

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-prod"
    key            = "networking/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-locks"
    
    # SSE-KMS encryption
    kms_key_id = "arn:aws:kms:us-east-1:123456789012:key/xxxx-xxxx"
  }
}

# DynamoDB Table for State Locking
resource "aws_dynamodb_table" "terraform_state_locks" {
  name         = "terraform-state-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  
  attribute {
    name = "LockID"
    type = "S"
  }
  
  server_side_encryption {
    enabled = true
  }
  
  lifecycle {
    prevent_destroy = true
  }
}
```

## HCL Examples

### Remote State Data Source

```hcl
# Reference state from another environment
data "terraform_remote_state" "network" {

# Access outputs
locals {
  vpc_id     = data.terraform_remote_state.network.outputs.vpc_network_id
  db_host    = data.terraform_remote_state.database.outputs.private_ip
}
```

## CLI Examples

### State Operations

```bash
# Initialize with backend
terraform init

# List all resources in state
terraform state list

# Show specific resource
terraform state show google_compute_instance.web_server

# Show resource in module
terraform state show module.mig.google_compute_instance_template.template

# Pull state (remote to local)
terraform state pull > local-state.tfstate

# Push state (local to remote)
terraform state push local-state.tfstate

# Rename resource
terraform state mv google_compute_instance.old google_compute_instance.new

# Move to module
terraform state mv google_compute_instance.server module.server

# Remove from state (orphan resource)
terraform state rm google_compute_instance.orphaned

# Force unlock (emergency only)
terraform force-unlock <lock-id>
```

### State Lock Troubleshooting

```bash
# Check current lock
terraform apply
# Error: Error acquiring the state lock

# View lock info
terraform state pull | jq '.metadata'

# Force unlock after verifying no other operations
terraform force-unlock <lock-id-from-error>

# Verify unlock
terraform plan
```

### State Versioning (GCS)

```bash
# List state versions
gsutil ls -a gs://my-project-terraform-state/networking/

# View specific version
gsutil cat gs://my-project-terraform-state/networking/default.2024-01-15T10:30:00.0~1

# Restore previous version
gsutil cp gs://my-project-terraform-state/networking/default.2024-01-15T10:30:00.0~1 /tmp/terraform.tfstate
terraform state push /tmp/terraform.tfstate
```

## Best Practices

### 1. Always Use Remote Backend

```hcl
# versions.tf
terraform {
  backend "gcs" {
    bucket = "project-terraform-state"
    prefix = "prod"
  }
}
```

### 2. Enable State Encryption

```hcl
terraform {
  backend "gcs" {
    bucket = "project-terraform-state"
    encryption_key = google_kms_key.terraform_state_key.id
  }
}
```

### 3. Enable State Versioning

```bash
# GCS bucket versioning
gsutil versioning set on gs://project-terraform-state

# S3 bucket versioning
resource "aws_s3_bucket_versioning" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id
  
  versioning_configuration {
    status = "Enabled"
  }
}
```

### 4. Separate State Per Environment

```hcl
# environments/prod/versions.tf
terraform {
  backend "gcs" {
    bucket = "my-project-terraform-state"
    prefix = "prod"  # Different prefix per environment
  }
}

# environments/dev/versions.tf
terraform {
  backend "gcs" {
    bucket = "my-project-terraform-state"
    prefix = "dev"
  }
}
```

### 5. State File Access Control

```hcl
# IAM for Terraform service account
resource "google_project_iam_member" "tf_state_admin" {
  project = var.project_id
  role    = "roles/storage.objectAdmin"
  member  = "serviceAccount:${google_service_account.terraform.email}"
}

# Restrict bucket access
resource "google_storage_bucket_iam_member" "tf_state" {
  bucket = google_storage_bucket.terraform_state.name
  role   = "roles/storage.objectAdmin"
  member = "serviceAccount:${var.terraform_service_account}"
  
  condition {
    title       = "terraform-state-access"
    expression  = "resource.name.startsWith(\"projects/${var.project_id}/buckets/${google_storage_bucket.terraform_state.name}\")"
  }
}
```

## Security

### State Contains Sensitive Data

```bash
# State may contain:
# - Resource passwords
# - Private keys
# - API tokens
# - Connection strings

# NEVER commit state to version control
echo "*.tfstate" >> .gitignore
echo "*.tfstate.*" >> .gitignore

# Pre-commit hook to prevent accidental commits
# .git/hooks/pre-commit
if git diff --cached --name-only | grep -q "\.tfstate"; then
  echo "ERROR: Cannot commit state files"
  exit 1
fi
```

### Encryption at Rest

```hcl
# GCS with CMEK
resource "google_storage_bucket" "terraform_state" {
  encryption {
    default_kms_key_name = google_kms_key.terraform_state_key.id
  }
}

# S3 with SSE-KMS
resource "aws_s3_bucket_server_side_encryption_configuration" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id
  
  rule {
    apply_server_side_encryption_by_default {
      kms_key_id = aws_kms_key.terraform_state.arn
      sse_algorithm = "aws:kms"
    }
  }
}
```

## Troubleshooting

### State Lock Timeout

```
Error: Error acquiring the state lock: condition not met
```

**Solution:**
```bash
# 1. Wait and retry (lock may be released)
terraform plan

# 2. Check if another operation is stuck
# Via GCS
gsutil cat gs://bucket/prefix/.terraform.lock.info

# 3. Force unlock (ONLY if no other operations running)
terraform force-unlock <lock-id>

# 4. Manual intervention (GCS)
# Delete .terraform.lock.info from bucket
gsutil rm gs://bucket/prefix/.terraform.lock.info
```

### State Drift

```bash
# Detect drift
terraform plan -detailed-exitcode
# Exit code 2 = drift detected

# Refresh state
terraform refresh

# Review differences
terraform show

# Sync state with actual infrastructure
terraform apply -auto-approve -refresh-only
```

### Corrupted State

```bash
# 1. Pull current state
terraform state pull > state.backup.tfstate

# 2. Restore from version
gsutil cp gs://bucket/prefix/default.2024-01-15T10:00:00.0~1 state.restore.tfstate

# 3. Push restored state
terraform state push state.restore.tfstate

# 4. Verify
terraform plan
```

### State File Not Found

```bash
# Error: No state file was found
# Solution: Initialize with correct backend
terraform init -backend-config="bucket=my-bucket"
terraform init -backend-config="prefix=prod"
```

### Large State Performance

```bash
# Split large state into modules
terraform state mv google_compute_instance.server module.server

# Verify state size
terraform state list | wc -l
# > 1000 resources = consider splitting

# Optimize with -target flag
terraform plan -target=module.database
```

## Interview Questions

### Q1: What is Terraform state and why is it needed?
**A:** State is a JSON file that maps configuration to actual resources. It tracks resource IDs, dependencies, and current attribute values. Required because cloud APIs don't expose full resource graphs - state acts as the source of truth for what exists.

### Q2: Why shouldn't state be committed to Git?
**A:** State may contain sensitive data (passwords, keys), it creates merge conflicts in teams, and local state doesn't support locking. Use remote backend instead.

### Q3: How does state locking work in GCS?
**A:** GCS doesn't support object locking natively. Terraform creates a `.terraform.lock.info` file when acquiring a lock and deletes it when releasing. Lock is considered held if the file exists and is older than expected timeout.

### Q4: How do you recover from a corrupted state file?
**A:** Use state versioning to restore previous version: `gsutil cp gs://bucket/prefix/default.2024-01-15T10:00:00.0~1 terraform.tfstate`, then `terraform state push`.

### Q5: What is state serialization and lineage?
**A:** Serial number increments on each apply - used to detect concurrent modifications. Lineage is a UUID assigned at state creation - identifies the state's identity.

### Q6: How do you handle sensitive data in state?
**A:** Mark variables as `sensitive = true`, enable encryption at rest and in transit, use remote backend with proper IAM, never commit to version control.

### Q7: What's the difference between `terraform refresh` and `terraform apply -refresh-only`?
**A:** `terraform refresh` updates state only without making changes. `terraform apply -refresh-only` updates state and highlights differences without planning changes.

### Q8: How do you manage state in a team?
**A:** Use remote backend with state locking, separate state per environment, restrict access via IAM, enable versioning for rollback capability.
