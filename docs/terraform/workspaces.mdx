# Terraform Workspaces

## Overview

Terraform workspaces allow you to manage multiple environments (dev, staging, prod) using the same configuration with isolated state files. Each workspace has its own state, enabling you to deploy identical infrastructure to multiple environments without duplicating configuration files.

## When To Use

- **Environment isolation**: Dev/staging/prod with same config
- **Branch-based deployments**: Feature branches with isolated state
- **Testing changes**: Test in isolation before applying to shared environments
- **Tenant isolation**: Multi-tenant infrastructure with shared configuration

## When NOT To Use

- **Different configurations**: If environments differ significantly, use separate directories
- **Production isolation**: Some teams prefer directory-based separation for stricter control
- **Complex environments**: Workspaces don't support different variable values elegantly

## Core Concepts

### Workspace States

```
workspaces/
├── default/        # Default workspace
├── dev/
├── staging/
└── prod/
```

### Workspace Commands

```bash
terraform workspace new dev
terraform workspace select dev
terraform workspace list
```

terraform {
  required_version = ">= 1.5.0"
  
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
  }
  
  # State in GCS with workspace prefix
  backend "gcs" {
    bucket = "my-project-terraform-state"
    # prefix set per workspace: workspace name
  }
}

      enable_cloud_armor      = false
      deletion_protection     = false
      backup_retention_days   = 7
      monitoring_level        = "basic"
    }
    staging = {
      enable_cdn              = true
      enable_cloud_armor      = true
      deletion_protection     = true
      backup_retention_days   = 14
      monitoring_level        = "detailed"
    }
    prod = {
      enable_cdn              = true
      enable_cloud_armor      = true
      deletion_protection     = true
      backup_retention_days   = 30
      monitoring_level        = "detailed"
    }
  }
  
  current_env_config = lookup(local.env_config, local.workspace, local.env_config.dev)
  
  common_labels = {
    environment = local.workspace
    managed_by  = "terraform"
    workspace   = local.workspace
  }
  
  # Environment-specific tags
  instance_tags = merge(local.common_labels, {
    cost-center = local.workspace == "prod" ? "production" : "development"
    team        = "platform"
  })
}

### CI/CD Pipeline

```yaml
# .github/workflows/terraform.yml
name: Terraform

on:
  push:
    branches: [main, 'feature/**', 'hotfix/**']
  pull_request:

jobs:
  terraform:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      
      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v2
        with:
          terraform_version: 1.5.0
      
      - name: Configure GCP
        uses: google-github-actions/auth@v1
        with:
          credentials_json: ${{ secrets.GCP_SA_KEY }}
      
      # Determine workspace from branch
      - name: Determine workspace
        id: workspace
        run: |
          BRANCH_NAME=${{ github.ref_name }}
          if [[ "$BRANCH_NAME" == "main" ]]; then
            echo "workspace=prod" >> $GITHUB_OUTPUT
          elif [[ "$BRANCH_NAME" == "staging" ]]; then
            echo "workspace=staging" >> $GITHUB_OUTPUT
          elif [[ "$BRANCH_NAME" == "develop" ]]; then
            echo "workspace=dev" >> $GITHUB_OUTPUT
          else
            echo "workspace=dev" >> $GITHUB_OUTPUT
          fi
      
      - name: Terraform Init
        run: |
          terraform init -reconfigure -backend-config="bucket=${{ secrets.TF_STATE_BUCKET }}"
      
      - name: Select Workspace
        run: terraform workspace select ${{ steps.workspace.outputs.workspace }} || terraform workspace new ${{ steps.workspace.outputs.workspace }}
      
      - name: Terraform Plan
        run: terraform plan -out=tfplan
      
      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply -auto-approve tfplan
```

## CLI Examples

### Workspace Management

```bash
# List all workspaces
terraform workspace list

# Show current workspace
terraform workspace show

# Create new workspace
terraform workspace new staging

# Create from existing state
terraform workspace new dev -use-existing

# Select workspace
terraform workspace select prod

# Delete empty workspace
terraform workspace delete staging

# Delete workspace (force if not empty)
terraform workspace delete staging -force

# Show workspace variables
terraform workspace show
```

## Best Practices

### 1. Use Descriptive Workspace Names

```bash
# Good
terraform workspace new production
terraform workspace new staging
terraform workspace new development

# Avoid
terraform workspace new env1
terraform workspace new p
```

### 2. Workspace-Specific Backend Configuration

```hcl
terraform {
  backend "gcs" {
    bucket = "my-terraform-state"
    # prefix automatically set to workspace name
  }
}
```

## Interview Questions

### Q1: What are Terraform workspaces?
**A:** Isolated state files within the same configuration. Each workspace has its own state, enabling management of multiple environments with identical code.

### Q2: How do workspaces differ from directories?
**A:** Workspaces share the same configuration with different state. Directories have different configurations (and optionally different state).

### Q3: How does workspace state get stored?
**A:** With GCS backend, state is stored at `gs://bucket/workspace_name/terraform.tfstate`.

### Q4: How do you set environment-specific variables with workspaces?
**A:** Use maps with `terraform.workspace` as key: `lookup(var.config, terraform.workspace, "default")`.

### Q5: How do you protect production workspace?
**A:** Use CI/CD with manual approval, restrict IAM access, use separate state buckets, implement policy-as-code.

### Q6: Can workspaces have different variables?
**A:** Yes, using lookup functions or conditional expressions based on `terraform.workspace`.

### Q7: What happens when you delete a workspace?
**A:** The state file is deleted. Resources managed by that workspace's state are NOT destroyed - they're orphaned.

### Q8: When should you NOT use workspaces?
**A:** When environments have significantly different configurations, when compliance requires complete isolation, or when variable complexity becomes unmanageable.
